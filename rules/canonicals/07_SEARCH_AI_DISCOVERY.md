## 1. Purpose

This canonical defines **how buyers discover listings** on Twicely. It extends the existing V3 keyword-only Typesense search with AI-powered capabilities:

- **Hybrid search** -- keyword relevance + vector semantic similarity, combined
- **Natural language queries** -- "red dress for a summer wedding" returns relevant results
- **Visual search** -- find similar listings by uploading a photo
- **Query understanding** -- intent extraction, spell correction, synonym expansion
- **Personalized ranking** -- user behavior signals influence result ordering
- **Smart autocomplete** -- type-ahead with intent detection
- **AI recommendations** -- personalized "For You" feed
- **Search analytics** -- zero-result tracking, query-to-purchase funnel, popular searches

The existing V3 keyword search, eligibility rules, trust gating, promoted listing cap, and PostgreSQL fallback are **preserved and extended**, not replaced. When AI is unavailable, the system degrades to V3 keyword-only behavior transparently.

### 1.1 Design Goals

1. **Buyer intent first** -- relevance > seller preferences > monetization.
2. **Backward compatible** -- existing keyword search works identically; AI layers are additive.
3. **Graceful degradation** -- AI failure = keyword-only, not search failure.
4. **Search is deterministic** -- same inputs produce the same ordered results (within freshness windows). Semantic results are deterministic for the same embedding model version.
5. **Eligibility before ranking** -- listings must pass all hard gates before any ranking (keyword, semantic, or personalized).
6. **No dark patterns** -- sponsored content always explicitly labeled; AI re-ranking never hides eligible results that match the query.
7. **Privacy-preserving personalization** -- user signals used for recommendations are explicit or clearly behavioral (not inferred demographics).

| # | Principle |
|---|-----------|
| S-1 | **Eligibility gates are unchanged from V3.** Status=ACTIVE, quantity>0, requiredAttributes complete, enforcementState=CLEAR, seller search-eligible. |

| S-8 | **AI-enhanced, not AI-dependent.** Every search path works without AI/embeddings. |

## 4. Architecture Overview

### 4.1 Search Pipeline (V4)

```
User Query
  |
  v
[1. Query Understanding] ---------> @twicely/ai/query-understanding
  | intent, filters, spelling correction, synonyms
  v
[2. Eligibility Filter] ----------> Hard gates (unchanged from V3)
  | Only ACTIVE, in-stock, enforcement-clear listings
  v
[3. Hybrid Search] ----------------> Typesense multi_search
  | keyword_score + vector_score (weighted)
  v
[4. Re-Ranking Pipeline]
  | keyword_score * keywordWeight
  | + vector_score * vectorWeight
  | + personalization_score * personalizationWeight
  | + freshness_score * freshnessWeight
  | * trust_multiplier
  | * promo_boost (capped)
  v
[5. Promoted Cap] -----------------> enforcePromotedCap() (D2.4, 30%)
  |
  v
[6. Search Analytics] -------------> Log query, results, latency
  |
  v
Final Results
```

### 4.2 Fallback Cascade

| Level | Condition | Behavior |
|---|---|---|
| Full AI | AI + Typesense available | Hybrid search with query understanding |
| Keyword-only | AI unavailable, Typesense available | V3 keyword search (current behavior) |
| PostgreSQL | Typesense unavailable | ILIKE search (current fallback) |

```ts
function buildEmbeddingText(doc: ListingDocument): string {
  return [
    doc.title,
    doc.brand,
    doc.categoryName,
    doc.condition,
    doc.description?.slice(0, 500),
    doc.tags?.join(' '),
  ].filter(Boolean).join(' | ');

```

### 5.2 Why Client-Side Embedding (Not Typesense Auto-Embedding)

Typesense v0.25+ supports `auto_embedding` fields that generate embeddings server-side. We use **client-side embedding via `@twicely/ai/embeddings`** instead because:
1. Centralized cost control and caching through the AI module
2. Same embedding model used for query embedding, visual search, and recommendations
3. Provider-agnostic (can switch from OpenAI to another embedding model without Typesense reconfiguration)
4. Embedding generation happens in BullMQ jobs (async), not blocking Typesense indexing

### 6.2 Hybrid Search (Default for V4)

Fuses keyword text_match score with vector distance. Typesense supports this natively.

```ts
async function searchHybrid(filters: SearchFilters): Promise<SearchResult> {
  const queryEmbedding = await embedQuery(filters.q!);
  const alpha = await getPlatformSetting('search.hybrid.alpha', 0.3);
  const client = getTypesenseClient();

  const result = await client
    .collections(LISTINGS_COLLECTION)
    .documents()
    .search({
      q: filters.q!,
      query_by: DEFAULT_QUERY_BY,
      query_by_weights: DEFAULT_QUERY_WEIGHTS,
      vector_query: `embedding:([${queryEmbedding.join(',')}], k:200, alpha:${alpha})`,
      filter_by: buildFilterString(filters),
      sort_by: '_text_match:desc,_vector_distance:asc,sellerScore:desc',
      per_page: limit,
      page: filters.page ?? 1,
      num_typos: 2,
      prioritize_exact_match: true,
    });

  return mapSearchResult(result, filters);
}
```

**Alpha parameter** controls keyword vs. vector weight. `alpha: 0.3` means 70% keyword, 30% vector.

### 6.4 Visual Search

User uploads a photo, system finds visually similar listings.

```ts
// packages/search/src/visual-search.ts

export interface VisualSearchRequest {
  imageUrl: string;
  categoryId?: string;
  limit?: number;              // default 24
  page?: number;
}

export async function searchByImage(req: VisualSearchRequest): Promise<SearchResult> {
  // 1. Call @twicely/ai/visual-search to extract attributes + generate embedding
  const { extractedAttributes } = await visualSearchAnalyze(req.imageUrl);

  // 2. Build text from extracted attributes, then embed
  const searchText = [
    extractedAttributes.category,
    extractedAttributes.brand,
    extractedAttributes.color,
    extractedAttributes.style,
  ].filter(Boolean).join(' | ');
  const embedding = await embedQuery(searchText);

  // 3. Vector search in Typesense
  const client = getTypesenseClient();
  const filterParts = ['availableQuantity:>0'];
  if (req.categoryId) filterParts.push(`categoryId:=${req.categoryId}`);

  const result = await client
    .collections(LISTINGS_COLLECTION)
    .documents()
    .search({
      q: '*',
      vector_query: `embedding:([${embedding.join(',')}], k:${req.limit ?? 24})`,
      filter_by: filterParts.join(' && '),
      per_page: req.limit ?? 24,
      page: req.page ?? 1,
    });

  return mapSearchResult(result, { page: req.page });
}
```

**Rate limit:** 30 visual searches per user per day (Canonical 30, section 4.2).

### 6.6 When Each Mode Activates

```ts
async function searchListings(filters: SearchFilters): Promise<SearchResult> {
  const hybridEnabled = await getPlatformSetting('search.hybrid.enabled', true);
  const minWords = await getPlatformSetting('discovery.search.minWordsForVector', 2);

  const hasQuery = filters.q && filters.q.trim() !== '' && filters.q !== '*';
  const wordCount = hasQuery ? filters.q!.trim().split(/\s+/).length : 0;
  const useVector = hybridEnabled && hasQuery && wordCount >= minWords;

  try {
    if (useVector) {
      return await searchHybrid(filters);
    }
    return await searchWithTypesense(filters);  // existing V3 keyword search
  } catch {
    return await searchWithPostgres(filters);   // existing V3 fallback
  }
}
```

| Query Type | Mode | Rationale |
|---|---|---|
| Empty query (`*`) | Keyword | Browse mode -- no semantic intent |
| Single word ("nike") | Keyword | Keyword sufficient for brand/term |
| Multi-word ("nike air max 90") | Hybrid | Semantic helps find variations |
| Natural language ("warm winter jacket") | Hybrid | Semantic is primary value-add |
| Filter-only (category browse) | Keyword | Faceted navigation, no text query |
| Image upload | Vector-only | Pure visual similarity |

## 7. Query Understanding

### 7.1 Integration with AI Module

Before executing the Typesense query, the search pipeline calls `@twicely/ai/query-understanding`:

```ts
import { understandQuery } from '@twicely/ai/query-understanding';

async function enrichQuery(rawQuery: string): Promise<EnrichedQuery> {
  const enabled = await getPlatformSetting('search.nlQuery.enabled', true);
  if (!enabled || !rawQuery.trim()) {
    return { query: rawQuery, filters: {}, synonyms: [], spellCorrection: null };
  }

  try {
    const result = await understandQuery({ query: rawQuery });
    return {
      query: result.expandedQuery,
      filters: result.extractedFilters,
      synonyms: result.synonyms,
      spellCorrection: result.spellCorrection,
    };
  } catch {
    return { query: rawQuery, filters: {}, synonyms: [], spellCorrection: null };
  }
}
```

### 7.2 Enrichment Effects

| Enrichment | Effect on Search |
|---|---|
| `expandedQuery` | Used as the `q` parameter (spell-corrected, normalized) |
| `extractedFilters.category` | Added to `filter_by` if not already set by user |
| `extractedFilters.brand` | Added to `filter_by` if not already set by user |
| `extractedFilters.priceRange` | Added to `filter_by` if not already set by user |
| `extractedFilters.condition` | Added to `filter_by` if not already set by user |
| `synonyms` | Appended to Typesense synonym rules for the query session |
| `spellCorrection` | Returned to UI as "Did you mean?" suggestion |

### 10.2 Recommendation Generation

```ts
// packages/search/src/recommendations.ts
export interface RecommendationRequest {
  userId: string;
  context: 'home_feed' | 'listing_detail' | 'cart' | 'post_purchase';
```

**Algorithm:**
1. Compute **user embedding** from weighted average of interest embeddings + recent search/browse embeddings (via `@twicely/ai/recommendations`).
2. Vector search in Typesense using the user embedding.
3. Apply **diversity filter**: max 3 listings per seller, max 40% from any single category.
4. Apply existing eligibility gates and trust gates.
5. Interleave with **exploration items** (10% random from underrepresented categories) to prevent filter bubbles.

## 11. Personalized Ranking

### 11.1 Personalization Signals

Personalization is a **post-Typesense re-ranking step**. Signals collected from browsing history.

### 11.2 Personalization Score

```ts
function computePersonalizationScore(
  listing: ListingCardData,
  ctx: PersonalizationContext,
): number {
  let score = 1.0;
  if (ctx.recentViewedCategoryIds.includes(listing.categoryId)) score += categoryBoost;
  if (listing.brand && ctx.recentSavedBrands.includes(listing.brand)) score += brandBoost;
  if (ctx.priceRangeCents && isInRange(listing.priceCents, ctx.priceRangeCents)) score += priceBoost;
  return Math.min(score, maxBoost);
}
```

### 11.3 Settings

| Key | Default | Description |
|---|---|---|
| `discovery.personalization.enabled` | `true` | Master toggle |
| `discovery.personalization.categoryBoost` | `0.1` | Category affinity boost |
| `discovery.personalization.brandBoost` | `0.05` | Brand affinity boost |
| `discovery.personalization.priceBoost` | `0.03` | Price range match boost |
| `discovery.personalization.maxBoost` | `1.2` | Cap on total multiplier |

## 12. Re-Ranking Pipeline (V4 Complete)

### 12.1 Pipeline Stages

```
Query Input -> [1. Query Understanding] -> [2. Retrieval (Hybrid)]
  -> [3. Eligibility Gates] -> [4. Trust Gates] -> [5. Scoring]
  -> [6. Personalization] -> [7. Promoted Cap] -> [8. Presentation]
```

### 12.2 Scoring Signals

| Signal | Weight | Source | Notes |
|---|---|---|---|
| Text relevance | HIGH | Typesense `_text_match` | title(10) > brand(5) > category(4) > description(3) > tags(2) |
| Vector similarity | MEDIUM | Typesense `_vector_distance` | Controlled by `search.hybrid.alpha` |
| Freshness | MEDIUM | `activatedAt` field | Tie-breaker in sort_by |
| Seller trust | LOW-MED | `sellerScore` field | Gated multiplier (V2 preserved) |
| Promo boost | ADDITIVE | `boostPercent` field | Capped at 1.5x, 30% of results max (D2.4) |
| Personalization | LOW | Server-side re-rank | 1.0-1.2 multiplier, logged-in only |

## 13. Embedding Generation Pipeline

### 13.1 Triggers

| Trigger | Action |
|---|---|
| Listing activated (new or relisted) | BullMQ job: generate embedding, upsert to Typesense |
| Listing updated (title, description, category, brand change) | BullMQ job: regenerate embedding, upsert |
| Listing deactivated/removed | Delete from Typesense (no embedding cleanup needed) |
| Nightly backfill cron | Bulk regenerate for active listings missing embeddings |
| Model change (`ai.model.embedding` updated) | Admin triggers full re-embed via hub action |

### 13.2 Embedding Text Construction

```ts
// packages/search/src/embedding-text.ts
export function buildEmbeddingText(doc: ListingDocument): string {
  const parts: string[] = [];
  if (doc.title) parts.push(doc.title);
  if (doc.brand) parts.push(doc.brand);
  if (doc.categoryName) parts.push(doc.categoryName);
  if (doc.condition) parts.push(doc.condition);
  if (doc.description) parts.push(doc.description.slice(0, 500));
  if (doc.tags?.length) parts.push(doc.tags.join(', '));
  return parts.join(' | ');
}
```

### 13.3 BullMQ Jobs

**listing-embedding** (on-demand):
- Queue: `listing-embedding`
- Triggered by: listing create/update server actions
- Concurrency: 5
- Retry: 3 attempts, exponential backoff

**listing-embedding-backfill** (cron):
- Pattern: from `jobs.cron.listingEmbeddingBackfill.pattern` (default `0 3 * * *`)
- tz: `UTC`
- Batch size: from `search.embedding.backfillBatchSize` (default 100)
- Delay between batches: 200ms

## 20. Graceful Degradation

| Component | Failure | Degraded Behavior |
|---|---|---|
| Embedding generation | AI provider down | Index without embedding; keyword search only |
| Hybrid search | Embeddings missing | Falls back to keyword-only (V3) |
| Query understanding | AI provider down | Treat as raw keyword search |
| Visual search | AI provider down | Show "temporarily unavailable" |
| Recommendations | AI/signals unavailable | Show "Trending" (newest popular items) |
| Similar items | No embedding on listing | Same-category browse fallback |
| Autocomplete | Valkey down | Disable autocomplete; search works |
| Typesense | Typesense down | PostgreSQL ILIKE fallback (V3) |

---

## 21. Migration Strategy (Keyword-Only to Hybrid)

1. **Phase 1 (schema):** Add `embedding` field as `optional: true`. Zero downtime.
2. **Phase 2 (backfill):** Run nightly job to generate embeddings for all active listings.
3. **Phase 3 (enable):** Set `search.hybrid.enabled = true`. Hybrid activates for multi-word queries.
4. **Phase 4 (tune):** Adjust `search.hybrid.alpha` based on CTR/conversion analytics.
5. **Rollback:** Set `search.hybrid.enabled = false`. Instant keyword-only. No data loss.

## 24. Final Rule

Discovery must always favor:
**valid inventory -> buyer intent -> fairness -> transparency**.

AI enhances discovery but never gates it. If embeddings, recommendations, or any AI component is unavailable, the system MUST fall back to keyword search gracefully. Users must never see an error page because AI is down.

If behavior is not defined here, it must be rejected or added as a new canonical version.
