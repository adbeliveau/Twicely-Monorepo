## 2. SCOPE -- EXACTLY WHAT TO BUILD

### 2.1 Architecture Overview

The Feature Lock-in Section 38 specifies:
> "Client-side: flags fetched on page load via API, cached in React context."

The build tracker notes:
> "V2 has `FeatureFlagContext` with `{ enabled, isLoading }`. V3 flags are admin-only -- no client consumption."

This step bridges that gap. The architecture is:

1. **API Route** (`GET /api/flags`) -- server evaluates flags for the current user (authenticated or guest) and returns a `Record<string, boolean>` map. This is the ONLY new data-fetching endpoint. No raw flag data (type, percentage, targeting rules) is ever exposed to the client.

2. **React Context Provider** (`FeatureFlagProvider`) -- wraps the app, fetches flags on mount via the API route, stores the resolved map in React state.

3. **React Hook** (`useFeatureFlag(key)`) -- consumes the context, returns `{ enabled: boolean, isLoading: boolean }` per flag key.

4. **Server Component Helper** (`getEvaluatedFlags`) -- for server components that want to pass initial flag state as props to avoid the client-side fetch waterfall. This uses the existing `isFeatureEnabled()` directly.

### 2.2 Database

**No schema changes.** The existing `featureFlag` table (Section 14.3) is sufficient. The new API route reads from the existing evaluation engine which already uses Valkey cache.

### 2.3 API Route

**`GET /api/flags`**

- **Auth:** Optional. If the user has a Better Auth session, pass `{ userId }` to `isFeatureEnabled()` for percentage rollout and targeted flags. If no session (guest), evaluate without userId (BOOLEAN flags only will resolve meaningfully).
- **No CASL gate.** This route does NOT require any specific CASL permission. All users (including guests) may read the evaluated boolean results of flags. Raw flag configuration (type, percentage, targeting rules) is NEVER returned. Only the resolved `true`/`false` per key is returned.
- **Request:** `GET /api/flags?keys=feature.newCheckout,gate.marketplace,kill.checkout` -- optional `keys` query param. If provided, only those keys are evaluated. If omitted, ALL active flags are evaluated (for initial page load).
- **Response shape:**

```typescript
// 200 OK
{
  flags: Record<string, boolean>
  // e.g. { "feature.newCheckout": true, "gate.marketplace": false, "kill.checkout": true }
}
```

- **Implementation:** Use `isFeatureEnabled()` from `src/lib/services/feature-flags.ts` for each flag. This already handles the full evaluation order (cache -> DB -> type-specific logic) and the Valkey cache layer.
- **Performance:** When no `keys` param is provided, fetch all flag keys from DB in one query, then evaluate each. The Valkey cache means most evaluations are cache hits. Expect <50ms for typical flag counts (<50 flags).
- **Error handling:** If evaluation fails for a specific flag, default to `false` for that key (consistent with `isFeatureEnabled` behavior). Never expose error details to client.
- **Cache headers:** Set `Cache-Control: private, max-age=30` (matches the Valkey TTL from `featureFlags.cacheSeconds` platform setting). The `private` directive is critical because flag evaluation is user-specific for PERCENTAGE and TARGETED types.

### 2.4 React Context Provider

**`FeatureFlagProvider`** -- a `'use client'` component.

```typescript
interface FeatureFlagContextValue {
  flags: Record<string, boolean>;
  isLoading: boolean;
}
```

- Creates a React context with `createContext<FeatureFlagContextValue>`.
- On mount, fetches `GET /api/flags` (no `keys` param -- fetch all).
- Stores the `flags` map in state. Sets `isLoading` to `false` when fetch completes.
- Accepts an optional `initialFlags` prop (type `Record<string, boolean>`) for SSR hydration -- when a server component pre-evaluates flags and passes them down, the provider uses those immediately (no loading state, no fetch). The fetch still fires to refresh in the background after hydration.
- If the fetch fails, `flags` remains the `initialFlags` (or empty `{}`). `isLoading` is set to `false`. No error is thrown to the UI.
- The provider does NOT poll or use WebSocket for live updates. Flags update on next page navigation (which remounts the provider in Next.js App Router). This is consistent with the 30-second Valkey TTL design.

### 2.5 React Hook

**`useFeatureFlag(key: string): { enabled: boolean; isLoading: boolean }`**

- Reads from the `FeatureFlagContext`.
- Returns `{ enabled: flags[key] ?? false, isLoading }` where `flags` is the context's flag map.
- If the key is not in the map, `enabled` defaults to `false` (consistent with server-side `isFeatureEnabled` behavior for unknown keys).
- If the context is not available (hook used outside provider), throw a descriptive error: `"useFeatureFlag must be used within a FeatureFlagProvider"`.

### 2.6 Server Component Helper

**`getEvaluatedFlags(keys: string[], userId?: string): Promise<Record<string, boolean>>`**

- A thin wrapper that calls `isFeatureEnabled(key, { userId })` for each key in parallel (`Promise.all`).
- Returns `Record<string, boolean>`.
- This is for server components that want to pre-populate the `FeatureFlagProvider.initialFlags` prop, avoiding the client-side fetch waterfall.
- This function is NOT a server action (no `'use server'` directive). It is a plain async utility function that runs on the server only (because it imports from `src/lib/services/feature-flags.ts` which uses Node.js DB/cache APIs).

### 2.7 Provider Wiring

The `FeatureFlagProvider` must be added to the **root layout** (`src/app/layout.tsx`) so both marketplace and hub pages can consume flags. However, the root layout is currently a server component and does not import any client providers.

**Approach:** Create a `Providers` wrapper component (`'use client'`) that wraps `FeatureFlagProvider` (and can later wrap other providers). The root layout renders `<Providers>{children}</Providers>`.

Alternatively, if the team prefers to keep the root layout minimal, the provider can be added to the marketplace layout and the hub layout independently. **Owner decision needed: should the provider live in the root layout (simpler, one place) or in each route group layout (marketplace + hub separately)?** For this prompt, we specify the root layout approach since it is simpler and ensures all routes have flag access.

The root layout is a server component. It can call `getEvaluatedFlags()` to pre-populate flags, then pass them as `initialFlags` to the client `Providers` wrapper.

**Important:** The root layout currently does NOT have an auth session check. Adding one would add latency to every page load. For the initial implementation, pass `initialFlags` as an empty `{}` (no SSR pre-evaluation). The client-side fetch in `FeatureFlagProvider` will resolve flags after mount. If specific pages need flags at SSR time, those page server components can call `getEvaluatedFlags()` and pass them to child client components directly.

**Revised wiring:** Do NOT modify the root layout for SSR flag evaluation. Only add the `FeatureFlagProvider` (with no `initialFlags`) to the `Providers` wrapper in the root layout. Individual pages that need SSR flags will call `getEvaluatedFlags()` themselves and pass the values as props.

## 3. CONSTRAINTS -- WHAT NOT TO DO

- **Do NOT expose raw flag data to the client.** The API route returns only `Record<string, boolean>`. Never return `type`, `percentage`, `targetingJson`, `createdByStaffId`, or any other column.
- **Do NOT use Zustand, Redux, or any external state library.** CLAUDE.md mandates React context + server state.
- **Do NOT create a polling mechanism or WebSocket subscription.** Flags refresh on page navigation. The 30-second Valkey TTL is sufficient.
- **Do NOT add a new CASL subject or permission.** The API route is public (returns only evaluated booleans, no sensitive data).
- **Do NOT import server-only modules (`@/lib/db`, `@/lib/cache/valkey`) in `'use client'` files.** This causes 500 errors in Next.js due to Node.js module resolution.
- **Do NOT hardcode any flag keys in the provider or hook.** Keys are dynamic strings passed by consumers.
- **Do NOT create any new database tables or columns.**
- **Do NOT add `as any`, `@ts-ignore`, or `@ts-expect-error`.**
- **Max 300 lines per file.**
- **Do NOT use `tRPC`.** This is a plain Next.js API route.
- **Banned terms check:** None of the banned vocabulary terms (SellerTier, FVF, wallet, etc.) are likely to appear in this feature, but scan output before committing.

## 4. ACCEPTANCE CRITERIA

### Functional

1. `GET /api/flags` returns `{ flags: Record<string, boolean> }` with status 200 for both authenticated and unauthenticated requests.
2. `GET /api/flags?keys=kill.checkout,gate.marketplace` returns only the specified keys.
3. `GET /api/flags` without `keys` param returns ALL flag keys from the `featureFlag` table, each evaluated.
4. For PERCENTAGE flags, `GET /api/flags` with an authenticated user returns deterministic results (same user always gets same bucket for same flag).
5. For PERCENTAGE flags, `GET /api/flags` without authentication returns `false` (no userId to hash).
6. For TARGETED flags with user overrides, `GET /api/flags` with authenticated user returns the override value.
7. `useFeatureFlag('some.key')` returns `{ enabled: true, isLoading: false }` after the provider has loaded and the flag is enabled.
8. `useFeatureFlag('nonexistent.key')` returns `{ enabled: false, isLoading: false }` (unknown key defaults to false).
9. `useFeatureFlag('any.key')` returns `{ enabled: false, isLoading: true }` before the API response arrives.
10. Using `useFeatureFlag()` outside of `FeatureFlagProvider` throws a descriptive error.
11. `FeatureFlagProvider` with `initialFlags` prop renders immediately with those flags (no loading state).
12. `getEvaluatedFlags(['kill.checkout', 'gate.marketplace'], userId)` returns correct evaluations using the existing server-side engine.

### Non-Functional

13. The API route response includes `Cache-Control: private, max-age=30`.
14. The API route does NOT return any raw flag metadata (type, percentage, targeting rules, staff IDs).
15. No `'use client'` file imports from `@/lib/db`, `@/lib/cache/valkey`, or any server-only module.
16. All files are under 300 lines.
17. TypeScript strict mode: zero errors.
18. No banned terms in any created/modified file.
19. Test count >= baseline (7322 per latest tracker).

## 8. IMPLEMENTATION NOTES

### API Route Pattern

Follow the existing pattern from `src/app/api/local/tokens/route.ts`:
- Import `auth` from `@/lib/auth/server` for optional session
- Import `headers` from `next/headers`
- Use `NextResponse.json()` for responses
- Return `{ success: true, ... }` shape is NOT required here -- the flags endpoint uses `{ flags: {...} }` which is simpler and specific to this use case
- Set response headers using `NextResponse.json(body, { headers: { 'Cache-Control': '...' } })`

### Auth for the API Route

The flags API does NOT use `authorize()` from CASL because:
1. It is not gated to any specific role
2. Guests should be able to fetch flags (for UI gating on public pages)
3. We only need the userId for PERCENTAGE/TARGETED evaluation, not for authorization

Instead, use a lightweight session check:
```typescript
const betterAuthSession = await auth.api.getSession({ headers: await headers() });
const userId = betterAuthSession?.user?.id ?? undefined;
```

### Fetching All Flag Keys

When no `keys` param is provided, query all flag keys from the DB:
```typescript
const allFlags = await db.select({ key: featureFlag.key }).from(featureFlag);
```
Then evaluate each using `isFeatureEnabled(flag.key, { userId })`. Most will be Valkey cache hits.

### Provider Component

The `FeatureFlagProvider` should:
1. Accept `initialFlags?: Record<string, boolean>` prop
2. Initialize state: `flags = initialFlags ?? {}`, `isLoading = !initialFlags`
3. On mount (`useEffect`), fetch `GET /api/flags`
4. On response, merge: `setFlags(prev => ({ ...prev, ...response.flags }))`, `setIsLoading(false)`
5. On error, just set `setIsLoading(false)` -- keep whatever flags were already loaded

### Root Layout Modification

The root layout change is minimal. Before:
```tsx
<body>{children}</body>
```
After:
```tsx
<body><Providers>{children}</Providers></body>
```

The `Providers` component is `'use client'` and wraps `FeatureFlagProvider` with no `initialFlags` (empty initial state, client fetch on mount).

### Why Not Server Action?

Server actions are for mutations. The flag fetch is a read operation. An API route (`GET /api/flags`) is the correct pattern per CLAUDE.md ("Next.js API routes + server actions").

### Why Not Pass Flags as Server Component Props Everywhere?

While `getEvaluatedFlags()` exists for SSR, the primary pattern is the client-side hook. This is because:
1. Most flag checks happen in interactive client components (button visibility, form sections, feature gates)
2. The client hook provides a clean `{ enabled, isLoading }` API matching V2's `FeatureFlagContext`
3. Server components that need flags can use `isFeatureEnabled()` directly (already exists from G10.4)
4. The `getEvaluatedFlags()` helper is for the specific case where a server component wants to batch-evaluate and hydrate a client component
