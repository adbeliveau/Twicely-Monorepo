### 2.3 Server Actions

#### Action 1: `applyForInfluencer` (Seller-Side)

**File:** `src/lib/actions/affiliate-influencer.ts` (NEW)

A seller who is ALREADY a community affiliate OR not yet an affiliate applies for the influencer tier.

**Logic:**
1. `authorize()` — must be authenticated seller
2. CASL check: `ability.can('create', sub('Affiliate', { userId: session.userId }))`
3. Feature flag: check `affiliate.enabled` AND `affiliate.influencer.enabled` — both must be `true`
4. Parse input with `applyInfluencerSchema`
5. Look up existing affiliate record by `session.userId`:
   - **If NO existing affiliate:** Create new affiliate record with `tier: 'INFLUENCER'`, `status: 'PENDING'`, `applicationNote: <serialized data>`, `commissionRateBps` from `affiliate.influencer.defaultCommissionRateBps` (2500), `cookieDurationDays` from `affiliate.influencer.cookieDays` (60)
   - **If existing COMMUNITY affiliate with status ACTIVE:** Update existing record: set `tier: 'INFLUENCER'`, `status: 'PENDING'`, `applicationNote: <serialized data>`. NOTE: While PENDING, the affiliate RETAINS their COMMUNITY-level access (existing referral links and promo codes continue to work). The tier field changes to INFLUENCER but status goes to PENDING.
   - **If existing INFLUENCER affiliate:** Return error "You have already applied"
   - **If existing affiliate with status SUSPENDED or BANNED:** Return error "Your affiliate account is not eligible"
6. Audit event: `INFLUENCER_APPLICATION_SUBMITTED`
7. Notify admin (placeholder — call `notify()` with a system notification to FINANCE/ADMIN staff, or log for now)
8. `revalidatePath('/my/selling/affiliate')`
9. Return `{ success: true }`

**IMPORTANT BUSINESS RULE:** When a COMMUNITY affiliate applies for INFLUENCER:
- Their status changes to PENDING
- BUT their existing community features MUST continue working (promo codes stay active, referral links still track)
- This means the promo-codes-affiliate.ts action already checks `affiliateRecord.status !== 'ACTIVE'` which would BLOCK their promo code creation while PENDING
- **Resolution:** The `createAffiliatePromoCode` action in `promo-codes-affiliate.ts` MUST be updated to also allow status `PENDING` for existing community affiliates upgrading. Check: `if (affiliateRecord.status !== 'ACTIVE' && affiliateRecord.status !== 'PENDING')` — BUT only if the affiliate has existing active promo codes (meaning they were previously COMMUNITY ACTIVE).
- **Simpler alternative (RECOMMENDED):** Do NOT change `status` to PENDING for COMMUNITY->INFLUENCER upgrades. Instead, add a new nullable column... BUT that requires schema changes.
- **Simplest (RECOMMENDED — NO SCHEMA CHANGE):** Keep existing COMMUNITY affiliate record's `status` as `ACTIVE`. Store the influencer application in the `applicationNote` field. Create a service function that checks if an active COMMUNITY affiliate has a pending influencer application by checking `tier === 'COMMUNITY' AND applicationNote IS NOT NULL AND applicationNote contains structured application JSON`.
- **WAIT — re-reading the canonical:** The canonical says influencer status `PENDING` means "Applied, awaiting approval (influencer only)". For a NEW influencer application (not existing community), the flow is clear: create with PENDING. For a COMMUNITY-to-INFLUENCER upgrade, the spec does not explicitly address this transition.

**OWNER DECISION NEEDED: How should COMMUNITY-to-INFLUENCER upgrade work?**

Option A: Change tier to INFLUENCER + status to PENDING (breaks promo code creation until approved). Clean. Matches canonical enum semantics exactly.

Option B: Leave tier as COMMUNITY + status as ACTIVE. Store application data in applicationNote. Admin approves → set tier to INFLUENCER. No disruption to existing community features. Slightly less clean semantically.

**FOR THIS PROMPT, USE OPTION A** (matches canonical exactly) **WITH MITIGATION:** Update `createAffiliatePromoCode` to allow PENDING status affiliates who have existing active promo codes (i.e., they were previously COMMUNITY ACTIVE and their promo codes should continue working). The `updateAffiliatePromoCode` and `deleteAffiliatePromoCode` should also work for PENDING affiliates.

#### Action 2: `approveInfluencerApplication` (Admin-Side)

**File:** `src/lib/actions/affiliate-admin.ts` (NEW)

Admin approves a PENDING influencer application, setting the negotiated commission rate.

**Logic:**
1. `staffAuthorize()` — must be authenticated hub staff
2. CASL check: `ability.can('manage', 'Affiliate')` — FINANCE or ADMIN scope
3. Parse input with `approveInfluencerSchema`
4. Load affiliate record by `affiliateId`
5. Verify `affiliate.status === 'PENDING'` — if not, return error
6. Verify `affiliate.tier === 'INFLUENCER'` — double-check (could be PENDING for other reasons in the future)
7. Commission rate bounds check: `commissionRateBps` must be between 2000 (20%) and 3000 (30%) per Canonical Section 2.1
8. Update affiliate record:
   - `status: 'ACTIVE'`
   - `commissionRateBps: input.commissionRateBps`
   - `cookieDurationDays: input.cookieDurationDays`
   - `commissionDurationMonths: input.commissionDurationMonths`
   - `updatedAt: new Date()`
9. Audit event: `INFLUENCER_APPLICATION_APPROVED` with metadata `{ affiliateId, commissionRateBps, approvedBy: staffSession.staffUserId }`
10. Notify the affiliate user: placeholder notify() call with template `INFLUENCER_APPROVED` (email: "Congratulations! Your influencer application has been approved.")
11. `revalidatePath('/usr/affiliates')`
12. Return `{ success: true }`

#### Action 3: `rejectInfluencerApplication` (Admin-Side)

**File:** `src/lib/actions/affiliate-admin.ts` (SAME FILE as Action 2)

Admin rejects a PENDING influencer application.

**Logic:**
1. `staffAuthorize()` — authenticated hub staff
2. CASL check: `ability.can('manage', 'Affiliate')`
3. Parse input with `rejectInfluencerSchema`
4. Load affiliate by `affiliateId`
5. Verify `affiliate.status === 'PENDING'` and `affiliate.tier === 'INFLUENCER'`
6. **Determine post-rejection state:**
   - If affiliate had promo codes or referrals (was previously COMMUNITY): revert to `tier: 'COMMUNITY'`, `status: 'ACTIVE'`, preserve all commission/referral data
   - If affiliate was NEW (no prior COMMUNITY activity — check if they have any referrals or commissions): set `status: 'ACTIVE'`, `tier: 'COMMUNITY'` (downgrade to community), OR delete the record if truly brand new
   - **Simplest approach:** Always revert to `tier: 'COMMUNITY'`, `status: 'ACTIVE'`. This way rejected influencer applicants become (or remain) community affiliates.
7. Append rejection reason to `applicationNote`
8. `updatedAt: new Date()`
9. Audit event: `INFLUENCER_APPLICATION_REJECTED` with metadata `{ affiliateId, rejectionReason, rejectedBy: staffSession.staffUserId }`
10. Notify the affiliate user: placeholder notify() with template `INFLUENCER_REJECTED` (email: "Your influencer application was not approved at this time. You remain an active Community affiliate.")
11. `revalidatePath('/usr/affiliates')`
12. Return `{ success: true }`

#### Action 4: `suspendAffiliate` (Admin-Side)

**File:** `src/lib/actions/affiliate-admin.ts` (SAME FILE)

Admin suspends any affiliate (community or influencer).

**Logic:**
1. `staffAuthorize()`
2. CASL: `ability.can('manage', 'Affiliate')`
3. Validate input: `{ affiliateId: string, reason: string }`
4. Load affiliate, verify `status !== 'BANNED'` and `status !== 'SUSPENDED'`
5. Update: `status: 'SUSPENDED'`, `suspendedAt: new Date()`, `suspendedReason: reason`, `warningCount: +1`
6. Audit event: `AFFILIATE_SUSPENDED`
7. Notify affiliate user
8. Return `{ success: true }`

#### Action 5: `unsuspendAffiliate` (Admin-Side)

**File:** `src/lib/actions/affiliate-admin.ts` (SAME FILE)

1. `staffAuthorize()`, CASL manage Affiliate
2. Validate: `{ affiliateId: string }`
3. Load affiliate, verify `status === 'SUSPENDED'`
4. Update: `status: 'ACTIVE'`, `suspendedAt: null`, `suspendedReason: null`
5. Audit event: `AFFILIATE_UNSUSPENDED`
6. Return `{ success: true }`

#### Action 6: `banAffiliate` (Admin-Side)

**File:** `src/lib/actions/affiliate-admin.ts` (SAME FILE)

1. `staffAuthorize()`, CASL manage Affiliate
2. Validate: `{ affiliateId: string, reason: string }`
3. Load affiliate, verify `status !== 'BANNED'`

- Staff auth is SEPARATE from Better Auth (staffUser table, `twicely.staff_token` cookie)

### Architecture
- `as any` / `@ts-ignore` / `@ts-expect-error` — ZERO occurrences

- Explicit field mapping on DB updates — never spread request body
- `role`, `status` NEVER settable from client request body (server-side only)

- Commission rate for influencers is 20-30% NEGOTIATED per partner (Canonical Section 2.1) — admin sets the rate on approval, not the applicant

- PENDING affiliates must NOT lose access to existing community promo codes
- Three strikes policy: warning count tracked on affiliate record (Canonical Section 2.9)
- Rejection does NOT ban — it reverts to COMMUNITY tier
- Self-referral is forbidden (Canonical Section 2.3) — already enforced in G1.6

- Hub admin pages under `/usr/affiliates` (NOT `/admin/affiliates`, NOT `/dashboard/affiliates`)
- Seller pages under `/my/selling/affiliate` (NOT `/dashboard/affiliate`)

## 6. ACCEPTANCE CRITERIA

### Positive Cases
1. A seller with an active COMMUNITY affiliate account can submit an influencer application with a note of at least 20 characters
2. A seller who is NOT yet an affiliate can apply directly for INFLUENCER tier (goes to PENDING)
3. While PENDING, the affiliate's existing promo codes remain active and functional
4. While PENDING, the affiliate can still create/update/delete promo codes (community limits apply)
5. Admin (FINANCE or ADMIN role) can see a list of all affiliates at `/usr/affiliates`
6. Admin can filter the affiliate list by status (Pending tab shows applications awaiting review)
7. Admin can click into an affiliate detail page at `/usr/affiliates/[id]`
8. Admin can approve a PENDING influencer with a custom commission rate (20-30% range)
9. After approval, the affiliate's tier becomes INFLUENCER and status becomes ACTIVE
10. After approval, the affiliate's commission rate, cookie duration, and commission duration are set to the admin-specified values
11. Admin can reject a PENDING influencer application with a reason
12. After rejection, the affiliate reverts to COMMUNITY tier with ACTIVE status
13. Admin can suspend an ACTIVE affiliate (community or influencer) with a reason
14. Admin can unsuspend a SUSPENDED affiliate
15. Admin can ban an affiliate (deactivates all promo codes)
16. All admin actions create audit events
17. The `/my/selling/affiliate` page shows the influencer application form when `affiliate.influencer.enabled` is `true`
18. The `/my/selling/affiliate` page shows "Application Under Review" when the user's affiliate status is PENDING

### Negative Cases
1. A non-seller cannot apply for influencer (must be isSeller)
2. A SUSPENDED or BANNED affiliate cannot apply for influencer
3. An existing INFLUENCER affiliate (active or pending) cannot re-apply
4. Admin cannot approve a non-PENDING affiliate
