### SUB-STEP G4.1-B: Score Calculation Engine (Pure Function)

#### File: `src/lib/scoring/calculate-seller-score.ts`

This MUST be a **pure function** — same inputs always produce the same output. No database access, no side effects, no randomness. Under 300 lines.

**Input types:**
```typescript
interface MetricValues {
  onTimeShippingPct: number;      // 0.0-1.0 (fraction, not percentage)
  inadClaimRatePct: number;       // 0.0-1.0
  reviewAverage: number;          // 1.0-5.0 (DSR weighted average)
  responseTimeHours: number;      // median response time in hours
  returnRatePct: number;          // 0.0-1.0 (seller-fault only)
  cancellationRatePct: number;    // 0.0-1.0
}

interface CategoryThresholds {
  onTimeShipping: { ideal: number; steepness: number };
  inadRate: { ideal: number; steepness: number };
  responseTime: { ideal: number; steepness: number };
  returnRate: { ideal: number; steepness: number };
  cancellationRate: { ideal: number; steepness: number };
}

interface MetricWeights {
  onTimeShipping: number;    // 0.25
  inadRate: number;          // 0.20
  reviewAverage: number;     // 0.20
  responseTime: number;      // 0.15
  returnRate: number;        // 0.10
  cancellationRate: number;  // 0.10
}

interface ScoreInput {
  metrics: MetricValues;
  thresholds: CategoryThresholds;
  weights: MetricWeights;
  orderCount: number;
  platformMean: number;
  smoothingFactor: number;
  trendData: { avg30day: number; avg90day: number } | null;
  trendModifierMax: number;
}

interface ScoreResult {
  score: number;                  // 0-1000, clamped
  band: PerformanceBand;          // EMERGING | ESTABLISHED | TOP_RATED | POWER_SELLER
  searchMultiplier: number;       // 0.60-1.25
  perMetricScores: {
    shippingScore: number;
    inadScore: number;
    reviewScore: number;
    responseScore: number;
    returnScore: number;
    cancellationScore: number;
  };
  rawScore: number;               // before Bayesian smoothing
  adjustedScore: number;          // after Bayesian, before trend
  trendModifier: number;          // -0.05 to +0.05
  bayesianSmoothing: number;      // how much score was pulled toward mean
}
```

**Algorithm (Seller Score Canonical Section 2):**

1. **Per-metric normalization (Section 2.3):** For each metric, compute sigmoid score:
   - For "higher is better" metrics (onTimeShipping, reviewAverage): `metricScore = 1000 / (1 + e^(-k * (value - threshold)))`
   - For "lower is better" metrics (inadRate, responseTime, returnRate, cancellationRate): INVERT the sigmoid — scores ABOVE ideal threshold are GOOD. `metricScore = 1000 / (1 + e^(k * (value - threshold)))` (note: positive k, value above threshold = bad = low score)
   - reviewAverage normalization: normalize 1-5 stars to 0-1000 range first, THEN apply sigmoid centered on 4.5 (configurable threshold). DSR weighted average = average of 4 DSR dimensions and overall rating.
   - responseTime: Lower hours = better. Ideal 4h = score ~1000. 48h = score ~0.

2. **Weighted sum:** `rawScore = sum(metricScore_i * weight_i)` for all 6 metrics. Clamp 0-1000.

3. **Bayesian smoothing (Section 2.4):** `adjustedScore = (rawScore * orderCount + platformMean * smoothingFactor) / (orderCount + smoothingFactor)`

4. **Trend modifier (Section 2.5):** If trendData provided: `trendMod = clamp((avg30day - avg90day) / avg90day, -trendModifierMax, +trendModifierMax)`. `finalScore = adjustedScore * (1 + trendMod)`. If no trendData (no history), trendMod = 0.

5. **Clamp final score** to 0-1000.

6. **Derive band from score (Section 3.1):**
   - score >= 900 -> POWER_SELLER
   - score >= 750 -> TOP_RATED
   - score >= 550 -> ESTABLISHED
   - score < 550 -> EMERGING
   Band thresholds read from config, not hardcoded.

7. **Search multiplier (Section 3.3):** `searchMultiplier = clamp(score / 800, 0.60, 1.25)`. SUSPENDED check happens OUTSIDE this pure function (since SUSPENDED is admin-driven, not score-derived).

**Export:** `calculateSellerScore(input: ScoreInput, bandThresholds: BandThresholds): ScoreResult`

Also export: `calculateSearchMultiplier(score: number, isSuspended: boolean): number`

Also export: `deriveBand(score: number, bandThresholds: BandThresholds): PerformanceBand`

Also export: `calculateTrend(scores: number[]): TrendState` where TrendState = `'SURGING' | 'CLIMBING' | 'STEADY' | 'SLIPPING' | 'DECLINING'`
- Computed from snapshot history, NOT stored in DB (Canonical Section 4.1)
- delta = avg30day - avg90day
- delta >= 50 -> SURGING, >= 10 -> CLIMBING, <= -50 -> DECLINING, <= -10 -> SLIPPING, else STEADY

#### File: `src/lib/scoring/metric-queries.ts`

Database queries to gather the 6 metric inputs for a given seller. Each query covers the 90-day rolling window.

**Functions:**
1. `getOnTimeShippingRate(userId: string, windowDays: number): Promise<number>` — % of orders shipped within seller's handlingTimeDays. Query orders with status SHIPPED/DELIVERED/COMPLETED where completedAt > 90 days ago. Compare shipment.shippedAt to order.createdAt + handlingTimeDays.

2. `getInadClaimRate(userId: string, windowDays: number): Promise<number>` — % of orders with INAD claims. Query returnRequest where reason = 'INAD' and sellerId = userId in window, divided by total completed orders.

3. `getReviewAverage(userId: string, windowDays: number): Promise<number>` — Weighted average of DSR dimensions + overall rating. Query reviews where sellerId = userId, status = APPROVED, createdAt in window.

4. `getMedianResponseTime(userId: string, windowDays: number): Promise<number>` — Median time in hours between buyer message and seller reply. Query conversations where sellerId = userId. For each first buyer message in conversation, measure time to seller's first reply.

5. `getReturnRate(userId: string, windowDays: number): Promise<number>` — % of orders returned where bucket = 'SELLER_FAULT'. Query returnRequests where sellerId in window / total completed orders.

6. `getCancellationRate(userId: string, windowDays: number): Promise<number>` — % of orders cancelled by seller (cancelInitiator = 'SELLER'). Query orders with status CANCELED, cancelInitiator SELLER / total orders in window.

7. `getPrimaryFeeBucket(userId: string, windowDays: number): Promise<string>` — The fee bucket with the highest order volume. Query orders, join to listing, join to category, group by category.feeBucket, return the max.

8. `getCompletedOrderCount(userId: string, windowDays: number): Promise<number>` — Count of completed orders.

9. `getPlatformMeanScore(): Promise<number>` — Average sellerScore across all active sellers with isNew = false.

### SUB-STEP G4.1-C: Daily Recalculation Job + Auto-Enforcement

#### File: `src/lib/jobs/seller-score-recalc.ts`

**Queue:** `seller-score-recalc`
**Schedule:** Daily at 3 AM UTC (from `score.recalcSchedule` platform setting)
**Concurrency:** 10 workers (Canonical Section 12.1)

**Job Flow (Canonical Section 12.1):**
1. Read platform settings: smoothingFactor, trendModifierMax, bandThresholds, weights, enforcement thresholds, newSellerOrderThreshold, transitionOrderThreshold, downgradeGraceDays
2. Fetch all sellers with `sellerProfile.status != 'SUSPENDED'` AND `sellerPerformance.completedOrders >= 1`
3. Get platform mean score (from last weekly recalc, or compute inline for first run)
4. For each seller (batched, concurrency 10):
   a. Query 90-day metrics via metric-queries.ts
   b. If completedOrders < newSellerOrderThreshold (10): mark `isNew = true`, skip scoring, continue
   c. Set `isNew = false` if completedOrders >= newSellerOrderThreshold
   d. Determine primary fee bucket
   e. Load category-specific thresholds from platform settings
   f. Calculate score via calculateSellerScore()
   g. Load last 7 daily snapshots to check downgrade grace period
   h. **Downgrade grace (Section 3.4):** If new band is LOWER than current band, check if seller has been below threshold for `downgradeGraceDays` (7) consecutive days. If not, keep current band. If yes, apply downgrade.
   i. **Upward transitions are IMMEDIATE** (no grace period)
   j. **Warning lockout (Section 6.3):** If seller has active Warning (enforcementLevel = 'WARNING' AND warningExpiresAt > now), they CANNOT achieve TOP_RATED or POWER_SELLER even if score qualifies. Clamp band to ESTABLISHED max.
   k. **Band override check:** If `bandOverride` is set and `bandOverrideExpiresAt` > now, use override band instead of score-derived band. If override expired, clear it.
   l. Update `sellerProfile`: sellerScore, sellerScoreUpdatedAt, performanceBand, isNew
   m. Insert `sellerScoreSnapshot` row with all metric values, scores, and band transition data
   n. If band changed: queue band transition notification
   o. Run auto-enforcement check (see below)
5. After all sellers: if it's Sunday (or `score.platformMeanRecalcSchedule` matches), update platform mean

**Auto-Enforcement Integration (Canonical Section 6.1):**

After score is calculated, check enforcement thresholds from platform settings:
- score < `score.enforcement.preSuspensionBelow` (100): set enforcementLevel = 'PRE_SUSPENSION'
- score < `score.enforcement.restrictionBelow` (250): set enforcementLevel = 'RESTRICTION'
- score < `score.enforcement.warningBelow` (400): set enforcementLevel = 'WARNING'
- score < `score.enforcement.coachingBelow` (550): set enforcementLevel = 'COACHING'
- score >= `score.enforcement.coachingBelow`: clear enforcementLevel (set to null)

**Enforcement level changes trigger:**
1. Update `sellerProfile.enforcementLevel` and `enforcementStartedAt`
2. If WARNING: set `warningExpiresAt = now + warningDurationDays`
3. Create an `enforcementAction` record with trigger = 'SCORE_BASED'
4. Send enforcement notification (reuse existing templates: enforcement.coaching, enforcement.warning, enforcement.restriction)
5. If enforcementLevel cleared (improved): send enforcement.lifted notification

**Enforcement is ADDITIVE** (Section 6.2): a seller can have enforcement AND a positive trend.

**RESTRICTION side effects (Section 6.1):**
- Search multiplier forced to 0.60 (handled in calculateSearchMultiplier when enforcementLevel = 'RESTRICTION')
- Boosting disabled (check in boost creation flow — NOT wired here, just the data flag)
- No new listings beyond current count (check in listing creation flow — NOT wired here)

**PRE_SUSPENSION (Section 6.1):** All RESTRICTION effects + account under review. If no improvement in `preSuspensionDays` (30), flag for admin review (create an enforcementAction with type PRE_SUSPENSION and details noting escalation needed).

**Band transition notifications (Section 3.4):**
- Upward: "Congratulations! You've reached {band} status"
- Downward: "Your performance status has changed. Here's how to improve."
- Use existing notification templates or add 2 new ones:
  - `enforcement.band_upgrade`: "Congratulations! You've reached {bandName} status"
  - `enforcement.band_downgrade`: "Your performance status has changed"

## 4. CONSTRAINTS — WHAT NOT TO DO

### Explicitly Banned
- Do NOT add `SUSPENDED` as a score-derived band outcome. SUSPENDED is ADMIN-ONLY (Section 3.2). The scoring engine must never set `performanceBand = 'SUSPENDED'`.
- Do NOT show any negative indicators to buyers. EMERGING sellers show NO badge, not a "New" or "Below Standard" badge (Section 1, Section 8).
- Do NOT couple TF rates to performance bands. TF uses progressive volume brackets — NO per-band TF discounts (Section 1, Decision #53).
- Do NOT show "Below Standard" badge to buyers — this term is banned (Canonical Section 1: "No Negative Labels to Buyers").
- Do NOT hardcode fee bucket thresholds. All thresholds from platform_settings.
- Do NOT implement boost credit issuance in this step. Seed the settings, defer the job.
- Do NOT implement the monthly boost credit job. Defer to a later step.
- Do NOT add any `as any`, `@ts-ignore`, or `@ts-expect-error`.
- Do NOT create files over 300 lines.
- Do NOT hardcode any rates, thresholds, or weights. ALL from platform_settings.
- Do NOT use storeId, sellerProfileId, or staffId as ownership key. Always userId.
- Do NOT show numeric score to buyers. Only badge tier is visible (Section 8).
- Do NOT show trend to buyers. Trend is internal/seller-only (Section 4.3).

### Gotchas from Canonical
- **Sigmoid direction matters:** "Higher is better" metrics (onTimeShipping, reviews) and "lower is better" metrics (INAD, cancellation, return, response time) need OPPOSITE sigmoid orientations.
- **reviewAverage is on 1-5 scale**, not 0-1000. Normalize before sigmoid.
- **responseTimeHours:** LOWER is better. A seller responding in 1 hour scores nearly 1000. A seller at 48 hours scores nearly 0.
- **New seller (< 10 orders):** Score calculated but NOT displayed. Band shows as "NEW" (UI label, not enum value). Search multiplier fixed at 1.0.
- **Transition seller (10-49 orders):** Score displayed. Search multiplier CLAMPED to 0.95-1.10 (Section 2.6).
- **Downgrade grace period:** 7 consecutive days below threshold before band changes down (Section 3.4). This means the recalc job must check the last 7 snapshots.
- **Warning lockout (Section 6.3):** Active warning prevents TOP_RATED/POWER_SELLER even if score qualifies.
- **Band override takes precedence** over score-derived band (Section 9.3). Check override first.
- **SUSPENDED overrides everything** (Section 3.2). If seller status is SUSPENDED, band is SUSPENDED regardless of score.
- **enforcementAction.trigger must be 'SCORE_BASED'** for auto-enforcement, not 'SYSTEM_AUTO' or 'ADMIN_MANUAL'.

### CLAUDE.md Rules That Apply
- Money as integer cents (boostCreditCents)
- Fee rates from platform_settings, never hardcoded
- Ownership via userId always
- Zod validation on all inputs
- Max 300 lines per file
- `strict: true` TypeScript

## 5. ACCEPTANCE CRITERIA

### Score Calculation Engine
- [ ] calculateSellerScore is a pure function: same inputs always produce same output
- [ ] Sigmoid normalization correctly handles "higher is better" vs "lower is better" metrics
- [ ] Bayesian smoothing pulls low-volume sellers toward platform mean (2 orders -> ~86% mean, 30 orders -> 50%, 500+ orders -> <6%)
- [ ] Trend modifier is clamped to +/-5%
- [ ] Final score clamped to 0-1000
- [ ] Band derived from configurable thresholds (not hardcoded 550/750/900)
- [ ] Search multiplier = clamp(score/800, 0.60, 1.25) for non-SUSPENDED sellers
- [ ] SUSPENDED sellers return multiplier 0.0
- [ ] New sellers (< 10 orders) get isNew = true, score = 0, multiplier = 1.0
- [ ] Transition sellers (10-49 orders) get multiplier clamped to 0.95-1.10

### Daily Recalculation Job
- [ ] Job registered in cron-jobs.ts, runs daily at 3 AM UTC
- [ ] Job processes all sellers with >= 1 completed order
- [ ] Job skips sellers with status = SUSPENDED
- [ ] Job inserts a sellerScoreSnapshot row per seller per run
- [ ] Job updates sellerProfile.sellerScore, performanceBand, sellerScoreUpdatedAt
- [ ] Downgrade grace: band does NOT decrease until seller is below threshold for 7 consecutive days
- [ ] Upward band transitions are immediate (no grace period)
- [ ] Warning lockout: sellers with active warning cannot reach TOP_RATED or POWER_SELLER
- [ ] Band override: if set and not expired, overrides score-derived band
- [ ] Expired overrides are cleared automatically

### Auto-Enforcement
- [ ] Score < 550 triggers COACHING (notification, no restrictions)
- [ ] Score < 400 triggers WARNING (email + in-app, 30-day window, enforcementStartedAt set)
- [ ] Score < 250 triggers RESTRICTION (visibility reduced, boost disabled)
- [ ] Score < 100 triggers PRE_SUSPENSION (all restrictions + admin review flag)
- [ ] Score improvement above threshold clears enforcement (enforcementLevel set to null)
- [ ] Enforcement level change creates enforcementAction with trigger = 'SCORE_BASED'
- [ ] Enforcement change sends correct notification template
- [ ] Enforcement is additive — seller can be in enforcement AND have positive trend
- [ ] All enforcement thresholds read from platform_settings, never hardcoded

### Performance Dashboard
- [ ] Page renders at `/my/selling/performance`
- [ ] Page requires SELLER auth gate (non-sellers see 403/redirect)
- [ ] Hero shows: band badge, score (if not new), trend arrow, progress bar, points to next band
- [ ] New sellers see "Complete X orders to earn your first performance badge"
- [ ] Score chart shows 90-day history with band threshold lines
- [ ] 6 metric breakdown cards with per-metric scores, values, ideal thresholds, tips
- [ ] Score projection shown only when CLIMBING or SURGING
- [ ] Rewards summary shows current + locked next-tier rewards
- [ ] Enforcement banner shown if enforcementLevel is set
- [ ] NO numeric score shown if seller is new (< 10 orders)
- [ ] NO "Below Standard" or negative labels anywhere

### Platform Settings
- [ ] All score.* settings seeded in extended settings file
- [ ] Weights sum to 1.0
- [ ] 4 fee bucket x 5 metric x 2 parameter = 40 category threshold settings seeded
- [ ] Coaching tips seeded as string arrays
- [ ] Reward configuration settings seeded (even though job is deferred)

### Negative Cases (MUST NOT be possible)
- [ ] Score calculation NEVER outputs a value outside 0-1000
- [ ] Score calculation NEVER depends on current time or randomness
- [ ] Buyers NEVER see numeric score, trend, or enforcement status
- [ ] Auto-enforcement NEVER sets performanceBand to SUSPENDED (admin-only)
- [ ] No banned terms: SellerTier, SubscriptionTier, FVF, BASIC, ELITE, PLUS, MAX, PREMIUM, STANDARD, RISING, Twicely Balance, wallet, Withdraw, Below Standard
