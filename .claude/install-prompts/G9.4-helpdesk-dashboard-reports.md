### 2.3 Queries to Create / Modify

#### A. MODIFY: `src/lib/queries/helpdesk-dashboard.ts` (~182 lines currently)

**Changes:**

1. **`getHelpdeskDashboardStats()`** — Expand return type to include:
   - `csatScore: number | null` — Average CSAT rating from `caseCsat` where `respondedAt` is not null, rolling 30 days
   - `csatCount: number` — Number of CSAT responses in rolling 30 days (for "X ratings" label)
   - `avgResolutionMinutes: number` — Average time from `createdAt` to `resolvedAt` for resolved cases, rolling 30 days
   - `slaCompliancePct: number` — Percentage of resolved cases (30d) where NEITHER `slaFirstResponseBreached` NOR `slaResolutionBreached` is true

2. **`getHelpdeskDashboardDeltas()`** — NEW function. Returns period-over-period comparison:
   - Compare current period (today for "resolved today", rolling 30d for averages) vs previous equivalent period
   - Returns: `{ openCasesDelta, resolvedTodayDelta, avgResponseDelta, slaBreachedDelta, csatDelta }`
   - Delta for "open cases": current open count minus open count 24 hours ago (approximate via cases created/resolved in last 24h)
   - Delta for "resolved today": today's resolved minus yesterday's resolved
   - Delta for "avg response": current 30d avg minus previous 30d avg (rolling 30-60 days)
   - Delta for "SLA breached": current breached count minus yesterday's breached count
   - Delta for "CSAT": current 30d average minus previous 30d average

   Implementation approach for deltas:
   - "Open Cases" delta: `(cases created today) - (cases resolved today)` gives net change. But simpler: query count of open cases where `createdAt >= today` minus count resolved today. Just use: resolved yesterday count vs resolved today count. **Simplest correct approach:** `resolvedTodayDelta = resolvedToday - resolvedYesterday`.
   - "Avg Response" delta: rolling 30d avg minus rolling 30-60d avg. Two separate AVG queries.
   - "CSAT" delta: same approach — 30d avg minus 30-60d avg.
   - "SLA Breached" delta: simply `currentBreached - yesterdayBreached`. Since breached is a running count, approximate by checking how many new breaches happened today.

3. **`getHelpdeskRecentActivity()`** — Fix agent name resolution:
   - Join `staffUser` on `caseEvent.actorId = staffUser.id` where `actorType = 'agent'`
   - For user actors, join `user` table to get display name
   - Also resolve `caseNumber` from `helpdeskCase` (currently returns raw `caseId`)

**Estimated file size after changes:** ~260 lines. If it exceeds 300, split delta queries into `helpdesk-dashboard-deltas.ts`.

## 8. IMPLEMENTATION NOTES

### Date Range Utility

For the reports page date range presets, implement a simple helper (can be inline in the page or in a utility):

```typescript
function getDateRange(preset: string): { from: Date; to: Date } {
  const now = new Date();
  const to = new Date(now.getFullYear(), now.getMonth(), now.getDate(), 23, 59, 59, 999);
  switch (preset) {
    case 'today': return { from: new Date(now.getFullYear(), now.getMonth(), now.getDate()), to };
    case 'this_week': { const d = new Date(to); d.setDate(d.getDate() - d.getDay()); d.setHours(0,0,0,0); return { from: d, to }; }
    case 'this_month': return { from: new Date(now.getFullYear(), now.getMonth(), 1), to };
    case 'last_30_days': default: { const d = new Date(to); d.setDate(d.getDate() - 30); d.setHours(0,0,0,0); return { from: d, to }; }
  }
}
```

### Reports Page Date Range — Server Component Pattern

The reports page is a server component. Date range selection uses search params:
- `/hd/reports` — defaults to last 30 days
- `/hd/reports?preset=today` — today only
- `/hd/reports?preset=this_week` — this week
- `/hd/reports?preset=this_month` — this month

The date range selector is a client component that navigates via `router.push()` with the new search params. The page re-renders server-side with the new range.

### CSAT Score Display

- When null (no data): show "—"
- Color coding: >= 4.0 green, >= 3.0 amber, < 3.0 red (can be done via the existing StatCard color prop or a wrapper)

### Delta Display Logic

- Positive delta + `isPositiveGood=true` = green arrow up (good)
- Positive delta + `isPositiveGood=false` = red arrow up (bad — e.g., more SLA breaches)
- Negative delta + `isPositiveGood=true` = red arrow down (bad)
- Negative delta + `isPositiveGood=false` = green arrow down (good — e.g., fewer breaches)
- Zero delta = neutral/hidden
- Null delta = hide badge entirely

This logic already exists in `StatCard` via the `isPositiveGood` prop — just ensure `delta={null}` case is handled.

### SQL Patterns for Common Queries

**Rolling 30-day CSAT average:**
```sql
SELECT AVG(rating) FROM case_csat WHERE responded_at IS NOT NULL AND responded_at >= NOW() - INTERVAL '30 days'
```

**SLA compliance (30d):**
```sql
SELECT
  COUNT(*) FILTER (WHERE NOT sla_first_response_breached AND NOT sla_resolution_breached) * 100.0 / NULLIF(COUNT(*), 0)
FROM helpdesk_case
WHERE resolved_at >= NOW() - INTERVAL '30 days'
  AND sla_first_response_due_at IS NOT NULL
```

**Period-over-period delta (resolved yesterday vs today):**
```sql
-- Today
SELECT COUNT(*) FROM helpdesk_case WHERE status = 'RESOLVED' AND resolved_at >= CURRENT_DATE
-- Yesterday
SELECT COUNT(*) FROM helpdesk_case WHERE status IN ('RESOLVED','CLOSED') AND resolved_at >= CURRENT_DATE - 1 AND resolved_at < CURRENT_DATE
```

Use Drizzle ORM equivalents with `sql` template literals for these aggregate queries.
