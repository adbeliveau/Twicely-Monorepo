## ROOT CAUSE ANALYSIS

After thorough code review, here are the exact root causes for each bug:

### Bug 1: Agent Replies Not Appearing (reported as "Replies 404")

**Root cause: NOT a 404.** The API route at `src/app/api/hub/helpdesk/cases/[id]/messages/route.ts` EXISTS and works. The actual problem is in `case-workspace.tsx` (lines 43-67):

```typescript
// case-workspace.tsx uses raw fetch() instead of the server action
const handleSendReply = useCallback(async (body: string) => {
  setIsSubmitting(true);
  try {
    await fetch(`/api/hub/helpdesk/cases/${caseDetail.id}/messages`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ body, direction: "OUTBOUND" }),
    });
  } finally {
    setIsSubmitting(false);
  }
}, [caseDetail.id]);
```

**Three problems with this:**
1. **No response error handling.** The fetch does not check `response.ok` or read error responses. If the API returns 401/403 (cookie not forwarded, wrong domain), the error is silently swallowed.
2. **No UI refresh after send.** There is no `router.refresh()` call after a successful send. The new message is written to the database but the RSC page data is never re-fetched, so the message does not appear in the timeline. The user sees nothing happen.
3. **Duplicate code path.** There is ALREADY a server action `addAgentReply` in `src/lib/actions/helpdesk-agent-cases.ts` that does the exact same thing AND calls `revalidatePath('/hd/cases/${caseId}')`. The workspace should use the server action, not the raw fetch.

**Fix:** Replace the raw `fetch()` calls in `case-workspace.tsx` with calls to the `addAgentReply` server action from `helpdesk-agent-cases.ts`. This eliminates the duplicate API route, gets proper error handling, and triggers `revalidatePath` to refresh the page.

### Bug 2: Context Panel Passes Undefined

**Root cause: Date serialization between server and client components.** The `getCaseContext` function (in `helpdesk-context.ts`) returns an object with Date fields (`slaFirstResponseDueAt`, `slaResolutionDueAt`, `firstResponseAt`, `resolvedAt`). These are passed from the server component (`page.tsx`) through to the client component (`CaseWorkspace` -> `ContextPanel`). Next.js RSC serialization converts Dates to strings during the server-to-client boundary crossing. But the `CaseContextData` interface declares these as `Date | null`, and the `ContextPanel` code (e.g., `SlaCountdown` at line 194-204) calls `new Date(due).getTime()` on them — which works whether they're Date objects or ISO strings.

**The ACTUAL issue is more subtle:** The `getCaseContext` function's `tags` field is declared as `string[]` in the `CaseContextData` interface (required, not optional). But the database column `helpdeskCase.tags` has a default of `'{}'::text[]` which Drizzle returns as `string[]`. This should work. However, looking more carefully at `getAgentCaseDetail`:

```typescript
export async function getAgentCaseDetail(caseId: string): Promise<CaseDetail | null> {
  const cases = await db.select().from(helpdeskCase).where(...).limit(1);
  const caseRecord = cases[0];
  // ...
  return { ...caseRecord, messages, events };
}
```

The `db.select()` with no explicit column selection returns ALL columns from `helpdeskCase`. Drizzle returns column values using the column's TypeScript type. The `tags` column is `text('tags').array()` which can return `null` if the database row has a null value (despite the schema default). If a case was created without explicitly setting `tags`, and the seed data or creation code doesn't always provide it, `tags` could be `null`.

**When `tags` is null**, `getCaseContext` sets `contextData.tags = null` (from `caseRecord.tags`). Then in `ContextPanel`, line 77: `{data.tags.length > 0 && ...}` throws a TypeError because `null.length` is not valid. This causes the entire server component render to fail, and Next.js returns an error boundary or blank render — which looks like "undefined" to the developer.

**Secondary issue:** The `requesterStats` query in `getCaseContext` can also fail silently if the `order` table import is wrong or the query errors. A `Promise.all` rejection in `getCaseContext` would prevent the context data from ever being created.

**Fix:**
1. In `getCaseContext`, add null-coalescing on `tags`: `tags: caseRecord.tags ?? []`
2. Add null-safe access in `ContextPanel` for all potentially-null fields
3. Add error handling around the `getCaseContext` call in the page component

### Bug 3: Sidebar Badge Counts Always 0

**Root cause: Two issues.**

**Issue A — Drawer sidebar missing badges prop.** In `helpdesk-layout-client.tsx` (lines 71-84), when the sidebar is in "drawer" mode (case detail pages auto-collapse the sidebar, and toggling opens a drawer overlay), the `HelpdeskSidebar` component is rendered WITHOUT the `badges` prop:

```typescript
// Line 76-79: Drawer sidebar — NO badges prop!
<HelpdeskSidebar
  agent={agent}
  onToggleStatus={setIsOnline}
  permissions={permissions}
  // badges is MISSING here
/>
```

Compare with the non-drawer sidebar (line 61-66) which correctly passes `badges={badges}`. This means on case detail pages, if the user opens the drawer sidebar, all badges show 0 (the default from `badges = {}`).

**Issue B — No seed cases with matching statuses.** The `getHelpdeskBadges` query filters by `OPEN_STATUSES = ['NEW', 'OPEN', 'PENDING_USER', 'PENDING_INTERNAL', 'ON_HOLD', 'ESCALATED']`. If the seed data only creates cases with status `RESOLVED` or `CLOSED`, all badge counts would correctly be 0. The seed file `seed-helpdesk-cases.ts` needs to include cases in open statuses.

**Fix:**
1. Pass `badges` to the drawer sidebar in `helpdesk-layout-client.tsx`
2. Verify seed data includes cases with open statuses (NEW, OPEN, etc.)

### Additional Issues Found During Code Review

**Issue 4 — StatusBadge missing ON_HOLD.** The `CaseStatus` type in `helpdesk-badges.tsx` (line 65-72) is missing `ON_HOLD` from the union type and `statusConfig` map. This means cases with ON_HOLD status would crash the StatusBadge component.

**Issue 5 — Duplicate API route should be deleted.** The POST route at `src/app/api/hub/helpdesk/cases/[id]/messages/route.ts` duplicates the `addAgentReply` server action. Once the workspace uses the server action, the API route should be removed to avoid code duplication and the risk of the two implementations diverging.

**Issue 6 — Case workspace sends without updating status.** Per Helpdesk Canonical section 7.2, when an agent sends a reply, if the case is in `NEW` status it should transition to `OPEN`. The `addAgentReply` server action does NOT do this — it only updates timestamps and `firstResponseAt`. The API route also doesn't do it. This is a separate issue but should be noted.

## SPEC INCONSISTENCIES (for owner decision)

1. **Helpdesk Canonical section 7 says case detail has a 3-column layout (left: properties, center: conversation, right: context).** The implementation uses a DIFFERENT 3-column layout (left: case queue, center: conversation, right: context). Properties like status/priority/tags/assignment are NOT shown as a left column with inline dropdowns — instead, the status badge is in the header bar and the CaseStatusControl is only used on the user-facing support page. This is a larger structural difference that goes beyond G9.1 scope but should be noted for a future alignment pass.

2. **Helpdesk Canonical section 9.2 says macros have a `category` enum field.** The schema (`helpdeskMacro.category`) exists as a text field, not an enum. The `getAgentMacros` query hardcodes `category: 'General'` for all macros instead of reading from the database. This is a minor data wiring issue — the macro category from the DB should be used. However, this is low-priority since macros work functionally.

3. **Helpdesk Canonical section 6.1 says sidebar items are gated by role (HELPDESK_AGENT sees Dashboard/My Cases/All Cases/Views; HELPDESK_LEAD adds Macros/Reports; HELPDESK_MANAGER adds Teams/Routing/SLA/Automation/Settings).** The current `helpdesk-sidebar.tsx` has `permission` strings on items but the layout passes `session.platformRoles` as `permissions`, and the sidebar checks `permissions.includes(permission)`. The permission strings like `helpdesk.macros.view` don't match the platformRole values like `HELPDESK_LEAD`. This means role-gating on sidebar items is currently broken — all items show for all helpdesk roles. This is out of scope for G9.1 but should be tracked.
