### 2.2 Two-Phase Monthly Job

The monthly affiliate payout is a **two-phase** process:

**Phase A: Commission Graduation (runs first)**
Transition PENDING commissions whose `holdExpiresAt < now()` to PAYABLE status. Update `affiliate.pendingBalanceCents` (decrement) and `affiliate.availableBalanceCents` (increment).

**Phase B: Payout Execution (runs second, same job)**
For each ACTIVE affiliate with `availableBalanceCents >= minPayoutCents`:
1. Check affiliate has valid `payoutMethod` and `taxInfoProvided` (if applicable)
2. Create `affiliatePayout` record with status PENDING
3. Execute Stripe Connect transfer (if method = 'stripe_connect')
4. Update payout status to COMPLETED or FAILED
5. Update commissions to PAID
6. Update affiliate balance columns
7. Post ledger entry (AFFILIATE_COMMISSION_PAYOUT)
8. Send notification

### 2.3 Commission Graduation Service

**File:** `src/lib/affiliate/commission-graduation.ts`

```
Purpose: Transition PENDING commissions whose hold has expired to PAYABLE.

Logic:
1. Read affiliate.holdDays from platform_settings (default: 30)
2. Query affiliateCommission WHERE status = 'PENDING' AND holdExpiresAt <= now()
3. Process in batches of 100 (same pattern as expire-free-lister-tier.ts)
4. For each batch:
   a. Group commissions by affiliateId
   b. For each affiliate:
      - Sum commissionCents of graduating commissions
      - UPDATE commissions SET status = 'PAYABLE'
      - UPDATE affiliate SET pendingBalanceCents = pendingBalanceCents - sum,
                            availableBalanceCents = availableBalanceCents + sum,
                            updatedAt = now()
5. Log total graduated count and amount
6. Return { graduatedCount, totalCents }
```

**Business rules:**
- The `holdExpiresAt` is set at commission creation time (30 days from commission date, per Canonical Section 2.6)
- This service reads `affiliate.holdDays` from platform_settings but DOES NOT recalculate hold — it just uses the pre-set `holdExpiresAt` on each commission
- SUSPENDED and BANNED affiliates' commissions still graduate to PAYABLE (money is owed), but they won't be paid out until status returns to ACTIVE
- Idempotent: safe to run multiple times

### 2.4 Affiliate Payout Service

**File:** `src/lib/affiliate/affiliate-payout-service.ts`

```
Purpose: Execute payouts for all eligible affiliates.

Eligibility criteria (ALL must be true):
1. affiliate.status = 'ACTIVE' (not PENDING, SUSPENDED, or BANNED)
2. affiliate.availableBalanceCents >= minPayoutCents (read from platform_settings, default $25.00 = 2500 cents)
3. affiliate.payoutMethod is not null
4. affiliate.taxInfoProvided = true (required before first payout, per Canonical Section 2.6)
5. For stripe_connect: affiliate.stripeConnectAccountId is not null

Processing per eligible affiliate:
1. Calculate payout amount = affiliate.availableBalanceCents
2. Get all PAYABLE commissions for this affiliate
3. Create affiliatePayout record:
   - affiliateId
   - amountCents = payout amount
   - method = affiliate.payoutMethod
   - status = 'PROCESSING'
   - periodStart = first day of previous month (UTC)
   - periodEnd = last day of previous month (UTC)
4. Execute Stripe transfer:
   - stripe.transfers.create({
       amount: payoutAmountCents,
       currency: 'usd',
       destination: affiliate.stripeConnectAccountId,
       description: 'Twicely affiliate commission payout',
       metadata: { affiliateId, payoutId, periodStart, periodEnd }
     })
   - On success: externalPayoutId = transfer.id
   - On failure: mark payout FAILED with failedReason
5. On success:
   a. UPDATE affiliatePayout SET status = 'COMPLETED', externalPayoutId, completedAt = now()
   b. UPDATE affiliateCommission SET status = 'PAID', paidAt = now() WHERE affiliateId AND status = 'PAYABLE'
   c. UPDATE affiliate SET availableBalanceCents = 0, totalPaidCents = totalPaidCents + amount, updatedAt = now()
   d. INSERT ledgerEntry (AFFILIATE_COMMISSION_PAYOUT, negative amount, POSTED)
   e. Send payout notification to affiliate
6. On failure:
   a. UPDATE affiliatePayout SET status = 'FAILED', failedReason = error.message
   b. DO NOT update commission status (remain PAYABLE for next cycle)
   c. DO NOT update affiliate balance
   d. Log error, continue to next affiliate (do not abort batch)
```

**Critical: PayPal payouts are NOT implemented in this step.** Per Canonical Section 2.6, PayPal is an option but Stripe Connect is the primary/preferred method. If `payoutMethod = 'paypal'`, skip with a log warning. PayPal Batch Payout integration is deferred to G5 (Tax & Compliance) where 1099-NEC handling requires it.

**Period calculation:**
- periodStart = first day of the month BEFORE the current month, 00:00:00 UTC
- periodEnd = last day of the month BEFORE the current month, 23:59:59.999 UTC
- Example: job runs March 15 → periodStart = Feb 1, periodEnd = Feb 28/29

### 2.9 Commission Reversal (Refund/Chargeback Handler)

**File:** `src/lib/affiliate/commission-reversal.ts`

Per Canonical Section 2.3: "If a subscription payment is refunded, the corresponding commission is reversed in the next payout cycle." And: "Chargeback → Commission reversed + affiliate flagged for review if pattern emerges."

```
Function: reverseAffiliateCommission(invoiceId: string, reason: string)

Logic:
1. Find affiliateCommission WHERE invoiceId AND status IN ('PENDING', 'PAYABLE')
2. If not found or status = 'PAID': return (no clawback on already-paid, per Canonical §11)
3. UPDATE commission SET status = 'REVERSED', reversedAt = now(), reversalReason = reason
4. UPDATE affiliate balance:
   - If was PENDING: pendingBalanceCents -= commissionCents
   - If was PAYABLE: availableBalanceCents -= commissionCents
5. Log the reversal
```

This function is NOT called by the payout job itself. It should be called from:
- The Stripe subscription webhook handler when a refund event is received
- The chargeback handler when a chargeback event is received

**IMPORTANT:** The actual webhook integration is NOT part of G3.3. G3.3 only creates the reversal function. Wiring it to the subscription webhook is deferred to G3.5 (Anti-fraud checks) since chargeback pattern detection is in scope there.

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Terms
- Never use `FVF`, `Final Value Fee` — use `TF` / `Transaction Fee`
- Never use `wallet` — use `payout`
- Never use `withdraw` — use `Request payout`
- Never use `SellerTier`, `SubscriptionTier` — use `StoreTier` or `ListerTier`
- Never use `Twicely Balance` — use `Available for payout`
- Never use `balance` in user-facing strings for affiliate payouts — use "available for payout" or "pending commissions"

### Tech Stack
- Use `createQueue` / `createWorker` from `src/lib/jobs/queue.ts` — never instantiate BullMQ directly
- Use Valkey connection from queue.ts — never reference Redis
- Use `logger` from `src/lib/logger` — never use `console.log`

### Business Logic Gotchas
- **No clawback on PAID commissions** (Canonical Section 11): Once a commission is PAID, it cannot be reversed. Only PENDING and PAYABLE can be reversed.
- **SUSPENDED/BANNED affiliates**: Commissions still graduate (PENDING → PAYABLE), but payouts are NOT executed. Money accumulates. If unsuspended, payout happens in next cycle.
- **PayPal not implemented**: Skip affiliates with `payoutMethod = 'paypal'`. Log a warning. Do not fail the batch.
- **Tax info gating**: Affiliates earning $600+/year need tax info. Check `taxInfoProvided` before payout. Skip with warning if false.
- **First-touch attribution**: This job does not touch attribution logic. Commissions are already attributed when created (by the subscription webhook).
- **No commission on trial period**: Commissions only exist after paid conversion. This is enforced at commission creation time (subscription webhook), not at payout time.
- **Period boundaries**: Use UTC. The 15th of each month processes the PRIOR month's confirmed commissions. But graduation happens for ALL eligible commissions regardless of when they were created.

### CLAUDE.md Rules
- All money in integer cents
- `strict: true` TypeScript, zero `as any`, zero `@ts-ignore`
- Max 300 lines per file
- Zod `.strict()` on any input schemas
- Explicit field mapping on DB updates (never spread)
- Fee rates from `platform_settings`, never hardcoded

## 4. ACCEPTANCE CRITERIA

### Positive Cases
1. **Commission graduation works**: A commission created >30 days ago with `holdExpiresAt` in the past transitions from PENDING to PAYABLE when the job runs.
2. **Balance updates on graduation**: When a $10.00 commission graduates, `affiliate.pendingBalanceCents` decreases by 1000 and `affiliate.availableBalanceCents` increases by 1000.
3. **Payout executes for eligible affiliate**: An ACTIVE affiliate with `availableBalanceCents >= 2500`, valid `payoutMethod = 'stripe_connect'`, valid `stripeConnectAccountId`, and `taxInfoProvided = true` receives a Stripe transfer.
4. **Payout record created**: An `affiliatePayout` row is created with correct `amountCents`, `method`, `periodStart`, `periodEnd`, and `status = 'COMPLETED'`.
5. **Commissions marked PAID**: All PAYABLE commissions for the affiliate are updated to `status = 'PAID'` and `paidAt` is set.
6. **Affiliate balance zeroed**: After successful payout, `availableBalanceCents = 0` and `totalPaidCents` increased by payout amount.
7. **Ledger entry posted**: An `AFFILIATE_COMMISSION_PAYOUT` ledger entry is created with negative `amountCents` (platform expense), `status = 'POSTED'`.
8. **Notification sent**: Affiliate receives 'affiliate.payout_sent' notification with correct amount and period.
9. **Multiple affiliates processed**: Job processes all eligible affiliates, not just the first one.
10. **Cron schedule correct**: Job is registered as a BullMQ repeatable with pattern `0 6 15 * *` (6:00 UTC on the 15th of each month).

### Negative Cases
11. **SUSPENDED affiliate skipped**: A SUSPENDED affiliate with $100 available balance does NOT receive a payout. Balance unchanged.
12. **BANNED affiliate skipped**: Same as above for BANNED.
13. **PENDING affiliate skipped**: An influencer with status = PENDING (application under review) does NOT receive a payout.
14. **Below minimum skipped**: An ACTIVE affiliate with `availableBalanceCents = 1500` (< 2500 minimum) does NOT receive a payout.
15. **No payout method skipped**: An ACTIVE affiliate with `payoutMethod = null` does NOT receive a payout.
16. **No Stripe account skipped**: An ACTIVE affiliate with `payoutMethod = 'stripe_connect'` but `stripeConnectAccountId = null` does NOT receive a payout.
17. **PayPal skipped (V1)**: An ACTIVE affiliate with `payoutMethod = 'paypal'` is skipped with a log warning.
18. **Tax info missing skipped**: An ACTIVE affiliate with `taxInfoProvided = false` and earnings >= $600 equivalent does NOT receive a payout.
19. **Stripe failure does not abort batch**: If Stripe transfer fails for affiliate A, affiliate B still gets processed.
20. **Failed payout preserves balance**: On Stripe failure, `availableBalanceCents` is NOT decremented and commissions remain PAYABLE.
21. **Already-paid commissions not reversed**: `reverseAffiliateCommission` for a commission with status 'PAID' returns without action.
22. **Master switch disabled**: If `affiliate.enabled` is false, the entire job exits early with no processing.

### Data Integrity
23. **All monetary values stored as integer cents**: `amountCents`, `commissionCents`, `pendingBalanceCents`, `availableBalanceCents`, `totalPaidCents`, `totalEarnedCents`.
24. **Ledger entry is negative**: The `AFFILIATE_COMMISSION_PAYOUT` entry has negative `amountCents` (it's a platform outflow).
25. **Balance consistency**: After graduation + payout, `affiliate.pendingBalanceCents + affiliate.availableBalanceCents + affiliate.totalPaidCents = affiliate.totalEarnedCents` minus any reversals.

### Authorization
26. **Manual trigger requires staff auth**: `triggerAffiliatePayoutManually` requires `staffAuthorize()` with `can('manage', 'AffiliatePayout')`.
27. **FINANCE role can trigger**: A staff member with FINANCE role can trigger manual payout.
28. **Non-FINANCE role cannot trigger**: A staff member with only SUPPORT or MODERATION role gets "Forbidden".

### Vocabulary
29. **No banned terms**: No occurrences of `FVF`, `wallet`, `withdraw`, `Twicely Balance`, `SellerTier`, `SubscriptionTier` in any created/modified file.

## 8. OWNER DECISION NEEDED

### Decision 1: PayPal Payout Implementation
The canonical lists PayPal as a payout method, but there is no PayPal SDK in the codebase and no PayPal integration exists anywhere in V3. This prompt instructs the installer to SKIP PayPal payouts (with a log warning) and defer to G5 (Tax & Compliance).

**If the owner wants PayPal in G3.3, additional scope is needed: PayPal SDK integration, PayPal Batch Payout API, PayPal OAuth, and PayPal webhook handling.**

### Decision 2: Tax Info Enforcement
The canonical says "Affiliates earning $600+/year receive 1099-NEC. Must provide tax info before first payout." The current `taxInfoProvided` field is a simple boolean with no UI to collect tax info. This prompt enforces `taxInfoProvided = true` as a payout gate. The actual tax collection UI/form is deferred to G5 (Tax & Compliance). For testing, `taxInfoProvided` can be manually set to true in the database.

### Decision 3: Commission Creation
This prompt does NOT implement commission creation (the `affiliateCommission` INSERT that happens when a referred user's subscription invoice is paid). That logic belongs in the Stripe subscription webhook handler (`invoice.paid` event). The existing `subscription-webhooks.ts` does not currently create affiliate commissions. This is a prerequisite gap — either:
- (a) Commission creation should be added to subscription-webhooks.ts as part of G3.3, OR
- (b) Commission creation is deferred to a separate step (e.g., G3.5 anti-fraud checks where it can be validated)

**Recommendation:** Add commission creation (webhook → affiliateCommission INSERT) as a sub-step of G3.3, since without it the payout job has nothing to process. This would add approximately 1 more file (commission creation service) and modify the subscription webhook handler.

## 9. SPEC INCONSISTENCY LOG

### Inconsistency 1: affiliatePayout.status is plain text, not enum
The `affiliatePayout` table in the schema uses `text('status')` with string values `'PENDING' | 'PROCESSING' | 'COMPLETED' | 'FAILED'`, while `affiliateCommission.status` uses a proper `commissionStatusEnum`. This is intentional per the schema (Section 21.6) — no change needed, just be aware that status validation must be done in application code, not DB-level.

### Inconsistency 2: Page Registry Missing `/fin/affiliate-payouts`
The Affiliate Canonical (Section 7.2) lists `/fin/affiliate-payouts` as a hub route. The Page Registry does NOT include this route. The Build Tracker assigns this route to G3.4 (Admin Affiliate Management), not G3.3. **This install prompt does NOT create any hub UI pages** — that is G3.4's scope. The manual trigger action is sufficient for G3.3.

### Inconsistency 3: Ledger userId for Platform Expenses
The Finance Engine Canonical (Section 9 of Affiliate Canonical) says affiliate commission payouts create ledger entries but does not specify `userId`. Standard ledger entries have `userId` = the seller. For affiliate payouts (a platform expense), `userId` should be NULL since it's not a seller transaction. The `ledgerEntry.userId` column is nullable (`references(() => user.id)` without `.notNull()`), so NULL is valid.
