## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2.1 vestiaire-types.ts — TypeScript Interfaces

Create `src/lib/crosslister/connectors/vestiaire-types.ts`.

Model after `therealreal-types.ts`. Vestiaire Collective is a luxury peer-to-peer marketplace. Key differences from TRR:
- **Sellers set their own price** (unlike TRR where platform sets consignment prices)
- **Multi-currency**: prices can be EUR, GBP, USD, CHF, SEK, DKK, NOK
- **Different condition labels**: "Never worn", "Very good condition", "Good condition", "Fair condition"
- **No authentication_status** (unlike TRR which authenticates luxury items)
- **URL pattern**: `https://www.vestiairecollective.com/*/p-{id}.html`

Define these interfaces:

```typescript
/** Vestiaire product image */
export interface VestiaireImage {
  id: string;
  url: string;
  position: number;
  is_primary?: boolean;
}

/** Vestiaire brand */
export interface VestiaireBrand {
  id: number;
  name: string;
  slug?: string;
}

/** Vestiaire category */
export interface VestiaireCategory {
  id: number;
  name: string;
  path?: string;
}

/** Vestiaire condition labels */
export type VestiaireCondition =
  | 'Never worn'
  | 'Never worn, with tag'
  | 'Very good condition'
  | 'Good condition'
  | 'Fair condition'
  | string;

/** Vestiaire listing from internal API — modeled after observed JSON-LD + internal API shape */
export interface VestiaireListing {
  id: string;
  title: string;
  description?: string;
  /** Price as decimal string, e.g. "450.00" or "1299.50" */
  price: string;
  /** ISO 4217 currency code — EUR, GBP, USD, CHF, etc. */
  currency?: string;
  condition?: VestiaireCondition;
  /** Listing status */
  status?: 'on_sale' | 'sold' | 'reserved' | 'withdrawn' | 'pending_moderation';
  brand?: VestiaireBrand;
  category?: VestiaireCategory;
  images?: VestiaireImage[];
  /** ISO 8601 timestamps */
  created_at?: string;
  sold_at?: string;
  /** Size information (e.g. "EU 38", "M", "One Size") */
  size?: string;
  /** Color */
  color?: string;
  /** Material */
  material?: string;
  /** Seller-facing URL handle/slug */
  slug?: string;
}

/** Vestiaire paginated response */
export interface VestiaireListingsResponse {
  items: VestiaireListing[];
  page: number;
  per_page: number;
  total: number;
  has_more: boolean;
}

/** Session data stored in crosslisterAccount.sessionData */
export interface VestiaireSessionData {
  sessionToken: string;
  userId: string;
  email: string;
  /** Timestamp when session was captured by extension */
  detectedAt?: number;
  [key: string]: unknown;
}
```

**Critical rule**: The `VestiaireSessionData` shape MUST match what the extension content script (`extensions/chrome/src/content/vestiaire.ts`) produces in `captureSession()`. That function returns `{ sessionToken, userId, email, detectedAt }`.

### 2.3 vestiaire-normalizer.ts — Normalizer

Create `src/lib/crosslister/connectors/vestiaire-normalizer.ts`.

Model after `therealreal-normalizer.ts`. Must implement:
- `normalizeVestiaireListing(raw: VestiaireListing): VestiaireNormalizedData`
- `toExternalListing(normalized: VestiaireNormalizedData): ExternalListing`
- `parseVestiairePrice(value: string): number` — decimal string to integer cents (same as TRR's `parseTrrPrice`)

**Condition mapping** (Vestiaire condition labels to Twicely listing conditions):

```typescript
const VESTIAIRE_CONDITION_MAP: Record<string, string | null> = {
  'Never worn': 'NEW_WITHOUT_TAGS',
  'Never worn, with tag': 'NEW_WITH_TAGS',
  'Very good condition': 'VERY_GOOD',
  'Good condition': 'GOOD',
  'Fair condition': 'ACCEPTABLE',
};
```

**Status mapping** (Vestiaire status to ExternalListing status):

```typescript
function mapVestiaireStatus(status: string | undefined): 'ACTIVE' | 'SOLD' | 'ENDED' | 'DRAFT' {
  switch (status) {
    case 'on_sale': return 'ACTIVE';
    case 'sold': return 'SOLD';
    case 'reserved': return 'ACTIVE'; // Reserved is still active on Vestiaire
    case 'withdrawn':
    case 'pending_moderation':
    default: return 'ENDED';
  }
}
```

**Multi-currency handling**: The normalizer does NOT convert currencies. It passes the currency code through as-is. The import pipeline stores the original `currencyCode` on the ExternalListing. Currency conversion (if needed) happens at listing creation time, not normalization time. This matches the TRR pattern (TRR also passes `currencyCode` through).

**Key fields**:
- `quantity`: Always 1 (luxury single items, same as TRR)
- `brand`: From `raw.brand?.name`
- `category`: From `raw.category?.name`
- `url`: Construct from ID: `https://www.vestiairecollective.com/products/p-${raw.id}.html` OR from slug if available
- `shippingType`, `shippingPriceCents`, `weight`, `dimensions`: All null (not available from Vestiaire API)
- `itemSpecifics`: Include `size`, `color`, `material`, `vestiaireCondition` (original condition label) if present

**VestiaireNormalizedData interface**: Same shape as `TrrNormalizedData` — mirrors ExternalListing but with explicit null types for shipping fields.

Create `src/lib/crosslister/connectors/vestiaire-connector.ts`.

Model CLOSELY after `therealreal-connector.ts`. Implements `PlatformConnector` interface.

```
readonly channel: ExternalChannel = 'VESTIAIRE';
readonly tier: ConnectorTier = 'C';
readonly version = '1.0.0';

```

**Capabilities** (must match channel-registry.ts line 301-307):
```typescript
const VESTIAIRE_CAPABILITIES: ConnectorCapabilities = {
  canImport: true,
  canPublish: true,
  canUpdate: false,     // Tier C — no update via API
  canDelist: true,
  hasWebhooks: false,   // No API, no webhooks
  hasStructuredCategories: false,
  canAutoRelist: false,
  canMakeOffers: false,
  canShare: false,
  maxImagesPerListing: 12,
  maxTitleLength: 80,
  maxDescriptionLength: 2000,
  supportedImageFormats: ['jpg', 'jpeg', 'png'],
};
```

**Config loading** — `loadVestiaireConfig()`:
Read from `platform_settings` where category = 'crosslister':
- `crosslister.vestiaire.apiBase` (default: `'https://www.vestiairecollective.com/api'`)
- `crosslister.vestiaire.userAgent` (default: `'Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X) AppleWebKit/605.1.15'`)

**Session data extraction** — `extractSessionData(account)`:
Extract `VestiaireSessionData` from `account.sessionData`. The extension content script saves:
```typescript
{
  sessionToken: string,  // from _vc_session / _vestiaire_session / vc_token cookie
  userId: string,
  email: string,
  detectedAt: number
}
```

The connector uses `sessionToken` as the auth cookie. Check that `sessionToken` is a non-empty string. Return null if missing.

**Methods to implement** (all following the TRR pattern):

1. **`authenticate(credentials: AuthInput): Promise<AuthResult>`**
   - Only accepts `method: 'SESSION'`
   - Calls `POST {apiBase}/sessions` with email/password
   - Returns session data on success
   - On error returns `{ success: false, error: 'message' }`

2. **`refreshAuth(account): Promise<AuthResult>`**
   - Tests session by fetching 1 listing: `GET {apiBase}/items?per_page=1`
   - Headers: `Cookie: _vc_session={sessionToken}`, `User-Agent`, `X-CSRF-Token` if available
   - If 200: session still valid, return success
   - If not OK: return `{ success: false, error: 'Session expired. Please reconnect.' }`

3. **`revokeAuth(account): Promise<void>`**
   - Best-effort DELETE to sessions endpoint
   - Swallow errors (same as TRR)

4. **`fetchListings(account, cursor?): Promise<PaginatedListings>`**
   - This is the CORE import method
   - Endpoint: `GET {apiBase}/items?page={page}&per_page=50`
   - Use page-based pagination (same as TRR): cursor = page number string
   - Headers: `Cookie: _vc_session={sessionToken}`, `User-Agent`
   - Parse response as `VestiaireListingsResponse`
   - Map each item through `normalizeVestiaireListing()` then `toExternalListing()`
   - Filter to ACTIVE status only
   - Handle 401 (session expired): return empty result
   - Handle network errors: return empty result, log error

5. **`fetchSingleListing(account, externalId): Promise<ExternalListing>`**
   - Endpoint: `GET {apiBase}/items/{externalId}`
   - Parse, normalize, return
   - Throw on non-OK response

6. **`createListing(account, listing): Promise<PublishResult>`**
   - POST to `{apiBase}/items`
   - 2-8 second human-like delay before API call (Tier C safeguard, Lister Canonical Section 9.4)
   - Build body: title, description, price_cents, brand, condition, images, category
   - Return PublishResult with externalId and URL on success

7. **`updateListing(): Promise<UpdateResult>`**
   - Tier C — always returns `{ success: false, error: 'Vestiaire Collective does not support listing updates via API', retryable: false }`

8. **`delistListing(account, externalId): Promise<DelistResult>`**
   - PUT to `{apiBase}/items/{externalId}` with `{ status: 'withdrawn' }`
   - 2-8 second delay before API call
   - Return success/failure

9. **`verifyListing(): Promise<VerificationResult>`**
   - Return stub: `{ exists: false, status: 'UNKNOWN', priceCents: null, quantity: null, lastModifiedAt: null, diff: null }`

10. **`healthCheck(account): Promise<HealthResult>`**
    - Fetch 1 item to test connectivity
    - Return latency measurement

**Self-registration at module load**:
```typescript
registerConnector(new VestiaireConnector());
```

**File size**: Target under 300 lines. If it exceeds, extract `fetchVestiaireListings()` and `fetchSingleVestiaireItem()` into `vestiaire-import.ts` (same pattern as `shopify-import.ts`).

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Patterns
- NO `as any` — fix the type
- NO `@ts-ignore` / `@ts-expect-error`
- NO hardcoded fee rates — use platform_settings
- NO currency conversion in the normalizer — pass currencyCode through as-is
- NO files over 300 lines
- NO `console.log` — use `logger` from `@/lib/logger`
- NO direct HTTP calls from content scripts — content scripts capture session data; the connector makes server-side calls
- NO `'use server'` on any of these files — they are plain TypeScript modules

### Gotchas
- **Multi-currency**: Vestiaire prices can be EUR, GBP, CHF, etc. The normalizer MUST preserve the original `currencyCode` from the API response. Do NOT assume USD. Default to `'EUR'` (Vestiaire is a French company, EUR is most common) if currency field is absent, NOT `'USD'` like TRR.
- **Session token field name**: The content script saves `sessionToken` (camelCase), NOT `sessionId` like TRR. Check the correct field name.
- **Price format**: Vestiaire prices from the internal API are decimal strings like TRR ("450.00"). European prices use comma-decimal (e.g., "1.299,50") in the UI but the API typically returns period-decimal. The `parseVestiairePrice()` function should handle both formats.
- **URL construction**: Vestiaire product URLs follow the pattern `/*/p-{id}.html`. Include the `p-` prefix in the URL.
- **`canUpdate: false`**: Unlike eBay/Etsy, Vestiaire does not support remote listing updates. The `updateListing()` method must immediately return failure.
- **Status values differ from TRR**: Vestiaire uses `on_sale` (not `listed`), `withdrawn` (not `expired`), and has `reserved` and `pending_moderation`.

### Route Prefixes
No routes are created in this step (connector-only). No UI changes.

### Tech Stack
- Drizzle for DB access (platform_settings reads)
- Zod for schema validation
- `fetch()` for HTTP calls to Vestiaire internal API
- `@/lib/logger` for logging

## 8. SPEC GAPS — Owner Decision Needed

### Gap 1: Vestiaire Internal API Shape
**NOT SPECIFIED** — Vestiaire Collective has no public API documentation. The API endpoint paths (`/api/items`), response fields, and pagination format are inferred from the extension content script patterns and modeled after TheRealReal's observed API. The installer should implement the interfaces as specified in this prompt, but production values may need adjustment once real API responses are observed.

### Gap 2: Default Currency
**DECISION MADE IN THIS PROMPT**: Default to `'EUR'` when Vestiaire API omits currency, because Vestiaire Collective is a French company headquartered in Paris and EUR is the most common currency on the platform. This differs from TRR which defaults to `'USD'`. If the owner disagrees, change the default in `normalizeVestiaireListing()`.

### Gap 3: Vestiaire Authentication Endpoint
**NOT SPECIFIED** — The `authenticate()` method assumes Vestiaire has a `POST /api/sessions` endpoint (modeled after TRR). In practice, Vestiaire authentication will primarily happen through the browser extension session capture (`captureSession()` in vestiaire.ts). The direct login endpoint may not exist or may have CAPTCHA protection. The connector should still implement `authenticate()` for completeness, but session capture via extension will be the primary auth flow.

### Gap 4: Price Format in API Responses
**ASSUMED**: The Vestiaire internal API returns prices as decimal strings (e.g., "450.00") in period-decimal format, same as most JSON APIs. The European comma-decimal format ("450,00") is only used in the HTML UI, not the API JSON. If this assumption is wrong, `parseVestiairePrice()` needs to handle comma-decimal format too.

### Gap 5: canShare Capability
**CONFIRMED FALSE** — The channel-registry (line 302) has `canShare: false` for VESTIAIRE. This matches: Vestiaire does not have a social "share" feature like Poshmark. No change needed.
