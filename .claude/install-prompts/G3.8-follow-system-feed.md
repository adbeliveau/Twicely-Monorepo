### 2.3 Queries (New File)

**File: `src/lib/queries/follow.ts`**

1. `getFollowedSellers(userId: string)` — Returns list of followed sellers with profile info (storeName, storeSlug, avatarUrl, followerCount, listingCount, performanceBand, memberSince, lastListedAt). Used by `/my/buying/following` page.
   - JOIN follow -> user -> sellerProfile -> sellerPerformance
   - Include listing count (active only) and follower count
   - Include most recent listing date (`lastListedAt`) for "X new listings" badge
   - Order by `follow.createdAt DESC` (most recently followed first)
   - Return type: `FollowedSellerData[]`

2. `getFollowedSellerNewListings(userId: string, limit?: number)` — Returns recent ACTIVE listings from all followed sellers, newest first. Used by feed Section 1.
   - JOIN follow -> listing (where listing.ownerUserId = follow.followedId AND listing.status = 'ACTIVE')
   - JOIN listing -> listingImage (primary), listing -> user (seller info), listing -> sellerProfile -> sellerPerformance
   - Filter: listings created in last 7 days (configurable? Keep hardcoded for V1)
   - Order by `listing.createdAt DESC`
   - Limit: read from `social.feed.followedSellerListingLimit` platform setting, default 20
   - Return type: `ListingCardData[]` (reuse existing type)

3. `getFollowerCount(userId: string)` — Returns the number of followers for a user. (Simple count query on follow table. Already done inline in storefront-public.ts but not as standalone.)

**File: `src/lib/queries/feed.ts`**

4. `getForYouFeed(userId: string, page?: number, pageSize?: number)` — The V1 personalized feed query per Personalization Canonical Section 13.
   - **Section 1 — Social override:** New listings from followed sellers (per query #2). These appear FIRST regardless of interest match. Following does NOT add interest weight (Personalization Canonical Section 9).
   - **Section 2 — Watchlist price drops:** Listings on user's watchlist where `listing.priceCents < watchlistItem_snapshot_price` — DEFER this section. Watchlist price tracking requires a `previousPriceCents` on watchlistItem or a separate price history table. Mark as `// TODO G3.9: watchlist price drops` and return empty array for this section.
   - **Section 3 — Interest-matched listings:** The core personalized feed query from Personalization Canonical Section 13. Adapted from the canonical pseudo-SQL:
     - CTE: aggregate user's interest weights by tag (SUM weight, exclude expired)
     - JOIN listing -> category -> interestTag (via `categoryId = ANY(interestTag.categoryIds)`)
     - LEFT JOIN to user_weights CTE on tag_slug
     - WHERE listing.status = 'ACTIVE'
     - Exclude listings already in Section 1 (followed seller listings)
     - ORDER BY relevance_score DESC, createdAt DESC
     - Limit: read from `social.feed.interestListingLimit`, default 40
   - **Section 4 — Boosted listings matching interests:** Listings with active promotions (`promotedListing.isActive = true AND promotedListing.expiresAt > NOW()`) that also match user interests. Lowest priority. Boosted listings that DON'T match interests are excluded from "For You" (Personalization Canonical Section 10).
   - Returns: `FeedData` containing:
     - `followedListings: ListingCardData[]` (Section 1)
     - `watchlistDrops: ListingCardData[]` (Section 2 — empty for V1)
     - `matchedListings: ListingCardData[]` (Section 3)
     - `boostedListings: ListingCardData[]` (Section 4)
     - `hasInterests: boolean` — whether user has any interest weights (determines whether to show For You or redirect to Explore)

5. `getFollowingCount(userId: string)` — Count of sellers the user follows. Used in sidebar badge.

- No `wallet`, `Twicely Balance`, `Withdraw` in any UI text
- No `SellerTier`, `FVF`, `BASIC`, `ELITE`, `PREMIUM`

- `/my/buying/following` — NOT `/following`, NOT `/my/following`, NOT `/dashboard/following`
- `/my/feed` — NOT `/feed`, NOT `/my/buying/feed`, NOT `/dashboard/feed`
- `/st/` prefix for store links — NOT `/store/`, NOT `/s/`

### Architecture Constraints
- Following does NOT add interest weight (Personalization Canonical Section 9: "Following does NOT add interest weight. It's a separate feed source.")
- Feed is NOT a social media feed. Feature Lock-in Section 12: "Lightweight — not a social media feed. Just new listing notifications from followed sellers."
- No algorithmic feed based on social activity (Feature Lock-in Section 12: "No algorithmic feed based on social activity")
- No follower requirements for visibility
- No gamification around social actions
- No "share to promote" algorithm (Poshmark's toxic model)
- Search and categories are NEVER interest-filtered (Personalization Canonical Sections 8)
- Interest tags are platform-curated, not user-generated

### Tech Stack
- Use Drizzle ORM for all queries (NOT Prisma)
- Use `authorize()` from `@/lib/casl` for auth checks (NOT `auth.api.getSession()` directly in actions)
- Pages can use `auth.api.getSession()` for SSR (existing pattern)
- No Zustand/Redux — use server state with Suspense
- No tRPC — use server actions + queries

### Code Quality
- Max 300 lines per file
- Money in integer cents only
- Zod `.strict()` on all input schemas
- Explicit field mapping (no spread of request bodies)
- Ownership via `userId` (never `storeId` or `sellerProfileId`)
- No `as any`, `@ts-ignore`, `@ts-expect-error`

## 4. ACCEPTANCE CRITERIA

### Positive Cases

1. **Following page shows all followed sellers.** A user who follows 3 sellers sees all 3 on `/my/buying/following` with correct store names, avatars, listing counts, and follower counts.

2. **Unfollowing works from Following page.** Clicking "Unfollow" on a seller card removes them from the page after server action completes.

3. **Feed shows followed seller listings first.** A user who follows seller A sees seller A's new listings in the "New from sellers you follow" section at the top of `/my/feed`, regardless of whether those listings match the user's interests.

4. **Feed shows interest-matched listings.** A user with "Sneakers" interest (weight 10.0) sees sneaker listings ranked higher than random electronics in the "Picked for you" section.

5. **Feed respects interest weights.** Listings matching a higher-weight interest appear before listings matching a lower-weight interest.

6. **Feed excludes expired interest signals.** A CLICK signal with `expiresAt` in the past does not influence feed ranking.

7. **Boosted listings only appear if matching interests.** A boosted electronics listing does NOT appear in the feed of a user who has only fashion interests.

8. **Empty state: no follows, no interests.** A brand-new user who skipped onboarding sees the empty state message with a CTA to pick interests.

9. **Empty state: follows but no new listings.** A user who follows sellers that haven't listed anything recently sees an empty "New from sellers you follow" section (with a message like "No new listings from sellers you follow").

10. **Feed does not show user's own listings.** A seller viewing their own feed does NOT see their own listings in any section.

11. **Sidebar nav entry for Following is enabled.** The nav item is clickable and navigates to `/my/buying/following`.

12. **Feed page requires authentication.** Unauthenticated users are redirected to login.

13. **Social override crosses categories.** A fashion buyer who follows an electronics seller sees that seller's electronics listings in Section 1 of the feed.

### Negative Cases

14. **Cannot follow yourself.** `toggleFollow()` with own userId returns error (already implemented).

15. **Duplicate follows impossible.** Unique constraint on `(followerId, followedId)` prevents duplicate entries.

16. **Unauthenticated users cannot access `/my/buying/following`.** Must redirect to login.

17. **Unauthenticated users cannot access `/my/feed`.** Must redirect to login.

18. **Feed does not filter search results.** This step does NOT modify search behavior. Search remains unfiltered by interests.

19. **Feed does not filter category pages.** This step does NOT modify category browsing.

20. **No notification spam.** The notification template is defined but NOT wired to listing creation. No notifications are sent in this step.

### Data Integrity

21. **All listing prices displayed in correct cents format.** Feed listing cards use the existing `ListingCardData` type which stores `priceCents`.

22. **Follow records use CUID2 IDs.** The `follow` table uses `$defaultFn(() => createId())` (already implemented).

23. **Feed query handles users with zero interests gracefully.** The CTE returns empty results; the query falls back to newest listings.

## 8. OPEN QUESTIONS FOR OWNER

### Q1: `/my/feed` Route

The Page Registry does NOT list `/my/feed`. The Personalization Canonical places the personalized feed as a homepage tab (For You / Explore / Categories on `/`). The Build Tracker calls this step "Follow system + feed (`/my/feed`)." Which is correct?

- **Option A:** Build `/my/feed` as a standalone page (matching build tracker). The homepage remains unchanged until G3.9.
- **Option B:** Skip `/my/feed` and build the feed as homepage tabs at `/` (matching Personalization Canonical Section 5). This would be a larger change involving homepage refactoring.

**Recommended:** Option A — build `/my/feed` now, refactor homepage in G3.9 (Explore/Discovery page). The feed query is the same either way; only the rendering location differs.

### Q2: Dashboard Widget

Feature Lock-in Section 12 says: `/my` dashboard widget: "Sellers you follow listed 12 new items today." The current `/my/page.tsx` is a minimal stub (just "Dashboard" + sign out button). Should this step add the widget to the dashboard, or defer to when the dashboard is properly built?

**Recommended:** Defer. The dashboard page is a stub. Adding a widget to a stub page adds no value. When the dashboard is properly built (separate step), the widget can be added using the `getFollowedSellerNewListings()` query from this step.

### Q3: Watchlist Price Drops in Feed

The Personalization Canonical Section 5 lists "Price drops on watchlist" as Section 2 of the "For You" feed. However, there is no mechanism to track previous prices on watchlist items. The `watchlistItem` table has no `snapshotPriceCents` column, and the `listing` table has `originalPriceCents` but that's the retail price, not the price when the item was watchlisted.

**Recommended:** Return empty array for Section 2 with a `// TODO G3.9: watchlist price drops — requires snapshotPriceCents on watchlistItem` comment. This avoids a schema change in G3.8.

### Q4: Feed vs. Homepage Tabs Architecture

When G3.9 builds the Explore/Discovery page, the feed query from this step will move from `/my/feed` to be one of three homepage tabs. Should the query be designed with this migration in mind (e.g., accept a `mode: 'for_you' | 'explore'` parameter)?

**Recommended:** No. Build the simplest correct thing now. G3.9 can import and reuse `getForYouFeed()` wherever it renders. The query function is independent of where it's rendered.
