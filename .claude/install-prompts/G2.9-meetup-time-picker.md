## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2A. SCHEMA CHANGES

**CRITICAL ISSUE:** The current `localTransaction.scheduledAt` column is `notNull()`:
```typescript
scheduledAt: timestamp('scheduled_at', { withTimezone: true }).notNull(),
```

Per Addendum A6, `scheduledAt` is **null** after checkout and gets set when both parties agree. The column must be changed to **nullable**:
```typescript
scheduledAt: timestamp('scheduled_at', { withTimezone: true }),
```

This is a breaking change. The `createLocalTransaction()` function in `src/lib/commerce/local-transaction.ts` currently passes `scheduledAt` at creation. After this change, it should pass `null` (or omit the field). The `checkInToMeetupAction` must enforce that `scheduledAt` is non-null before allowing check-in (see Section 2C).

**New columns on `localTransaction`** (per Addendum A6):

```typescript
// After the existing scheduledAt column:
scheduledAtConfirmedAt:   timestamp('scheduled_at_confirmed_at', { withTimezone: true }),
schedulingProposedBy:     text('scheduling_proposed_by'),  // userId of who proposed the accepted time
```

**New indexes:** None required (these columns are not queried by).

### 2D. SERVER ACTIONS

Create `src/lib/actions/local-scheduling.ts` (`'use server'`):

**Action 1: `proposeMeetupTimeAction`**

Input: `z.infer<typeof proposeMeetupTimeSchema>`
Output: `{ success: boolean; error?: string }`

Logic:
1. `authorize()` — must be authenticated
2. Validate input with `proposeMeetupTimeSchema.safeParse()`
3. Fetch transaction by ID from DB
4. Verify caller is either `buyerId` or `sellerId` (return `'Not found'` otherwise)
5. CASL check: `ability.can('update', sub('LocalTransaction', { buyerId/sellerId }))`
6. Check `isUserSuspendedFromLocal(session.userId)` — if suspended, return error
7. Validate transaction state:
   - `scheduledAtConfirmedAt` must be `null` (if already confirmed, scheduling is done — changes go through G2.10 Reschedule Flow)
   - Transaction status must NOT be terminal (COMPLETED, CANCELED, NO_SHOW)
8. Validate proposed time:
   - Parse `proposedAt` as Date
   - Must be >= 1 hour from now (minimum lead time)
   - Must be <= 30 days from now (maximum lead time)
9. Update `localTransaction`:
   - Set `scheduledAt` to the proposed datetime
   - Set `schedulingProposedBy` to `session.userId`
   - Set `scheduledAtConfirmedAt` to `null` (clear any previous state — should already be null for initial scheduling)
   - Set `updatedAt` to now
10. Send in-app notification to the OTHER party:
    - Use `notify(otherPartyId, 'local.schedule.proposal', { ... })`
    - The other party is: if caller is buyer, other = sellerId; if caller is seller, other = buyerId
11. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`
12. Return `{ success: true }`

**Action 2: `acceptMeetupTimeAction`**

Input: `z.infer<typeof acceptMeetupTimeSchema>`
Output: `{ success: boolean; error?: string }`

Logic:
1. `authorize()` — must be authenticated
2. Validate input with `acceptMeetupTimeSchema.safeParse()`
3. Fetch transaction by ID from DB
4. Verify caller is either `buyerId` or `sellerId` (return `'Not found'` otherwise)
5. CASL check: `ability.can('update', sub('LocalTransaction', { buyerId/sellerId }))`
6. Validate transaction state:
   - `scheduledAt` must NOT be null (there must be a proposed time to accept)
   - `scheduledAtConfirmedAt` must be `null` (not already confirmed)
   - `schedulingProposedBy` must NOT equal `session.userId` (you cannot accept your own proposal)
7. Validate the proposed time is STILL valid:
   - `scheduledAt` must be >= 1 hour from now (if the proposed time has become stale, reject)
8. Update `localTransaction`:
   - Set `scheduledAtConfirmedAt` to now
   - Set `updatedAt` to now
   - (Do NOT change `scheduledAt` or `schedulingProposedBy` — those were set by the proposer)
9. Enqueue BullMQ jobs (see Section 2E)
10. Send in-app notification to the OTHER party:
    - Use `notify(otherPartyId, 'local.schedule.accepted', { ... })`
11. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`
12. Return `{ success: true }`

**IMPORTANT:** Both actions must be unexported helpers kept as private functions where needed. Only the two action functions themselves should be `export`ed.

### 2E. BULLMQ JOB ENQUEUE ON ACCEPTANCE

When `acceptMeetupTimeAction` succeeds, the following BullMQ jobs must be enqueued. These use EXISTING queue/worker infrastructure:

1. **Auto-cancel job** — `enqueueLocalAutoCancel()` from `src/lib/jobs/local-auto-cancel.ts`
   - Already exists and is called at `createLocalTransaction()` time
   - Currently fires based on `autoCancelHours` (48) from creation time
   - After G2.9: the auto-cancel should fire at `scheduledAt + 48hr`. Since `scheduledAt` is now set at acceptance time (not creation), the job must be enqueued HERE, not at creation time.
   - **Action required:** Remove the `enqueueLocalAutoCancel()` call from `createLocalTransaction()` in `src/lib/commerce/local-transaction.ts`. Move it to `acceptMeetupTimeAction`.

2. **No-show check job** — `enqueueNoShowCheck()` from `src/lib/jobs/local-noshow-check.ts`
   - Currently triggered at check-in time, NOT at scheduling time
   - Per Addendum A6, the no-show check fires at `scheduledAt + 30min`. But this is the *existing G2.3 logic* which already fires at check-in. No change needed here — the existing no-show check at check-in is correct.

3. **24hr + 1hr reminder jobs** — These are specified in Addendum A10 (G2.13) as BullMQ delayed jobs. **G2.13 is a SEPARATE step.** For G2.9, we prepare the integration point by creating a placeholder function `enqueueSchedulingReminders()` that the G2.13 step will implement. For now, it logs and returns.

4. **Scheduling nudge** — Per Decision #117: "If `scheduledAt` still null 24hr after checkout — nudge notification to both parties." This is a separate BullMQ job that should be enqueued at `createLocalTransaction()` time (replacing the removed auto-cancel). Create `src/lib/jobs/local-schedule-nudge.ts`:
   - Queue name: `local-schedule-nudge`
   - Delay: `scheduleReminderHours` (24) hours from platform_settings
   - Worker checks: if `scheduledAtConfirmedAt` is still null, sends nudge notification to both parties
   - If `scheduledAtConfirmedAt` is set (time was agreed), job is a no-op

### 2I. UPDATE `createLocalTransaction()`

Modify `src/lib/commerce/local-transaction.ts`:

1. Remove `scheduledAt` from the `CreateLocalTransactionInput` interface (make it optional or remove it)
2. In the `insert` call, pass `scheduledAt: null` (or omit — column is now nullable)
3. Remove the `enqueueLocalAutoCancel()` call — this now happens at scheduling acceptance time
4. Keep token generation — tokens are still generated at creation time per A4 (tokens are generated with a placeholder expiry; they get regenerated at scheduling confirmation if needed)

**IMPORTANT DECISION POINT:** Tokens require `scheduledAt` for their `expiresAt` calculation (currently `expiresAt = scheduledAt + tokenExpiryHours`). With `scheduledAt` now null at creation time, tokens CANNOT be generated at creation. Two options:

- **Option A (RECOMMENDED):** Defer token generation to `acceptMeetupTimeAction`. Tokens are generated when scheduling is confirmed. The `createLocalTransaction()` function sets placeholder values for the confirmation code columns (they're `notNull()`). Tokens are regenerated with real expiry when `scheduledAt` is confirmed.
- **Option B:** Generate tokens with a far-future expiry at creation, then regenerate with correct expiry at scheduling. More complex, same outcome.

**Use Option A.** The `createLocalTransaction()` function already uses placeholder codes (`pending-${Date.now()}-s`) that get replaced. Keep that pattern — placeholders at creation, real tokens at scheduling confirmation.

**NOT SPECIFIED — Owner decision needed:** The existing code sets `scheduledAt` as `notNull()` and passes it at transaction creation time. Multiple downstream systems (auto-cancel, token expiry) depend on `scheduledAt` being set at creation. Changing to nullable requires auditing all consumers. The installer should search for all references to `localTransaction.scheduledAt` and update null handling.

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Terms

- Never use "escrow" in UI copy (use "SafeTrade" when referencing the payment hold)
- Never use "wallet", "withdraw", "Twicely Balance" in any UI text
- Never use "FVF" or "Final Value Fee" — use "TF" / "Transaction Fee"
- Never use any of the banned tier names (BASIC, ELITE, PLUS, MAX, PREMIUM, STANDARD, RISING)

### Tech Stack

- Do NOT use any date picker library (react-datepicker, etc.). Use native `<input type="datetime-local">`.
- Do NOT use Zustand/Redux for state. Use `useState` / `useTransition`.
- Do NOT use tRPC. Use server actions.

### Code Patterns

- Do NOT hardcode the 1-hour minimum lead time or 30-day maximum. Read from constants defined in the scheduling service file, not from platform_settings (these are UX constraints, not admin-configurable rates).
- Do NOT spread request body into DB update. Use explicit field mapping.
- Do NOT use `as any` or `@ts-ignore`.
- Keep all files under 300 lines.
- All money values in integer cents.
- All Zod schemas use `.strict()`.

### Business Logic

- Do NOT implement the Reschedule Flow (G2.10) — once `scheduledAtConfirmedAt` is set, the time picker is locked. Reschedule is a separate step.
- Do NOT implement the 24hr/1hr BullMQ reminder notifications (G2.13) — just enqueue the auto-cancel. Create the placeholder function for reminder enqueue.
- Do NOT implement the day-of confirmation request (G2.12) — that's separate.
- Do NOT implement SafeTrade payment flow — SafeTrade is not yet built. The time picker works for ALL local transactions regardless of payment method.
- The time picker is NOT gated by SafeTrade. Per Addendum A0, the time picker is a SafeTrade-only feature. **HOWEVER**, SafeTrade is not yet implemented, and the scheduling mechanism is needed for the entire local flow to function. **Implement the time picker for all local transactions now. The SafeTrade gate will be added when SafeTrade is built.** Flag this with a `// TODO(SafeTrade): Gate behind SafeTrade when A0 is implemented` comment.

### Gotchas

1. **`scheduledAt` is currently `notNull()`.** The schema change to nullable is the most impactful part of this step. Every consumer of `scheduledAt` must handle null.
2. **Token generation depends on `scheduledAt`.** Tokens cannot have a valid `expiresAt` without a scheduled time. Token generation must be deferred to acceptance.
3. **Auto-cancel timing changes.** Currently enqueued at creation with a 48hr delay from creation time. After G2.9, it should be `scheduledAt + 48hr`. The delay calculation must use the confirmed `scheduledAt`, not the current time.
4. **The `LocalMeetupCard` is 277 lines.** Adding the time picker integration risks exceeding 300 lines. The time picker MUST be a separate component, not inlined.
5. **Both order detail pages (buyer and seller) need the `otherPartyName` prop.** The buyer page already has `seller.storeName ?? seller.name`. The seller page already computes `buyerDisplayName`.

## 4. ACCEPTANCE CRITERIA

### Positive Cases

1. Either buyer or seller can propose a meetup time via the time picker on the order detail page.
2. The other party sees the proposal and can accept it, which sets `scheduledAtConfirmedAt`.
3. The other party can counter-propose by selecting a different time instead of accepting.
4. After acceptance, the check-in button becomes available.
5. After acceptance, the auto-cancel BullMQ job is enqueued with delay = `scheduledAt + 48hr - now`.
6. The scheduling nudge BullMQ job fires if `scheduledAt` is still not confirmed 24 hours after order creation.
7. Both parties receive in-app notifications for proposals and acceptances.
8. Time picker shows 30-minute increments.
9. Proposed times must be at least 1 hour in the future.
10. Proposed times must be at most 30 days in the future.
11. The `scheduledAt` column is nullable and is `null` after local transaction creation.
12. `scheduledAtConfirmedAt` and `schedulingProposedBy` columns exist on `localTransaction`.

### Negative Cases

13. A user CANNOT accept their own proposal (must be accepted by the other party).
14. A user CANNOT check in before `scheduledAtConfirmedAt` is set.
15. A user CANNOT propose a time in the past or within the next hour.
16. A user CANNOT propose a time more than 30 days away.
17. A user who is suspended from local transactions CANNOT propose a time (error returned).
18. A user who is NOT the buyer or seller CANNOT propose or accept (returns "Not found").
19. Once `scheduledAtConfirmedAt` is set, the time picker is no longer interactive (shows confirmed state).
20. Unauthenticated users cannot call the scheduling actions (returns "Unauthorized").

### Data Integrity

21. All timestamps stored with timezone (`{ withTimezone: true }`).
22. `schedulingProposedBy` stores a userId (CUID2), not a role string.
23. `scheduledAtConfirmedAt` is set to server time, never client time.
24. Zod schemas use `.strict()` — unknown keys rejected.

### Authorization

25. Both actions use `authorize()` from `@/lib/casl`.
26. CASL check uses `sub('LocalTransaction', { buyerId/sellerId })` for condition matching.
27. Error messages never reveal internal IDs — always "Not found".

### Vocabulary

28. No banned terms appear in any UI text or code comments.
29. UI says "Schedule Your Meetup", not "Set meetup time" or "Pick a time".

- Transaction created before G2.9 migration may have `scheduledAt` set but `scheduledAtConfirmedAt` null — treat as unconfirmed. **Migration note:** existing local transactions with `scheduledAt` set should have `scheduledAtConfirmedAt` backfilled to their `createdAt` value, OR the check-in guard should allow check-in when `scheduledAt` is set even if `scheduledAtConfirmedAt` is null for pre-migration records.

## 9. SPEC INCONSISTENCIES AND OPEN QUESTIONS

1. **Schema vs. Addendum conflict on `scheduledAt` nullability:** The schema doc (v2.1.0, section 19.1) shows `scheduledAt` as `notNull()`. The Addendum A6 says "`localTransaction.scheduledAt` is null" after checkout. The Addendum supersedes per its header. **Resolution: Make nullable per Addendum.**

2. **Token generation timing:** The existing code generates Ed25519 tokens at `createLocalTransaction()` time, using `scheduledAt` for expiry. With `scheduledAt` now null at creation, token generation must be deferred. This is a functional change to G2.7's implementation. **The installer must move token generation to `acceptMeetupTimeAction` or a separate `generateTokensOnSchedulingConfirmation()` function.**

3. **SafeTrade gate:** Addendum A0 lists the time picker as SafeTrade-only. SafeTrade is not yet implemented. The time picker must work without SafeTrade for now. **Flag with TODO comment.**

4. **Migration for existing data:** If any local transactions were created before this migration, they will have `scheduledAt` set but `scheduledAtConfirmedAt` null. The check-in guard must handle this gracefully. **Recommended: Add a migration note that backfills `scheduledAtConfirmedAt = createdAt` for existing records where `scheduledAt IS NOT NULL`.**
