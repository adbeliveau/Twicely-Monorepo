### 2.6 New Service: `src/lib/commerce/local-reliability.ts`

This replaces the logic in `local-noshow-strikes.ts`. The old file should be DELETED (or emptied and re-exported from this new file for any remaining imports).

**Functions to implement:**

#### `postReliabilityMark(params)`

```typescript
interface PostReliabilityMarkParams {
  userId: string;
  transactionId: string;
  eventType: LocalReliabilityEventType; // the enum values
  marksApplied: number; // e.g., -1, -2, -3
}
```

Logic:
1. Read `commerce.local.markDecayDays` from platform_settings (default: 180)
2. Compute `decaysAt = now + markDecayDays` days
3. Insert row into `localReliabilityEvent` table
4. Call `recalculateReliabilityMarks(userId)` to update the user's cached fields

#### `recalculateReliabilityMarks(userId)`

Logic:
1. Query `localReliabilityEvent` for this userId where `decaysAt > now`
2. Sum `marksApplied` (these are negative numbers, so the sum will be negative or zero)
3. The total active marks = `Math.abs(sum)` (display as positive count; marks are "demerits")
4. Read `commerce.local.suspensionMarkThreshold` (default: 9) and `commerce.local.suspensionDays` (default: 90)
5. If active marks >= threshold AND `localSuspendedUntil` is null or in the past: set `localSuspendedUntil = now + suspensionDays`
6. Update `user` table: `localReliabilityMarks`, `localSuspendedUntil`
7. Also update `localTransactionCount` and `localCompletionRate` by querying `localTransaction` (count where buyerId or sellerId = userId with status COMPLETED / total attempted)

IMPORTANT: `marksApplied` in the DB is stored as a negative integer (e.g., -1, -2, -3). The `localReliabilityMarks` on the user table stores the absolute accumulated value (positive integer). This distinction matters: the mark table records negative events, but the user-facing count is "you have 6 marks."

#### `getReliabilityDisplay(userId)`

Returns the public-facing reliability tier for display on meetup screens.

```typescript
interface ReliabilityDisplay {
  tier: 'RELIABLE' | 'INCONSISTENT' | 'UNRELIABLE';
  completedCount: number;
  completionRate: number; // 0.0 to 1.0
  isSuspended: boolean;
  suspendedUntil: Date | null;
}
```

Derivation logic (from Addendum A5 + A14):
- `RELIABLE`: `localReliabilityMarks < 3` (or no marks at all)
- `INCONSISTENT`: `localReliabilityMarks >= 3 && localReliabilityMarks < 9`
- `UNRELIABLE`: `localReliabilityMarks >= 9` (at or above suspension threshold)
- `isSuspended`: `localSuspendedUntil !== null && localSuspendedUntil > now`

NOT SPECIFIED: The exact boundary between RELIABLE and INCONSISTENT is not numerically defined in the canonical. The addendum says "cancellations in last 90 days" for INCONSISTENT and "multiple no-shows" for UNRELIABLE. Using marks < 3 / 3-8 / 9+ as the boundaries is a reasonable interpretation based on the mark weights (one no-show = 3 marks). **Owner decision needed: confirm the exact mark thresholds for RELIABLE/INCONSISTENT/UNRELIABLE display tiers.** Default implementation uses 0-2 / 3-8 / 9+.

#### `isUserSuspendedFromLocal(userId)`

Replaces the function in `local-noshow-strikes.ts`.

```typescript
async function isUserSuspendedFromLocal(userId: string): Promise<{
  suspended: boolean;
  resumesAt?: Date;

```

Logic:
1. Read `user.localSuspendedUntil` for the given userId
2. If null or in the past: `{ suspended: false }`
3. If in the future: `{ suspended: true, resumesAt: localSuspendedUntil }`

This is now a simple column check instead of the old query-based strike counting approach.

### 2.9 Mark Weights Reference

Per Addendum A5 and Decision #114:

| Behavior | Event Type | Marks |
|----------|-----------|-------|
| Cancel 24hr+ before | `BUYER_CANCEL_GRACEFUL` / `SELLER_CANCEL_GRACEFUL` | 0 (no event recorded) |
| Cancel under 24hr | `BUYER_CANCEL_LATE` / `SELLER_CANCEL_LATE` | -1 |
| Cancel same day under 2hr | `BUYER_CANCEL_SAMEDAY` / `SELLER_CANCEL_SAMEDAY` | -2 |
| No-show | `BUYER_NOSHOW` / `SELLER_NOSHOW` | -3 |
| Seller dark on meetup day | `SELLER_DARK` | -1 |
| Reschedule excess (3+ on same tx) | `RESCHEDULE_EXCESS` | -1 |

IMPORTANT: Graceful cancellations (24hr+ before) do NOT create a reliability event at all — 0 marks means nothing is recorded. The function should be a no-op for graceful cancellations.

## 8. SPEC INCONSISTENCIES AND OWNER DECISIONS NEEDED

### 8.1 Display Tier Thresholds (NEEDS DECISION)

The canonical describes three display tiers but does NOT specify exact mark count boundaries:
- Addendum A5: "Reliable — X local meetups, Y% completion" / "Inconsistent — cancellations in last 90 days" / "Unreliable — multiple no-shows, proceed with caution"
- Decision #114: Same descriptions, no numeric boundaries

**Default implementation:** RELIABLE = 0-2 marks, INCONSISTENT = 3-8 marks, UNRELIABLE = 9+ marks.

**Rationale:** One no-show = 3 marks = INCONSISTENT. Three no-shows = 9 marks = UNRELIABLE = suspension. This aligns with the "multiple no-shows" description for UNRELIABLE.

**Decision needed:** Confirm or override these thresholds. Consider adding platform settings (`commerce.local.reliabilityTierInconsistentThreshold` and `commerce.local.reliabilityTierUnreliableThreshold`) for configurability.

### 8.2 Mark Counting Window

Decision #114 says "9 marks in 90 days → suspended" but marks "decay after 180 days." These are different windows:
- **Suspension check**: 9 marks in ANY active (non-decayed, within 180 days) window
- **NOT**: 9 marks specifically within the last 90 days

The suspension check uses `decaysAt > now` (180-day decay window), NOT a separate 90-day rolling window. The `suspensionDays: 90` is how LONG the suspension lasts, not the window for counting marks.

**Implementation:** Count all marks where `decaysAt > now`. If count >= threshold, suspend for `suspensionDays` days. This is consistent with the Decision Rationale text "9-mark threshold differentiates genuine bad actors."

### 8.3 Suspension Re-Extension

The canonical does not specify what happens if a user accumulates MORE marks after being suspended (e.g., they get suspended, wait 90 days, immediately no-show again). The implementation should:
1. Allow marks to be posted even while suspended (they participated somehow, or the mark is from a transaction that started before suspension)
2. If new marks push the total back above threshold after suspension expires, re-suspend for another `suspensionDays`

### 8.4 `localCompletionRate` Calculation Scope

Decision #114 says `localCompletionRate = completed / total attempted`. The implementation should query `localTransaction` where the user is either buyerId or sellerId. "Attempted" = all transactions not in DRAFT state. "Completed" = status = 'COMPLETED'.

### 8.5 Schema Doc Missing

The `local_reliability_event` table and the 4 new `user` columns are defined in Decision #114 of `TWICELY_V3_DECISION_RATIONALE.md` but are NOT present in `TWICELY_V3_SCHEMA_v2_1_0.md`. The Decision Rationale doc is the authoritative source for this schema since it supersedes the base Local Canonical. The schema doc should be updated by the owner after G2.8 is complete.
