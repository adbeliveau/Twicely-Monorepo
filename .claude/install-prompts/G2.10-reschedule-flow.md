- `RESCHEDULE_PENDING` does NOT allow transition to check-in states or `RECEIPT_CONFIRMED` — you must resolve the reschedule first

### 2.4 Scheduling Service Extension

Add a new helper function to `src/lib/commerce/local-scheduling.ts`:

```typescript
/**
 * Statuses from which reschedule is allowed.
 * Per Addendum A7: SCHEDULED, SELLER_CHECKED_IN, BUYER_CHECKED_IN.
 * NOT available from BOTH_CHECKED_IN or later.
 */
const RESCHEDULE_ELIGIBLE_STATUSES = ['SCHEDULED', 'SELLER_CHECKED_IN', 'BUYER_CHECKED_IN'] as const;

export function canRequestReschedule(tx: LocalTransactionRow): boolean {
  if (isTerminalStatus(tx.status)) return false;
  if (tx.scheduledAtConfirmedAt === null) return false;  // scheduling not confirmed yet — use propose flow, not reschedule
  if (tx.status === 'RESCHEDULE_PENDING') return false;  // already a pending reschedule
  return (RESCHEDULE_ELIGIBLE_STATUSES as readonly string[]).includes(tx.status);
}
```

### 2.5 Server Actions

Create a new file `src/lib/actions/local-reschedule.ts` with two server actions:

#### 2.5.1 `proposeRescheduleAction`

**Purpose:** Either party proposes a new meetup time after the original scheduling has been confirmed.

**Input:** `proposeRescheduleSchema` (`{ localTransactionId: string, proposedAt: string (ISO datetime) }`)

**Logic:**
1. `authorize()` — must be authenticated
2. Validate input with `proposeRescheduleSchema`
3. Fetch the `localTransaction` by ID
4. Verify caller is buyer or seller on this transaction
5. CASL check: `ability.can('update', sub('LocalTransaction', { buyerId/sellerId }))`
6. Check suspension: `isUserSuspendedFromLocal(session.userId)`
7. Verify `tx.scheduledAtConfirmedAt !== null` — scheduling must be confirmed before reschedule is relevant
8. Verify `canRequestReschedule(tx)` — checks eligible statuses (not BOTH_CHECKED_IN or later, not already RESCHEDULE_PENDING)
9. Validate proposed time: `validateProposedTime(new Date(proposedAt))` — same 1-hour minimum, 30-day maximum
10. Update the `localTransaction`:
    - Set `scheduledAt` to the newly proposed time
    - Set `schedulingProposedBy` to `session.userId`
    - Set `scheduledAtConfirmedAt` to `null` (pending acceptance of the new time)
    - Set `status` to `'RESCHEDULE_PENDING'`
    - Reset check-in states: `sellerCheckedIn: false`, `sellerCheckedInAt: null`, `buyerCheckedIn: false`, `buyerCheckedInAt: null` (if the reschedule is from a checked-in state, those check-ins are invalidated)
    - Set `updatedAt` to `new Date()`
11. Send notification to other party: `notify(otherPartyId, 'local.reschedule.proposal', { ... })`
12. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`
13. Return `{ success: true }`

**Error responses:**
- Unauthorized -> `{ success: false, error: 'Unauthorized' }`
- Invalid input -> `{ success: false, error: '<Zod error>' }`
- Not found / not buyer/seller -> `{ success: false, error: 'Not found' }`
- CASL denied -> `{ success: false, error: 'Not found' }`
- Suspended -> `{ success: false, error: 'Your local transaction access is temporarily suspended' }`
- Not yet confirmed -> `{ success: false, error: 'Scheduling not yet confirmed. Use the scheduling flow instead.' }`
- Ineligible status -> `{ success: false, error: 'Reschedule is not available at this stage' }`
- Already RESCHEDULE_PENDING -> `{ success: false, error: 'A reschedule is already pending' }`
- Invalid time -> `{ success: false, error: '<time validation error>' }`

#### 2.5.2 `respondToRescheduleAction`

**Purpose:** The other party accepts or declines a pending reschedule.

**Input:** `respondToRescheduleSchema` (`{ localTransactionId: string, accept: boolean }`)

**Logic:**
1. `authorize()` — must be authenticated
2. Validate input with `respondToRescheduleSchema`
3. Fetch the `localTransaction` by ID
4. Verify caller is buyer or seller on this transaction
5. CASL check
6. Verify `tx.status === 'RESCHEDULE_PENDING'` — can only respond to a pending reschedule
7. Verify `tx.schedulingProposedBy !== session.userId` — cannot accept your own reschedule proposal
8. Verify `tx.scheduledAt !== null` — a proposed time must exist

**If `accept === true` (Reschedule Accepted):**

9. Re-validate the proposed time is still valid: `validateProposedTime(tx.scheduledAt)` — the proposed time may have become invalid (e.g., less than 1 hour from now) since it was proposed
10. Compute the new `rescheduleCount` = `tx.rescheduleCount + 1`
11. Determine who initiated: `lastRescheduledBy = tx.schedulingProposedBy === tx.buyerId ? 'BUYER' : 'SELLER'`
12. Determine `originalScheduledAt`: if `tx.originalScheduledAt === null` (first reschedule), set it to the PREVIOUS `scheduledAt` value that was confirmed (i.e., the time that was just replaced). If `tx.originalScheduledAt` is already set, do NOT overwrite it.

    **IMPORTANT:** The `originalScheduledAt` should capture the VERY FIRST confirmed time. On the first reschedule, this is the `scheduledAt` value that was active before the reschedule was proposed. We need to capture it BEFORE the reschedule proposal overwrites `scheduledAt`. But since `proposeRescheduleAction` already overwrites `scheduledAt` with the new proposed time, and `scheduledAtConfirmedAt` is set to null, we do NOT have the old confirmed time available in the transaction row at response time.

    **Resolution:** Move the `originalScheduledAt` preservation to `proposeRescheduleAction` instead. When proposing a reschedule, if `tx.originalScheduledAt === null`, set `originalScheduledAt = tx.scheduledAt` (the current confirmed time that is about to be replaced). This way the original time is captured at proposal time, before it's overwritten.

13. Update `localTransaction`:
    - Set `scheduledAtConfirmedAt` to `new Date()` (confirmed)
    - Set `status` to `'SCHEDULED'` (back to scheduled, check-ins already reset by propose action)
    - Set `rescheduleCount` to the new count
    - Set `lastRescheduledAt` to `new Date()`
    - Set `lastRescheduledBy` to the determined value
    - Set `updatedAt` to `new Date()`
14. Read `commerce.local.rescheduleMaxCount` (default 2) from platform settings
15. If `rescheduleCount > rescheduleMaxCount` (i.e., 3rd+ reschedule), post a reliability mark:
    - `postReliabilityMark({ userId: <initiating party's userId>, transactionId: tx.id, eventType: 'RESCHEDULE_EXCESS', marksApplied: -1 })`
    - The initiating party is determined by `tx.schedulingProposedBy` (the person who proposed the reschedule, NOT the person who accepted it)
16. Regenerate tokens: `regenerateTokensOnConfirmation(tx.id, tx.orderId, tx.buyerId, tx.sellerId, tx.scheduledAt)`
17. Remove old BullMQ jobs and re-enqueue:
    - Remove existing auto-cancel job: `localAutoCancelQueue.remove('local-auto-cancel-' + tx.id)` (try/catch — may not exist)
    - Re-enqueue auto-cancel: `enqueueAutoCancelAtScheduledTime(tx.id, tx.orderId, tx.buyerId, tx.sellerId, tx.scheduledAt)`
    - Placeholder call for reminder re-enqueue (same pattern as G2.9 `enqueueSchedulingReminders`)
18. Send notification to the proposer: `notify(proposerUserId, 'local.reschedule.accepted', { ... })`
19. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`
20. Return `{ success: true }`

**If `accept === false` (Reschedule Declined):**

9. Restore the original confirmed state:
    - Set `status` back to `'SCHEDULED'` (the pre-reschedule status; note: the propose action already reset check-ins, so we go back to SCHEDULED regardless)
    - Set `scheduledAtConfirmedAt` to its previous confirmed value — BUT we don't have the old confirmed timestamp anymore since `proposeRescheduleAction` cleared it. **Resolution:** Instead of trying to restore the old `scheduledAtConfirmedAt`, store the PREVIOUS confirmed `scheduledAt` and restore it on decline.

    **Revised approach for propose action:** When proposing a reschedule, save the PREVIOUS `scheduledAt` and `scheduledAtConfirmedAt` somewhere so they can be restored on decline. The cleanest way: add a `rescheduleProposedNewTime` column? No — that adds a column not in the spec.

    **Simpler approach:** On proposal, do NOT overwrite `scheduledAt`. Instead, store the proposed new time in `schedulingProposedBy` context, and only overwrite `scheduledAt` on acceptance.

    **Final approach (spec-aligned):** Re-read Addendum A7 carefully. The spec says:
    - "Either party can propose a reschedule" -> proposeRescheduleAction
    - "Other party must accept or decline in-app"
    - "On acceptance: scheduledAt updated, ..."

    This implies `scheduledAt` is updated on ACCEPTANCE, not on proposal. So the flow is:
    1. Propose: status -> RESCHEDULE_PENDING, store proposed time somewhere, do NOT change scheduledAt yet
    2. Accept: scheduledAt updated to the proposed time, jobs re-queued
    3. Decline: status returns to previous status, nothing changes

    **But we need somewhere to store the proposed time.** The `scheduledAt` column itself can serve double duty: the addendum says to store it there. Looking at how G2.9 works: `proposeMeetupTimeAction` stores the proposed time in `scheduledAt` and sets `scheduledAtConfirmedAt = null`. The accept action then sets `scheduledAtConfirmedAt` to now.

    **For reschedule, we can do the same pattern but we need to be able to restore the old time on decline.** We need a temporary column or a different strategy.

    **DECISION (spec-compliant, minimal columns):** Use the same pattern as G2.9 but save the old confirmed time. Since the spec already defines `originalScheduledAt` for tracking purposes, we can use a DIFFERENT approach:

    - On PROPOSE: Set `scheduledAt = newProposedTime`, `scheduledAtConfirmedAt = null`, `status = RESCHEDULE_PENDING`. Also set `originalScheduledAt` to the old `scheduledAt` IF `originalScheduledAt` is currently null (first reschedule). Reset check-ins.
    - On ACCEPT: Set `scheduledAtConfirmedAt = now`, `status = SCHEDULED`, increment rescheduleCount, etc.
    - On DECLINE: We need to restore the old `scheduledAt`. Problem: `originalScheduledAt` only stores the FIRST ever time. On 2nd reschedule, we need the time from the 1st reschedule, not the original.

    **SIMPLEST SPEC-COMPLIANT APPROACH:** Add NO extra columns beyond what the spec defines. Store the proposed reschedule time in `scheduledAt` (overwriting the old confirmed time). On decline, the user needs to re-schedule via the time picker.

    Actually, re-reading the spec more carefully: "Other party must accept or decline in-app." On decline, the spec does NOT say "restore old time." Looking at the cancellation flow (A8): "Available while status is SCHEDULED, SELLER_CHECKED_IN, BUYER_CHECKED_IN, or RESCHEDULE_PENDING." This means RESCHEDULE_PENDING is a valid state for cancellation. On decline, we should just go back to SCHEDULED with the OLD confirmed time.

    **FINAL APPROACH:** We need to preserve the old scheduledAt on propose so we can restore it on decline.

    We will store the proposed new time in a new column `rescheduleProposedAt` (timestamp, nullable). This is NOT in the spec's schema additions. However, without it, we cannot implement the decline path correctly.

    **NOT SPECIFIED — Owner decision needed:** The addendum A7 specifies 4 new columns (`rescheduleCount`, `lastRescheduledAt`, `lastRescheduledBy`, `originalScheduledAt`) but does not specify how to store the proposed reschedule time separately from the confirmed `scheduledAt`. The two options are:

    **Option A (Recommended):** Do NOT overwrite `scheduledAt` on proposal. Add a `rescheduleProposedAt` timestamp column (nullable) to store the proposed new time. On acceptance, copy `rescheduleProposedAt` -> `scheduledAt` and clear `rescheduleProposedAt`. On decline, clear `rescheduleProposedAt` and return status to `SCHEDULED`. This keeps `scheduledAt` as the source of truth for the confirmed time at all times.

    **Option B:** Overwrite `scheduledAt` on proposal (like G2.9 does for initial scheduling). On decline, there is no way to restore the old confirmed time without an additional column. The transaction effectively loses its confirmed time and must be re-scheduled from scratch.

    **IMPLEMENT OPTION A** (adds 1 extra column not in spec, but required for correct decline behavior). If the owner objects, fall back to Option B.

Add to `localTransaction` in `local.ts`:
```typescript
rescheduleProposedAt: timestamp('reschedule_proposed_at', { withTimezone: true }),
```

## 4. ACCEPTANCE CRITERIA

### Positive Cases
1. Buyer can propose a reschedule when status is SCHEDULED and scheduling is confirmed
2. Seller can propose a reschedule when status is SCHEDULED and scheduling is confirmed
3. Buyer can propose a reschedule when status is SELLER_CHECKED_IN (seller checked in but buyer hasn't)
4. Seller can propose a reschedule when status is BUYER_CHECKED_IN
5. Other party can accept a pending reschedule — status returns to SCHEDULED, scheduledAt updated, tokens regenerated, auto-cancel re-queued
6. Other party can decline a pending reschedule — status returns to SCHEDULED, original scheduledAt preserved, no token changes
7. rescheduleCount increments by 1 on each accepted reschedule
8. originalScheduledAt is set to the first confirmed scheduledAt on the first reschedule, and never overwritten on subsequent reschedules
9. lastRescheduledAt and lastRescheduledBy are updated on each accepted reschedule
10. On 3rd+ accepted reschedule (rescheduleCount > rescheduleMaxCount), a RESCHEDULE_EXCESS reliability mark (-1) is posted against the INITIATING party (not the accepting party)
11. Proposed time must be at least 1 hour in the future and at most 30 days out
12. Notifications are sent to the other party on proposal, acceptance, and decline
13. Check-in states (sellerCheckedIn, buyerCheckedIn, timestamps) are reset to false/null on reschedule proposal
14. Check-in button is disabled when status is RESCHEDULE_PENDING

### Negative Cases
15. Reschedule is NOT available when status is BOTH_CHECKED_IN
16. Reschedule is NOT available when status is ADJUSTMENT_PENDING
17. Reschedule is NOT available when status is RECEIPT_CONFIRMED, COMPLETED, CANCELED, NO_SHOW, or DISPUTED
18. Cannot propose reschedule if scheduling is not yet confirmed (scheduledAtConfirmedAt is null)
19. Cannot accept or decline your own reschedule proposal
20. Unauthenticated users cannot call any reschedule action
21. Users who are not buyer or seller on the transaction get "Not found"
22. Suspended users cannot propose a reschedule
23. Proposed time in the past returns validation error
24. Proposed time less than 1 hour from now returns validation error
25. Proposed time more than 30 days out returns validation error

### Data Integrity
26. All monetary values remain as integer cents (no reschedule flow touches money)
27. rescheduleCount is stored as integer, default 0
28. The RESCHEDULE_PENDING status exists in the enum and state machine
29. State machine transitions are bidirectional: SCHEDULED -> RESCHEDULE_PENDING -> SCHEDULED
30. rescheduleMaxCount is read from platform_settings, never hardcoded

### Authorization
31. CASL check on LocalTransaction with buyerId/sellerId condition
32. No route or action is accessible without authentication

### Vocabulary
33. No banned terms appear in any new code or UI text
34. No references to "wallet", "withdraw", "FVF", "SellerTier", etc.

## 8. SPEC INCONSISTENCIES / OWNER DECISIONS NEEDED

### 8.1 `rescheduleProposedAt` Column Not in Spec

**Issue:** Addendum A7 specifies 4 new columns (`rescheduleCount`, `lastRescheduledAt`, `lastRescheduledBy`, `originalScheduledAt`) but does NOT specify how the proposed reschedule time is stored separately from the confirmed `scheduledAt`. Without a `rescheduleProposedAt` column, the decline path cannot restore the original confirmed time because proposing a reschedule would overwrite `scheduledAt`.

**Recommendation:** Add `rescheduleProposedAt` (timestamp, nullable) to `localTransaction`. This is the minimal addition needed for correct decline behavior.

**Alternative:** Overwrite `scheduledAt` on proposal (matching the G2.9 initial scheduling pattern). On decline, the old confirmed time is lost and the transaction returns to an unscheduled state, requiring both parties to re-schedule from scratch. This is a worse UX but requires no extra column.

**Adopted in this prompt:** Option A (add `rescheduleProposedAt`). Owner should confirm.

### 8.2 Proposer Cannot Cancel Their Own Proposal

**Issue:** The spec says "Other party must accept or decline in-app." It does not specify whether the proposer can withdraw/cancel their own reschedule proposal before the other party responds. This prompt assumes the proposer CANNOT cancel — they must wait for the other party to respond. The messaging quick-reply "Need to reschedule" exists for informal coordination.

**Owner decision needed if:** The proposer should be able to withdraw their own reschedule proposal.

### 8.3 Check-In State Reset on Propose

**Issue:** The spec says reschedule is available from SELLER_CHECKED_IN and BUYER_CHECKED_IN. When a reschedule is proposed from these states, the check-in data should be reset (since the meetup time is changing and the old check-in is no longer relevant). This prompt resets `sellerCheckedIn`, `sellerCheckedInAt`, `buyerCheckedIn`, `buyerCheckedInAt` to false/null on proposal. The spec does not explicitly state this, but it is logically required.
