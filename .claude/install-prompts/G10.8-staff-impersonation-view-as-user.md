## 3. SCOPE — EXACTLY WHAT TO BUILD

### 3.1 Session Mechanism

**Approach:** Short-lived HMAC-signed cookie containing `targetUserId`, `staffUserId`, and `expiresAt`. The cookie is stateless — no new database table is needed. The schema v2.1.3 has no `staff_impersonation_session` table.

Cookie name: `twicely.impersonation_token`

**Token payload (JSON, base64url-encoded):**
```typescript
interface ImpersonationTokenPayload {
  targetUserId: string;     // The marketplace user being viewed
  staffUserId: string;      // The staff member who started the session
  staffDisplayName: string; // For banner display
  expiresAt: number;        // Unix timestamp in milliseconds
}
```

**Token structure:**
`base64url(JSON.stringify(payload)) + '.' + hmac_sha256_hex(base64url(JSON.stringify(payload)), IMPERSONATION_SECRET)`

**Token TTL:** 15 minutes from start. NOT SPECIFIED in any canonical. Owner may override.

**Cookie attributes:**
- `HttpOnly: true`
- `SameSite: Lax`
- `Secure: true` in production
- `Path: /`
- `Domain: .twicely.co` in production, `.twicely.local` in dev

#### NOT SPECIFIED — Owner Decision 1: Session Storage Mechanism

The schema doc v2.1.3 has NO impersonation session table. Two valid approaches:

- **Option A (implemented here):** Stateless HMAC-signed cookie. No DB table. Cannot be remotely revoked mid-session.
- **Option B:** New `staff_impersonation_session` DB table. Allows server-side revocation. Requires schema addendum.

**Installer proceeds with Option A.** If owner wants Option B, a schema addendum is required before execution.

#### NOT SPECIFIED — Owner Decision 2: Redirect Target

When impersonation starts, where does the staff member land?

- **Option A (implemented here):** Redirect to `hub.twicely.co/usr/[targetUserId]`. The existing hub user detail page already shows all user data (orders, listings, balance, cases). The banner renders on hub pages. This matches V2 `ImpersonationBanner` behavior — it was a hub-side preview, NOT a full session swap.
- **Option B:** Redirect to the marketplace `/my` dashboard as the target user. Requires minting a marketplace session (`twicely.session_token`) for the target user — a security-sensitive operation outside the current spec.

**Installer proceeds with Option A.** If owner requires Option B, the prompt must be rewritten. Option B must NEVER be implemented without explicit owner decision because it grants the staff member an active Better Auth session as the target user.

#### NOT SPECIFIED — Owner Decision 3: Session TTL

15 minutes is the assumed TTL. No canonical specifies this.

### 3.3 Server-Side Utilities

**New file: `src/lib/auth/impersonation.ts`**

Pure utility functions — NO `'use server'` directive. Under 300 lines.

```typescript
// All exports:

export interface ImpersonationTokenPayload {
  targetUserId: string;
  staffUserId: string;
  staffDisplayName: string;
  expiresAt: number; // ms since epoch
}

// Creates a signed token string. Throws if IMPERSONATION_SECRET is not set.
export function createImpersonationToken(payload: ImpersonationTokenPayload): string

// Verifies signature and expiry. Returns null on any failure.
export function verifyImpersonationToken(token: string): ImpersonationTokenPayload | null

// Reads twicely.impersonation_token from Next.js cookies(). Returns null if absent/invalid.
// Must be called from a Server Component, Route Handler, or Server Action (uses next/headers).
export async function getImpersonationSession(): Promise<ImpersonationTokenPayload | null>
```

Implementation notes:
- Use `import { createHmac } from 'crypto'` (Node.js built-in, not a package import).
- Use `Buffer.from(json).toString('base64url')` for encoding.
- Validate `expiresAt > Date.now()` inside `verifyImpersonationToken`.
- If `process.env.IMPERSONATION_SECRET` is `undefined` or empty string: throw `new Error('IMPERSONATION_SECRET is not configured')`.

### 3.4 API Route Handlers

**Why route handlers instead of server actions for start/end:**
Next.js server actions run on the same origin as the page. Setting `HttpOnly` cookies with a `Domain` attribute spanning subdomains requires a `NextResponse` or `Response` from a Route Handler. Server actions can set cookies via `next/headers` `cookies().set()`, but the `Domain` attribute behavior is constrained. Using Route Handlers gives full control over the `Set-Cookie` header.

**New file: `src/app/api/hub/impersonation/start/route.ts`**

```
POST /api/hub/impersonation/start
Content-Type: application/json
Body: { targetUserId: string }
Cookie: twicely.staff_token required

Steps:
1. Read staff token from cookies() — return Response 401 JSON if missing
2. Call getStaffSession(token) — return Response 401 JSON if null
3. Load custom role permissions via loadCustomRolePermissions (same pattern as staff-authorize.ts)
4. Build CASL ability from staff session
5. Check ability.can('impersonate', 'User') — return Response 403 JSON if false
6. Parse body with Zod .strict(): { targetUserId: z.string().min(1) }
   Return 400 JSON on validation failure
7. SELECT id, name, email FROM user WHERE id = targetUserId
   Return 404 JSON if not found
8. Verify IMPERSONATION_SECRET is set — return 500 JSON if missing
9. Create ImpersonationTokenPayload: { targetUserId, staffUserId, staffDisplayName, expiresAt: Date.now() + 15*60*1000 }
10. Sign token via createImpersonationToken()
11. Insert auditEvent: { actorType: 'STAFF', actorId: staffUserId, action: 'IMPERSONATE_USER_START',
    subject: 'User', subjectId: targetUserId, severity: 'HIGH',
    detailsJson: { targetUserName: user.name, targetUserEmail: user.email } }
12. Build NextResponse.redirect(302) to hub URL: `${HUB_BASE_URL}/usr/${targetUserId}`
13. Set cookie on response: name='twicely.impersonation_token', value=token,
    httpOnly=true, secure=(NODE_ENV=production), sameSite='lax',
    maxAge=900, path='/', domain=COOKIE_DOMAIN
14. Return the redirect response

HUB_BASE_URL: process.env.NODE_ENV === 'production' ? 'https://hub.twicely.co' : 'http://hub.twicely.local'
COOKIE_DOMAIN: process.env.NODE_ENV === 'production' ? '.twicely.co' : '.twicely.local'
```

**New file: `src/app/api/hub/impersonation/end/route.ts`**

```
POST /api/hub/impersonation/end
No auth required (reads from impersonation token cookie)

Steps:
1. Call getImpersonationSession() — return Response 400 JSON if null
2. Extract { targetUserId, staffUserId } from payload
3. Insert auditEvent: { actorType: 'STAFF', actorId: staffUserId, action: 'IMPERSONATE_USER_END',
   subject: 'User', subjectId: targetUserId, severity: 'HIGH', detailsJson: {} }
4. Build NextResponse.redirect(302) to `${HUB_BASE_URL}/usr/${targetUserId}`
5. Clear cookie on response: Set-Cookie with maxAge=0 (same name/path/domain)
6. Return the redirect response
```

### 3.7 User Actions Component Changes

**File: `src/components/admin/actions/user-actions.tsx`** — MODIFY (currently 76 lines)

Add a new `canImpersonate: boolean` prop. When `true`, render a "View as user" button.

The button submits a form POST to `/api/hub/impersonation/start`:
```tsx
{canImpersonate && (
  <form
    onSubmit={(e) => {
      e.preventDefault();
      if (!confirm(`View as this user? You will be in read-only mode.`)) return;
      (e.currentTarget as HTMLFormElement).submit();
    }}
    action="/api/hub/impersonation/start"
    method="POST"
  >
    <input type="hidden" name="targetUserId" value={userId} />
    <button type="submit" className="rounded bg-blue-600 px-3 py-1.5 text-xs text-white hover:bg-blue-700">
      View as user
    </button>
  </form>
)}
```

Note: Forms that POST JSON require either a custom `fetch` or the server to parse `multipart/form-data`. Since the route handler reads JSON body via `req.json()`, the form submit must use `fetch` from `useTransition` instead of a raw form submit. Use the same `startTransition` pattern already in the component. On success, the route handler redirects — the fetch will follow the redirect.

**Revised approach for the button handler:**
```typescript
function handleImpersonate() {
  if (!confirm('View as this user? You will be in read-only mode.')) return;
  startTransition(async () => {
    // Route handler returns a redirect — follow it
    const res = await fetch('/api/hub/impersonation/start', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ targetUserId: userId }),
      redirect: 'follow',
    });
    if (res.ok || res.redirected) {
      window.location.href = res.url;
    } else {
      setResult('Failed to start impersonation session');
    }
  });
}
```

## 4. CONSTRAINTS — WHAT NOT TO DO

### Banned Code Patterns
- `as any`, `as unknown as T`, `@ts-ignore`, `@ts-expect-error` — zero occurrences.
- **Never set `twicely.session_token` in the impersonation flow.** This would create an unauthorized marketplace session.
- **Never expose `IMPERSONATION_SECRET` in client code, error messages, or response bodies.**
- **Never skip the CASL `ability.can('impersonate', 'User')` check** in the start handler.
- Files over 300 lines — split if necessary.
- `console.log` in production code.

### Banned Technology
- No Prisma, no NextAuth, no Redis, no tRPC, no Zustand/Redux.

### Banned Business Logic
- Do NOT grant the impersonating staff member write access to user resources.
- Do NOT create a new DB table without a schema addendum from the owner.

### Gotchas to Avoid
- **`action-types.ts` must be updated FIRST** before `platform-abilities.ts`. TypeScript will error on `can('impersonate', 'User')` until the Action type includes `'impersonate'`.
- **`'impersonate'` must be in the `ACTIONS as const` array** (not just the type), or CASL may reject it at runtime.
- **Cookie Domain in dev vs prod:** Use `.twicely.local` in dev, `.twicely.co` in prod. Read from `NODE_ENV`.
- **Exported helpers in a future `'use server'` file become server actions.** Keep `createImpersonationToken` and `verifyImpersonationToken` in `impersonation.ts` (no `'use server'`) — NOT in the actions file.
- **The `(hub)/layout.tsx` also wraps `/my/*` pages** (returns `<>{children}</>` for non-hub subdomain). The banner only renders inside the hub shell, which is already guarded by `if (subdomain !== 'hub') return <>{children}</>`.
- **`getImpersonationSession()` uses `cookies()` from `next/headers`** — it can only be called from server-side code (Server Components, Route Handlers, Server Actions). Never import it in client components.
- **`fetch(..., { redirect: 'follow' })` in the user-actions component** — the start route handler returns a 302 redirect, so the fetch must follow it (`redirect: 'follow'` is the default in the browser; confirm it is set explicitly).
