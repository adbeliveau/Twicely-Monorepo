## SCOPE -- EXACTLY WHAT TO BUILD

### 1. CASL Rules Update

**File**: `src/lib/casl/ability.ts`

In `defineSellerAbilities`, change the existing Expense rule:
```
CURRENT:  can('read', 'Expense', { userId });
CHANGE TO: can('manage', 'Expense', { userId });
```

This grants sellers full CRUD (create, read, update, delete) on their own expenses. The `{ userId }` condition ensures ownership enforcement.

**Delegation remains read-only**. Staff with `finance.view` scope only gets `can('read', 'Expense', ...)`. No new delegation scope for expense creation -- expenses are owner-managed. This matches the Actors Canonical which defines `finance.view` as "View payouts, balance, ledger" (read-only).

### 3. Expense Queries

**File**: `src/lib/queries/finance-center-expenses.ts` (NEW -- split from detail file per 300-line limit)

Implement these query functions:

```typescript
// Types
export interface ExpenseRow {
  id: string;
  category: string;
  amountCents: number;
  currency: string;
  vendor: string | null;
  description: string | null;
  receiptUrl: string | null;
  expenseDate: Date;
  isRecurring: boolean;
  recurringFrequency: string | null;
  recurringEndDate: Date | null;
  parentExpenseId: string | null;
  createdAt: Date;
  updatedAt: Date;
}

export interface ExpenseListResult {
  expenses: ExpenseRow[];
  total: number;
  page: number;
  pageSize: number;
}

export interface ExpenseCategoryBreakdown {
  category: string;
  totalCents: number;
  count: number;
  percentOfTotal: number; // 0-100
}

export interface CogsSummary {
  totalCogsCents: number;         // sum of cogsCents for sold items in period
  totalSoldRevenueCents: number;  // sum of soldPriceCents for items with COGS in period
  grossProfitCents: number;       // totalSoldRevenueCents - totalCogsCents
  cogsMarginPercent: number;      // (grossProfitCents / totalSoldRevenueCents) * 100
  itemCount: number;              // number of items with COGS that were sold
}
```

Functions to implement:

1. **`getExpenseList(userId, opts)`** -- Paginated, filterable expense list. Supports category filter, date range, sort by any column. Uses `expense` table. Conditions: `eq(expense.userId, userId)` + optional filters.

2. **`getExpenseById(userId, expenseId)`** -- Single expense fetch. Returns `ExpenseRow | null`. MUST verify `eq(expense.userId, userId)` AND `eq(expense.id, expenseId)` -- never return another user's expense.

3. **`getExpenseCategoryBreakdown(userId, days)`** -- Group expenses by category for the period, with total cents and percent of total. Used for the pie/bar chart on the expenses page.

4. **`getCogsSummary(userId, days)`** -- Calculate COGS-based profit metrics. Query: join `order` (as seller, status COMPLETED, completedAt within period) to `orderItem` to `listing` where `listing.cogsCents IS NOT NULL`. Sum `listing.cogsCents` for totalCogsCents, sum `order.totalCents` for revenue. Calculate gross profit and margin. **NOTE**: `listing.cogsCents` is optional -- only items where the seller entered a cost are included. Items without COGS are excluded from this calculation (not zero-filled).

### 4. Expense Actions

All actions follow the existing pattern in `finance-center.ts`:

1. **`createExpenseAction(input)`**
   - Validate with `createExpenseSchema`
   - CASL: `ability.can('create', sub('Expense', { userId }))`
   - Tier gate: `financeTier === 'PRO'` required
   - Insert into `expense` table with explicit field mapping
   - Return: `{ success: true; expense: ExpenseRow } | { success: false; error: string }`

2. **`updateExpenseAction(input)`**
   - Validate with `updateExpenseSchema`
   - CASL: `ability.can('update', sub('Expense', { userId }))`
   - Tier gate: `financeTier === 'PRO'` required
   - Fetch existing expense first, verify `expense.userId === userId`
   - Update with explicit field mapping (only provided fields)
   - Return: `{ success: true; expense: ExpenseRow } | { success: false; error: string }`

3. **`deleteExpenseAction(input)`**
   - Validate with `deleteExpenseSchema`
   - CASL: `ability.can('delete', sub('Expense', { userId }))`
   - Tier gate: `financeTier === 'PRO'` required
   - Verify ownership before delete
   - Soft consideration: if expense has `parentExpenseId` (is a recurring-generated entry), still allow delete
   - Return: `{ success: true } | { success: false; error: string }`

4. **`listExpensesAction(input)`**
   - Validate with `listExpensesSchema`
   - CASL: `ability.can('read', sub('Expense', { userId }))`
   - No tier gate for READ -- FREE tier can see seed/migrated expenses (read-only view per canonical)
   - Calls `getExpenseList()`
   - Return: `{ success: true; data: ExpenseListResult } | { success: false; error: string }`

5. **`getCogsSummaryAction(days?)`**
   - CASL: `ability.can('read', sub('Analytics', { sellerId: userId }))`
   - No tier gate -- COGS is part of the dashboard P&L (FREE tier sees it)
   - Calls `getCogsSummary()`
   - Return: `{ success: true; cogs: CogsSummary } | { success: false; error: string }`

## CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms (scan ALL output before committing)
- `SellerTier`, `SubscriptionTier` -- use `StoreTier` or `ListerTier`
- `FVF`, `Final Value Fee` -- use `TF`, `Transaction Fee`
- `Twicely Balance` -- use `Available for payout`
- `wallet` (in seller UI) -- use `payout`
- `Withdraw` -- use `Request payout`
- `FinanceTier` as an enum -- it is derived from subscription state, stored as `financeTier` on `sellerProfile`
- `Twicely fees` -- use `Transaction Fee` for the TF line item (per Pricing Canonical SS3.1)
- `Stripe fee` -- use `Payment processing fee`

### Tech Stack Violations
- NO Prisma (use Drizzle)
- NO Redux/Zustand (use React context + server state)
- NO tRPC (use server actions)
- NO recharts for the category chart (too heavy for simple bars -- use Tailwind CSS bars)

### Code Pattern Violations
- NO `as any`, `as unknown as T`, `@ts-ignore`, `@ts-expect-error`
- NO floating-point money (integer cents ONLY)
- NO hardcoded fee rates or expense categories -- categories from `EXPENSE_CATEGORIES` constant (which mirrors `platform_settings` key `finance.expenseCategories`)
- NO `console.log` in production code
- NO files over 300 lines (split if needed)
- NO spreading request body into DB -- explicit field mapping only
- NO ownership via `storeId` or `sellerProfileId` -- always `userId`
- All Zod schemas use `.strict()` to reject unknown keys
- `role`, `status`, `ownerUserId` -- never settable from client input

### Business Logic Violations
- NO expense CRUD for FREE tier users (read-only view allowed)
- NO expense creation by delegated staff (only `finance.view` scope exists, no `finance.manage`)
- COGS is OPTIONAL per listing -- never default to 0. Exclude items without COGS from COGS calculations entirely.
- Receipt upload is D4.2 (NOT this step) -- the `receiptUrl` field exists in schema but is NOT populated by D4.1 actions. Do not build receipt upload UI.
- Recurring expense generation (auto-creating future entries from recurring templates) is NOT in D4.1 scope. The `isRecurring` flag and `recurringFrequency` are stored but no cron/job generates child entries. That is a future step.

### Route Violations
- The expense page MUST be at `/my/selling/finances/expenses` (under the existing finances section)
- NOT at `/my/selling/expenses` or `/my/finances/expenses`

## ACCEPTANCE CRITERIA

### Positive Cases
1. A Finance PRO seller can create an expense with category, amount (in cents), vendor, description, and date
2. A Finance PRO seller can view a paginated list of their expenses, filtered by category and date range
3. A Finance PRO seller can edit any field of their own expense
4. A Finance PRO seller can delete their own expense (with confirmation)
5. The P&L summary on the dashboard includes COGS as a line item when sold items have `cogsCents` set
6. The P&L shows the canonical formula: Revenue - COGS = Gross Profit - Fees - Shipping - Expenses - Mileage = Net Profit
7. The expenses page shows a category breakdown visualization
8. The expense summary card on the dashboard links to `/my/selling/finances/expenses`
9. Delegation: a staff user with `finance.view` scope can view the seller's expenses (read-only)
10. The COGS summary correctly excludes items without `cogsCents` (never treats null as 0)

### Negative Cases
11. A FREE tier seller CANNOT create, update, or delete expenses -- sees `FinanceProGate` upgrade CTA
12. A FREE tier seller CAN see the expense summary on the dashboard (read-only, if seed data exists)
13. An unauthenticated user is redirected to `/auth/login`
14. A non-seller user sees "Start Selling First" message
15. A seller CANNOT create/update/delete another seller's expenses -- CASL `{ userId }` condition enforced
16. A delegated staff with `finance.view` CANNOT create or delete expenses -- only `read` granted
17. Expense amount must be positive integer cents -- negative or zero amounts are rejected by Zod
18. Expense category must be one of the 16 preset categories -- arbitrary strings rejected
19. Creating a recurring expense without `recurringFrequency` fails validation

### Data Integrity
20. All monetary values stored as integer cents (`amountCents`)
21. All expenses have `userId` ownership -- never `storeId` or `sellerProfileId`
22. Zod `.strict()` on all input schemas -- unknown fields rejected
23. `expenseDate` stored with timezone (`withTimezone: true`)
24. COGS calculations use `listing.cogsCents` (integer cents) -- no floats

### Vocabulary
25. No banned terms appear anywhere in UI text, component names, or variable names
26. The expense page header says "Expenses" (not "Business Expenses" or "Operating Expenses")
27. P&L uses "Cost of goods sold" label (not "COGS" in UI), "Gross profit", "Net earnings"
28. Finance PRO gate says "Finance Pro" (not "Finance Premium" or "Finance Plus")

## SPEC NOTES AND CLARIFICATIONS

### SPEC INCONSISTENCY: Finance Tier Count
Decision #45 originally described 5 finance tiers (FREE/Lite/Plus/Pro/Enterprise). The schema doc and Pricing Canonical v3.2 section 7 define only 2 tiers: `financeTierEnum = ['FREE', 'PRO']`. **The schema and Pricing Canonical are authoritative.** Build for 2 tiers only (FREE / PRO). Decision #45 tier table is outdated.

### SPEC INCONSISTENCY: Decision #46 Store Tier Names
Decision #46 uses old tier names (Basic, Elite) in the "Finance Included" table. The correct v3.2 tier names are NONE/STARTER/PRO/POWER/ENTERPRISE. All Store tiers >= STARTER include Finance Pro.

### NOT IN SCOPE for D4.1
- Receipt scanning / AI extraction (D4.2)
- Mileage tracking CRUD (D4.2)
- Balance sheet report (D4.3)
- Cash flow statement (D4.3)
- CSV/PDF export (D4.3)
- QuickBooks/Xero integration (G10.3)
- Recurring expense auto-generation cron job (future)
- Receipt photo upload to R2 (D4.2)

### `finance.manage` Delegation Scope
The Actors Canonical does NOT define a `finance.manage` delegation scope. Only `finance.view` exists ("View payouts, balance, ledger"). This means delegated staff CANNOT create/edit/delete expenses. This is intentional -- expense management is owner-only for financial integrity. Do NOT invent a `finance.manage` scope.

### Expense Categories Source
Financial Center Canonical section 9 defines `finance.expenseCategories` as a platform_settings key. In D4.1, use the hardcoded `EXPENSE_CATEGORIES` constant (matching the canonical's 16 categories). A future admin settings UI could allow customization. The `expense.category` column is `text` (not enum) specifically to support future platform-configurable categories.
