## 2. SCOPE -- EXACTLY WHAT TO BUILD

### 2.1 Conceptual Overview

The existing affiliate system (G1-G3.5) handles **user-acquisition referrals**: affiliate shares `/ref/{CODE}` link, new user signs up, cookie tracks attribution, commission earned on subscription payments.

G3.6 extends this with **listing-level referrals**: affiliate shares `twicely.co/i/{slug}?ref={CODE}`, buyer clicks, cookie set (7-day window), buyer purchases, affiliate earns commission on the **sale price** (not subscriptions). This is a fundamentally different commission model -- it's paid by the seller (deducted from their payout), not by the platform.

### 2.2 SCHEMA CHANGES

**OWNER DECISION NEEDED -- Schema additions required:**

The following columns DO NOT exist in the schema doc but are required by the Feature Lock-in Social & Discovery spec. The installer should add them:

#### A. `sellerProfile` table -- add 2 columns:

```typescript
affiliateOptIn:          boolean('affiliate_opt_in').notNull().default(true),

```

**Canonical basis:** Feature Lock-in Social & Discovery, "Creator Affiliate Extension":
- "Seller opt-in: sellers can enable/disable affiliate commission on their listings"
- "Seller can set custom commission rate (2-10%) to attract more affiliate promotion"
- "Seller Settings -> Affiliate: 'Allow affiliates to earn commission on my listings' (default: ON)"
- `affiliate.sellerOptInDefault: true` confirms default is ON

#### B. `referral` table -- add 1 column:

```typescript
// In src/lib/db/schema/affiliates.ts, add to referral:
listingId:  text('listing_id').references(() => listing.id),  // nullable -- NULL for signup referrals, set for listing referrals
```

**Canonical basis:** G3.6 generates links to specific listings. The referral record must track WHICH listing was clicked. Without this, we cannot attribute listing-click referrals vs signup referrals.

#### C. New cookie for listing attribution

The existing `twicely_ref` cookie stores a referral ID for signup attribution (30/60-day lifetime). G3.6 needs a SEPARATE cookie for listing-level attribution:

```typescript
export const LISTING_REF_COOKIE_NAME = 'twicely_listing_ref';
// Value: JSON string { affiliateId, affiliateCode, clickedAt }
// Max-age: 7 days (from affiliate.attributionWindowDays platform setting)
```

**Why separate:** The signup referral cookie has 30/60-day lifetime for subscription attribution. The listing referral cookie has 7-day lifetime per spec ("7-day attribution window, matching boosting"). Conflating them would break the different durations.

#### D. Drizzle migration

Create migration file: `drizzle/XXXX_add-creator-affiliate-columns.sql`

```sql

ALTER TABLE seller_profile ADD COLUMN affiliate_commission_bps integer;
ALTER TABLE referral ADD COLUMN listing_id text REFERENCES listing(id);
```

### 2.4 API ROUTE -- Listing Click Handler

**New file:** `src/app/api/affiliate/listing-click/route.ts`

This is a POST API route (NOT the listing page itself). The listing page calls this when `?ref=` is present.

**Why POST, not middleware intercept:** The listing page is SSR-rendered and public. We need to:
1. Validate the affiliate code
2. Check seller opt-in
3. Insert referral record
4. Set cookie
5. Return whether the click was attributed (for UI feedback)

Doing this in middleware would slow down every `/i/` page load. Instead, the listing page client component fires a POST to this endpoint when `?ref=` is detected.

```typescript
// POST /api/affiliate/listing-click
// Input (Zod strict):
{
  referralCode: string,  // The affiliate's code from ?ref= param
  listingId: string,     // The listing being viewed
  listingSlug: string,   // For audit/logging
}
// Response:
{ success: boolean, attributed: boolean, error?: string }
```

**Business logic (in order):**

1. Check `affiliate.listingLinkEnabled` platform setting -- if false, return `{ success: true, attributed: false }`
2. Look up affiliate by `referralCode` (uppercase) via `getAffiliateByReferralCode()`
3. If not found or status !== 'ACTIVE', return `{ success: true, attributed: false }`
4. Look up listing by `listingId` -- verify it exists and status === 'ACTIVE'
5. Look up seller (listing.ownerUserId) sellerProfile -- check `affiliateOptIn === true`
6. If seller opted out, return `{ success: true, attributed: false }`
7. **Self-referral check:** affiliate.userId must NOT equal listing.ownerUserId (affiliate cannot promote their own listings)
8. Extract IP and userAgent from request headers (same pattern as `/ref/[code]/route.ts`)
9. Run `checkSelfReferralByIp()` from fraud-detection.ts -- if flagged, log audit event, do NOT set cookie
10. Check for existing `twicely_listing_ref` cookie -- **first-touch wins** (same as signup referral)
11. If no existing cookie: insert `referral` record with `listingId` set, `status: 'CLICKED'`, set cookie
12. Return `{ success: true, attributed: true }`

**Cookie format:**
```typescript
cookies().set(LISTING_REF_COOKIE_NAME, JSON.stringify({
  referralId: newReferral.id,
  affiliateId: aff.id,
  affiliateCode: normalizedCode,
  clickedAt: new Date().toISOString(),
}), {
  httpOnly: true,
  secure: process.env.NODE_ENV === 'production',
  sameSite: 'lax',
  path: '/',
  maxAge: attributionWindowDays * 24 * 60 * 60,
});
```

### 2.5 CLIENT COMPONENT -- AffiliateClickTracker

**New file:** `src/components/affiliate/affiliate-click-tracker.tsx`

A `'use client'` component rendered on the listing detail page that:
1. Reads `?ref=` from `useSearchParams()`
2. If present, fires a POST to `/api/affiliate/listing-click` with the listing ID and ref code
3. Fires once per page load (useEffect with ref dependency)
4. Does not block page render (fire-and-forget pattern)
5. Strips `?ref=` from the URL bar using `window.history.replaceState()` after tracking (clean URL)

```tsx
'use client';

import { useEffect } from 'react';
import { useSearchParams } from 'next/navigation';

interface AffiliateClickTrackerProps {
  listingId: string;
  listingSlug: string;
}

export function AffiliateClickTracker({ listingId, listingSlug }: AffiliateClickTrackerProps) {
  const searchParams = useSearchParams();
  const refCode = searchParams.get('ref');

  useEffect(() => {
    if (!refCode) return;

    fetch('/api/affiliate/listing-click', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        referralCode: refCode,
        listingId,
        listingSlug,
      }),
    }).catch(() => {});

    // Clean URL
    const url = new URL(window.location.href);
    url.searchParams.delete('ref');
    window.history.replaceState({}, '', url.toString());
  }, [refCode, listingId, listingSlug]);

  return null;
}
```

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- NO `SellerTier` or `SubscriptionTier` anywhere
- NO `FVF` or `Final Value Fee` -- this commission is called "affiliate commission" or just "commission"
- NO `wallet` in any seller-facing UI -- use `payout`
- NO `Twicely Balance` -- use `Available for payout`

### Architecture Constraints
- NO changes to the existing `/ref/[code]/route.ts` -- that handles signup referrals and must remain independent
- NO middleware-level interception for `?ref=` -- would slow down all `/i/` page loads
- NO hardcoded commission rates -- all rates from `platform_settings` or `sellerProfile.affiliateCommissionBps`
- NO commission calculation on the frontend -- server-side only
- NO changes to the `twicely_ref` (signup referral) cookie -- listing attribution uses a SEPARATE cookie
- NO per-listing opt-in toggle -- opt-in is per-SELLER (Feature Lock-in spec: "sellers can enable/disable affiliate commission on their **listings**" plural, controlled from Seller Settings, not per-listing)
- NO fee deduction at checkout time -- the commission ledger entry happens post-purchase (this wiring deferred to when order completion is connected to affiliate tracking)
- DO NOT spread request body into DB updates -- explicit field mapping only
- All files must be under 300 lines

### Business Logic Constraints
- Commission is paid by the SELLER (deducted from their payout), NOT by the platform (Canonical: "Paid by seller, deducted from payout, same line as TF and auth fee -- settles in THIS transaction")
- Attribution window: 7 days (NOT 30/60 days like signup referrals)
- First-touch attribution (same as signup referrals -- Canonical Section 2.3)
- Self-referral is forbidden -- affiliate cannot generate links to their OWN listings
- Commission rate range for sellers: 2-10% (200-1000 bps) per Feature Lock-in Social & Discovery
- Default commission: 3% (300 bps) per Feature Lock-in Social & Discovery

### IMPORTANT: Commission Settlement is NOT Wired Yet

The spec says commission "settles in THIS transaction" -- meaning it should be deducted from the seller's payout alongside TF and auth fees. However, the order completion flow (`src/lib/commerce/order-completion.ts`) does NOT currently check for affiliate listing attribution.

**This step (G3.6) builds the infrastructure:**
- Click tracking and cookie setting
- Seller opt-in controls
- Affiliate link generation UI
- Platform settings for commission rates

**A future step must wire:**
- Order completion reads `twicely_listing_ref` cookie to find the attributing affiliate
- Creates an `affiliateCommission` record with the listing commission amount
- Deducts from seller's payout (new ledger entry type may be needed, or reuse `AFFILIATE_COMMISSION_PAYOUT`)

Flag this as a tracked gap in the build tracker notes.

## 4. ACCEPTANCE CRITERIA

### Functional -- Listing Click Attribution
- [ ] Visiting `twicely.co/i/{slug}?ref=VALIDCODE` sets the `twicely_listing_ref` cookie with 7-day lifetime

- [ ] A `referral` record is inserted with `listingId` set and `status: 'CLICKED'`

- [ ] Visiting `twicely.co/i/{slug}?ref=SUSPENDEDAFFILIATECODE` does NOT set any cookie
- [ ] First-touch wins: if `twicely_listing_ref` cookie already exists, new click does NOT overwrite it
- [ ] Self-referral blocked: if affiliate's userId equals listing's ownerUserId, no cookie set
- [ ] IP-based fraud check runs on listing clicks (reuses existing `checkSelfReferralByIp`)

### Data Integrity
- [ ] All monetary values stored as integer cents / basis points
- [ ] Commission rates read from `platform_settings` table, never hardcoded
- [ ] `sellerProfile.affiliateCommissionBps` is nullable (null = platform default)
- [ ] Referral records with `listingId` are distinguishable from signup referrals (which have `listingId: NULL`)

### Authorization
- [ ] `/api/affiliate/listing-click` POST route is PUBLIC (no auth required -- visitors clicking affiliate links may not be logged in)
- [ ] `updateAffiliateOptIn` requires authenticated seller (CASL check on SellerProfile)
- [ ] `updateAffiliateCommissionRate` requires authenticated seller (CASL check on SellerProfile)
- [ ] Affiliate link button visibility is gated by affiliate status check (server-side data, client-side render)

### Vocabulary
- [ ] No banned terms appear in any UI text, variable names, or comments
- [ ] Commission is called "commission" (never "FVF", "fee", or "Final Value Fee")
- [ ] Route uses `/i/` prefix for listings (never `/l/`, `/listing/`, `/listings/`)

## 8. SPEC INCONSISTENCIES AND OWNER DECISIONS NEEDED

### 8.1 INCONSISTENCY: Commission Type Mismatch

The Affiliate Canonical (Section 2.2) says commissions are on **subscription revenue** only. The Feature Lock-in Social & Discovery says creator affiliate commissions are on the **sale price** and **paid by the seller**. These are fundamentally different commission models:

- Subscription commissions: platform pays, from subscription revenue
- Listing commissions: seller pays, from sale price, deducted from payout

**Resolution for G3.6:** Build the infrastructure (click tracking, opt-in, settings) without wiring the commission settlement into order completion. The commission settlement model needs an explicit owner decision since it's a different financial flow than existing affiliate commissions.

### 8.2 MISSING: Commission Settlement Wiring

When a buyer who clicked an affiliate listing link completes a purchase, the system needs to:
1. Read the `twicely_listing_ref` cookie
2. Verify the referral is still valid (within 7-day window)
3. Calculate commission (sale price * commission rate)
4. Create `affiliateCommission` record
5. Deduct from seller's payout via ledger entry

**This wiring does NOT exist and is NOT built in G3.6.** It belongs in a future step when order completion is extended. Flag this in the build tracker.

### 8.3 MISSING: Per-Listing vs Per-Seller Opt-In

The spec says "Seller opt-in: sellers can enable/disable affiliate commission on **their listings**" -- this is ambiguous. Does it mean:
- (A) Per-seller toggle (all listings on/off together) -- simpler, what this prompt implements
- (B) Per-listing toggle (each listing can be individually controlled) -- requires a column on the `listing` table

**Decision taken in this prompt:** Option A (per-seller). The spec uses "their listings" as a possessive ("the listings that belong to them"), not as "individual listings." The seller settings UI says "Allow affiliates to earn commission on my listings" which is a blanket toggle.

If the owner wants per-listing control, it requires adding `affiliateOptIn: boolean` to the `listing` table, which is a schema change beyond what's specified.

### 8.4 Affiliate Dashboard "Sales from affiliates" Stat

The spec says the seller dashboard should show "Sales from affiliates: X this month, $Y commission paid." This requires:
- Completed orders with affiliate attribution
- Commission records linked to orders (not subscriptions)

Neither exists yet. The stat is deferred with placeholder text. This should be tracked for when commission settlement is wired.
