## G10.11.4 — Real-Time: Centrifugo Client + Typing Indicator

### What V3 currently lacks (verified by exploring the codebase)
1. No Centrifugo client subscription infrastructure. `centrifugo-publisher.ts` handles server-side publish only. There is no client-side Centrifugo hook anywhere in the codebase.
2. No `/api/realtime/subscribe` route exists.
3. After `sendMessage`, the thread uses `router.refresh()` (full page reload). Spec §19 and §15 require instant delivery via Centrifugo.
4. No typing indicator. Spec §19: "Typing indicator: 'Seller is typing...' via Centrifugo."

### Architecture for G10.11.4

**Channel name**: `private-conversation.{conversationId}` — per Feature Lock-in §15 channel architecture. All channels require server-side authorization before subscription.

**Server-side token endpoint**: `POST /api/realtime/subscribe`
- Auth: require authenticated session.
- Input body (Zod validated): `{ channel: string }` — strict schema.
- Validation: `channel` must match one of the allowed patterns:
  - `private-conversation.{cuid2}` — user must be a participant in that conversation.
  - `private-user.{userId}` — user must be authenticating for their own user channel.
- For `private-conversation.{conversationId}`: call `fetchConversation(conversationId)` from `messaging-helpers.ts`. Check `isParticipant(conv, session.userId, session.onBehalfOfSellerId)`. Return `{ success: false, error: 'Not found.' }` if not a participant.
- Generate a Centrifugo subscription token using HMAC-SHA256 with `CENTRIFUGO_TOKEN_HMAC_SECRET` (env var). Token payload: `{ sub: session.userId, channel, exp: Math.floor(Date.now()/1000) + 3600 }`.
- Response: `{ success: true, token: string }`.
- If `CENTRIFUGO_TOKEN_HMAC_SECRET` is not set, return `{ success: false, error: 'Realtime not configured.' }`.
- CASL: `authorize()` called. Ability check: `ability.can('read', 'Conversation')` for conversation channels. Inline — no new CASL subjects needed.

**`src/hooks/use-conversation-realtime.ts` — CREATE**
- 'use client' hook. Manages Centrifugo subscription for one conversation thread.
- Props: `conversationId: string`, `currentUserId: string`, `onNewMessage: (msg: MessageItem) => void`, `onTyping: (userId: string) => void`.
- Uses `centrifuge` npm package (`import { Centrifuge } from 'centrifuge'`).
- On mount:
  1. Fetch subscription token via `POST /api/realtime/subscribe { channel: 'private-conversation.{conversationId}' }`.
  2. If token received: create `Centrifuge` instance pointing to `process.env.NEXT_PUBLIC_CENTRIFUGO_URL`. Connect with the subscription token.
  3. Subscribe to `private-conversation.{conversationId}`. On publication: if `data.type === 'message'`, call `onNewMessage(data.message as MessageItem)`. If `data.type === 'typing'` and `data.userId !== currentUserId`, call `onTyping(data.userId)`.
- On unmount: disconnect Centrifuge.
- Returns `{ isConnected: boolean }`.
- If `NEXT_PUBLIC_CENTRIFUGO_URL` is not set (falsy), log a warning and return `{ isConnected: false }` — graceful degradation, no crash.

**`src/app/api/messaging/typing/route.ts` — CREATE**

- Auth: `authorize()`. Require authenticated session.
- Input (Zod validated, strict): `{ conversationId: string (cuid2) }`.
- Participant check: call `fetchConversation(conversationId)`. Call `isParticipant(conv, session.userId, session.onBehalfOfSellerId)`. Return 403 if not a participant. Return 404 if conversation not found.
- Status check: only publish if `conv.status === 'OPEN'`. Return 200 silently if READ_ONLY (don't error — just no-op).
- Publish to Centrifugo: `publishToChannel('private-conversation.{conversationId}', { type: 'typing', userId: session.userId })`.
- Rate limit: max 1 publish per 2 seconds per user per conversation (client-side debounce handles this — server is fire-and-forget via existing `centrifugo-publisher.ts`).
- Response: `{ success: true }`.
- This route is called by the composer's `onKeyDown` handler (debounced at 1500ms client-side).

## CONSTRAINTS — What NOT to Do

### No banned terms
- Never use `wallet`, `Withdraw`, `Twicely Balance`, `FVF`, `SellerTier`, `SubscriptionTier` in any UI text or code.
- Messaging has no fee implications — the payout vocabulary constraint does not apply here, but general banned-term rules apply to any UI copy.

### No schema changes beyond what is explicitly specified
- Do NOT create a `sellerQuickReply` table. Feature Lock-in §19 says sellers manage a quick reply library, but the schema doc (§9) has no such table. The static quick reply implementation in G10.11.3 is the spec-compliant approach until the owner adds this to the schema doc.
- Do NOT add any columns to `conversation` or `message`. Both tables are fully aligned with schema §9.1 and §9.2.

### No new pages
- G10.11 adds NO new pages. All changes are to existing pages (`/my/messages`, `/my/messages/[conversationId]`) and components.
- `/api/realtime/subscribe` and `/api/messaging/typing` are new API routes — these are NOT in the Page Registry (which covers user-facing pages). API routes do not need page registry entries.

### No hardcoded values
- Image attachment max size must come from `platform_settings` key `comms.messaging.attachmentMaxBytes`. The seed value (10 MB = 10_485_760) is the default. The upload handler must read from DB, not a constant.
- Message rate limit already reads from `comms.messaging.rateLimitPerHour` in `messaging-helpers.ts`. Do not change this.

### No banned tech
- Do not use Redis. Centrifugo is the real-time layer.
- Do not use Zustand or Redux. Local `useState` is correct for message list state.
- The `centrifuge` npm package is the official Centrifugo JavaScript client. This is correct.

### Code pattern rules
- All new `'use server'` files: `authorize()` before any DB access, Zod `.strict()` on all inputs, explicit field mapping (no spread).
- All new API routes: return `{ success: boolean, error?: string, ... }` shape.
- All new client components: `'use client'` directive at top.
- No `as any`, no `@ts-ignore`, no `@ts-expect-error`.
- All files max 300 lines.

### Route correctness
- Conversation route: `/my/messages/[conversationId]` — NOT `/m/[conversationId]`. The `/m` route redirects to `/my/messages` (per Unified Hub Canonical §2.1). Do not change redirect behavior.
- Real-time subscribe API: `/api/realtime/subscribe` (POST).
- Typing API: `/api/messaging/typing` (POST).

### Ownership model
- Do not use `storeId`, `sellerProfileId`, or `businessId` as ownership keys. Conversations are owned by `buyerId` and `sellerId` — both are `user.id` values. This is correct in the existing code.

### Real-time degradation
- If Centrifugo is not configured (`NEXT_PUBLIC_CENTRIFUGO_URL` is falsy), the hook must silently skip connection and return `{ isConnected: false }`. The thread must still work (SSR renders messages, `router.refresh()` fallback via existing compose flow). No crash, no error banner.

## Summary of What Will Be Built

G10.11 upgrades Twicely's messaging UI from 4 basic functional components to a polished chat-app experience:

- **G10.11.1**: The inbox gets "All/Buying/Selling" filter tabs and last-message preview with "You:" sender prefix — exactly matching the spec's conversation list requirements.
- **G10.11.2**: Conversation threads show the other party's name and avatar prominently, display "Read" receipts under the last read own message, and add archive and report actions accessible from the thread header.
- **G10.11.3**: The Paperclip button is fully enabled — users can upload up to 4 images per message via R2, with file size enforcement from platform settings. Five static quick reply chips appear above the compose area for one-click message templates.
- **G10.11.4**: Centrifugo WebSocket integration makes messages appear instantly without page refresh. A typing indicator shows "Typing..." when the other party is composing. A new `/api/realtime/subscribe` endpoint issues scoped channel tokens. All real-time features degrade gracefully when Centrifugo is not configured.
