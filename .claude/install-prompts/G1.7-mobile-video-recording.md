## 3. WHAT EXISTS (from G3.11) -- Do Not Duplicate

The following infrastructure is complete and working. G1.7 MUST reuse it, not rebuild it:

**Upload pipeline:**
- `/api/upload` route handles `type=video` and `type=video-thumbnail` -- routes to `video-handler.ts`
- `video-handler.ts` validates magic bytes, duration, size; uploads to R2 or local fallback
- `video-service.ts` handles R2 key generation (`videos/listings/{listingId}/`)
- `validate-video.ts` provides `detectVideoType()`, `validateVideoBytes()`, constants

**Client-side flow (existing in `video-uploader.tsx`):**
1. User picks file via `<input type="file">` or drag-and-drop
2. Client validates size (100MB) and duration (15-60s) via `<video>` element
3. Client uploads video to `/api/upload` with `type=video`
4. Client extracts thumbnail via `<canvas>` at 0.5s mark
5. Client uploads thumbnail to `/api/upload` with `type=video-thumbnail`
6. `onVideoChange` callback fires with `{ url, thumbUrl, durationSeconds }`

G1.7 adds a SECOND entry path (camera recording) that produces the same `File` object, then feeds into the same upload pipeline. The trim tool operates BEFORE upload.

## 4. SCOPE -- EXACTLY WHAT TO BUILD

### 4.1 Overview of User Flow

**Mobile user on listing creation form, "Video (optional)" card:**

1. Card shows TWO entry points (instead of just the current drop zone):
   - "Record Video" button (camera icon) -- NEW, G1.7
   - "Upload from Gallery" button (upload icon) -- existing behavior, refactored
2. Tapping "Record Video":
   - Request camera permission via `navigator.mediaDevices.getUserMedia({ video: true, audio: true })`
   - Show live camera preview with recording controls
   - User taps record button -- starts recording
   - Duration timer counts up; max 60s auto-stop
   - User taps stop -- recording ends
   - Transition to trim screen
3. Trim screen (applies to BOTH recorded and gallery-uploaded videos):
   - Shows video preview
   - Two draggable handles (start/end) on a timeline scrubber
   - Current selection duration displayed (must be 15-60s)
   - "Use This Clip" button -- commits the trim selection
4. After trim (or if video is already within 15-60s and user skips trim):
   - Create a `File` object from the trimmed segment
   - Feed into existing `handleFile()` flow in `video-uploader.tsx`
   - Thumbnail extraction + upload proceeds as normal

### 4.2 Component: `video-recorder.tsx` (NEW)

**Path:** `src/components/pages/listing/video-recorder.tsx`

**Purpose:** Full-screen camera recording overlay for mobile video capture.

**Props:**
```typescript
interface VideoRecorderProps {
  maxDurationSeconds: number; // from platform settings (default 60)
  minDurationSeconds: number; // from platform settings (default 15)
  onRecordingComplete: (file: File) => void;
  onCancel: () => void;
}
```

**Implementation details:**
- Uses `navigator.mediaDevices.getUserMedia({ video: { facingMode: 'environment' }, audio: true })` for rear camera
- Camera flip button to toggle `facingMode` between `'environment'` and `'user'`
- Live `<video>` preview element with `autoPlay`, `muted`, `playsInline`
- Record button (large, centered, red circle -- standard mobile recording UX)
- Recording state shows:
  - Elapsed time counter (MM:SS format)
  - Red pulsing dot indicator
  - Stop button replaces record button
- Auto-stop at `maxDurationSeconds`
- When recording stops:
  - Stop all `MediaRecorder` chunks, create Blob
  - Determine MIME type: prefer `video/mp4` if `MediaRecorder.isTypeSupported('video/mp4')`, fallback to `video/webm`
  - Convert Blob to `File` with appropriate extension
  - Call `onRecordingComplete(file)`
- Cancel button (X or "Cancel" text) at top-left stops recording and calls `onCancel()`
- Full-screen overlay (`fixed inset-0 z-50 bg-black`) -- covers everything while recording
- Stop and release all media tracks on unmount (cleanup in `useEffect` return)
- Handle permission denied: show error message "Camera access is needed to record video. Please allow camera access in your browser settings."
- Handle `getUserMedia` not available (desktop/older browsers): do not render the record button at all; check `typeof navigator.mediaDevices?.getUserMedia === 'function'`

**MediaRecorder configuration:**
```typescript
const mimeType = MediaRecorder.isTypeSupported('video/mp4')
  ? 'video/mp4'
  : MediaRecorder.isTypeSupported('video/webm;codecs=vp9')
    ? 'video/webm;codecs=vp9'
    : 'video/webm';

const recorder = new MediaRecorder(stream, {
  mimeType,
  videoBitsPerSecond: 2_500_000, // 2.5 Mbps -- good quality under 100MB for 60s
});
```

**Constraints / rules:**
- Max file length: 300 lines
- `'use client'` directive required
- No server-side imports (no `fs`, no `path`, no `db`)
- Must handle all MediaRecorder lifecycle events: `ondataavailable`, `onstop`, `onerror`
- Must clean up: stop tracks, revoke object URLs, on unmount AND on cancel

### 4.3 Component: `video-trimmer.tsx` (NEW)

**Path:** `src/components/pages/listing/video-trimmer.tsx`

**Purpose:** Basic trim tool with start/end point selection. Displayed after recording OR after selecting a long video from gallery that needs trimming.

**Props:**
```typescript
interface VideoTrimmerProps {
  videoFile: File;
  minDurationSeconds: number;
  maxDurationSeconds: number;
  onTrimComplete: (trimmedFile: File) => void;
  onCancel: () => void;
}
```

**Implementation details:**
- Loads video file into `<video>` element via `URL.createObjectURL()`
- Shows video preview (playable, with controls)
- Timeline scrubber below video:
  - Full width bar representing total video duration
  - Two drag handles: start (left) and end (right)
  - Colored region between handles shows selected segment
  - Duration label shows selected duration: "Selected: {N}s"
  - Validation message if selection < `minDurationSeconds` or > `maxDurationSeconds`
- "Preview Clip" button: seeks video to start handle position and plays to end handle position, then pauses
- "Use This Clip" button (primary, disabled if invalid duration):
  - Uses `MediaRecorder` + `captureStream()` to re-record the trimmed segment:
    1. Seek `<video>` to `startTime`
    2. Call `video.captureStream()` to get a `MediaStream`
    3. Create `MediaRecorder` on captured stream
    4. Play video from `startTime`
    5. When `video.currentTime >= endTime`, stop recorder
    6. Assemble chunks into `File`
    7. Call `onTrimComplete(trimmedFile)`
  - Show loading state during re-recording ("Trimming...")
- "Cancel" button: calls `onCancel()`, revokes object URL
- If video is already within min/max duration range, start and end handles default to 0 and full duration (user can still adjust)
- If video exceeds max duration, auto-set end handle to `maxDurationSeconds` from start

**IMPORTANT -- Trim is client-side re-recording, not byte-level cutting:**
The trim tool plays the video between start/end and re-records via `captureStream()` + `MediaRecorder`. This produces a new valid video file without needing server-side ffmpeg. Quality is maintained because we're capturing the decoded video stream. The output format follows the same MIME type negotiation as the recorder (MP4 preferred, WebM fallback).

**Constraints / rules:**
- Max file length: 300 lines
- `'use client'` directive required
- Must revoke all object URLs on unmount
- Touch-friendly drag handles (minimum 44x44px per WCAG)
- `prefers-reduced-motion`: disable auto-preview animations

### 4.4 Hook: `use-camera-support.ts` (NEW)

**Path:** `src/hooks/use-camera-support.ts`

**Purpose:** Detect whether the browser supports in-app camera recording.

```typescript
export function useCameraSupport(): {
  hasCamera: boolean;
  hasMediaRecorder: boolean;
  isSupported: boolean; // hasCamera && hasMediaRecorder
  isLoading: boolean;
}
```

**Implementation:**
- Check `typeof navigator !== 'undefined'` (SSR guard)
- Check `typeof navigator.mediaDevices?.getUserMedia === 'function'`
- Check `typeof MediaRecorder !== 'undefined'`
- Use `navigator.mediaDevices.enumerateDevices()` to check for at least one `videoinput` device
- `isSupported = hasCamera && hasMediaRecorder`
- `isLoading = true` until `enumerateDevices()` resolves
- This hook determines whether to show the "Record Video" button. On desktop browsers without cameras or older browsers, only the upload option appears.
- Note: `enumerateDevices()` may return devices without labels if permission hasn't been granted yet. We only need to check `.kind === 'videoinput'` exists, not the label.

## 5. SECURITY -- CRITICAL NOTE

### Permissions-Policy Conflict

The Actors/Security Canonical (Section 5.5, line 1085) specifies:
```
Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(self)
```

`camera=()` means "deny camera access for all origins including self." This would BLOCK `getUserMedia()`.

**Current state:** This header is NOT implemented anywhere in the codebase. Neither `next.config.ts` nor `middleware.ts` sets security headers. The canonical header specification is aspirational/planned (Phase G6 or later).

**What to do now:** G1.7 implements camera recording assuming NO `Permissions-Policy` header is set. When security headers are implemented (likely G6), the header MUST be updated to:
```
Permissions-Policy: camera=(self), microphone=(self), geolocation=(), payment=(self)
```

**Action for installer:** Add a code comment in `video-recorder.tsx` noting this dependency:
```typescript
// NOTE: Actors/Security Canonical specifies Permissions-Policy camera=() which would block
// getUserMedia. When security headers are implemented (G6), update to camera=(self).
```

**NOT SPECIFIED -- Owner Decision Needed:** When security headers (Phase G6) are implemented, should `camera` be allowed on `(self)` unconditionally, or only on specific pages like `/my/selling/listings/new`? The `Permissions-Policy` header is set globally (per response) and cannot be page-specific without middleware logic. Recommend `camera=(self)` globally since camera is only invoked by user action (never ambient).

## 11. EDGE CASES TO HANDLE

| Edge Case | Expected Behavior |
|---|---|
| User denies camera permission | Show error message in recorder overlay: "Camera access is needed to record video. Please allow camera access in your browser settings." Return to video card. |
| Browser has no getUserMedia (desktop Firefox ESR, etc.) | `useCameraSupport` returns `isSupported=false`. Record button never appears. Upload-only mode. |
| Camera is in use by another app | `getUserMedia` rejects with `NotReadableError`. Show error: "Camera is currently in use by another application." |
| MediaRecorder doesn't support video/mp4 | Fall back to `video/webm`. Both are in the allowed formats list for the upload pipeline. |
| User records < 15 seconds and stops | In the trim tool, "Use This Clip" is disabled with message "Video must be at least 15 seconds." If they recorded and stopped within the recorder, the trimmer still opens but the clip is too short to use. |
| User records exactly 60 seconds (auto-stop) | Auto-stop fires, transition to trimmer. Entire clip is pre-selected. User can trim shorter if desired. |
| User toggles camera during recording | Camera flip re-acquires stream. Recording restarts (not stitched). Display warning: "Switching cameras will restart recording." |
| Orientation change during recording | The `<video>` element and recorder should handle orientation via CSS `object-fit: cover`. No explicit orientation lock (browser handles this). |
| Very large recorded file (>100MB for 60s at high bitrate) | Client-side size check before upload (existing in `handleFile`). If exceeded, show error "Video must be less than 100MB." Target bitrate of 2.5 Mbps keeps 60s under ~20MB. |
| `captureStream()` not available (very old browsers) | Trim tool fallback: show message "Trimming is not supported in your browser. Please record a video between 15 and 60 seconds." Disable trim functionality, only allow full-length clips within valid range. |
| User navigates away during recording | `useEffect` cleanup stops all tracks and releases recorder. |
| User backgrounded the tab during recording | Recording continues (browser behavior). Timer continues. No special handling needed. |

## 13. SPEC INCONSISTENCIES / OWNER DECISIONS

### 13.1 Camera Flip During Recording

**NOT SPECIFIED:** The canonical says "In-app camera access" but does not specify whether camera flip is available during recording. Switching cameras mid-recording requires stopping and restarting the MediaRecorder, which means the recording is lost.

**Recommended:** Allow camera flip BEFORE recording starts (during preview), but disable flip WHILE recording is active. This avoids the complexity of stream stitching.

**Owner decision needed:** Should camera flip be available during recording (requires recording restart)?

### 13.2 Trim Tool for Gallery-Uploaded Videos

**NOT SPECIFIED:** The canonical says "Basic trim tool (set start/end point)" but does not explicitly say it applies to gallery-uploaded videos. The trim tool was designed alongside camera recording.

**Recommended:** Apply trim tool to gallery-uploaded videos that exceed `maxDurationSeconds`. This prevents the current "Video must be between 15 and 60 seconds" hard rejection for slightly-too-long videos. Users can trim down instead of re-recording.

**Owner decision needed:** Should the trim tool be available for ALL gallery uploads (not just over-length ones)?

### 13.3 Audio Recording

**NOT SPECIFIED:** The canonical says "Record directly" but does not mention audio. Videos without audio are generally acceptable for item showcases, but ambient audio adds authenticity.

**Recommended:** Request both video and audio (`{ video: true, audio: true }`). If microphone access is denied, record video-only silently (no error). This mirrors standard mobile camera behavior.

### 13.4 Permissions-Policy Header Conflict

**DOCUMENTED ABOVE in Section 5.** When security headers are eventually implemented (G6), `camera=(self)` must replace `camera=()` to allow this feature to function.

- Resolution selection UI (let the browser pick optimal resolution)
- Video compression or quality adjustment UI

- Video in search results thumbnails (separate discovery feature)

- ffmpeg/wasm-based client-side processing
