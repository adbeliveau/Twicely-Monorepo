### 2.2 CASL Subject Addition

Add `'ListingQuestion'` to the CASL subjects list in `src/lib/casl/subjects.ts`.

**Current subjects.ts** has 37 entries. Add `'ListingQuestion'` to the array.

**CASL ability rules to add** in `src/lib/casl/ability.ts`:

- **Guest**: `can('read', 'ListingQuestion')` -- Q&A is public, visible to all (Amazon-style)
- **Buyer**: inherits Guest read + `can('create', 'ListingQuestion')` -- any authenticated user can ask a question
- **Seller**: inherits Buyer + `can('update', 'ListingQuestion', { answeredBy: userId })` + `can('delete', 'ListingQuestion', { answeredBy: userId })` -- seller can answer (update) and hide (delete) Q&A on their own listings. Also: `can('update', 'ListingQuestion', { listingOwnerId: userId })` for answering questions on their listings.

**IMPORTANT ownership note:** The `listingQuestion` table does NOT have a direct `sellerId` or `ownerUserId` column. The seller relationship is indirect via `listingQuestion.listingId -> listing.ownerUserId`. For CASL conditions:
- `askerId: userId` for asker-owns-question checks
- For "seller can answer/hide questions on their listings," the CASL condition must check listing ownership. Since CASL conditions operate on flat objects, the action layer must verify `listing.ownerUserId === session.userId` before allowing answer/hide operations. Use a generic condition: `can('update', 'ListingQuestion')` for sellers, with action-level ownership verification (same pattern used for offers).

**Simplified CASL rules:**

In `defineGuestAbilities`:
```typescript
can('read', 'ListingQuestion');
```

In `defineBuyerAbilities`:
```typescript
can('create', 'ListingQuestion');
can('read', 'ListingQuestion');
```

In `defineSellerAbilities`:
```typescript
// Sellers can answer/hide questions on their own listings
// Action-level check verifies listing.ownerUserId === session.userId
can('update', 'ListingQuestion');
can('delete', 'ListingQuestion');
```

### 2.5 Q&A Notifier Service

Create `src/lib/notifications/qa-notifier.ts` following the `category-alert-notifier.ts` pattern.

Two exported functions:

**`notifyQuestionAsked(questionId: string): Promise<void>`**
- Fetch the `listingQuestion` row by `questionId`
- If question not found or `isHidden`, return early
- Fetch the associated `listing` row to get `ownerUserId`, `title`, `slug`
- Fetch the asker's `user` row to get `name`
- If `listing.ownerUserId === question.askerId`, return early (don't notify seller about their own question)
- Call `notify(listing.ownerUserId, 'qa.new_question', { ... })`
- Fire-and-forget: wrap in try/catch, log errors, never throw

**`notifyQuestionAnswered(questionId: string): Promise<void>`**
- Fetch the `listingQuestion` row by `questionId`
- If question not found, `isHidden`, or no `answerText`, return early
- Fetch the associated `listing` row to get `title`, `slug`
- Fetch the asker's `user` row to get `name`
- If `question.answeredBy === question.askerId`, return early (don't notify asker if they somehow answered their own question)
- Call `notify(question.askerId, 'qa.answer_received', { ... })`
- Fire-and-forget: wrap in try/catch, log errors, never throw

**Data passed to `notify()` for `qa.new_question`:**
```typescript
{
  recipientName: sellerName,
  askerName: askerUser.name ?? 'Someone',
  itemTitle: listing.title ?? 'Item',
  questionText: question.questionText.slice(0, 200), // Truncate for notification
  listingUrl: `${BASE_URL}/i/${listing.slug}`,
}
```

**Data passed to `notify()` for `qa.answer_received`:**
```typescript
{
  recipientName: askerUser.name ?? 'there',
  itemTitle: listing.title ?? 'Item',
  answerText: question.answerText!.slice(0, 200), // Truncate for notification
  listingUrl: `${BASE_URL}/i/${listing.slug}`,
}
```

### 2.6 Q&A Server Actions (Ask + Answer)

**IMPORTANT SCOPE NOTE:** E1.3 builds the actions needed to ask questions and answer them, because the notification wiring depends on these actions existing to call the notifier functions. E1.4 builds the UI that calls these actions. The actions are the minimum needed to wire notifications.

Create `src/lib/actions/qa.ts` with `'use server'` directive.

**`askQuestion(input: AskQuestionInput): Promise<ActionResult>`**
- Zod schema (`askQuestionSchema`):
  ```typescript
  z.object({
    listingId: z.string().cuid2(),
    questionText: z.string().min(5).max(500),
  }).strict()
  ```
- `authorize()` -- must be authenticated (any buyer/seller can ask)
- CASL check: `ability.can('create', 'ListingQuestion')`
- Verify listing exists and is ACTIVE status
- Verify user has not already asked 3 unanswered questions on this listing (anti-spam: max 3 pending questions per user per listing)
- Insert into `listingQuestion` with explicit field mapping:
  - `listingId`, `askerId: session.userId`, `questionText`
- Call `notifyQuestionAsked(newQuestionId)` (fire-and-forget, no await needed for the notification -- but DO await the insert)
- `revalidatePath('/i/[slug]')` (for when E1.4 adds the Q&A section)
- Return `{ success: true, questionId }`

**`answerQuestion(input: AnswerQuestionInput): Promise<ActionResult>`**
- Zod schema (`answerQuestionSchema`):
  ```typescript
  z.object({
    questionId: z.string().cuid2(),
    answerText: z.string().min(1).max(1000),
  }).strict()
  ```
- `authorize()` -- must be authenticated
- Fetch the question row + its associated listing
- Verify `listing.ownerUserId === session.userId` (only listing owner can answer) OR check delegation (if `session.delegationId`, verify the delegate's seller is the listing owner)
- Verify question is not already answered (`answeredAt === null`)
- Verify question is not hidden
- Update `listingQuestion` with explicit field mapping:
  - `answerText`, `answeredAt: new Date()`, `answeredBy: session.userId`, `updatedAt: new Date()`
- Call `notifyQuestionAnswered(questionId)` (fire-and-forget)
- `revalidatePath('/i/[slug]')`
- Return `{ success: true }`

**`hideQuestion(input: HideQuestionInput): Promise<ActionResult>`**
- Zod schema:
  ```typescript
  z.object({
    questionId: z.string().cuid2(),
  }).strict()
  ```
- `authorize()` -- must be authenticated
- Fetch the question + listing
- Verify `listing.ownerUserId === session.userId` (only listing owner can hide)
- Update `isHidden: true, updatedAt: new Date()`
- No notification needed for hide
- `revalidatePath('/i/[slug]')`
- Return `{ success: true }`

**`pinQuestion(input: PinQuestionInput): Promise<ActionResult>`**
- Zod schema:
  ```typescript
  z.object({
    questionId: z.string().cuid2(),
    isPinned: z.boolean(),
  }).strict()
  ```
- `authorize()` -- must be authenticated
- Fetch the question + listing
- Verify `listing.ownerUserId === session.userId`
- If pinning (`isPinned: true`): count existing pinned questions on this listing, reject if >= 3 (schema note: "seller can pin up to 3")
- Update `isPinned`, `updatedAt: new Date()`
- No notification for pin/unpin
- `revalidatePath('/i/[slug]')`
- Return `{ success: true }`

## SPEC GAPS AND DECISIONS

### Confirmed from Canonicals
1. **Q&A is Amazon-style structured, NOT Poshmark-style comments** (Decision #55, #56)
2. **Q&A is public** -- visible to all, including guests (Decision #56: "buyer asks, seller answers, visible to all")
3. **Schema is finalized** -- `listingQuestion` table from Schema Doc Section 24.1, implemented in A2.2
4. **Step numbering** -- E1.3 = Q&A notifications, confirmed in Decision #63

### Anti-Spam Limit
- Max 3 unanswered questions per user per listing. This is NOT in the canonical docs. This is a reasonable default to prevent abuse. The asker can ask more once existing questions are answered.
- **NOT SPECIFIED -- owner decision needed:** Should there be a rate limit on questions (e.g., max 10 questions per user per day across all listings)? This prompt implements only the per-listing cap. Global rate limiting can be added in E1.4 or later.

### Delegation Support
- `answerQuestion` should respect delegation. If `session.delegationId` is set, the action should verify the delegate's seller (`session.sellerId`) matches `listing.ownerUserId`. The delegation system (D5) already populates `session.sellerId` with the delegating seller's userId.

### No Moderation Scan
- The canonical mentions "moderation scan" in E1.4 context. E1.3 does NOT implement moderation/content scanning. It only provides the `isHidden` flag for manual seller hiding. Automated moderation is a separate concern.
