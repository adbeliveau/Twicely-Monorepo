### 2.2 Poshmark Content Script (`extensions/chrome/src/content/poshmark.ts`)

Create a new module that handles all Poshmark-specific DOM interactions. This is a plain TypeScript module (NOT an IIFE) -- it exports functions that the bridge dispatcher will call.

**Session capture:**
Poshmark stores its JWT in a cookie named `_web_session` or in `localStorage` under `__posh_auth`. The content script must:
1. Read `document.cookie` and look for `_posh_session` or similar JWT cookie
2. Check `window.localStorage` for Poshmark auth tokens
3. Extract the JWT string, the username (from DOM element `.header__username` or `[data-username]` or from the Poshmark `__NEXT_DATA__` JSON blob in the page), and the userId
4. Build a `PoshmarkBrowserSession` object and send it via `SESSION_CAPTURED` message

**Key implementation details for session capture:**
- Poshmark is a React/Next.js app. It embeds a `<script id="__NEXT_DATA__">` JSON blob containing user info
- Parse this blob to extract `user.id`, `user.username`, and auth tokens
- The JWT may also be in `document.cookie` as `_posh_jwt` (exact cookie name varies; search for a JWT-shaped value starting with `eyJ`)
- If session data cannot be extracted (user not logged in), send NO message -- do not report failure

**Listing scrape:**
When the URL matches `/listing/<id>`, the content script scrapes the listing page DOM:

```typescript
function scrapePoshmarkListing(): ScrapedListing | null {
  // Only run on listing detail pages: poshmark.com/listing/<id>
  // Extract data from:
  //   - Title: .listing__title, h1[data-test="listing-title"], or __NEXT_DATA__
  //   - Price: .listing__price, [data-test="listing-price"], or __NEXT_DATA__
  //     Convert "$XX.XX" string to integer cents (multiply by 100, round)
  //   - Description: .listing__description, [data-test="listing-description"]
  //   - Condition: badge text containing NWT/NWOT/Good/Fair etc.
  //   - Brand: .listing__brand a, [data-test="listing-brand"]
  //   - Category: breadcrumb trail text
  //   - Size: .listing__size, [data-test="listing-size"]
  //   - Images: all img elements in the gallery carousel (.gallery img, [data-test="listing-image"])
  //     Extract full-size URLs, not thumbnails (look for url_fullsize in data attributes or srcset)
  //   - External ID: extract from URL pathname (/listing/<ID>)
  //   - URL: window.location.href
}
```

**CRITICAL: Poshmark uses `__NEXT_DATA__` extensively.** The most reliable scrape path is:
1. Find `<script id="__NEXT_DATA__" type="application/json">`
2. Parse the JSON
3. Navigate to `props.pageProps.listing` (or similar path -- the exact shape may vary)
4. Extract title, price, description, brand, size, condition, images from the structured data
5. FALL BACK to DOM selectors only if `__NEXT_DATA__` is not available or missing the listing

**Auto-fill for creating a listing:**
When the URL matches Poshmark's listing creation page (`/create-listing` or `/create_listing` or `/closet/create`), the content script can fill form fields:

```typescript
function autofillPoshmarkListing(payload: AutofillPayload): boolean {
  // Fill form fields on the listing creation page:
  //   - Title input: set value, dispatch 'input' + 'change' events
  //   - Description textarea: set value, dispatch events
  //   - Price input: convert cents to dollars string, set value, dispatch events
  //   - Brand input: type brand name, select from autocomplete dropdown
  //   - Size selector: click the matching size option
  //   - Condition selector: click the matching condition radio/button
  //   - Category: navigate category tree by clicking breadcrumb options
  //
  // IMPORTANT: Poshmark React forms require dispatching native Input events
  //   to trigger React's synthetic event handlers. Use:
  //   const nativeInputValueSetter = Object.getOwnPropertyDescriptor(
  //     HTMLInputElement.prototype, 'value'
  //   )?.set;
  //   nativeInputValueSetter?.call(input, newValue);
  //   input.dispatchEvent(new Event('input', { bubbles: true }));
  //   input.dispatchEvent(new Event('change', { bubbles: true }));
  //
  // Return true if all required fields were filled, false otherwise.
  // Photo upload is NOT automated (browser security blocks programmatic file input).
}
```

**Photo upload note:** The content script CANNOT programmatically set file input values due to browser security restrictions. Auto-fill sets all text fields. Photos must be manually added by the seller. The auto-fill response should include a note: `photoUploadRequired: true`.

### 2.3 Facebook Marketplace Content Script (`extensions/chrome/src/content/fb-marketplace.ts`)

Create a new module for Facebook Marketplace DOM interactions.

**Session capture:**
Facebook stores its access token differently than Poshmark:
1. Check for `document.cookie` for `c_user` (Facebook user ID) and session cookies
2. Facebook's Graph API tokens are NOT available in the DOM for regular Marketplace users. The extension captures the `c_user` cookie value (user ID) and the `xs` cookie (session cookie) as a session indicator
3. For sellers who have connected via Facebook Commerce API (OAuth, Tier B), the `accessToken` is stored server-side. The extension session capture for FB is informational only -- it confirms the user is logged in and captures their Facebook user ID
4. Build a `FbMarketplaceBrowserSession` and send via `SESSION_CAPTURED`

**Key difference from Poshmark:** FB Marketplace is officially Tier B (OAuth/Graph API) in the codebase. The extension's role for FB is lighter than for Poshmark:
- Session capture = confirm user identity (c_user) + detect login state
- Scraping = read listing data from the DOM (Facebook does NOT have `__NEXT_DATA__`)
- Auto-fill = fill the Marketplace listing creation form

**Listing scrape:**
When URL matches `/marketplace/item/<id>`, scrape the listing:

```typescript
function scrapeFbMarketplaceListing(): ScrapedListing | null {
  // Facebook Marketplace listing pages use dynamic React rendering
  // Extract data from DOM elements (NO __NEXT_DATA__ equivalent):
  //   - Title: main heading in the listing detail container
  //     Selector strategy: [role="main"] h1, span[dir="auto"] within the listing container
  //   - Price: text containing "$" near the title
  //     Parse "$XX" or "$X,XXX" to integer cents
  //   - Description: large text block below title/price area
  //   - Condition: badge/label text (New, Used - Like New, Used - Good, Used - Fair)
  //   - Category: breadcrumb or category label
  //   - Images: carousel images in the media gallery
  //     Extract from img[src] or background-image styles
  //     Look for high-res versions (scontent URLs with large dimensions)
  //   - External ID: extract from URL pathname (/marketplace/item/<ID>/)
  //   - URL: window.location.href
  //
  // Facebook heavily obfuscates class names (random hashes).
  // Strategy: use ARIA roles, data-testid attributes, and structural
  // selectors (nesting depth, element types) rather than class names.
  // Fall back to text content pattern matching.
}
```

**Facebook DOM scraping challenges:**
- Facebook uses randomized class names that change between deployments
- Rely on: `[role="main"]`, `[data-testid]`, `aria-label`, semantic HTML (h1, img, etc.)
- The listing price may be in a `<span>` without any identifying attribute -- locate by proximity to the title heading
- Images are lazy-loaded; the initial page may have placeholder URLs. Use `MutationObserver` to wait for real image URLs to load, with a 3-second timeout

**Auto-fill for creating a listing:**
When URL matches `/marketplace/create/item/` or `/marketplace/create/`, fill the creation form:

```typescript
function autofillFbMarketplaceListing(payload: AutofillPayload): boolean {
  // Facebook Marketplace listing creation form:
  //   - Title: find the title input field (aria-label="Title" or placeholder="Title")
  //   - Price: find price input (aria-label="Price" or placeholder="Price")
  //     Convert cents to whole dollars string
  //   - Description: find the description textarea/contenteditable
  //   - Condition: click the matching condition radio button
  //   - Category: Facebook auto-suggests category; may need to type and select
  //
  // Same React event dispatch technique as Poshmark (native setter + input event).
  // Facebook may use contenteditable divs instead of textareas -- handle both.
  //
  // Return true if required fields filled, false otherwise.
```

### 2.4 Bridge Dispatcher (modify `extensions/chrome/src/content/bridge.ts`)

Replace the current minimal bridge with a dispatcher that:
1. Detects the current platform (same hostname matching as H1.1)
2. Imports and calls the platform-specific module
3. Sends `PLATFORM_DETECTED` message (preserved from H1.1)
4. Automatically triggers session capture on page load
5. Automatically triggers listing scrape if on a listing detail page
6. Handles `EXECUTE_ACTION` messages by routing to the correct platform handler

```typescript
// bridge.ts -- Platform dispatcher
(function twicelyCrosslisterBridge() {
  const hostname = window.location.hostname;
  const pathname = window.location.pathname;

  type PlatformHandler = {
    captureSession: () => Promise<Record<string, unknown> | null>;
    scrapeListing: () => ScrapedListing | null;
    autofillListing: (payload: AutofillPayload) => boolean;
  };

  let channel: ExtensionChannel | null = null;
  let handler: PlatformHandler | null = null;

  // Platform detection + handler assignment
  if (hostname === 'poshmark.com' || hostname === 'www.poshmark.com') {
    channel = 'POSHMARK';
    // Import poshmark handler (bundled by esbuild)
    handler = poshmarkHandler; // Direct reference, not dynamic import
  } else if (hostname === 'www.facebook.com' && pathname.startsWith('/marketplace')) {
    channel = 'FB_MARKETPLACE';
    handler = fbMarketplaceHandler;
  }
  // THEREALREAL and VESTIAIRE left as stubs for H1.3/H4.1

  if (!channel) return;

  // 1. Report platform detection
  chrome.runtime.sendMessage({
    type: 'PLATFORM_DETECTED',
    channel,
    url: window.location.href,
  });

  // 2. Auto-capture session (non-blocking)
  if (handler) {
    handler.captureSession().then((sessionData) => {
      if (sessionData) {
        chrome.runtime.sendMessage({
          type: 'SESSION_CAPTURED',
          channel: channel!,
          sessionData,
        });
      }
    }).catch(() => { /* Silent -- session capture is best-effort */ });
  }

  // 3. Auto-scrape if on a listing detail page
  if (handler) {
    // Poshmark: /listing/<id>
    // FB: /marketplace/item/<id>/
    const isListingPage =
      (channel === 'POSHMARK' && pathname.startsWith('/listing/')) ||
      (channel === 'FB_MARKETPLACE' && /^\/marketplace\/item\/\d+/.test(pathname));

    if (isListingPage) {
      // Delay to allow page to render (SPA content may not be in DOM immediately)
      setTimeout(() => {
        const listing = handler!.scrapeListing();
        if (listing) {
          chrome.runtime.sendMessage({
            type: 'LISTING_SCRAPED',
            channel: channel!,
            listing,
          });
        }
      }, 2000); // 2-second delay for SPA hydration
    }
  }

  // 4. Listen for EXECUTE_ACTION from service worker
  chrome.runtime.onMessage.addListener(
    (message: { type: string; action?: string; payload?: Record<string, unknown> }) => {
      if (message.type !== 'EXECUTE_ACTION' || !handler || !channel) return;

      if (message.action === 'SCRAPE_LISTING') {
        const listing = handler.scrapeListing();
        chrome.runtime.sendMessage({
          type: listing ? 'LISTING_SCRAPED' : 'ACTION_RESULT',
          channel,
          ...(listing
            ? { listing }
            : { action: 'SCRAPE_LISTING', success: false, error: 'Could not scrape listing from this page' }),
        });
      }

      if (message.action === 'CAPTURE_SESSION') {
        handler.captureSession().then((sessionData) => {
          if (sessionData) {
            chrome.runtime.sendMessage({ type: 'SESSION_CAPTURED', channel: channel!, sessionData });
          } else {
            chrome.runtime.sendMessage({
              type: 'ACTION_RESULT', channel: channel!, action: 'CAPTURE_SESSION',
              success: false, error: 'User not logged in or session not found',
            });
          }
        }).catch(() => {
          chrome.runtime.sendMessage({
            type: 'ACTION_RESULT', channel: channel!, action: 'CAPTURE_SESSION',
            success: false, error: 'Session capture failed',
          });
        });
      }

      if (message.action === 'AUTOFILL_LISTING' && message.payload) {
        const success = handler.autofillListing(message.payload as AutofillPayload);
        chrome.runtime.sendMessage({
          type: 'ACTION_RESULT', channel, action: 'AUTOFILL_LISTING',
          success, error: success ? undefined : 'Could not fill all required fields',
        });
      }
    },
  );
})();
```

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Patterns
- Do NOT make direct API calls from content scripts to the Twicely backend. ALL communication goes through `chrome.runtime.sendMessage` to the service worker, which then uses `apiClient`
- Do NOT use `eval()`, `new Function()`, or `innerHTML` with untrusted content in the extension (MV3 CSP violation)
- Do NOT hardcode any Poshmark or Facebook API URLs in the content scripts. The content scripts only interact with the DOM; API calls are the connector's job (server-side)
- Do NOT store passwords, API keys, or OAuth tokens in the content script or `window` scope. Session data flows: DOM read -> `chrome.runtime.sendMessage` -> service worker -> Twicely API
- Do NOT use `document.write` or synchronous XHR
- Do NOT import any Node.js modules or modules that transitively depend on Node.js (postgres, fs, etc.)
- Do NOT use `as any`, `@ts-ignore`, or `@ts-expect-error`

### Platform-Specific Gotchas
- **Poshmark class names:** Poshmark uses styled-components with stable class prefixes (e.g., `listing__title`). But they CAN change between deployments. Always try `__NEXT_DATA__` first, DOM selectors as fallback
- **Facebook class names:** Facebook uses randomized hash-based class names (e.g., `x1i10hfl`). NEVER rely on class names. Use `[role]`, `[aria-label]`, `[data-testid]`, `h1`, `img`, and structural queries
- **React state vs DOM:** Both platforms are React SPAs. Setting `input.value` does NOT trigger React state updates. You MUST use the native value setter trick and dispatch native events (see Section 2.2)
- **Content Security Policy:** MV3 content scripts run in the page's JS context but with the extension's CSP. They CAN read the DOM but CANNOT make cross-origin fetches (that is the service worker's job)
- **SPA navigation:** Both platforms are SPAs. The content script loads once at `document_idle`. If the user navigates within the SPA (e.g., from feed to listing), the content script does NOT re-run. Use `MutationObserver` on the URL or body to detect SPA navigation and re-trigger scrape logic

### Vocabulary Enforcement
- `ScrapedListing.priceCents` -- integer cents, never dollars as float
- No banned terms from CLAUDE.md (SellerTier, FVF, wallet, etc.) -- though these are unlikely in extension code

## 4. BUSINESS LOGIC

### Session Capture Priority
From Lister Canonical Section 9.4: "Sessions stored encrypted in database." The extension captures the raw session tokens from the browser. The server-side `/api/extension/session` route stores them in `crosslisterAccount.sessionData` (JSONB column). Encryption at rest is a TODO noted in the existing Poshmark connector.

### Poshmark Session Data Shape
From `src/lib/crosslister/connectors/poshmark-types.ts`:
```typescript
interface PoshmarkSessionData {
  jwt: string;
  username: string;
  [key: string]: unknown;
}
```
The content script MUST capture at minimum `jwt` and `username` to match the server-side expectation in `extractSessionData()` within poshmark-connector.ts.

### FB Marketplace Session Data Shape
From `src/lib/crosslister/connectors/fb-marketplace-connector.ts`: FB Marketplace uses OAuth (Tier B). The connector uses `account.accessToken` (from OAuth flow), NOT `sessionData`. The extension session capture for FB is a secondary data channel -- it confirms the user is logged into Facebook and captures their user ID. The primary auth flow remains OAuth via `/api/crosslister/fb-marketplace/callback`.

The session data sent by the extension for FB should be:
```typescript
{
  fbUserId: string;       // From c_user cookie
  isLoggedIn: boolean;    // Cookie presence check
  detectedAt: number;     // Date.now()
}
```
This is informational, not auth-critical.

### Scrape-to-Import Flow (Architecture)
1. Content script scrapes listing data from platform page
2. Sends `LISTING_SCRAPED` to service worker
3. Service worker stores locally + sends to `/api/extension/scrape`
4. Server caches scraped data with 1-hour TTL
5. Seller opens Twicely UI, sees "Import from extension" option (H1.4 scope)
6. Seller confirms -> scraped data is fed into the existing import pipeline

This step (H1.2) builds steps 1-4. Steps 5-6 are H1.4 scope.

### Human-Like Timing
From Lister Canonical Section 9.4 and Section 16.3: "Randomized timing (2-8 second delays between actions)." The auto-fill functions should add small random delays between filling each form field to avoid detection:
```typescript
function randomDelay(minMs: number, maxMs: number): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, minMs + Math.random() * (maxMs - minMs)));
}
// Use between field fills: await randomDelay(300, 800);
```

## 5. ACCEPTANCE CRITERIA

### Functional
1. When a user browses to `poshmark.com` while logged in, the extension captures the Poshmark JWT and username from the page and sends it via `SESSION_CAPTURED` to the service worker
2. When a user browses to a Poshmark listing page (`poshmark.com/listing/<id>`), the extension scrapes title, price (as integer cents), description, condition, brand, category, size, images, and external ID
3. When a user browses to `www.facebook.com/marketplace` while logged in, the extension detects the Facebook user ID from cookies and sends session status
4. When a user browses to a Facebook Marketplace listing page (`/marketplace/item/<id>`), the extension scrapes title, price (as integer cents), description, condition, images, and external ID
5. When the service worker sends `EXECUTE_ACTION` with `action: 'AUTOFILL_LISTING'` and the user is on Poshmark's listing creation page, the content script fills title, price, description, brand, condition, and size fields
6. When the service worker sends `EXECUTE_ACTION` with `action: 'AUTOFILL_LISTING'` and the user is on Facebook Marketplace's creation page, the content script fills title, price, description, and condition
7. The `POST /api/extension/scrape` endpoint accepts valid scrape data, validates with Zod `.strict()`, and returns 200
8. The scrape endpoint returns 400 for invalid/missing fields
9. The scrape endpoint returns 401 for missing/expired tokens
10. All scraped prices are in integer cents (never floats, never dollar strings)

### Negative Cases
11. The extension does NOT make any direct HTTP calls to Twicely or external APIs from content scripts
12. Session capture does NOT fire if the user is not logged in to the platform
13. Listing scrape does NOT fire on non-listing pages (e.g., Poshmark feed, FB Marketplace search results)
14. Auto-fill does NOT attempt to upload photos programmatically
15. The scrape endpoint rejects unknown fields (Zod `.strict()`)
16. The scrape endpoint rejects channels not in the allowed enum

### Data Integrity
17. All monetary values in `ScrapedListing.priceCents` are integer cents
18. All image URLs in `ScrapedListing.imageUrls` are valid URL strings
19. `ScrapedListing.externalId` is never empty string
20. Poshmark session data includes both `jwt` (string) and `username` (string) -- matches server expectation

### Security
21. Content scripts never access `chrome.storage` directly -- all storage goes through service worker messages
22. No raw tokens or passwords are logged to console
23. Extension JWT validation uses `jose` library (per Decision #138)
24. Scrape API route validates JWT with `purpose: 'extension-session'` check (same as other extension routes)

## 10. SPEC GAPS AND OWNER DECISIONS NEEDED

The following items are NOT specified in canonical documents. The prompt provides recommendations but the installer should flag these for the owner:

1. **Poshmark `__NEXT_DATA__` JSON path:** The exact path to listing data within `__NEXT_DATA__` (e.g., `props.pageProps.listing`) is not in specs because it depends on Poshmark's internal implementation. The content script should try multiple known paths and fall back to DOM scraping. This is inherently fragile and expected to break periodically.

2. **Facebook DOM selectors:** Facebook's randomized class names mean the content script may need updating when Facebook deploys changes. The recommendation is to use ARIA/semantic selectors and accept that scraping accuracy may degrade. A health-check mechanism (H1.4 scope) should detect degradation.

3. **Scrape data persistence:** Should scraped listings be stored in Valkey (1-hour TTL) or a DB table? Recommendation: Valkey cache for now, DB table when H1.4 import-from-extension UI is built.

4. **Auto-fill photo upload:** Browser security prevents programmatic file input. Should the auto-fill response indicate which photos the seller needs to manually upload? Recommendation: Return `{ success: true, photoUploadRequired: true, photoUrls: [...] }` so the UI can show a checklist.

5. **SPA navigation detection:** Should the content script use `MutationObserver` to detect SPA navigation within Poshmark/Facebook, or rely on the one-time `document_idle` run? Recommendation: Add a `MutationObserver` on the `<title>` element or URL changes to re-trigger detection. This prevents missing scrape opportunities when users navigate within the SPA.
