### 2.1 Keyboard Navigation ([ and ] for case queue traversal)

**Canonical source:** Helpdesk Canonical §6.2 — `←` / `→` = Previous/next case in queue.

The build tracker says `[/]` but the canonical says `←` / `→`. **Use both**: arrow keys AND bracket keys for case navigation. This matches V2 behavior where both worked.

**Current state:** The case list page (`/hd/cases/page.tsx`) has J/K/Enter keyboard nav for the full case list. The case DETAIL page (`/hd/cases/[id]/case-workspace.tsx`) has NO keyboard navigation between cases.

**What to build:**
- In `CaseWorkspace`, add a `useHotkeys` hook (new file) that listens for `[`, `]`, `ArrowLeft`, `ArrowRight` when NOT focused on a text input/textarea.
- `[` or `ArrowLeft` = navigate to previous case in the queue (by index in `caseQueue` array).
- `]` or `ArrowRight` = navigate to next case in the queue.
- Navigation uses `router.push(`/hd/cases/${nextCaseId}`)`.
- When at first/last case, wrap around (or do nothing — canonical does not specify; do nothing).
- Add additional shortcuts per canonical §6.2:
  - `R` = focus reply textarea (set mode to "reply")
  - `N` = focus reply textarea (set mode to "internal")
  - `E` = set status to ESCALATED (call updateCaseStatus)
  - `M` = toggle macro picker
  - `Cmd+Shift+R` = resolve case (call updateCaseStatus with RESOLVED)
  - `1`–`5` = set priority (1=CRITICAL, 2=URGENT, 3=HIGH, 4=NORMAL, 5=LOW)
  - `?` = toggle keyboard shortcut help panel
- All shortcuts MUST check: `if (e.target instanceof HTMLInputElement || e.target instanceof HTMLTextAreaElement || e.target instanceof HTMLSelectElement) return;`

**Implementation pattern:** Create a reusable `useHelpdeskHotkeys` hook that accepts a config object mapping keys to callbacks. This avoids duplicating event listener logic.

**2.3a Merge Dialog (Canonical §28):**

**Schema already supports merge:** `helpdeskCase.mergedIntoCaseId` column EXISTS. `caseMessage.fromMergedCaseId` and `caseEvent.fromMergedCaseId` columns EXIST.

**New server action needed:** `mergeCases({ sourceCaseId, targetCaseId })` in `src/lib/actions/helpdesk-merge.ts`.

Merge action logic (per §28.2–28.3):
1. Validate: source !== target.
2. Validate: target is not CLOSED.
3. Validate: source is not already merged (mergedIntoCaseId is null).
4. Validate: target has fewer than 5 merged sources.
5. Copy all messages from source to target (set `fromMergedCaseId` = source case ID on copies).
6. Copy all events from source to target (set `fromMergedCaseId` = source case ID on copies).
7. Copy all watchers from source to target (skip duplicates).
8. Union all tags from source into target.
9. Union all commerce links from source into target (orderId, listingId, etc.).
10. Set source case status = CLOSED, mergedIntoCaseId = target.id.
11. Create caseEvent on source: eventType = 'merged_into', data = { targetCaseId }.
12. Create caseEvent on target: eventType = 'merged_from', data = { sourceCaseId, sourceCaseNumber }.
13. Revalidate paths for both cases.

**Merge dialog UI:**
- Modal triggered by "Merge" toolbar button.
- Search input: searches cases by case number or subject (client-side filter of a query result, or debounced server fetch).
- For the search, add a `searchCasesForMerge(query: string)` query that returns top 10 matching cases (exclude current case, exclude CLOSED cases).
- Results list shows case number, subject, requester email, status.
- On select, show confirmation: "Merge {source case number} into {target case number}? This cannot be undone."
- On confirm, execute merge action.

**New Zod schema:** `mergeCasesSchema` in `src/lib/validations/helpdesk.ts`:
```typescript
export const mergeCasesSchema = z.object({
  sourceCaseId: z.string().cuid2(),
  targetCaseId: z.string().cuid2(),
}).strict();
```

**What to build:**
- Create a `substituteMacroVariables` utility function (pure, synchronous, client-side).
- It takes the template string and a context object, replaces all `{{variable_name}}` placeholders.
- Supported variables (per canonical §9.3):

| Variable | Source |

- Show a **preview** below the macro picker item: the body with variables substituted, so agent sees the resolved text before selecting.

### 2.5 Optimistic Replies

**Canonical source:** Build tracker description: "optimistic replies — show reply immediately, confirm async."

**Current state:** `handleSendReply` in `CaseWorkspace` calls `addAgentReply`, waits for success, then calls `router.refresh()`. During the wait, the compose button shows "Sending..." but the message does NOT appear in the timeline until the full page refresh completes. This creates a ~500ms-2s gap where the agent sees nothing.

**What to build:**
- Add an `optimisticMessages` state array to `CaseWorkspace`.
- When agent clicks Send:
  1. Immediately push a new optimistic message to the array: `{ kind: 'message', id: 'optimistic-' + Date.now(), createdAt: new Date(), direction: mode === 'reply' ? 'OUTBOUND' : 'INTERNAL', senderName: agentName, body: body.trim() }`.
  2. Clear the textarea immediately.
  3. Fire the `addAgentReply` action in background.
  4. On success: call `router.refresh()` — server data replaces optimistic.
  5. On failure: remove the optimistic message from the array, restore the body text, show error.
- The timeline rendering merges `timeline` (from server) with `optimisticMessages` (from state).
- Optimistic messages render with a subtle "Sending..." indicator (e.g., a pulsing opacity or a small spinner next to the timestamp).
- Pass `agentName` as a prop to `CaseWorkspace` (sourced from `session.displayName` in the page server component).

## 4. ACCEPTANCE CRITERIA

### Positive Cases

1. **Bracket/arrow navigation**: On `/hd/cases/[id]`, pressing `]` or `ArrowRight` navigates to the next case in the queue panel. Pressing `[` or `ArrowLeft` navigates to the previous case.
2. **Keyboard shortcuts**: `R` focuses reply textarea in reply mode. `N` focuses in internal note mode. `E` sets status to ESCALATED. `M` toggles macro picker. `Cmd+Shift+R` resolves the case. `1`–`5` set priority. `?` toggles help overlay.
3. **Shortcuts respect focus**: None of the single-key shortcuts (R, N, E, M, 1-5, ?, [, ]) fire when an input, textarea, or select element is focused.
4. **Status dropdown**: In the workspace header, clicking the status badge opens a dropdown. Selecting a new status calls `updateCaseStatus` and the badge updates optimistically.
5. **Priority dropdown**: Same as status — inline dropdown, optimistic update, calls `updateCasePriority`.
6. **Quick action buttons**: Resolve, Escalate, Merge, and "Assign to me" buttons appear in the toolbar. Each fires the correct server action.
7. **Merge flow**: Clicking Merge opens a search dialog. Searching returns matching cases. Selecting a target and confirming executes the merge. Source case is CLOSED with `mergedIntoCaseId` set. Target case gains all messages, events, watchers, tags, and commerce links from source.
8. **Merge validation**: Cannot merge a case into itself. Cannot merge into a CLOSED case. Cannot merge a case that is already merged.
9. **Macro variable substitution**: When applying a macro containing `{{buyer_name}}` and `{{case_number}}`, the textarea receives text with actual values substituted.
10. **Unresolvable variables**: `{{listing_title}}` resolves to empty string when no listing is linked.
11. **Optimistic replies**: After clicking Send, the message immediately appears in the timeline with a "Sending..." indicator. After server confirmation, the indicator disappears and the message is confirmed.
12. **Optimistic failure**: If the server action fails, the optimistic message is removed from the timeline, the textarea is restored with the original text, and an error message appears.
13. **Shortcut help overlay**: Pressing `?` shows a floating panel with all shortcuts. Pressing `?` again or Escape dismisses it.
14. **Cmd+Enter**: Pressing Cmd+Enter (or Ctrl+Enter on Windows/Linux) sends the current reply/note regardless of focus location.

### Negative Cases

15. **No shortcut conflicts**: Cmd+K does NOT trigger any helpdesk shortcut (it's reserved for global search per Feature Lock-in §23).
16. **No shortcuts on closed cases**: R, N, E, Cmd+Shift+R do NOT work when case status is CLOSED.
17. **No merge on closed target**: Attempting to merge into a CLOSED case shows an error.
18. **No self-merge**: Attempting to merge a case into itself shows an error.
19. **Unauthenticated access**: All server actions require `staffAuthorize()` — no bypass.
20. **No snooze UI**: No snooze button, dialog, or functionality anywhere.

### Data Integrity

21. **Merge preserves timestamps**: Copied messages retain their original `createdAt` from the source case.
22. **Merge creates events on both cases**: Source gets `merged_into` event, target gets `merged_from` event.
23. **Merge sets mergedIntoCaseId**: Source case's `mergedIntoCaseId` column is set to the target case ID.
24. **Revalidation**: After any server action (status change, priority change, merge), affected paths are revalidated.

### Vocabulary

25. **No banned terms** in any UI text or code comments.
26. **No banned tech** (no Zustand, no tRPC, etc.).

## 7. IMPLEMENTATION NOTES

### Keyboard Shortcut Pattern

Use a single `useEffect` in `CaseWorkspace` that registers a `keydown` handler on `window`. Pattern:

```typescript
useEffect(() => {
  const handler = (e: KeyboardEvent) => {
    // Skip if focused on form elements
    const tag = (e.target as HTMLElement).tagName;
    if (tag === 'INPUT' || tag === 'TEXTAREA' || tag === 'SELECT') return;
    if ((e.target as HTMLElement).isContentEditable) return;

    // Cmd/Ctrl combos (fire even in inputs)
    if ((e.metaKey || e.ctrlKey) && e.shiftKey && e.key === 'r') {
      e.preventDefault();
      handleResolve();
      return;
    }

    // Single-key shortcuts (only outside inputs)
    switch (e.key) {
      case '[': case 'ArrowLeft': navigatePrev(); break;
      case ']': case 'ArrowRight': navigateNext(); break;
      case 'r': focusReply('reply'); break;
      case 'n': focusReply('internal'); break;
      case 'e': handleEscalate(); break;
      case 'm': toggleMacros(); break;
      case '?': toggleShortcutHelp(); break;
      case '1': handleSetPriority('CRITICAL'); break;
      case '2': handleSetPriority('URGENT'); break;
      case '3': handleSetPriority('HIGH'); break;
      case '4': handleSetPriority('NORMAL'); break;
      case '5': handleSetPriority('LOW'); break;
    }
  };
  window.addEventListener('keydown', handler);
  return () => window.removeEventListener('keydown', handler);
}, [/* stable refs */]);
```

### Optimistic Reply Pattern

Use React's `useOptimistic` if available in the project's React version, otherwise use a manual state array:

```typescript
const [optimisticMessages, setOptimisticMessages] = useState<TimelineItem[]>([]);

// Merge into timeline for rendering
const fullTimeline = [...timeline, ...optimisticMessages]
  .sort((a, b) => a.createdAt.getTime() - b.createdAt.getTime());
```

Optimistic messages have `id` prefixed with `optimistic-` so they can be identified and removed on failure or replaced on refresh.

### Merge Action — Server-Side Validation Order

1. Parse with `mergeCasesSchema.safeParse()`
2. `staffAuthorize()` + `ability.can('manage', 'HelpdeskCase')`
3. Fetch both cases in parallel
4. Validate: source exists, target exists
5. Validate: source !== target (compare IDs)
6. Validate: target status !== 'CLOSED'
7. Validate: source.mergedIntoCaseId is null
8. Count existing merges into target (SELECT COUNT WHERE mergedIntoCaseId = targetId); must be < 5
9. Execute merge in a single transaction if possible (or sequential writes)
10. Return `{ success: true }`

### What NOT to Implement

- **Collision detection** ("Sarah is also viewing" / "Sarah is typing a reply") — mentioned in Page Registry but requires Centrifugo real-time, which is not wired for helpdesk. Defer to G10.
- **Agent assignment dropdown population** — The `getHelpdeskAgentsAndTeams` query returns all staff with helpdesk role. Do NOT attempt to filter by team membership or availability for the dropdown — that level of filtering is for G9.2 (routing/teams management).
- **Snooze** — Not in any canonical document. Do not build.
- **Bulk actions from workspace** — Bulk actions are on the case LIST page, not the case DETAIL workspace.
