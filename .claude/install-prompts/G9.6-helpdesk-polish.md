## 2. SCOPE -- EXACTLY WHAT TO BUILD

### 2A. Agent Online/Away Persistence

**Problem:** The sidebar shows Online/Offline status but it is `useState(true)` in `helpdesk-layout-client.tsx`. It resets to "Online" on every page load. There is no server persistence.

**Solution:** Use `helpdeskTeamMember.isAvailable` as the persistence layer. When agent toggles status, update ALL their team memberships simultaneously. On layout load, read their current status.

**Server Action — `toggleAgentOnlineStatus`** (add to `src/lib/actions/helpdesk-agent.ts`):
```
Input: { isOnline: boolean }
Auth: staffAuthorize() — any HELPDESK_* role
Logic:
  1. Get session.staffUserId
  2. Update ALL helpdeskTeamMember rows where staffUserId = session.staffUserId, set isAvailable = isOnline
  3. revalidatePath('/hd')
Output: { success: boolean }
```

Zod schema: `z.object({ isOnline: z.boolean() }).strict()`

**Query — `getAgentOnlineStatus`** (add to `src/lib/queries/helpdesk-agents.ts`):
```
Input: staffUserId: string
Logic:
  1. Select isAvailable from helpdeskTeamMember where staffUserId = staffUserId, limit 1
  2. If no membership rows, return true (default online)
  3. Return the isAvailable value
Output: boolean
```

**Wiring Changes:**
- `src/app/(helpdesk)/hd/layout.tsx`: Call `getAgentOnlineStatus(session.staffUserId)` and pass `initialIsOnline` to `HelpdeskLayoutClient`.
- `src/app/(helpdesk)/hd/helpdesk-layout-client.tsx`: Change `useState(true)` to `useState(initialIsOnline)`. Wire `onToggleStatus` to call `toggleAgentOnlineStatus` via server action import.
- `src/app/(helpdesk)/hd/helpdesk-sidebar.tsx`: No changes needed (already accepts `agent.isOnline` and `onToggleStatus` props).

### 2B. Resolved Cases Archive Page

**Problem:** The sidebar "Archive" section links "Resolved Cases" to `/hd/views` (the saved views management page). Per Helpdesk Canonical ss8.1, "Resolved" is a default queue view showing `status = RESOLVED`, sorted by resolved newest first. There is no dedicated archive page with retention countdown badges.

**Solution:** Create `/hd/resolved` page showing RESOLVED + CLOSED cases with deletion countdown badges. Fix sidebar link.

**Page: `src/app/(helpdesk)/hd/resolved/page.tsx`**
- SSR page, `staffAuthorize()`, CASL check `ability.can('read', 'HelpdeskCase')`
- Title: "Resolved Cases | Twicely Hub"
- Two tabs: "Resolved" (RESOLVED status) and "Closed" (CLOSED status), default "Resolved"
- Table columns: Case #, Subject, Requester, Type (badge), Priority (badge), Resolved/Closed Date, Retention Countdown (badge), Agent
- Each row links to `/hd/cases/{id}`

**Query: `getResolvedCases`** (add to `src/lib/queries/helpdesk-cases.ts`):
```
Input: { tab: 'resolved' | 'closed', limit?: number }
Logic:
  1. Select from helpdeskCase where status = tab value (RESOLVED or CLOSED)
  2. Join or batch-load requester name (from user table) and agent name (from staffUser table)
  3. Order by resolvedAt DESC (for RESOLVED) or closedAt DESC (for CLOSED)
  4. Limit 100
Output: Array of { id, caseNumber, subject, requesterEmail, type, priority, resolvedAt, closedAt, assignedAgentId, createdAt }
```

**Retention Countdown Badge Component: `src/components/helpdesk/retention-badge.tsx`**
- Input: `closedAt: Date | null`, `resolvedAt: Date | null`, `retentionDays: number`
- Logic:
  - If case is CLOSED: countdown = closedAt + retentionDays - now (in days)
  - If case is RESOLVED: countdown = "Closing in X days" (resolvedAt + autoCloseDays - now)
  - Color coding: green (>90 days), amber (30-90 days), red (<30 days), flashing red (<7 days)
  - Text: "Deletes in X days" (CLOSED), "Auto-closes in X days" (RESOLVED)
- No clock icon emoji — use text only per project convention.

**Sidebar Fix:**
- In `src/app/(helpdesk)/hd/helpdesk-sidebar.tsx`: Change the Archive section "Resolved Cases" href from `/hd/views` to `/hd/resolved`.

### 2E. Retention Purge Job

**Problem:** Cases are auto-closed after 7 days in RESOLVED (by helpdesk-auto-close.ts). But there is no job to actually delete/anonymize case data after the full retention period (helpdesk.retentionDays = 365 days from closure).

**New BullMQ Job: `src/lib/jobs/helpdesk-retention-purge.ts`**

Per Helpdesk Canonical ss26 and ss24.1 (`helpdesk.retentionDays`):
```
Queue: helpdesk-retention-purge
Schedule: Daily (once per day)
Logic:
  1. Read helpdesk.retentionDays from platform settings (default 365)
  2. Find all CLOSED cases where closedAt < (now - retentionDays)
  3. For each case:
     a. Delete caseMessage rows (CASCADE will handle via FK, but explicit for clarity)
     b. Delete caseEvent rows (CASCADE)
     c. Delete caseWatcher rows (CASCADE)
     d. Delete caseCsat rows (FK, not cascade — delete explicitly)
     e. Delete the helpdeskCase row itself
  4. Log count of purged cases
  5. Limit to 200 per run (avoid long-running queries)
```

**IMPORTANT:** This is a data DELETION job, not anonymization. Per GDPR spec (G8), helpdesk cases are covered under the general data retention policy. The canonical says "resolved_retention_days" in email config and "helpdesk.retentionDays" in platform settings. Use the platform setting as authoritative.

**Registration:** Add the cron schedule to the worker init file (wherever BullMQ cron jobs are registered for helpdesk).

### 2F. Wire Watcher Notifications in Agent Actions

**Utility: `notifyCaseWatchers`** (add as unexported helper in a new file `src/lib/helpdesk/notify-watchers.ts`):
```typescript
// NOT 'use server' — this is a plain utility, not a server action
Input: caseId: string, excludeStaffUserId: string, eventDescription: string
Logic:
  1. Query caseWatcher where caseId, filter out excludeStaffUserId
  2. Query helpdeskCase for caseNumber and subject
  3. For each watcher: call notify() with template 'helpdesk.case.watcher_update'
     Variables: { caseNumber, subject, eventDescription, caseUrl: `/hd/cases/${caseId}` }
```

**Wire into actions:**
- `addAgentReply` (helpdesk-agent-cases.ts): After successful reply, call `notifyCaseWatchers(caseId, session.staffUserId, isInternal ? 'New internal note added' : 'New reply sent')`
- `updateCaseStatus` (helpdesk-agent-cases.ts): After successful status change, call `notifyCaseWatchers(caseId, '', `Status changed to ${status}`)`

**CRITICAL:** `notifyCaseWatchers` must be a plain function, NOT exported from a `'use server'` file (that would make it an unintended server action). Put it in `src/lib/helpdesk/notify-watchers.ts` without `'use server'` directive.

## 8. SPEC INCONSISTENCIES (Flag for Owner)

1. **isAvailable is per-team, online status is global.** The Helpdesk Canonical ss10.2 defines `isAvailable` on `helpdeskTeamMember` (per-team). The sidebar shows a single global toggle. Solution in this prompt: toggle ALL team memberships simultaneously. This means an agent cannot be "online for General Support but offline for Trust & Safety." **OWNER DECISION NEEDED if per-team granularity is desired** (would require a different UI).

2. **resolvedRetentionDays on helpdeskEmailConfig vs helpdesk.retentionDays platform setting.** Both exist and store retention days. This prompt uses the platform setting as authoritative. The email config column is likely vestigial. **NOT SPECIFIED whether to remove the email config column** -- leaving it as-is.

3. **Retention purge deletes vs anonymizes.** The canonical ss24.1 says "Days to keep resolved case data" which implies deletion. G8 (GDPR) established anonymization patterns. This prompt implements hard deletion since the canonical does not mention anonymization for helpdesk cases specifically, and the cases table has no PII beyond requesterId/requesterEmail which are references, not embedded PII. **OWNER DECISION NEEDED if anonymization is preferred over deletion.**
