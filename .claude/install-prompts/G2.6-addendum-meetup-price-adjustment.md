**File:** `src/lib/commerce/local-state-machine.ts`

```typescript
const VALID_TRANSITIONS: Record<string, string[]> = {
  SCHEDULED: ['SELLER_CHECKED_IN', 'BUYER_CHECKED_IN', 'BOTH_CHECKED_IN', 'CANCELED'],
  SELLER_CHECKED_IN: ['BOTH_CHECKED_IN', 'RECEIPT_CONFIRMED', 'NO_SHOW', 'CANCELED'],
  BUYER_CHECKED_IN: ['BOTH_CHECKED_IN', 'RECEIPT_CONFIRMED', 'NO_SHOW', 'CANCELED'],
  BOTH_CHECKED_IN: ['ADJUSTMENT_PENDING', 'RECEIPT_CONFIRMED', 'NO_SHOW', 'CANCELED'],
  ADJUSTMENT_PENDING: ['BOTH_CHECKED_IN', 'RECEIPT_CONFIRMED'],  // decline -> BOTH_CHECKED_IN, accept -> proceed to RECEIPT_CONFIRMED
  RECEIPT_CONFIRMED: ['COMPLETED', 'DISPUTED'],
  COMPLETED: [],
  CANCELED: [],
  NO_SHOW: [],
  DISPUTED: ['COMPLETED', 'CANCELED'],
};
```

Key transitions:
- `BOTH_CHECKED_IN -> ADJUSTMENT_PENDING`: Seller initiates adjustment
- `ADJUSTMENT_PENDING -> BOTH_CHECKED_IN`: Buyer declines adjustment
- `ADJUSTMENT_PENDING -> RECEIPT_CONFIRMED`: Buyer accepts, then normal confirmation flow continues (buyer scans QR with new tokens)

Note: `ADJUSTMENT_PENDING` does NOT allow `CANCELED` or `NO_SHOW` transitions. Both parties are physically present (status was `BOTH_CHECKED_IN` before adjustment). If they want to cancel, the buyer must first decline the adjustment (returning to `BOTH_CHECKED_IN`).

### 2.4 Price Adjustment Service

**File:** `src/lib/commerce/local-price-adjustment.ts` (NEW -- ~120 lines)

This is a plain TypeScript module (NOT a `'use server'` file).

```typescript
/**
 * Local Meetup Price Adjustment Service (G2.6)
 *
 * Per Local Canonical Addendum v1.2, Section A3:
 * - Seller can reduce price at meetup (max 33% of original)
 * - Max 1 adjustment per transaction
 * - Twicely keeps TF on ORIGINAL amount (no fee reversal)
 * - Old tokens invalidated, new tokens generated on buyer acceptance
 * - Buyer decline returns status to BOTH_CHECKED_IN
 */
```

Functions to implement:

**`validateAdjustment(orderId: string, localTransactionId: string, adjustedPriceCents: number): Promise<{ valid: boolean; error?: string; originalPriceCents?: number }>`**
1. Fetch `order.itemSubtotalCents` for the given `orderId` -- this is the original price.
2. If `adjustedPriceCents >= originalPriceCents`, return `{ valid: false, error: 'Adjusted price must be less than original price' }`.
3. If `adjustedPriceCents <= 0`, return `{ valid: false, error: 'Adjusted price must be greater than zero' }`.
4. Read `commerce.local.maxAdjustmentPercent` from platform settings (default: 33).
5. Calculate minimum allowed price: `Math.ceil(originalPriceCents * (1 - maxPercent / 100))`.
6. If `adjustedPriceCents < minimumAllowed`, return `{ valid: false, error: 'Exceeds maximum allowed discount' }`.
7. Return `{ valid: true, originalPriceCents }`.

**`initiatePriceAdjustment(localTransactionId: string, adjustedPriceCents: number, reason: string): Promise<{ success: boolean; error?: string }>`**
1. Update `localTransaction` set: `adjustedPriceCents`, `adjustmentReason`, `adjustmentInitiatedAt: now`, `status: 'ADJUSTMENT_PENDING'`, `updatedAt: now`.
2. Use `canTransition()` to validate the state transition before updating.
3. Return success/failure.

**`acceptPriceAdjustment(localTransactionId: string): Promise<{ success: boolean; confirmationCode?: string; offlineCode?: string; error?: string }>`**
1. Update `localTransaction` set: `adjustmentAcceptedAt: now`, `status: 'BOTH_CHECKED_IN'`, `updatedAt: now`.
   - Status goes back to `BOTH_CHECKED_IN` because the buyer still needs to scan QR to confirm receipt.
2. Generate new confirmation code via `generateConfirmationCode()` and new offline code via `generateOfflineCode()`.
3. Update `localTransaction` set: `confirmationCode: newCode`, `offlineCode: newOfflineCode`.
4. Return the new codes so the UI can display the new QR.

**`declinePriceAdjustment(localTransactionId: string): Promise<{ success: boolean; error?: string }>`**
1. Update `localTransaction` set: `adjustmentDeclinedAt: now`, `status: 'BOTH_CHECKED_IN'`, `adjustedPriceCents: null`, `updatedAt: now`.
   - Clear `adjustedPriceCents` on decline so the original tokens remain valid.
2. Use `canTransition()` to validate the transition.
3. Return success/failure.

IMPORTANT: On `acceptPriceAdjustment`, the old tokens are invalidated by overwriting them with new codes. The old QR code string no longer exists in the DB, so scanning it will fail validation. This is the "invalidation" mechanism.

### 2.5 Ledger Entry for Price Adjustment

**File:** `src/lib/commerce/local-ledger.ts`

Add a new function:

```typescript
/**
 * Create a LOCAL_PRICE_ADJUSTMENT ledger entry.
 *
 * This records the Stripe partial refund for the delta between original and adjusted price.
 * TF is NOT recalculated -- Twicely keeps TF on the original amount per Addendum A3.
 *
 * The amountCents is POSITIVE (credit to buyer / refund delta).
 */
export async function createPriceAdjustmentLedgerEntry(input: {
  orderId: string;
  sellerId: string;
  originalPriceCents: number;
  adjustedPriceCents: number;
}): Promise<void>
```

Implementation:
1. Calculate `deltaCents = input.originalPriceCents - input.adjustedPriceCents`.
2. Insert one ledger entry:
   - `type: 'LOCAL_PRICE_ADJUSTMENT'`
   - `status: 'POSTED'`
   - `amountCents: -deltaCents` (negative = debit from seller's balance, reflecting the partial refund)
   - `userId: sellerId`
   - `orderId`
   - `reasonCode: 'local:{orderId}:price-adjustment'`
   - `memo: 'Meetup price adjustment — partial refund of ${formatCentsToDollars(deltaCents)}'`
   - `postedAt: now`
   - `createdAt: now`

Use `formatCentsToDollars` from `src/lib/finance/format.ts` for the memo.

Note: TF is NOT recalculated. The `postConfirmationEffects()` in the actions file will still read `order.itemSubtotalCents` and calculate TF on the original amount. This is correct per the addendum: "Twicely keeps TF on original amount. No fee reversal."

**`initiatePriceAdjustmentAction(data: z.infer<typeof initiatePriceAdjustmentSchema>): Promise<ActionResult>`**

Flow:
1. `authorize()` -- get session and ability.
2. Validate input with `initiatePriceAdjustmentSchema.safeParse(data)`.
3. Fetch the localTransaction by `parsed.data.localTransactionId`.
4. Verify `session.userId === transaction.sellerId` (only seller can initiate).

8. Call `validateAdjustment(transaction.orderId, transaction.id, parsed.data.adjustedPriceCents)`.
9. Call `initiatePriceAdjustment(transaction.id, parsed.data.adjustedPriceCents, parsed.data.adjustmentReason)`.
10. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`.
11. Return `{ success: true }`.

interface AdjustmentResponseResult {

  offlineCode?: string;

Flow:
1. `authorize()` -- get session and ability.
2. Validate input with `respondToAdjustmentSchema.safeParse(data)`.
3. Fetch the localTransaction by `parsed.data.localTransactionId`.
4. Verify `session.userId === transaction.buyerId` (only buyer can respond).

6. Verify `transaction.status === 'ADJUSTMENT_PENDING'`.

8. If `parsed.data.accept === false`:

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Patterns
- Do NOT hardcode the 33% discount limit. Read from `commerce.local.maxAdjustmentPercent` platform setting.
- Do NOT recalculate TF on the adjusted price. TF is always calculated on the ORIGINAL `order.itemSubtotalCents`. The spec explicitly says: "Twicely keeps TF on original amount. No fee reversal."
- Do NOT add an `amountCents` column to `localTransaction`. The original price comes from `order.itemSubtotalCents`.
- Do NOT allow more than 1 adjustment per transaction. Check `transaction.adjustmentInitiatedAt === null`.
- Do NOT allow adjustment from any status other than `BOTH_CHECKED_IN`.
- Do NOT show new QR tokens to the buyer before they explicitly accept. The `respondToAdjustmentAction` returns the new codes only on accept.
- Do NOT create a Stripe refund API call. The `LOCAL_PRICE_ADJUSTMENT` ledger entry records the intent. Actual Stripe partial refund execution is handled by the finance reconciliation pipeline (not in scope for G2.6).
- Do NOT allow the buyer to initiate a price adjustment. Seller only.
- Do NOT allow price increases. `adjustedPriceCents` must be strictly less than `order.itemSubtotalCents`.
- Do NOT use `console.log`. Use `logger` from `@/lib/logger`.
- Do NOT export helper functions from `'use server'` files.
- Do NOT spread request body into DB updates. Use explicit field mapping.
- All money values in integer cents only.

### Route Constraints
- This feature does NOT create any new pages or routes. It adds functionality to the existing order detail pages at `/my/buying/orders/[id]` and `/my/selling/orders/[id]`.
- No new API routes needed. Everything uses server actions.

### Vocabulary
- Use "Twicely fees" not "TF" or "commission" in UI copy.
- Use "price adjustment" not "discount" or "price reduction" in code.
- Use `formatCentsToDollars()` from `src/lib/finance/format.ts` for all dollar displays.

## 4. ACCEPTANCE CRITERIA

### Functional
1. A seller can initiate a price adjustment when `status === 'BOTH_CHECKED_IN'` and no prior adjustment exists on the transaction.
2. The adjustment form enforces the max discount limit read from `commerce.local.maxAdjustmentPercent` (default 33%).
3. On initiation, the transaction status transitions to `ADJUSTMENT_PENDING`.
4. The buyer sees the proposed price, original price (struck through), reason, and Accept/Decline buttons when `status === 'ADJUSTMENT_PENDING'`.
5. On buyer accept: new confirmation code and offline code are generated, old codes are overwritten, `adjustmentAcceptedAt` is set, status returns to `BOTH_CHECKED_IN`, a `LOCAL_PRICE_ADJUSTMENT` ledger entry is created.
6. On buyer decline: `adjustmentDeclinedAt` is set, `adjustedPriceCents` is cleared (set to null), status returns to `BOTH_CHECKED_IN`, original codes remain valid.
7. After acceptance, the seller's QR display shows the new confirmation code.
8. After decline, the seller's QR display shows the original (unchanged) confirmation code.
9. Max 1 adjustment per transaction -- the "Adjust Price" button does not appear if `adjustmentInitiatedAt` is not null, even after a decline.

### Authorization
10. Only the seller can call `initiatePriceAdjustmentAction`. Buyer calling it returns `{ success: false, error: 'Not found' }`.
11. Only the buyer can call `respondToAdjustmentAction`. Seller calling it returns `{ success: false, error: 'Not found' }`.
12. Non-participants (neither buyer nor seller) get `{ success: false, error: 'Not found' }` for both actions.
13. Unauthenticated users get `{ success: false, error: 'Unauthorized' }`.
14. CASL checks enforce ownership via `buyerId`/`sellerId` matching `session.userId`.

### State Machine
15. `BOTH_CHECKED_IN -> ADJUSTMENT_PENDING` is a valid transition.
16. `ADJUSTMENT_PENDING -> BOTH_CHECKED_IN` is a valid transition (decline or accept).
17. `ADJUSTMENT_PENDING -> RECEIPT_CONFIRMED` is a valid transition (accept, then buyer scans new QR).
18. `ADJUSTMENT_PENDING` does NOT allow transitions to `CANCELED` or `NO_SHOW`.
19. `SCHEDULED -> ADJUSTMENT_PENDING` is NOT a valid transition.

### Financial
20. `adjustedPriceCents` is stored as integer cents.
21. The `LOCAL_PRICE_ADJUSTMENT` ledger entry has `amountCents` equal to negative delta (original minus adjusted).
22. The existing `postConfirmationEffects()` still calculates TF on `order.itemSubtotalCents` (the original price), NOT on `adjustedPriceCents`. No changes to `postConfirmationEffects()` are needed for this.
23. Fee rates are never hardcoded -- `maxAdjustmentPercent` comes from platform settings.

### Data Integrity
24. All new columns on `localTransaction` are nullable.
25. `adjustmentInitiatedAt` is set exactly once per transaction (never overwritten).
26. After decline, `adjustedPriceCents` is cleared to null so the field accurately reflects current state.

## 8. SPEC NOTES AND OPEN QUESTIONS

### Confirmed Behaviors (from canonical cross-reference)
1. **TF on original amount**: Decision Rationale line ~199 confirms: "On all seller-initiated partial refunds (adjustments), Twicely keeps TF calculated on the FULL original sale price."
2. **SafeTrade-only feature**: Addendum A0 feature matrix confirms price adjustment is SafeTrade-only (not available for cash transactions). Since the G2 base only implements SafeTrade transactions (cash tracking is separate), this is automatically satisfied.
3. **Token invalidation**: The addendum says "Old tokens invalidated immediately." Implementation achieves this by overwriting `confirmationCode` and `offlineCode` in the DB -- the old values no longer exist for validation.
4. **Max 1 adjustment**: The addendum says "Max per transaction: 1 adjustment." Implementation checks `adjustmentInitiatedAt === null`. After a decline, `adjustmentInitiatedAt` remains set (only `adjustedPriceCents` is cleared), so no second attempt is possible.

### Design Decision Made in This Prompt
- **`ADJUSTMENT_PENDING -> RECEIPT_CONFIRMED` transition**: The addendum state machine diagram shows `ADJUSTMENT_PENDING` sitting between `BOTH_CHECKED_IN` and `RECEIPT_CONFIRMED`. However, the flow is: buyer accepts -> status goes to `BOTH_CHECKED_IN` -> buyer scans new QR -> status goes to `RECEIPT_CONFIRMED`. So the direct `ADJUSTMENT_PENDING -> RECEIPT_CONFIRMED` transition may not be needed in practice, but is included for correctness (in case the accept-and-confirm happen atomically in a future optimization).

### NOT SPECIFIED -- Owner Decision Needed
- **What happens to `order.itemSubtotalCents` after adjustment?** The addendum does not say whether the order table's price should also be updated. The current prompt keeps `order.itemSubtotalCents` unchanged (TF is calculated on it). The `adjustedPriceCents` on `localTransaction` tracks the actual amount the buyer will pay. If the finance team needs the order to reflect the adjusted price, this is a separate change. **Recommendation: leave `order.itemSubtotalCents` unchanged for now.** The delta is captured in the `LOCAL_PRICE_ADJUSTMENT` ledger entry.
- **Stripe partial refund timing**: The addendum says "Stripe partial refund for delta only" but does not specify when this happens. The current prompt creates a ledger entry (recording intent) but does NOT call the Stripe API. The actual Stripe partial refund would be executed by the escrow release pipeline or a separate reconciliation job. This is consistent with how other ledger entries work in the codebase. **Recommendation: defer Stripe API call to escrow release pipeline.**
