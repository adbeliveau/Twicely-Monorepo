## 2. SCOPE -- EXACTLY WHAT TO BUILD

### 2.1 Kill Switch System

**Concept:** Kill switches are a curated subset of feature flags with key prefix `kill.*`. They represent critical platform capabilities that must be disableable in an emergency. The existing E4 table already renders `kill.*` flags with a red background (see `feature-flag-table.tsx` line 70: `isKillSwitch = (key) => key.startsWith('kill.')`). G10.4 adds:

1. **Pre-seeded kill switch flags** -- a set of platform-critical kill switches seeded into the `featureFlag` table
2. **Dedicated Kill Switch panel** -- a visually distinct section on the `/flags` page that groups kill switches separately from regular flags
3. **One-click toggle with confirmation** -- kill switches require a confirmation dialog before disabling (destructive action)
4. **Server-side `isKillSwitchActive()` helper** -- convenience wrapper around `isFeatureEnabled()` for kill switch keys
5. **Valkey caching** -- add Valkey cache layer to `isFeatureEnabled()` with configurable TTL from `featureFlags.cacheSeconds`

### 2.2 Launch Gate System

**Concept:** Launch gates are feature flags with key prefix `gate.*` that control whether major platform features are available. Unlike kill switches (which START enabled and get disabled in emergencies), launch gates START disabled and get enabled when the feature is ready for production. They serve as a pre-launch checklist.

1. **Pre-seeded launch gate flags** -- gates for major platform modules
2. **Dedicated Launch Gates panel** -- a separate section on the `/flags` page showing gate status as a checklist
3. **Server-side `isLaunchGateOpen()` helper** -- convenience wrapper for gate keys
4. **Gate status summary** -- visual indicator showing "X of Y gates open"

### 2.3 Database -- NO Schema Changes

The existing `featureFlag` table is sufficient. Kill switches and launch gates are differentiated by key prefix convention:
- `kill.*` -- Kill switches (e.g., `kill.checkout`, `kill.crosslister`, `kill.messaging`)
- `gate.*` -- Launch gates (e.g., `gate.marketplace`, `gate.crosslister`, `gate.local`)
- Everything else -- regular feature flags (existing behavior)

### 2.6 Valkey Cache Layer

Upgrade `isFeatureEnabled()` in `src/lib/services/feature-flags.ts` to use Valkey caching:

1. On flag evaluation, check Valkey first: key = `ff:{flagKey}` (string, JSON-serialized flag row)
2. If cache hit, parse and evaluate without DB query
3. If cache miss, query DB, write to Valkey with TTL from `featureFlags.cacheSeconds` platform setting
4. On flag toggle/update/delete, invalidate the Valkey key `ff:{flagKey}` immediately (cache bust)
5. **Fallback:** If Valkey is unavailable, fall through to direct DB query (never fail open or closed based on cache failure -- always try the DB)

**Import the Valkey client from the existing infrastructure.** Look for the Valkey/ioredis import pattern used in `src/lib/jobs/queue.ts` and `src/lib/crosslister/queue/rate-limiter.ts`.

### 2.7 Server-Side Gate/Kill Helpers

Add to `src/lib/services/feature-flags.ts`:

```typescript
/**
 * Check if a kill switch is active (feature is ENABLED).
 * Returns true if the feature should be available (kill switch is ON).
 * Returns false if the feature has been killed (kill switch is OFF).
 * If the flag doesn't exist, defaults to true (feature available).
 */
export async function isKillSwitchActive(featureKey: string): Promise<boolean>;

/**
 * Check if a launch gate is open (feature is ready for production).
 * Returns true if the gate is open (enabled = true).
 * Returns false if the gate is closed (enabled = false or not found).
 */
export async function isLaunchGateOpen(gateKey: string): Promise<boolean>;
```

`isKillSwitchActive('checkout')` internally checks `kill.checkout`. Prepends the `kill.` prefix.
`isLaunchGateOpen('marketplace')` internally checks `gate.marketplace`. Prepends the `gate.` prefix.

## 3. CONSTRAINTS -- WHAT NOT TO DO

- **DO NOT** create a new database table. Kill switches and launch gates use the existing `featureFlag` table.
- **DO NOT** add new enums. The key prefix convention (`kill.*`, `gate.*`) is sufficient.
- **DO NOT** add a new CASL subject. Kill switches and launch gates are `FeatureFlag` subjects.
- **DO NOT** add a new route/page. Everything goes on the existing `/flags` page.
- **DO NOT** hardcode kill switch or gate names in application code. Use the seeded flags and the `isKillSwitchActive()`/`isLaunchGateOpen()` helpers.
- **DO NOT** make kill switches fail-closed on cache miss. If Valkey AND the DB are both unreachable, log an error but return true (feature available). The alternative (failing closed) would take down the entire platform.
- **DO NOT** use Redis -- use Valkey (CLAUDE.md tech stack rule).
- **DO NOT** introduce any banned terms (see CLAUDE.md vocabulary section).
- **DO NOT** create files over 300 lines. Split if needed.
- **DO NOT** use `as any`, `@ts-ignore`, or `@ts-expect-error`.
- **DO NOT** add `console.log` -- use the existing `logger` from `@/lib/logger`.
