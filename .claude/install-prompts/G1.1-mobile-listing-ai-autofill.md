// ai_autofill_usage — tracks monthly auto-fill usage per user
export const aiAutofillUsage = pgTable('ai_autofill_usage', {
  id:        text('id').primaryKey().$defaultFn(() => createId()),

}));

### 2.3 API Route: `POST /api/listings/ai-autofill`

Create at `src/app/api/listings/ai-autofill/route.ts`.

**Input (Zod schema, `.strict()`):**
```typescript
const aiAutofillRequestSchema = z.object({
  imageUrls: z.array(z.string().url()).min(1).max(12),
}).strict();
```

**Authorization:** Call `authorize()`. Require authenticated user. CASL check: `ability.can('create', 'Listing')`.

**Rate Limiting Flow:**
1. Read `ai.autofill.enabled` from platform settings. If `false`, return 503 `{ success: false, error: 'AI auto-fill is temporarily unavailable' }`.
2. Get the user's `storeTier` from `sellerProfile` table (via `userId`). Default to `'NONE'` if no profile.
3. Read the appropriate `ai.autofill.limit*` setting based on tier.
4. Get or create the `aiAutofillUsage` row for `(userId, currentMonthKey)`.
5. If `count >= limit` and `limit !== -1`, return 429 `{ success: false, error: 'Monthly auto-fill limit reached. Upgrade your Store tier for more.', remainingUses: 0 }`.
6. Increment `count` by 1 (upsert with `ON CONFLICT DO UPDATE SET count = count + 1`).
7. Call Claude Vision API.
8. Return structured result.

**Month key format:** `YYYY-MM` (e.g., `"2026-03"`). Generated as:
```typescript
const now = new Date();
const monthKey = `${now.getUTCFullYear()}-${String(now.getUTCMonth() + 1).padStart(2, '0')}`;
```

**Claude Vision API Call:**
```typescript
import Anthropic from '@anthropic-ai/sdk';

// Create client (reads ANTHROPIC_API_KEY from env automatically)
const anthropic = new Anthropic();
```

The service must:
1. Fetch each image URL and convert to base64 (the images are on our R2 / local upload, not publicly reachable from Anthropic — we must send the image data).
2. Send up to 4 images (first 4 from the array — Claude has context limits). If more than 4 provided, use only the first 4 and note this in the response.
3. Use the system prompt and user prompt specified below.

**IMPORTANT:** The image URLs come from R2 (production) or `/uploads/listings/` (dev). In dev, these are relative URLs. The service must handle both cases:
- If URL starts with `http`, fetch directly.
- If URL starts with `/`, construct full URL from `process.env.NEXT_PUBLIC_APP_URL` or use `http://localhost:3000`.

**System Prompt for Claude Vision:**
```
You are a product listing assistant for Twicely, a secondhand resale marketplace. Analyze the provided product photos and suggest listing details.

Return a JSON object with exactly these fields:
{
  "title": "Concise product title, max 80 characters",
  "description": "Detailed description of the item, mentioning condition, notable features, measurements if visible. Max 500 characters.",
  "category": "Best matching category name (e.g., 'Men's Shoes', 'Women's Dresses', 'Electronics')",
  "brand": "Brand name if identifiable, or empty string",
  "condition": "One of: NEW_WITH_TAGS, NEW_WITHOUT_TAGS, LIKE_NEW, VERY_GOOD, GOOD, ACCEPTABLE",
  "color": "Primary color of the item",
  "tags": ["up to 5 relevant search tags"],
  "suggestedPriceMinCents": 0,
  "suggestedPriceMaxCents": 0,
  "confidence": "HIGH, MEDIUM, or LOW"
}

Rules:
- Title should be descriptive but concise (include brand, key feature, size if visible)
- Description should focus on what a buyer needs to know
- For condition, look for wear signs, tags, packaging
- Price suggestions should be in cents (e.g., 2500 = $25.00) and reflect typical resale value
- If you cannot determine a field, use empty string or null
- Return ONLY the JSON object, no markdown formatting, no explanation
```

**User Prompt:**
```
Analyze these product photos and suggest listing details for a secondhand marketplace listing.
```

**Response Parsing:**
Parse the Claude response as JSON. Validate against a Zod schema:

```typescript
const aiAutofillResponseSchema = z.object({
  title: z.string().max(80).default(''),
  description: z.string().max(5000).default(''),
  category: z.string().default(''),
  brand: z.string().default(''),
  condition: z.enum([
    'NEW_WITH_TAGS', 'NEW_WITHOUT_TAGS', 'NEW_WITH_DEFECTS',
    'LIKE_NEW', 'VERY_GOOD', 'GOOD', 'ACCEPTABLE'
  ]).nullable().default(null),
  color: z.string().default(''),
  tags: z.array(z.string()).max(10).default([]),
  suggestedPriceMinCents: z.number().int().nonnegative().default(0),
  suggestedPriceMaxCents: z.number().int().nonnegative().default(0),
  confidence: z.enum(['HIGH', 'MEDIUM', 'LOW']).default('LOW'),
});
```

**API Response Shape:**
```typescript
// Success (200):
{
  success: true,
  suggestions: {
    title: string;
    description: string;
    category: string;
    brand: string;
    condition: string | null;
    color: string;
    tags: string[];
    suggestedPriceMinCents: number;
    suggestedPriceMaxCents: number;
    confidence: 'HIGH' | 'MEDIUM' | 'LOW';
  },
  remainingUses: number;  // -1 for unlimited
}

// Error (4xx/5xx):
{
  success: false,
  error: string;
  remainingUses?: number;
}
```

**Error Handling:**
- If Claude API call fails (network error, timeout, rate limit), return 502 `{ success: false, error: 'Auto-fill unavailable, please fill in manually' }`.
- If Claude returns unparseable JSON, return 502 with the same message.
- If image fetch fails, skip that image. If ALL images fail to fetch, return 400 `{ success: false, error: 'Could not load images. Please try again.' }`.
- Log all errors with `logger.error()`. Never expose internal error details to the client.
- NEVER block listing creation. This is an enhancement, not a gate.

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Terms (scan output before commit)
- No `SellerTier` — use `StoreTier`
- No `SubscriptionTier` — use `StoreTier`
- No `wallet`, `withdraw`, `Twicely Balance` in UI
- No `FVF` or `Final Value Fee`
- No `BASIC`, `ELITE`, `PLUS`, `MAX`, `PREMIUM` as tier names

### Banned Patterns
- No `as any` or `@ts-expect-error` or `@ts-ignore`
- No `console.log` — use `logger` from `@/lib/logger`
- No hardcoded rate limits — all from `platform_settings`
- No fee calculations in the client
- No files over 300 lines
- No spreading request body into DB updates — explicit field mapping
- No `tRPC`, no `Prisma`, no `NextAuth`, no `Redis` (use Valkey if caching needed)

### Specific Gotchas for This Feature
1. **Do NOT auto-publish.** AI suggestions are a starting point. Every field remains editable. The user must still click "Publish Listing" manually.
2. **Do NOT bypass Zod validation.** AI suggestions go through the same form validation as manual entry.
3. **Do NOT send images directly from client to Anthropic.** Images must go through our API route (server-side only).
4. **Do NOT hardcode the model name.** Read from `ai.autofill.model` platform setting.
5. **Do NOT block listing creation on AI failure.** If Claude is down, the button shows an error toast and the user fills in manually.
6. **Price suggestion is a RANGE (min/max in cents), never auto-set.** The form price field is not populated — only a hint is shown.
7. **Category suggestion is TEXT, not an auto-select.** The category picker has complex state (ID + slug + parentId). AI returns a category name string that the user can use as a search hint.
8. **Image URLs from the uploader may be relative** (in dev: `/uploads/listings/xxx.jpg`). The service must handle constructing absolute URLs.
9. **The Anthropic SDK reads `ANTHROPIC_API_KEY` from environment automatically.** Do NOT pass the key manually. Do NOT store it in platform_settings (it is a secret).
10. **The `aiAutofillUsage` table is NOT in the canonical schema doc.** This is an owner-approved addition for this feature only.
11. **Ownership key is `userId`.** The usage table uses `userId`, not `sellerProfileId`.
12. **`sellerProfile.id` is NOT the same as `userId`.** To get `storeTier`, query `sellerProfile` WHERE `userId = session.userId`. Use `sellerProfile.storeTier`, not a join through `sellerProfile.id`.

## 8. IMPLEMENTATION NOTES

### Image Base64 Encoding for Claude API

The Anthropic SDK expects image content in this format:
```typescript
{
  type: 'image',
  source: {
    type: 'base64',
    media_type: 'image/jpeg',  // or 'image/png', 'image/webp'
    data: '<base64 string>',
  },
}
```

The service must fetch each R2/local URL, read the response buffer, and encode as base64.

### Handling the Anthropic SDK Type

The `@anthropic-ai/sdk` package exports the `Anthropic` class as default. Usage:
```typescript
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();
// Client reads ANTHROPIC_API_KEY from process.env automatically
```

Create the client lazily (not at module scope) to avoid errors when the env var is not set during build/test:
```typescript
function getAnthropicClient(): Anthropic {
  return new Anthropic();
}
```

### Atomicity of Usage Increment

Use Drizzle's upsert with ON CONFLICT:
```typescript
await db
  .insert(aiAutofillUsage)
  .values({
    userId,
    monthKey,
    count: 1,
  })
  .onConflictDoUpdate({
    target: [aiAutofillUsage.userId, aiAutofillUsage.monthKey],
    set: {
      count: sql`${aiAutofillUsage.count} + 1`,
      updatedAt: new Date(),
    },
  });
```

This is atomic at the DB level. No need for a separate SELECT + UPDATE.

### Form Integration Pattern

The AI auto-fill button sits inside the Photos card. When suggestions come back:
1. Fields that are EMPTY get auto-filled silently.
2. Fields that already have user content: do NOT overwrite. Show a small note: "AI also suggested: [value]" that the user can click to accept.
3. This prevents destroying work the user already did.

Example placement in `listing-form.tsx`:
```tsx
{/* Inside the Photos Card, after ImageUploader */}
{formData.images.length > 0 && (
  <AiAutofillButton
    imageUrls={formData.images.filter(img => img.url).map(img => img.url)}
    onSuggestionsReceived={handleAiSuggestions}
    disabled={isSubmitting}
  />
)}
```
