### 2.4 Server Actions

**File: `src/lib/actions/enforcement-appeals.ts`** (new file, separate from enforcement.ts to stay under 300 lines)

**`submitEnforcementAppealAction(input: unknown)`**

Auth: `authorize()` (seller auth, NOT staff auth). Uses CASL: user must own the enforcement action (the enforcementAction.userId === session.user.id).

Business rules (all from Feature Lock-in Section 44 + Section 25 appeal pattern):
1. Parse input with `submitAppealSchema`.
2. Fetch the enforcement action by ID. Verify it exists.
3. Verify `enforcementAction.userId === session.user.id` (ownership check). Return generic "Not found" if mismatch.
4. Verify `enforcementAction.status === 'ACTIVE'` (can only appeal active actions).
5. Read `score.enforcement.appealWindowDays` (default 30). Verify `now - enforcementAction.createdAt <= appealWindowDays * 24 * 60 * 60 * 1000`. Return "Appeal window has expired" if exceeded.
6. Read `score.enforcement.maxAppealsPerAction` (default 1). Check if action already has `appealedAt` set. Return "This action has already been appealed" if so. (For V1, one appeal per action.)
7. Read `score.enforcement.appealableActionTypes` (default array). Verify `enforcementAction.actionType` is in the list. Return "This action type cannot be appealed" if not.
8. Update the enforcement action:
   - `status = 'APPEALED'`
   - `appealNote = parsed.data.appealNote`
   - `appealEvidenceUrls = parsed.data.appealEvidenceUrls ?? []`
   - `appealedAt = new Date()`
   - `appealedByUserId = session.user.id`
   - `updatedAt = new Date()`
9. Create audit event: `action = 'ENFORCEMENT_APPEAL_SUBMITTED'`, `subject = 'EnforcementAction'`, `subjectId = enforcementActionId`, `severity = 'HIGH'`, `detailsJson = { userId, actionType, appealNote }`.
10. Revalidate paths: `/my/selling/performance`, `/mod/enforcement`.
11. Return `{ success: true }`.

**`reviewEnforcementAppealAction(input: unknown)`**

Auth: `staffAuthorize()`. Uses CASL: `ability.can('update', 'EnforcementAction')`.

Business rules:
1. Parse input with `reviewAppealSchema`.
2. Fetch the enforcement action by ID. Verify it exists.
3. Verify `enforcementAction.status === 'APPEALED'` (can only review appealed actions).
4. If decision is `'APPROVED'`:
   - Update enforcement action: `status = 'APPEAL_APPROVED'`, `appealReviewedByStaffId = session.staffUserId`, `appealReviewNote = parsed.data.reviewNote`, `appealResolvedAt = new Date()`, `liftedAt = new Date()`, `liftedByStaffId = session.staffUserId`, `liftedReason = 'Appeal approved: ' + parsed.data.reviewNote`, `updatedAt = new Date()`.
   - Reverse side effects (same logic as `liftEnforcementActionAction`): if SUSPENSION or RESTRICTION, set sellerProfile.status back to 'ACTIVE'. If WARNING/COACHING/RESTRICTION/PRE_SUSPENSION, clear enforcementLevel and enforcementStartedAt. If LISTING_REMOVAL/LISTING_SUPPRESSION, restore listing enforcementState to 'CLEAR'.
   - Queue notification to seller: use `enforcement.appeal_approved` template.
5. If decision is `'DENIED'`:
   - Update enforcement action: `status = 'ACTIVE'`, `appealReviewedByStaffId = session.staffUserId`, `appealReviewNote = parsed.data.reviewNote`, `appealResolvedAt = new Date()`, `updatedAt = new Date()`.
   - Note: status returns to 'ACTIVE', NOT a new status. The appeal was denied so the enforcement remains active.
   - Queue notification to seller: use `enforcement.appeal_denied` template.
6. Create audit event: `action = 'ENFORCEMENT_APPEAL_REVIEWED'`, `subject = 'EnforcementAction'`, `subjectId = enforcementActionId`, `severity = 'HIGH'`, `detailsJson = { decision, reviewNote, userId: enforcementAction.userId }`.
7. Revalidate paths: `/mod/enforcement`, `/mod/enforcement/[enforcementActionId]`.
8. Return `{ success: true }`.

## 4. Acceptance Criteria

### Seller-Facing (Positive Cases)
1. A seller with an ACTIVE enforcement action of type WARNING, RESTRICTION, PRE_SUSPENSION, or SUSPENSION can see the "Appeal this action" form on `/my/selling/performance`.
2. A seller can submit an appeal with a note (10-2000 chars) and optional evidence URLs (up to 5).
3. After submitting an appeal, the enforcement action status changes to `APPEALED`.
4. After submitting, the seller sees "Your appeal has been submitted and will be reviewed within 48 hours" on the performance page.
5. A seller whose appeal was approved sees a success message and the enforcement action is lifted (status = `APPEAL_APPROVED`).
6. A seller whose appeal was denied sees a neutral message and the enforcement action returns to `ACTIVE`.

### Seller-Facing (Negative Cases)
7. A seller CANNOT appeal a COACHING action (not in appealable types).
8. A seller CANNOT appeal an ACCOUNT_BAN action.
9. A seller CANNOT appeal an action that has already been appealed (one appeal per action).
10. A seller CANNOT appeal an action outside the appeal window (30 days by default).
11. A seller CANNOT appeal another seller's enforcement action (ownership check).
12. An unauthenticated user CANNOT submit an appeal (auth gate).
13. A seller CANNOT set the enforcement action status directly (only the appeal action transitions the status).

### Staff-Facing (Positive Cases)
14. Staff with MODERATION or ADMIN role can see APPEALED enforcement actions in the Appealed tab on `/mod/enforcement`.
15. Staff can see appeal details (note, evidence, date) on the enforcement action detail page when the action has been appealed.
16. Staff can approve an appeal, which lifts the enforcement action and reverses side effects (seller status restored, listings restored, etc.).
17. Staff can deny an appeal, which returns the enforcement action to ACTIVE status.
18. Both approve and deny require a review note.

### Staff-Facing (Negative Cases)
19. SUPPORT role staff CANNOT approve or deny appeals (only `read` permission on EnforcementAction).
20. Staff CANNOT review an appeal on an action that is not in APPEALED status.
21. Staff CANNOT review an appeal without providing a review note.

### Data Integrity
22. Every appeal submission creates an audit event with `action = 'ENFORCEMENT_APPEAL_SUBMITTED'`.
23. Every appeal review creates an audit event with `action = 'ENFORCEMENT_APPEAL_REVIEWED'` and includes the decision.
24. When an appeal is approved and the action type was SUSPENSION, the sellerProfile.status is set back to 'ACTIVE'.
25. When an appeal is approved and the action type was RESTRICTION, the sellerProfile.status is set back to 'ACTIVE'.
26. When an appeal is approved and the enforcement had set enforcementLevel, the enforcementLevel and enforcementStartedAt are cleared.
27. When an appeal is approved and the action type was LISTING_REMOVAL/LISTING_SUPPRESSION, the linked listing's enforcementState is restored to 'CLEAR'.
28. All appeal-related timestamps are stored with timezone (`withTimezone: true`).

### Notification
29. When a seller submits an appeal, the `enforcement.appeal_submitted` template notification is available (staff-facing, IN_APP).
30. When staff approves an appeal, the seller receives `enforcement.appeal_approved` notification (EMAIL + IN_APP).
31. When staff denies an appeal, the seller receives `enforcement.appeal_denied` notification (EMAIL + IN_APP).

### Vocabulary
32. No banned terms appear in any code, UI text, or test descriptions.
33. The appeal denial message uses neutral language ("will remain in effect"), not negative language ("rejected").

### Platform Settings
34. `score.enforcement.appealWindowDays` is seeded with default 30.
35. `score.enforcement.maxAppealsPerAction` is seeded with default 1.
36. `score.enforcement.appealReviewSlaHours` is seeded with default 48.
37. `score.enforcement.appealableActionTypes` is seeded with the correct array of appealable types.
38. All appeal logic reads these settings at runtime, never hardcodes them.

## 6. Spec Inconsistencies

### INCONSISTENCY 1: No dedicated appeal page in Page Registry
The Page Registry (entry 73a) says `/my/selling/performance` has "band appeal form" but there is no separate `/my/selling/appeals` page and no `/mod/appeals` page. The Build Sequence Tracker for G4.2 just says "Appeal flow" with no route details.

**Resolution**: Appeal form is embedded in `/my/selling/performance` as the Page Registry indicates. Appeals are reviewed on the existing `/mod/enforcement/[id]` page. The APPEALED tab on `/mod/enforcement` serves as the queue. No new pages needed.

### INCONSISTENCY 2: No `enforcementAppeal` table in schema doc
The Schema doc has no `enforcementAppeal` table. The `dispute` table has inline appeal fields as a pattern. The `enforcementActionStatusEnum` already has `APPEALED` and `APPEAL_APPROVED` values.

**Resolution**: Appeal data lives inline on the `enforcementAction` table (new columns added via migration), following the dispute table pattern. This is the simplest approach and matches the existing enum design.

### INCONSISTENCY 3: Feature Lock-in Section 44 vs Seller Score Canonical on "Below Standard" badge
Feature Lock-in Section 44 says Restriction tier includes "'Below Standard' badge visible to buyers." Seller Score Canonical Section 1 says "No Negative Labels" to buyers.

**Resolution**: Already documented in G4.1 prompt. Seller Score Canonical wins (per its own law statement). This does NOT affect the appeal flow but is noted for completeness.

### INCONSISTENCY 4: Feature Lock-in Section 25 appeal rules vs enforcement appeals
Section 25 (Buyer Protection) defines appeal rules for dispute resolutions (30-day window, one appeal, 48-hour SLA). These apply to buyer-vs-seller disputes. The enforcement appeal flow is for seller-vs-platform actions. The same general pattern applies (30-day window, one appeal, 48-hour SLA) but enforcement appeals are a separate system.

**Resolution**: Apply the same 30-day/1-appeal/48-hour pattern to enforcement appeals via platform settings. The two systems share parameters but operate independently.

### INCONSISTENCY 5: "Appeal available" only for Suspension in Feature Lock-in Section 44
Feature Lock-in Section 44 enforcement tiers table says "Appeal available" only in the Suspension row. But logically, Warning, Restriction, and PRE_SUSPENSION should also be appealable (e.g., wrongful policy violation accusation at Warning level).

**OWNER DECISION NEEDED**: Should appeals be available for all enforcement tiers (except COACHING), or only for SUSPENSION as Feature Lock-in Section 44 literally states? The install prompt defaults to making WARNING, RESTRICTION, PRE_SUSPENSION, SUSPENSION, and listing-level actions (LISTING_REMOVAL, LISTING_SUPPRESSION, BOOST_DISABLED, LISTING_CAP, SEARCH_DEMOTION) all appealable, controlled by the `score.enforcement.appealableActionTypes` platform setting. The owner can narrow this by modifying the setting's default array.
