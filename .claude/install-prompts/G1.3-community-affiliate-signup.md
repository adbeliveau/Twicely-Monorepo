## CASL FIX REQUIRED BEFORE IMPLEMENTATION

**ISSUE:** The Affiliate Canonical Section 8 specifies `can('manage', 'Affiliate', { userId })` for sellers, but the current `ability.ts` line 229 only has `can('read', 'Affiliate', { userId })`. The signup action needs `can('create', 'Affiliate')` to work.

**FIX:** In `src/lib/casl/ability.ts`, change line 229 from:
```typescript
can('read', 'Affiliate', { userId });
```
to:
```typescript
can('manage', 'Affiliate', { userId });
```

This matches the canonical spec Section 8: `{ action: 'manage', subject: 'Affiliate', conditions: { userId } }`. The `manage` action is a CASL shorthand for create, read, update, and delete. Since the affiliate record has `affiliate.userId` as the ownership column, the condition `{ userId }` correctly scopes it to the seller's own record.

**`joinAffiliateSchema`:**
```

  referralCode: z.string()
```

Rules from Affiliate Canonical Section 2.4:
- Promo codes are 4-20 chars, alphanumeric + hyphens, auto-uppercased

- Must be unique across platform (checked server-side)
- Only alphanumeric, hyphens, underscores

The referral code is optional in the input -- if omitted, the server action derives it from the user's `username` (uppercased). If the user has no username, the action must return an error asking them to set a username first.

**Updated schema:**
```
joinAffiliateSchema = z.object({
  referralCode: z.string()
    .min(3, 'Referral code must be at least 3 characters')
    .max(30, 'Referral code must be at most 30 characters')
    .regex(/^[a-zA-Z0-9_-]+$/, 'Only letters, numbers, hyphens, and underscores allowed')
    .transform(val => val.toUpperCase())
    .optional(),
}).strict();
```

### 4. Server Action: joinAffiliateProgram

**`joinAffiliateProgram(input: z.infer<typeof joinAffiliateSchema>)`**

Flow:
1. `const { ability, session } = await authorize()`
2. If no session, return `{ success: false, error: 'Please sign in to continue' }`
3. If `!session.isSeller`, return `{ success: false, error: 'You must be a seller to join the affiliate program' }`
4. CASL check: `if (!ability.can('create', sub('Affiliate', { userId: session.userId })))` return forbidden
5. Read platform settings:
   - `affiliate.enabled` (fallback: true) -- if false, return `{ success: false, error: 'The affiliate program is currently disabled' }`
   - `affiliate.community.enabled` (fallback: true) -- if false, return `{ success: false, error: 'Community affiliate signup is currently disabled' }`
   - `affiliate.community.commissionRateBps` (fallback: 1500)
   - `affiliate.community.cookieDays` (fallback: 30)
   - `affiliate.commissionDurationMonths` (fallback: 12)
6. Check if user already has an affiliate record:
   - `getAffiliateByUserId(session.userId)` -- if exists, return `{ success: false, error: 'You are already registered as an affiliate' }`
7. Determine referral code:
   - Query user table for username: `db.select({ username: user.username }).from(user).where(eq(user.id, session.userId)).limit(1)` (CaslSession does NOT include username -- see Spec Notes below)
   - If `input.referralCode` provided: use it (already uppercased by Zod transform)
   - If not provided: derive from `userRow.username` uppercased. If `userRow.username` is null/undefined, return `{ success: false, error: 'Please set a username in your account settings before joining the affiliate program' }`
8. Check referral code uniqueness:
   - `getAffiliateByReferralCode(referralCode)` -- if exists, return `{ success: false, error: 'This referral code is already taken' }`
9. Insert into `affiliate` table with explicit field mapping:
   ```typescript
   await db.insert(affiliate).values({
     userId: session.userId,
     tier: 'COMMUNITY',
     status: 'ACTIVE',
     referralCode: referralCode,
     commissionRateBps: commissionRateBps,
     cookieDurationDays: cookieDays,
     commissionDurationMonths: commissionDurationMonths,
   });
   ```
   - NOTE: All other columns have defaults (pendingBalanceCents: 0, availableBalanceCents: 0, etc.)
   - NOTE: Community tier goes ACTIVE immediately -- no approval needed (per Section 2.1: "Self-serve signup in seller dashboard")
   - NOTE: Influencer tier requires application + manual approval -- that is G3.1, NOT this step
10. Write audit event:
    ```typescript
    await db.insert(auditEvent).values({
      actorType: 'USER',
      actorId: session.userId,
      action: 'AFFILIATE_JOINED',
      subject: 'Affiliate',
      subjectId: session.userId,
      severity: 'LOW',
      detailsJson: { tier: 'COMMUNITY', referralCode },
    });
    ```
11. `revalidatePath('/my/selling/affiliate')`
12. Return `{ success: true, referralCode }`

**Return type:** `{ success: boolean; error?: string; referralCode?: string }`

## SPEC NOTES & EDGE CASES

### Why Community Goes ACTIVE Immediately
Affiliate Canonical Section 2.1: Community tier join method is "Self-serve signup in seller dashboard." There is no approval queue for community. Influencer tier (G3.1) requires "Application -> manual approval by admin."

### Referral Code Uniqueness Race Condition
The uniqueness check (query before insert) has a theoretical TOCTOU race. The `affiliate.referralCode` column has a UNIQUE constraint in the DB, so a concurrent duplicate insert will fail with a DB error. The action should catch this and return a user-friendly error. Pattern:
```typescript
try {
  await db.insert(affiliate).values({ ... });
} catch (err: unknown) {
  if (err instanceof Error && err.message.includes('unique')) {
    return { success: false, error: 'This referral code is already taken' };
  }
  throw err;
}
```

### Session Username Access
**IMPORTANT:** The `CaslSession` type does NOT include `username`. The `authorize()` function (in `src/lib/casl/authorize.ts`) only populates `userId`, `email`, `isSeller`, `sellerId`, `sellerStatus`, delegation fields, and platform staff fields. The `username` field exists on the `user` table (`src/lib/db/schema/auth.ts` line 17) and on the Better Auth session user object, but NOT on `CaslSession`.

The action MUST query the user table to get the username:
```typescript
const [userRow] = await db
  .select({ username: user.username })
  .from(user)
  .where(eq(user.id, session.userId))
  .limit(1);

if (!userRow?.username && !input.referralCode) {
  return { success: false, error: 'Please set a username in your account settings before joining the affiliate program' };
}

const referralCode = input.referralCode ?? userRow!.username!.toUpperCase();
```

Do NOT attempt to add `username` to `CaslSession` or modify `authorize()` -- that is out of scope for this step.
