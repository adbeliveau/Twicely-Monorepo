## 2. CONTEXT -- WHY A BROWSER EXTENSION?

The Lister Canonical (Section 9.1) defines three connector tiers:

| Tier | Auth Method | Platforms |
|---|---|---|
| **Tier A** | OAuth / API Key | eBay, Etsy |
| **Tier B** | OAuth / API Key | Mercari, Depop, Grailed, FB Marketplace |
| **Tier C** | Session cookies | Poshmark, TheRealReal |

Currently, **Tier C connectors** (Poshmark, TheRealReal) use the platforms' undocumented internal APIs with username/password authentication stored server-side. This works for basic operations but has significant limitations:

1. **Session durability** -- Server-side sessions expire frequently, requiring re-authentication
2. **Detection risk** -- Server-to-platform requests from Twicely's IP range are easier to fingerprint
3. **Limited capabilities** -- Some platform actions (e.g., Poshmark sharing, closet browsing) require browser-context execution
4. **FB Marketplace** -- Facebook's Commerce API is being deprecated/restricted; browser-context listing may become the only viable path

The browser extension solves these problems by:
- Running in the seller's own browser session (authentic cookies, real IP, real user agent)
- Executing content scripts on platform pages to perform actions the seller would do manually
- Maintaining persistent sessions without server-side credential storage
- Communicating results back to the Twicely server via authenticated API

**H1.1 builds the infrastructure. H1.2-H1.4 build the platform-specific content scripts.**

### 3.3 Service Worker (Background Script)

File: `extensions/chrome/src/background/service-worker.ts`

Responsibilities:
1. **Receive messages from content scripts** via `chrome.runtime.onMessage`
2. **Relay data to Twicely API** via the `api-client.ts` HTTP client
3. **Manage auth state** -- store/retrieve JWT token from chrome.storage.local
4. **Heartbeat alarm** -- every 5 minutes, ping `POST /api/extension/heartbeat` to report extension health
5. **Handle extension install/update** via `chrome.runtime.onInstalled`

**Message types (service-worker handles):**

```typescript
// Messages FROM content scripts TO service worker
type ContentToBackground =
  | { type: 'PLATFORM_DETECTED'; channel: ExtensionChannel; url: string }
  | { type: 'SESSION_CAPTURED'; channel: ExtensionChannel; sessionData: Record<string, unknown> }
  | { type: 'LISTING_SCRAPED'; channel: ExtensionChannel; listing: ScrapedListing }
  | { type: 'ACTION_RESULT'; channel: ExtensionChannel; action: string; success: boolean; error?: string }
  | { type: 'GET_AUTH_STATE' };

// Messages FROM service worker TO content scripts (responses)
type BackgroundToContent =
  | { type: 'AUTH_STATE'; authenticated: boolean; userId: string | null }
  | { type: 'EXECUTE_ACTION'; action: string; payload: Record<string, unknown> }
  | { type: 'ACK'; success: boolean };
```

**On install handler:**

### 3.4 Popup (Login + Status)

File: `extensions/chrome/src/popup/popup.html` + `popup.ts` + `popup.css`

The popup is a simple HTML page (NO React, NO framework). It has two states:

**State 1: Not authenticated**
- Shows Twicely logo
- "Connect to Twicely" button that opens `https://twicely.co/api/extension/authorize` in a new tab
- Brief description: "Sign in to your Twicely account to enable crosslisting."

**State 2: Authenticated**
- Shows "Connected as {displayName}" with user avatar
- Shows extension version
- Shows connected platforms with status indicators:
  - Green dot: content script loaded on current tab's platform
  - Gray dot: platform not active on current tab
- "Disconnect" button (clears stored token)
- "Open Twicely" link to `/my/selling/crosslist`

**Auth flow (token-based, NOT cookie-based):**
1. User clicks "Connect to Twicely" in popup
2. New tab opens to `https://twicely.co/api/extension/authorize`
3. Server checks Better Auth session (user must already be logged into Twicely)
4. If authenticated, server generates a short-lived extension registration token (JWT, 5-minute expiry)
5. Server redirects to `https://twicely.co/api/extension/callback?token={registrationToken}`
6. The callback page uses `window.postMessage` or a custom protocol to pass the token to the extension
7. Extension exchanges the registration token for a long-lived extension session token via `POST /api/extension/register`
8. Long-lived token stored in `chrome.storage.local`
9. Popup updates to show authenticated state

### 3.5 Content Script Bridge

File: `extensions/chrome/src/content/bridge.ts`

The bridge is a minimal content script injected on all matched platform pages. In H1.1, it does NOT perform any platform-specific actions (that is H1.2/H1.3). It only:

1. **Detects the current platform** based on `window.location.hostname`:
   - `poshmark.com` -> `POSHMARK`
   - `www.facebook.com` (when URL contains `/marketplace/`) -> `FB_MARKETPLACE`
   - `www.therealreal.com` -> `THEREALREAL`
   - `www.vestiairecollective.com` -> `VESTIAIRE` (future, noop for now)

2. **Sends PLATFORM_DETECTED message** to the service worker on page load

3. **Listens for EXECUTE_ACTION messages** from the service worker (future -- H1.2 will add handlers)

4. **Captures basic session cookies/tokens** from the page context (if available) and sends SESSION_CAPTURED to the service worker. This replaces the username/password flow for Tier C platforms.

```typescript
// Minimal bridge -- H1.1 scaffold only
(function twicelyCrosslisterBridge() {
  const hostname = window.location.hostname;
  let channel: string | null = null;

  if (hostname === 'poshmark.com' || hostname === 'www.poshmark.com') {
    channel = 'POSHMARK';
  } else if (hostname === 'www.facebook.com' && window.location.pathname.startsWith('/marketplace')) {
    channel = 'FB_MARKETPLACE';
  } else if (hostname === 'www.therealreal.com') {
    channel = 'THEREALREAL';
  } else if (hostname === 'www.vestiairecollective.com') {
    channel = 'VESTIAIRE';
  }

  if (channel) {
    chrome.runtime.sendMessage({
      type: 'PLATFORM_DETECTED',
      channel,
      url: window.location.href,
    });
  }
})();
```

### 3.10 Server-Side API Routes (Next.js)

These are new API routes in the Next.js app that the extension communicates with.

#### 3.10.1 `GET /api/extension/authorize`

Checks if the user has a valid Better Auth session. If yes, generates a short-lived JWT registration token (5-minute expiry) containing `{ userId, purpose: 'extension-registration' }` and redirects to `/api/extension/callback?token={jwt}`. If not authenticated, redirects to `/auth/login?redirect=/api/extension/authorize`.

```typescript
// src/app/api/extension/authorize/route.ts
import { getSession } from '@/lib/auth-server';
import { NextResponse } from 'next/server';
import { SignJWT } from 'jose';

export async function GET(request: Request) {
  const session = await getSession();
  if (!session?.userId) {
    const loginUrl = new URL('/auth/login', request.url);
    loginUrl.searchParams.set('redirect', '/api/extension/authorize');
    return NextResponse.redirect(loginUrl);
  }

  // Generate short-lived registration token
  const secret = new TextEncoder().encode(process.env.EXTENSION_JWT_SECRET);
  const token = await new SignJWT({
    userId: session.userId,
    purpose: 'extension-registration',
  })
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('5m')
    .sign(secret);

  const callbackUrl = new URL('/api/extension/callback', request.url);
  callbackUrl.searchParams.set('token', token);
  return NextResponse.redirect(callbackUrl);
}
```

#### 3.10.2 `GET /api/extension/callback`

A simple HTML page that receives the registration token via query parameter and communicates it to the extension. Uses a well-known message format that the extension's service worker listens for.

```typescript
// src/app/api/extension/callback/route.ts
import { NextResponse } from 'next/server';

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const token = searchParams.get('token');

  if (!token) {
    return new NextResponse('Missing token', { status: 400 });
  }

  // Return an HTML page that sends the token to the extension
  const html = `<!DOCTYPE html>
<html>
<head><title>Twicely Extension Authorization</title></head>
<body>
  <h2>Connecting to Twicely Extension...</h2>
  <p id="status">Sending authorization to extension...</p>
  <script>
    // Post message that the extension content script or popup can receive
    const token = ${JSON.stringify(token)};
    // Method 1: chrome.runtime.sendMessage via externally_connectable
    // Method 2: postMessage for the popup window
    if (window.opener) {
      window.opener.postMessage({ type: 'TWICELY_EXTENSION_TOKEN', token }, '*');
      document.getElementById('status').textContent = 'Authorization sent! You can close this tab.';
      setTimeout(() => window.close(), 2000);
    } else {
      // Extension opened this tab directly -- use a storage key the extension polls
      try {
        localStorage.setItem('twicely_extension_token', token);
        document.getElementById('status').textContent = 'Authorization complete! You can close this tab.';
        setTimeout(() => window.close(), 2000);
      } catch (e) {
        document.getElementById('status').textContent = 'Authorization complete. Please close this tab and return to the extension.';
      }
    }
  </script>
</body>
</html>`;

  return new NextResponse(html, {
    headers: { 'Content-Type': 'text/html' },
  });
}
```

#### 3.10.3 `POST /api/extension/register`

Exchanges a short-lived registration token for a long-lived extension session token. Validates the registration token, then creates a long-lived JWT (30-day expiry) for ongoing API access.

```typescript
// src/app/api/extension/register/route.ts
import { NextResponse } from 'next/server';
import { jwtVerify, SignJWT } from 'jose';
import { db } from '@/lib/db';
import { user } from '@/lib/db/schema';
import { eq } from 'drizzle-orm';
import { z } from 'zod';
import { logger } from '@/lib/logger';

const registerSchema = z.object({
  registrationToken: z.string().min(1),
  extensionVersion: z.string().min(1),
}).strict();

export async function POST(request: Request) {
  let body: unknown;
  try {
    body = await request.json();
  } catch {
    return NextResponse.json({ success: false, error: 'Invalid JSON' }, { status: 400 });
  }

  const parsed = registerSchema.safeParse(body);
  if (!parsed.success) {
    return NextResponse.json({ success: false, error: 'Invalid input' }, { status: 400 });
  }

  const secret = new TextEncoder().encode(process.env.EXTENSION_JWT_SECRET);

  try {
    const { payload } = await jwtVerify(parsed.data.registrationToken, secret);

    if (payload['purpose'] !== 'extension-registration') {
      return NextResponse.json({ success: false, error: 'Invalid token purpose' }, { status: 403 });
    }

    const userId = payload['userId'] as string;
    if (!userId) {
      return NextResponse.json({ success: false, error: 'Invalid token' }, { status: 403 });
    }

    // Fetch user display info
    const [dbUser] = await db
      .select({ displayName: user.displayName, name: user.name, image: user.image })
      .from(user)
      .where(eq(user.id, userId))
      .limit(1);

    if (!dbUser) {
      return NextResponse.json({ success: false, error: 'User not found' }, { status: 404 });
    }

    // Generate long-lived extension token (30 days)
    const expiresAt = Math.floor(Date.now() / 1000) + 30 * 24 * 60 * 60;
    const extensionToken = await new SignJWT({
      userId,
      purpose: 'extension-session',
    })
      .setProtectedHeader({ alg: 'HS256' })
      .setIssuedAt()
      .setExpirationTime('30d')
      .sign(secret);

    logger.info('[extension/register] Extension registered', {
      userId,
      extensionVersion: parsed.data.extensionVersion,
    });

    return NextResponse.json({
      success: true,
      token: extensionToken,
      userId,
      displayName: dbUser.displayName ?? dbUser.name ?? 'Seller',
      avatarUrl: dbUser.image,
      expiresAt: expiresAt * 1000, // Convert to ms for client
    });
  } catch (err) {
    logger.error('[extension/register] Token validation failed', { error: String(err) });
    return NextResponse.json({ success: false, error: 'Invalid or expired token' }, { status: 403 });
  }
}
```

#### 3.10.4 `POST /api/extension/heartbeat`

Receives periodic heartbeats from the extension. Validates the extension token, returns server time and connected channel list for the seller.

```typescript
// src/app/api/extension/heartbeat/route.ts
import { NextResponse } from 'next/server';
import { jwtVerify } from 'jose';
import { db } from '@/lib/db';
import { crosslisterAccount } from '@/lib/db/schema';
import { eq, and } from 'drizzle-orm';
import { logger } from '@/lib/logger';

export async function POST(request: Request) {
  const authHeader = request.headers.get('Authorization');
  if (!authHeader?.startsWith('Bearer ')) {
    return NextResponse.json({ success: false, error: 'Unauthorized' }, { status: 401 });
  }

  const token = authHeader.slice(7);
  const secret = new TextEncoder().encode(process.env.EXTENSION_JWT_SECRET);

  try {
    const { payload } = await jwtVerify(token, secret);
    if (payload['purpose'] !== 'extension-session') {
      return NextResponse.json({ success: false, error: 'Invalid token' }, { status: 403 });
    }

    const userId = payload['userId'] as string;

    // Fetch connected channels for this seller
    const accounts = await db
      .select({ channel: crosslisterAccount.channel })
      .from(crosslisterAccount)
      .where(and(
        eq(crosslisterAccount.sellerId, userId),
        eq(crosslisterAccount.status, 'ACTIVE'),
      ));

    return NextResponse.json({
      success: true,
      serverTime: Date.now(),
      connectedChannels: accounts.map((a) => a.channel),
    });
  } catch (err) {
    logger.warn('[extension/heartbeat] Invalid token', { error: String(err) });
    return NextResponse.json({ success: false, error: 'Invalid token' }, { status: 401 });
  }
}
```

#### 3.10.5 `POST /api/extension/session`

Receives session data captured by the content script bridge from a Tier C platform page. This allows the extension to forward cookies/tokens from the seller's browser session to Twicely, replacing the username/password authentication flow for Tier C connectors.

```typescript
// src/app/api/extension/session/route.ts
import { NextResponse } from 'next/server';
import { jwtVerify } from 'jose';
import { db } from '@/lib/db';
import { crosslisterAccount } from '@/lib/db/schema';
import { eq, and } from 'drizzle-orm';
import { z } from 'zod';
import { logger } from '@/lib/logger';

const sessionSchema = z.object({
  channel: z.enum(['POSHMARK', 'FB_MARKETPLACE', 'THEREALREAL', 'VESTIAIRE']),
  sessionData: z.record(z.unknown()),
}).strict();

export async function POST(request: Request) {
  const authHeader = request.headers.get('Authorization');
  if (!authHeader?.startsWith('Bearer ')) {
    return NextResponse.json({ success: false, error: 'Unauthorized' }, { status: 401 });
  }

  const token = authHeader.slice(7);
  const secret = new TextEncoder().encode(process.env.EXTENSION_JWT_SECRET);

  let userId: string;
  try {
    const { payload } = await jwtVerify(token, secret);
    if (payload['purpose'] !== 'extension-session') {
      return NextResponse.json({ success: false, error: 'Invalid token' }, { status: 403 });
    }
    userId = payload['userId'] as string;
  } catch {
    return NextResponse.json({ success: false, error: 'Invalid token' }, { status: 401 });
  }

  let body: unknown;
  try {
    body = await request.json();
  } catch {
    return NextResponse.json({ success: false, error: 'Invalid JSON' }, { status: 400 });
  }

  const parsed = sessionSchema.safeParse(body);
  if (!parsed.success) {
    return NextResponse.json({ success: false, error: 'Invalid input' }, { status: 400 });
  }

  const { channel, sessionData } = parsed.data;

  try {
    // Upsert the crosslister account with session data from the extension
    const [existing] = await db
      .select({ id: crosslisterAccount.id })
      .from(crosslisterAccount)
      .where(and(
        eq(crosslisterAccount.sellerId, userId),
        eq(crosslisterAccount.channel, channel),
      ))
      .limit(1);

    if (existing) {
      await db.update(crosslisterAccount).set({
        sessionData,
        status: 'ACTIVE',
        lastAuthAt: new Date(),
        consecutiveErrors: 0,
        lastError: null,
        lastErrorAt: null,
        updatedAt: new Date(),
      }).where(eq(crosslisterAccount.id, existing.id));
    } else {
      await db.insert(crosslisterAccount).values({
        sellerId: userId,
        channel,
        authMethod: 'SESSION',
        status: 'ACTIVE',
        sessionData,
        capabilities: {}, // Will be enriched by connector framework
        lastAuthAt: new Date(),
      });
    }

    logger.info('[extension/session] Session data updated', { userId, channel });
    return NextResponse.json({ success: true });
  } catch (err) {
    logger.error('[extension/session] Failed to update session', { userId, channel, error: String(err) });
    return NextResponse.json({ success: false, error: 'Internal error' }, { status: 500 });
  }
}
```

#### 3.10.6 `POST /api/extension/detect`

Lightweight endpoint that records which platform the seller's browser is currently on. Used for analytics and the extension status UI (H1.4). Fire-and-forget from the extension side.

```typescript
// src/app/api/extension/detect/route.ts
import { NextResponse } from 'next/server';
import { jwtVerify } from 'jose';
import { logger } from '@/lib/logger';

export async function POST(request: Request) {
  const authHeader = request.headers.get('Authorization');
  if (!authHeader?.startsWith('Bearer ')) {
    return NextResponse.json({ success: false }, { status: 401 });
  }

  const token = authHeader.slice(7);
  const secret = new TextEncoder().encode(process.env.EXTENSION_JWT_SECRET);

  try {
    const { payload } = await jwtVerify(token, secret);
    if (payload['purpose'] !== 'extension-session') {
      return NextResponse.json({ success: false }, { status: 403 });
    }

    // For now, just log. H1.4 will add real tracking.
    logger.debug('[extension/detect] Platform detected', {
      userId: payload['userId'],
    });

    return NextResponse.json({ success: true });
  } catch {
    return NextResponse.json({ success: false }, { status: 401 });
  }
}
```

## 4. CONSTRAINTS -- WHAT NOT TO DO

### 4.1 Banned Patterns
- Do NOT use `as any`, `@ts-ignore`, or `@ts-expect-error` in extension code
- Do NOT use React, Zustand, or any framework in the extension -- plain TypeScript + DOM
- Do NOT store the user's Twicely password in the extension -- use JWT tokens only
- Do NOT store platform (Poshmark/TRR) passwords in the extension -- capture browser session tokens only
- Do NOT import from `src/` in the extension code -- no shared runtime
- Do NOT use `chrome.storage.sync` (syncs across devices = security risk for tokens) -- use `chrome.storage.local` only
- Do NOT add the extension to the pnpm workspace -- it is a standalone project

### 4.2 Security Constraints
- Extension JWT tokens are signed with a SEPARATE secret (`EXTENSION_JWT_SECRET`), NOT the Better Auth secret
- Short-lived registration tokens expire in 5 minutes -- enforce via JWT `exp` claim
- Long-lived session tokens expire in 30 days -- enforce via JWT `exp` claim
- All API routes validate the JWT `purpose` claim to prevent token type confusion
- Session data from Tier C platforms (cookies, tokens) is stored encrypted in the `crosslisterAccount.sessionData` column (encryption TODO noted for production, same as existing connectors)
- Content scripts must check `window.location` to confirm they are on the expected platform before capturing session data
- Never log tokens in plaintext in the extension or server

### 4.3 Vocabulary Enforcement
- Use `ListerTier` not `SellerTier` or `SubscriptionTier`
- Use `crosslister` not `crosslisting tool` in UI copy
- Platform names: "Poshmark", "Facebook Marketplace" (not "FB Marketplace" in user-facing text), "The RealReal"

### 4.4 Route Enforcement
- Extension API routes live under `/api/extension/` (NOT `/api/crosslister/extension/`)
- The extension opens `/my/selling/crosslist` (NOT `/dashboard/crosslist` or `/settings/crosslist`)

## 9. NOT SPECIFIED -- OWNER DECISIONS NEEDED

The following items are not specified in the canonical documents and require owner input:

1. **Extension distribution model** -- Will the extension be published to the Chrome Web Store, or distributed as an unlisted .crx file, or loaded unpacked during development only? This affects manifest fields like `update_url` and whether we need Chrome Web Store developer account setup.

2. **Firefox / Edge support** -- The Build Sequence Tracker only mentions "Browser Extension" generically. Should we also create a Firefox WebExtension manifest (Manifest V2/V3)? The current scaffold targets Chrome only.

3. **Extension-to-callback communication method** -- The prompt specifies `localStorage` + `window.opener.postMessage` as the token relay mechanism. An alternative is Chrome's `externally_connectable` manifest key which allows specific websites to send messages directly to the extension via `chrome.runtime.sendMessage`. This is more secure but requires the extension ID to be known at build time. Owner should decide which approach.

4. **Vestiaire Collective** -- The Build Sequence Tracker has H4.1 "Vestiaire extension scripts" but Vestiaire is not in the current `channelEnum` in the schema. The manifest includes Vestiaire in `host_permissions` for future-proofing but the server-side session endpoint does not accept it as a valid channel. Should we add `VESTIAIRE` to the `channelEnum` now, or defer?

5. **jose dependency** -- The server-side JWT handling uses the `jose` library. This is already commonly available in Next.js environments. Confirm this is acceptable vs. using `jsonwebtoken` or Better Auth's built-in JWT utilities.

6. **Token refresh strategy** -- The 30-day extension token does not auto-refresh. When it expires, the seller must re-authorize via the popup. Should we implement a silent refresh mechanism (extension checks token expiry and re-registers before it expires)?
