### 2.3 Server Actions

**File: `src/lib/actions/shipping-quote.ts`**

Three server actions:

#### `submitShippingQuote(input)`
- **Who calls:** Seller (or delegated staff with `orders.manage`)
- **Input schema (Zod .strict()):**
  ```typescript
  z.object({
    quoteId: z.string().min(1),
    quotedShippingCents: z.number().int().min(0).max(999999),
  }).strict()
  ```
- **Logic:**
  1. `authorize()` + CASL check: `ability.can('update', sub('CombinedShippingQuote', { sellerId: userId }))`
  2. Fetch quote by ID. Verify `sellerId === userId` (or `onBehalfOfSellerId`).
  3. Verify `status === 'PENDING_SELLER'` or `status === 'PENALTY_APPLIED'` (seller can still quote after penalty).
  4. Verify `quotedShippingCents <= maxShippingCents` (quote cannot exceed sum of individual rates).
  5. Update quote: set `quotedShippingCents`, `sellerQuotedAt = now()`.
  6. If status was `PENDING_SELLER` (before deadline): set `status = 'PENDING_BUYER'`.
  7. If status was `PENALTY_APPLIED` (after deadline): keep status `PENALTY_APPLIED` but set `quotedShippingCents`. The buyer will pay the LOWER of the seller's quote or the penalty-discounted price (see `resolveQuoteFinalPrice()`).
  8. Notify buyer: `notify(buyerId, 'shipping_quote.received', { sellerName, quotedAmountFormatted, savingsFormatted, orderNumber })`.
  9. `revalidatePath('/my/selling/orders/[id]')` and `revalidatePath('/my/buying/orders/[id]')`.
- **Returns:** `{ success: boolean; error?: string }`

#### `respondToShippingQuote(input)`
- **Who calls:** Buyer
- **Input schema (Zod .strict()):**
  ```typescript
  z.object({
    quoteId: z.string().min(1),
    action: z.enum(['ACCEPT', 'DISPUTE']),
  }).strict()
  ```
- **Logic:**
  1. `authorize()` + CASL check: `ability.can('update', sub('CombinedShippingQuote', { buyerId: userId }))`
  2. Fetch quote by ID. Verify `buyerId === userId`.
  3. Verify `status === 'PENDING_BUYER'`.
  4. If `action === 'ACCEPT'`:
     - Compute `finalShippingCents = quotedShippingCents`.
     - Compute `savingsCents = maxShippingCents - finalShippingCents`.
     - Set `status = 'ACCEPTED'`, `finalShippingCents`, `savingsCents`, `buyerRespondedAt = now()`.
     - Update `order.shippingCents` to `finalShippingCents`.
     - Recalculate `order.totalCents = order.itemSubtotalCents + finalShippingCents + order.taxCents - order.discountCents`.
     - Notify seller: `notify(sellerId, 'shipping_quote.accepted', { buyerName, orderNumber })`.
  5. If `action === 'DISPUTE'`:
     - Set `status = 'DISPUTED'`, `buyerRespondedAt = now()`.
     - Notify seller: `notify(sellerId, 'shipping_quote.disputed', { buyerName, orderNumber })`.
     - (Support case creation is manual -- buyer can open support from the order page.)
  6. `revalidatePath(...)` for both buyer and seller order pages.
- **Returns:** `{ success: boolean; error?: string }`

#### `resolveQuoteFinalPrice(quoteId)` -- internal helper (NOT a server action, not exported with 'use server')
- **Purpose:** After penalty is applied AND seller eventually quotes, determine the final price the buyer pays.
- **Logic per canonical:**
  - `penaltyPrice = Math.round(maxShippingCents * (1 - penaltyDiscountPercent / 100))`
  - If `quotedShippingCents !== null`: `finalShippingCents = Math.min(quotedShippingCents, penaltyPrice)`
  - If `quotedShippingCents === null` (seller never quoted): `finalShippingCents = penaltyPrice`
  - `savingsCents = maxShippingCents - finalShippingCents`
- Source: Feature Lock-in Addendum section 50, "Mode 5" flow, specifically: "Buyer pays lower of: seller quote OR penalty-discounted price."

```typescript
case 'QUOTED':
  // QUOTED mode: at order creation time, charge MAX shipping (sum of individual rates).
  // The actual combined quote will be provided by the seller post-purchase.
  // At checkout, we return the individual total as the hold ceiling.
  return {

    itemBreakdown: items.map((item) => ({
      listingId: item.listingId,
      originalCents: item.shippingCents * item.quantity,

      quantity: item.quantity,
    })),
  };
```

### 2.7 Penalty Processing (BullMQ Job)

**File: `src/lib/jobs/shipping-quote-deadline.ts`**

A repeatable BullMQ job that runs every 15 minutes to check for expired QUOTED shipping quotes.

**Logic:**
1. Query all `combinedShippingQuote` rows where `status = 'PENDING_SELLER'` AND `sellerDeadline < now()`.
2. For each expired quote:
   a. Read `penaltyDiscountPercent` from the quote row (was set at creation time from platform settings).
   b. Compute `penaltyPrice = Math.round(maxShippingCents * (1 - penaltyDiscountPercent / 100))`.
   c. Update quote: `status = 'PENALTY_APPLIED'`, `penaltyApplied = true`, `finalShippingCents = penaltyPrice`, `savingsCents = maxShippingCents - penaltyPrice`.
   d. Update `order.shippingCents = penaltyPrice`.
   e. Recalculate `order.totalCents`.
   f. Notify buyer: `notify(buyerId, 'shipping_quote.penalty_applied', { orderNumber, originalShippingFormatted, discountedShippingFormatted, savingsFormatted })`.
   g. Notify seller: `notify(sellerId, 'shipping_quote.deadline_missed', { orderNumber, penaltyPercent, discountedShippingFormatted })`.

**Job registration** in the existing job queue setup file (create or add to `src/lib/jobs/index.ts`).

**Important:** If the seller subsequently submits a quote that is LOWER than the penalty price, the buyer pays the seller's quote (not the penalty). This is handled by the `submitShippingQuote` action checking `resolveQuoteFinalPrice()`.

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- Do NOT use "balance" or "wallet" in any UI copy
- Do NOT use "FVF" or "Final Value Fee" -- use "Transaction Fee" or "TF"
- Do NOT use any V2 tier names (BASIC, ELITE, PLUS, MAX, PREMIUM, STANDARD, RISING)

### Tech Stack
- Do NOT use Prisma, NextAuth, Redis (use Valkey), Bull (use BullMQ), Zustand/Redux, tRPC
- BullMQ job for deadline checking -- NOT a cron setTimeout or setInterval

### Business Logic
- The `quotedShippingCents` CANNOT exceed `maxShippingCents` (seller cannot charge MORE than individual rates combined)
- The penalty discount percent is read from `penaltyDiscountPercent` on the quote row (which was populated from platform settings at creation time) -- NOT re-read from platform settings at penalty time. This prevents retroactive changes.
- The buyer ALWAYS pays the LOWER of: seller's quote OR penalty-discounted price. Never the higher.
- `resolveQuoteFinalPrice()` is an internal helper, NOT a server action. Do not put it in a `'use server'` file or if it must be in the same file, do NOT export it.
- Ownership via `userId` always. The `sellerId` and `buyerId` on the quote reference `user.id`, not `sellerProfile.id`.
- All monetary values are integer cents. The `penaltyDiscountPercent` is the one exception -- it is a percentage (real/float) because it is a discount rate, not money.
- Fee calculations are NOT affected by quoted shipping mode. TF is calculated on the original sale price at order time. Shipping quote resolution does not change TF.
- The Stripe PaymentIntent amount adjustment when a quote resolves is OUT OF SCOPE for D2.2. The spec says "Stripe auth hold at MAX shipping" at checkout, and the charge adjustment happens at capture time (which is in the existing payment flow). D2.2 only updates `order.shippingCents` and `order.totalCents` in the database. The actual Stripe amount update at capture is handled by the existing finalize-order flow.

### Code Patterns
- `strict: true` TypeScript -- zero `as any`, zero `@ts-ignore`
- Zod `.strict()` on all action input schemas
- Max 300 lines per file
- Explicit field mapping in all DB updates (never spread request body)
- All platform settings read via `getPlatformSetting()`, never hardcoded

### Route Enforcement
- Seller order detail: `/my/selling/orders/[id]` (NOT `/dashboard`, NOT `/admin`)
- Buyer order detail: `/my/buying/orders/[id]`

## 4. ACCEPTANCE CRITERIA

### Functional
1. When a seller has `combinedShippingMode = 'QUOTED'` on their default shipping profile and a buyer purchases 2+ items, a `combinedShippingQuote` record is created with `status = 'PENDING_SELLER'`.
2. Single-item orders from QUOTED-mode sellers do NOT create a shipping quote (no combination needed).
3. The `maxShippingCents` on the quote equals the sum of individual item shipping rates.
4. The `sellerDeadline` is set to `now + commerce.shipping.combinedQuoteDeadlineHours` (default 48 hours).
5. Seller can submit a quote that is >= $0 and <= maxShippingCents.
6. Seller CANNOT submit a quote > maxShippingCents.
7. After seller submits a quote (before deadline), quote status changes to `PENDING_BUYER`.
8. Buyer can accept the quote -- order.shippingCents updates, order.totalCents recalculates, quote status = `ACCEPTED`.
9. Buyer can dispute the quote -- status = `DISPUTED`, seller notified.
10. After the 48-hour deadline passes with no seller response, the BullMQ job sets `status = 'PENALTY_APPLIED'`, `penaltyApplied = true`, and applies the 25% discount to `finalShippingCents`.
11. After penalty is applied, seller can STILL submit a quote. If the seller's quote is LOWER than the penalty price, `finalShippingCents` is updated to the seller's lower quote.
12. If seller's post-penalty quote is HIGHER than the penalty price, `finalShippingCents` stays at the penalty price (buyer always pays lower).
13. All 6 notification templates fire at the correct times to the correct recipients.
14. The `combinedShippingQuote` row's `penaltyDiscountPercent` is snapshotted from platform settings at quote creation time, NOT re-read at penalty application time.

### Authorization
15. Unauthenticated users cannot access any shipping quote actions.
16. A seller can only submit quotes for their own orders (CASL condition on `sellerId`).
17. A buyer can only respond to quotes on their own orders (CASL condition on `buyerId`).
18. Delegated staff with `orders.manage` scope can submit quotes on behalf of the seller.
19. Delegated staff with `orders.view` scope can read but NOT submit quotes.
20. Platform staff (ADMIN, SUPPORT) can read all quotes.

### Data Integrity
21. All monetary values stored as integer cents (quotedShippingCents, maxShippingCents, finalShippingCents, savingsCents).
22. `savingsCents` is always `maxShippingCents - finalShippingCents`, never negative.
23. `order.totalCents` is always recalculated as `itemSubtotalCents + shippingCents + taxCents - discountCents` after any shipping update.
24. The 4 new platform settings keys exist in the seed data.

### Vocabulary
25. No banned terms appear in any UI component, notification template, or code comment.

### Negative Cases
26. Quote submission with `quotedShippingCents > maxShippingCents` is rejected.
27. Quote submission on a quote with `status !== 'PENDING_SELLER'` and `status !== 'PENALTY_APPLIED'` is rejected.
28. Buyer respond on a quote with `status !== 'PENDING_BUYER'` is rejected.
29. A seller's QUOTED mode profile with a single-item checkout does NOT create a quote.
30. A non-QUOTED mode seller's multi-item order does NOT create a quote.

```typescript
export interface QuoteResolutionInput {
  maxShippingCents: number;
  quotedShippingCents: number | null;
  penaltyDiscountPercent: number;
}

export interface QuoteResolutionResult {
  finalShippingCents: number;
  savingsCents: number;
  usedSellerQuote: boolean;
}

export function resolveQuoteFinalPrice(input: QuoteResolutionInput): QuoteResolutionResult {
  const { maxShippingCents, quotedShippingCents, penaltyDiscountPercent } = input;
  const penaltyPrice = Math.round(maxShippingCents * (1 - penaltyDiscountPercent / 100));

  let finalShippingCents: number;
  let usedSellerQuote: boolean;

  if (quotedShippingCents !== null) {
    finalShippingCents = Math.min(quotedShippingCents, penaltyPrice);
    usedSellerQuote = quotedShippingCents <= penaltyPrice;
  } else {
    finalShippingCents = penaltyPrice;
    usedSellerQuote = false;
  }

  return {
    finalShippingCents,
    savingsCents: maxShippingCents - finalShippingCents,
    usedSellerQuote,
  };
}
```

## 8. SPEC GAPS AND DECISIONS NEEDED

### SPEC GAP 1: Quote Status Type

The `combinedShippingQuote.status` column is typed as `text('status')` (not an enum). The schema doc lists the valid values as a comment: `'PENDING_SELLER' | 'PENDING_BUYER' | 'ACCEPTED' | 'DISPUTED' | 'EXPIRED' | 'PENALTY_APPLIED'`. Since the schema is already deployed, use a TypeScript union type constant to enforce valid values in code, but do NOT create a new Drizzle enum (that would require a migration for an existing column).

**Resolution:** Define a `QUOTE_STATUSES` constant in the validation or types file:
```typescript
export const QUOTE_STATUSES = ['PENDING_SELLER', 'PENDING_BUYER', 'ACCEPTED', 'DISPUTED', 'EXPIRED', 'PENALTY_APPLIED'] as const;
export type QuoteStatus = typeof QUOTE_STATUSES[number];
```

### SPEC GAP 2: Stripe PaymentIntent Adjustment

The Feature Lock-in Addendum says "Stripe auth hold at MAX shipping (sum of individual rates)" for QUOTED mode. The existing checkout flow creates a PaymentIntent for the order total. When the quote resolves (buyer accepts a lower price), the Stripe charge amount needs to be adjusted. This adjustment mechanism is NOT detailed in the QUOTED shipping spec.

**Resolution for D2.2:** D2.2 only updates `order.shippingCents` and `order.totalCents` in the database. The Stripe capture amount adjustment (capturing less than the authorized amount) is handled by the existing finalize-order flow at capture time. If the existing finalize-order reads `order.totalCents` at capture time, no additional Stripe work is needed for D2.2.

**OWNER DECISION NEEDED:** Confirm that the existing finalize-order/payment capture flow reads `order.totalCents` at capture time (not the original authorized amount). If it uses the authorized amount, additional work is needed.

### SPEC GAP 3: BullMQ Queue Registration

The codebase may or may not have an existing BullMQ queue registration pattern. The spec calls for a repeatable job. The installer should check for existing BullMQ setup in `src/lib/jobs/` or similar and follow the same pattern.

**Resolution:** If no BullMQ infrastructure exists yet, create a minimal setup that can be expanded later. The job is simple (query + update + notify loop) and can alternatively be implemented as a Next.js cron API route (`/api/cron/shipping-quote-deadline`) that runs every 15 minutes via external cron (Railway cron or GitHub Actions).

### SPEC GAP 4: Notification Template Bodies

The canonical docs do not specify exact email/notification copy for shipping quote notifications. The installer should follow the pattern of existing templates (subject line + body with interpolated variables) and use language consistent with the Payout UX Language Pack (no "balance", "wallet", etc.).
