### 3A. Database: Schema Changes

Add two columns to `localTransaction` in `src/lib/db/schema/local.ts`:

```typescript
// Photo evidence (A13)
meetupPhotoUrls:     text('meetup_photo_urls').array().notNull().default(sql`'{}'::text[]`),
meetupPhotosAt:      timestamp('meetup_photos_at', { withTimezone: true }),
```

Per Addendum A13:
- `meetupPhotoUrls`: text array, default empty `[]`. Stores R2 URLs of condition photos taken by buyer before receipt confirmation.
- `meetupPhotosAt`: nullable timestamp. Set when photos are uploaded. Null if buyer skipped photos.

Create a Drizzle migration for these two columns via `ALTER TABLE local_transaction ADD COLUMN`.

**Schema doc note:** These columns are specified in Addendum A13 but NOT yet in the schema doc (v2.1.1). The schema doc MUST be updated to include them in Section 19.1 after implementation. The installer should note this in their report but proceed with implementation -- the addendum takes precedence per "Where conflict exists, this addendum wins."

### 3C. Upload API Route: Add `meetup-photo` Type

Modify `src/app/api/upload/route.ts` to support `type: 'meetup-photo'`:

- CASL gate: `ability.can('update', sub('LocalTransaction', { buyerId: session.userId }))` -- only the buyer can upload meetup photos.
- Require `localTransactionId` from form data.
- Validate that the localTransaction exists and belongs to the requesting buyer.
- Validate transaction status: photos can be uploaded when status is `BOTH_CHECKED_IN` or `ADJUSTMENT_PENDING` (buyer is inspecting item before confirming). NOT after `RECEIPT_CONFIRMED` or `COMPLETED`.
- Max 5 photos per transaction (read from the existing photo URLs on the transaction row; if count >= 5, reject).
- Route to `uploadMeetupPhoto()` from image service.
- Response shape: `{ success: true, image: { id, url, position } }`

### 3D. Server Action: Upload Meetup Photos

Create `src/lib/actions/local-photo-evidence.ts` (`'use server'`):

**Action 1: `uploadMeetupPhotosAction`**

```typescript
export async function uploadMeetupPhotosAction(
  data: z.infer<typeof uploadMeetupPhotosSchema>
): Promise<{ success: boolean; photoUrls?: string[]; error?: string }>
```

- Input: `{ localTransactionId: string, photoUrls: string[] }` (URLs from the upload API -- photos already uploaded to R2 via the upload route; this action links them to the transaction)
- Validate: `authorize()`, Zod `.strict()`, buyer owns transaction, status is `BOTH_CHECKED_IN` or `ADJUSTMENT_PENDING`
- Appends photoUrls to existing `meetupPhotoUrls` array (deduplication: skip URLs already in array)
- Sets `meetupPhotosAt` to now (only on first photo upload; subsequent uploads don't overwrite the timestamp)
- Max total 5 URLs enforced (existing + new <= 5)
- Returns updated `photoUrls` array
- `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`

**Action 2: `removeMeetupPhotoAction`**

```typescript
export async function removeMeetupPhotoAction(
  data: z.infer<typeof removeMeetupPhotoSchema>
): Promise<{ success: boolean; error?: string }>
```

- Input: `{ localTransactionId: string, photoUrl: string }`
- Validate: `authorize()`, Zod `.strict()`, buyer owns transaction, status is `BOTH_CHECKED_IN` or `ADJUSTMENT_PENDING` (cannot remove after confirmation)
- Removes the URL from `meetupPhotoUrls` array
- If array becomes empty, reset `meetupPhotosAt` to null
- Does NOT delete the R2 object (photos are evidence -- never deleted even if buyer removes from UI before confirming)
- `revalidatePath` both order paths

## 5. ACCEPTANCE CRITERIA

### Positive Cases

1. **Photo upload works:** Buyer with status BOTH_CHECKED_IN can upload up to 5 photos to R2 (or local filesystem in dev) and they are linked to `localTransaction.meetupPhotoUrls`.
2. **Timestamp recorded:** After first photo upload, `localTransaction.meetupPhotosAt` is set to the upload time and does not change on subsequent uploads.
3. **Photo removal works:** Buyer can remove a photo from the transaction before confirming receipt; the URL is removed from `meetupPhotoUrls` but the R2 object persists.
4. **Photo display after confirmation:** After receipt is confirmed, both buyer and seller see the condition photos in the transaction record with the timestamp.
5. **Skip works:** Buyer can proceed to QR scan / code entry and confirm receipt without uploading any photos. Transaction completes normally with `meetupPhotoUrls: []` and `meetupPhotosAt: null`.
6. **Claim context helper:** `getMeetupPhotoContext(localTransactionId)` returns correct photo status for claim integration.
7. **Upload route accepts meetup-photo type:** POST to `/api/upload` with `type=meetup-photo` and `localTransactionId` succeeds for authenticated buyer.

### Negative Cases

8. **Seller cannot upload:** Seller attempting to upload meetup photos receives an error (CASL check fails on buyer ownership).
9. **Unauthenticated upload rejected:** Request without session returns 401.
10. **Wrong status rejected:** Upload attempt when status is SCHEDULED, SELLER_CHECKED_IN, BUYER_CHECKED_IN (only one party checked in), RECEIPT_CONFIRMED, COMPLETED, CANCELED, or NO_SHOW returns an error.
11. **Max photos enforced:** Attempt to upload a 6th photo returns an error with message indicating max reached.
12. **Post-confirmation removal blocked:** Attempt to remove a photo after `confirmedAt` is set returns an error.
13. **Non-owner rejected:** Buyer A cannot upload photos to Buyer B's transaction.
14. **Invalid file type rejected:** Non-image files (PDF, video, etc.) rejected by existing image validation.

### Data Integrity

15. **meetupPhotoUrls default:** New localTransaction rows default to empty array `[]`.
16. **meetupPhotosAt default:** New localTransaction rows default to `null`.
17. **Existing transactions unaffected:** Existing localTransaction rows get the default empty array and null timestamp via ALTER TABLE defaults.

### Vocabulary

18. **No banned terms** appear in any new or modified file.

## 9. OWNER DECISIONS NEEDED

### Decision #1: Maximum Photos Per Transaction

NOT SPECIFIED in Addendum A13. The addendum says photos are "timestamped and stored to R2, linked to localTransaction.meetupPhotoUrls" but does not specify a limit.

**Prompt default: 5 photos.** Rationale: matches common marketplace patterns (eBay allows 24, Poshmark 16 for listings; 5 is reasonable for quick meetup condition shots). This is a platform setting candidate if the owner wants it configurable.

**Owner question:** Is 5 photos correct? Should this be a platform_setting key (e.g., `commerce.local.maxMeetupPhotos: 5`) or hardcoded? If it should be a platform setting, the installer should add a seed entry.

### Decision #2: Upload via Existing Route vs. Dedicated Route

The prompt specifies using the existing `/api/upload` route with a new `type: 'meetup-photo'` value. Alternative: create a dedicated `/api/local/meetup-photos` route.

**Prompt default: existing route.** Rationale: the upload route already handles validation, rate limiting, R2/local fallback, and auth. Adding a type is consistent with the avatar/banner/receipt pattern.

**Owner question:** Confirm using existing upload route pattern.

### Decision #3: R2 Object Persistence on Photo Removal

The prompt specifies that R2 objects are NOT deleted when a buyer removes a photo from the UI before confirming. This means orphaned R2 objects exist if a buyer uploads and then removes.

**Prompt default: keep forever.** Rationale: photos are potential evidence. If a buyer uploads evidence of item damage, removes it, then later files a claim, staff might want to recover the original photos. R2 storage cost is negligible for individual photos.

**Owner question:** Confirm photos are never deleted from R2 (even on UI removal).

### Decision #4: Schema Doc Update

The schema doc (v2.1.1) does NOT include `meetupPhotoUrls` or `meetupPhotosAt` in the localTransaction definition. The addendum A13 specifies them. The installer will add the columns to the codebase schema, but the schema doc itself needs updating.

**Owner action:** Update `TWICELY_V3_SCHEMA_v2_1_0.md` Section 19.1 to include these columns, or delegate this to the installer as part of the prompt.

```typescript
'use server';

import { authorize, sub } from '@/lib/casl';
import { z } from 'zod';
import { revalidatePath } from 'next/cache';

export async function someAction(
  data: z.infer<typeof someSchema>
): Promise<{ success: boolean; error?: string }> {
  const { session, ability } = await authorize();
  if (!session) return { success: false, error: 'Unauthorized' };

  const parsed = someSchema.safeParse(data);
  if (!parsed.success) {
    return { success: false, error: parsed.error.issues[0]?.message ?? 'Invalid input' };
  }

  // Fetch transaction, verify ownership, check status...

  if (session.userId !== transaction.buyerId) return { success: false, error: 'Not found' };

  if (!ability.can('update', sub('LocalTransaction', { buyerId: transaction.buyerId }))) {
    return { success: false, error: 'Not found' };
  }

  // Business logic...

  revalidatePath('/my/buying/orders');
  revalidatePath('/my/selling/orders');
  return { success: true };
}
```

```typescript
  buffer: Buffer,
  position: number
): Promise<UploadResult> {
  const bytes = new Uint8Array(buffer);
  const validation = validateImageBytes(bytes, buffer.length);
  if (!validation.valid) {
    return { success: false, error: validation.error };
  }

  const imageType = detectImageType(bytes);
  if (!imageType) {
    return { success: false, error: 'Unable to detect image type' };
  }

  const ext = getExtension(imageType);
  const contentType = `image/${imageType}`;
  const timestamp = Date.now();
  const key = `meetup-photos/${localTransactionId}/${position}-${timestamp}.${ext}`;

  try {
    const url = await uploadToR2(key, buffer, contentType);
    return { success: true, url };
  } catch (error) {
    return {
      success: false,
      error: error instanceof Error ? error.message : 'Upload failed',
    };
  }
}
```

## 12. SPEC INCONSISTENCIES TO NOTE

1. **Schema doc vs. Addendum:** The schema doc (v2.1.1) does not include `meetupPhotoUrls` or `meetupPhotosAt`. The addendum A13 specifies them. Per addendum header: "Where conflict exists, this addendum wins." Installer proceeds with addendum spec.

2. **Max photo count unspecified:** Addendum A13 does not specify a maximum number of photos. The prompt assumes 5. Owner should confirm.

3. **`meetupPhotoUrls` Drizzle syntax:** The addendum shows `text[].default([])` which is pseudocode. The actual Drizzle syntax is `text('meetup_photo_urls').array().notNull().default(sql\`'{}'::text[]\`)` matching the existing `localHandlingFlags` pattern on the listing table.

4. **Status gate for photo upload:** The addendum says "Before confirming receipt (scanning QR or entering code), buyer is prompted to take condition photos." This implies `BOTH_CHECKED_IN` is the primary status. The prompt also allows `ADJUSTMENT_PENDING` because during a price adjustment the buyer is still inspecting the item. If the owner disagrees, remove ADJUSTMENT_PENDING from the allowed statuses.
