# G10.9 — Become Seller CTA Page (`/become-seller`)

**Phase & Step:** G10.9
**Feature Name:** Become Seller CTA Page
**One-line Summary:** A public marketing page at `/become-seller` that explains selling on Twicely, shows the pricing preview for all three subscription axes (Store, Crosslister, Automation), and routes visitors to the correct next action based on their authentication and seller status.
**Canonical Sources:**
- `TWICELY_V3_USER_MODEL.md` — seller activation model (NO registration flow; first listing activates selling; PERSONAL → BUSINESS upgrade flow)
- `TWICELY_V3_PRICING_PAYMENTS_PAYOUT_CANONICAL_v3_2.md` — Store tier pricing (§4), Crosslister pricing (§6), Automation pricing (§8), Bundle pricing (§9), TF marketing headline (§2.5)
- `TWICELY_V3_PAGE_REGISTRY.md` — route prefix conventions; existing routes; layout types
- `TWICELY_V3_FEATURE_LOCKIN_ALL_DOMAINS.md` — seller first-run onboarding (§13), empty states, 404/error patterns (§28)
- `TWICELY_V3_TESTING_STANDARDS.md` — test patterns

## CRITICAL SPEC FINDING — READ BEFORE STARTING

The User Model Canonical (§3, "Activation Rule") explicitly states:

> "There is NO 'Become a Seller' registration flow"
> "A user becomes a seller by action, not by registration."
> "First listing automatically enables selling"

This means the page is a **marketing/conversion page** (similar to `/about` or `/pricing`), NOT a registration funnel with a form. It has NO server action, NO database write, NO seller-activation logic. Its only job is to:

1. Explain how selling works on Twicely
2. Show the subscription tier pricing preview (read from `platform_settings`)
3. Route the visitor to the correct CTA based on their status

### 2.2 Page: `src/app/(marketplace)/become-seller/page.tsx`

**Gate:** PUBLIC — no auth required. However, the page MUST detect auth state to show the correct CTA.

**Metadata:**
```
title: 'Start Selling on Twicely | Twicely'
description: 'Turn your closet into income. Free to list, simple fees, and built-in tools to help you sell more.'
```

**Data loaded at render time (server-side):**
- Session via `auth.api.getSession()` (from `@/lib/auth`) — to determine auth/seller state for CTA routing
- If `session?.user?.id` and `isSeller === true`: additionally load `sellerProfile.sellerType` via a DB query
- Pricing data from `platform_setting` via `getPlatformSettingsByPrefix('store.pricing')`, `getPlatformSettingsByPrefix('crosslister.pricing')`, `getPlatformSettingsByPrefix('automation.pricing')`, `getPlatformSettingsByPrefix('bundle.')`, `getPlatformSettingsByPrefix('fees.insertion')`, `getPlatformSettingsByPrefix('fees.freeListings')` — to render real pricing, not hardcoded values

All DB calls MUST be parallelized via `Promise.all()`.

**CTA Routing Logic (server-resolved, passed as props to client components):**

| Visitor state | Primary CTA label | Primary CTA href | Notes |
|---|---|---|---|
| Guest (not authenticated) | "Sign up and start selling" | `/auth/signup` | Also show secondary: "Log in" → `/auth/login` |
| Authenticated, `isSeller === false` | "Create your first listing" | `/my/selling/listings/new` | First listing activates selling — no separate registration |
| Authenticated, `isSeller === true`, `sellerType === 'PERSONAL'` | "Go to my listings" | `/my/selling/listings` | Already a seller; show upgrade nudge to BUSINESS |
| Authenticated, `isSeller === true`, `sellerType === 'BUSINESS'` | "Go to selling dashboard" | `/my/selling` | Already a business seller; link to subscription page for upgrades |

The CTA routing logic is PURE — no server action, no DB write, no mutation. It is read-only.

### Gap 1: Route not in Page Registry (BLOCKING)

### Gap 2: "No Become a Seller registration flow" vs. conversion funnel
**Problem:** The User Model Canonical (§3) explicitly says there is no "Become a Seller" registration flow. The Build Sequence Tracker describes this as a "conversion funnel with benefits, pricing preview, one-click upgrade." **Resolution applied in this prompt:** The page is a marketing page that routes to listing creation (which activates selling). The page does NOT implement any seller registration form or activation server action. This interpretation is correct per the User Model canonical.

### Gap 3: Annual vs. monthly pricing display
**Problem:** The pricing canonical shows both annual/mo and monthly prices. The platform settings have both `*annualCents` and `*monthlyCents` keys. The page registry does not specify which to show by default.

**Recommended resolution:** Show MONTHLY prices as the default with a toggle or "(save X% annually)" callout. The installer should use monthly prices from `*.monthlyCents` keys as the primary display and show annual option as secondary. If owner wants annual-first, they should specify.
