### 3.2 Queries -- 2 new query files

#### A. `src/lib/queries/explore.ts` (~200 lines)

This is the core of G3.9. Four query functions, one for each Explore section per Personalization Canonical Section 5:

**1. `getTrendingListings(limit?: number): Promise<ListingCardData[]>`**

"Trending Now -- highest velocity items across all categories" (Personalization Canonical Section 5).

Trending is defined by velocity: the combination of recent sales + watchlist adds + browsing views within a configurable window. There is NO trending_score table in the schema -- compute at query time.

Implementation approach (V1 -- pure SQL, no materialized view):
- Count recent orders containing each listing in the last N days (from `orderItem` JOIN `order` WHERE `order.createdAt > now() - interval`)
- Count recent watchlist adds (from `watchlistItem` WHERE `createdAt > now() - interval`)
- Count recent browsing views (from `browsingHistory` WHERE `lastViewedAt > now() - interval`)
- Score: `(orderCount * 10) + (watchlistCount * 3) + (viewCount * 1)` -- server-side weights, no frontend calculation
- Filter: only ACTIVE listings, exclude REMOVED/SOLD
- Order by score DESC, then by `listing.createdAt` DESC as tiebreaker
- Read `discovery.explore.trendingWindowDays` (default 7) and `discovery.explore.trendingLimit` (default 24) from platform settings
- Return `ListingCardData[]` using `mapToListingCard()`

IMPORTANT: The trending weights (10/3/1) are hardcoded V1 approximations. They are NOT in the platform settings canonical. The installer should define them as constants at the top of the file with a comment: `// V1 trending weights -- move to platform_settings in Phase H if tuning is needed`.

IMPORTANT: The build tracker says "materialized view" for trending. A true PG materialized view is overkill for V1 and not in the schema doc. Instead, use a standard Drizzle query with subqueries/CTEs. If performance becomes an issue at scale, a materialized view can be added later (Personalization Canonical Section 13 mentions this: "If performance degrades at scale, materialize user_weights as a cached view refreshed hourly").

**2. `getStaffPickCollections(): Promise<ExploreCollection[]>`**

"Staff Picks -- editorially curated (admin tool)" (Personalization Canonical Section 5).

- Query `curatedCollection` WHERE `isPublished = true` AND (`startDate IS NULL OR startDate <= now()`) AND (`endDate IS NULL OR endDate > now()`)
- Order by `sortOrder ASC`
- For each collection, fetch up to 12 listings from `curatedCollectionItem` JOIN `listing` WHERE `listing.status = 'ACTIVE'`
- Return type:
```typescript
export interface ExploreCollection {
  id: string;
  title: string;
  slug: string;
  description: string | null;
  coverImageUrl: string | null;
  listings: ListingCardData[];
}
```
- Use `mapToListingCard()` for listing cards

NOTE: G3.10 builds the admin curation tool at `hub.twicely.co/mod/collections`. In G3.9, collections are query-only. The seed data may have zero published collections, so the section should gracefully hide when empty.

**3. `getSeasonalCollections(): Promise<ExploreCollection[]>`**

"Seasonal -- time-based features (back to school, holiday, etc.)" (Personalization Canonical Section 5).

Same query as staff picks but filtered: `startDate IS NOT NULL AND startDate <= now() AND endDate IS NOT NULL AND endDate > now()`. Seasonal collections have BOTH date bounds set. Staff picks have neither (or only startDate).

If this returns the same results as staff picks (because the date filters overlap), that's fine -- the component handles deduplication by collection ID.

NOTE: In practice, there may be zero seasonal collections at launch. The section hides when empty.

**4. `getRisingSellers(limit?: number): Promise<RisingSellerData[]>`**

"Rising Sellers -- new sellers with early traction" (Personalization Canonical Section 5).

- "New sellers with early traction" = sellers whose `user.createdAt` is within the last 90 days AND who have at least 3 ACTIVE listings AND at least 1 completed order
- Query `sellerProfile` JOIN `user` JOIN `sellerPerformance`
- Filter: `user.createdAt > now() - interval '90 days'`
- Subquery: `listing` count WHERE `ownerUserId = user.id AND status = 'ACTIVE'` >= 3
- Subquery: `order` count WHERE `sellerId = user.id AND status = 'COMPLETED'` >= 1
- Order by: total ACTIVE listing count DESC (most prolific new sellers first)
- Limit from `discovery.explore.risingSellerLimit` (default 8)
- Return type:
```typescript
export interface RisingSellerData {
  userId: string;
  storeName: string | null;
  storeSlug: string | null;
  avatarUrl: string | null;
  performanceBand: string;
  listingCount: number;
  followerCount: number;
  memberSince: Date;
}
```

Per Personalization Canonical Section 10: "Boosted listings ALWAYS appear in 'Explore' tab (not interest-filtered)."

## 5. ACCEPTANCE CRITERIA

### Positive Cases

1. **Guest user sees Explore as default:** Navigate to `/` without authentication. The Explore tab is the default. Trending, staff picks (if any published), rising sellers sections are visible.
2. **Authenticated user with interests sees For You as default:** Log in as a user with `userInterest` rows. Navigate to `/`. The "For You" tab is default. Personalized content appears.
3. **Authenticated user without interests sees Explore as default:** Log in as a user with zero `userInterest` rows. Navigate to `/`. The "Explore" tab is default. The "For You" tab shows a CTA to pick interests.
4. **Trending section shows velocity-ordered listings:** Trending listings are ordered by the composite velocity score (sales > watchlist > views), not by recency alone.
5. **Staff picks render correctly:** When `curatedCollection` rows exist with `isPublished = true`, they appear as horizontal scrollable rows with collection title headings.
6. **Empty sections hide gracefully:** If no curated collections exist, the Staff Picks section does not render. If no new sellers exist, Rising Sellers does not render.
7. **Rising sellers link to storefronts:** Each rising seller card links to `/st/{storeSlug}`.
8. **Promoted listings appear on Explore unfiltered:** Active promoted listings appear in the Promoted section regardless of any user's interest profile.
9. **Tab URL persistence:** Clicking a tab updates the URL query parameter (`/?tab=explore`, `/?tab=foryou`, `/?tab=categories`). Direct navigation to `/?tab=foryou` opens the For You tab.
10. **Standalone `/explore` page works:** Navigation to `/explore` renders Explore content without tabs. PUBLIC access (no login required).
11. **Categories tab shows full catalog:** The Categories tab renders the same content as the current homepage (category grid + recent listings).

### Negative Cases

12. **Explore content is NOT personalized:** The Explore sections (trending, staff picks, seasonal, rising sellers) show the same content for ALL users. No interest filtering.
13. **Search is NOT affected:** Search at `/s` continues to work identically. No interest filtering on search.
14. **Category pages are NOT affected:** `/c/{slug}` pages continue to show all listings in that category. No interest filtering.
15. **Guest users cannot see For You tab:** When not authenticated, the "For You" tab is hidden (not shown at all, not just disabled).
16. **No `as any` or `@ts-ignore`:** Zero type violations in all new/modified files.
17. **No files over 300 lines:** All new and modified files are 300 lines or fewer.
18. **No banned terms:** Zero occurrences of banned terms in all new/modified files.

### Data Integrity

19. **Only ACTIVE listings in trending:** Trending query filters `listing.status = 'ACTIVE'`. No SOLD, DRAFT, ENDED, REMOVED, or RESERVED listings appear.
20. **Only published collections shown:** Staff picks query filters `curatedCollection.isPublished = true`.
21. **Seasonal date bounds enforced:** Seasonal collections only show when `now()` is between `startDate` and `endDate`.
22. **Platform settings read correctly:** `trendingWindowDays` and `trendingLimit` are read from `platform_settings`, not hardcoded.
