### 2C. Receipt OCR Service (NEW FILE)

Create `src/lib/finance/receipt-ocr.ts`:

**Extracted receipt data shape (matches `expense.receiptDataJson`):**
```typescript
export interface ReceiptOcrResult {
  vendor: string | null;
  amountCents: number | null;
  date: string | null;            // ISO date string
  suggestedCategory: string | null; // One of EXPENSE_CATEGORIES
  confidence: number;             // 0-1 float
  rawText: string | null;         // Full OCR text for debugging
}
```

**Service function:**
```typescript
export async function extractReceiptData(
  imageUrl: string,
): Promise<ReceiptOcrResult>
```

**Logic:**
1. Read `RECEIPT_OCR_PROVIDER_URL` and `RECEIPT_OCR_API_KEY` from `process.env`
2. If either is missing, return a mock result:
   ```typescript
   {
     vendor: null,
     amountCents: null,
     date: null,
     suggestedCategory: null,
     confidence: 0,
     rawText: null,
   }
   ```
3. If configured, send a `fetch()` POST to the provider URL with:
   - The image URL in the request body
   - A system prompt instructing the AI to extract: vendor name, total amount (in dollars), date, and suggest one of the 16 expense categories
   - Parse the AI response JSON into `ReceiptOcrResult`
   - Convert the extracted dollar amount to integer cents
   - Catch errors and return the mock result on failure (log the error)
4. The prompt sent to the AI must include the 16 expense categories from `EXPENSE_CATEGORIES` so the AI can suggest one

**Platform settings key:** `finance.receiptScanAiCredits: 1` (from Financial Center Canonical Section 9). This is informational for now -- credit tracking is deferred. The platform setting exists for future gating.

### 2H. Mileage Entry CRUD Actions (NEW FILE)

Create `src/lib/actions/finance-center-mileage.ts`:

**`'use server'` file with these exports:**

```typescript
export async function createMileageAction(input: unknown): Promise<CreateMileageResponse>
export async function updateMileageAction(input: unknown): Promise<UpdateMileageResponse>
export async function deleteMileageAction(input: unknown): Promise<DeleteMileageResponse>
export async function listMileageAction(input: unknown): Promise<ListMileageResponse>
```

**Each action must:**
1. Parse input with Zod `.strict()` schema
2. Call `authorize()` to get `{ ability, session }`
3. Return `{ success: false, error: 'Unauthorized' }` if no session
4. Resolve `userId` via delegation pattern: `session.delegationId ? session.onBehalfOfSellerId! : session.userId`
5. Check CASL: `ability.can('create'/'update'/'delete'/'read', sub('MileageEntry', { userId }))`
6. Check `getFinanceTier(userId) === 'PRO'` -- mileage is PRO-only
7. Use explicit field mapping (never spread input)
8. `revalidatePath('/my/selling/finances')` and `revalidatePath('/my/selling/finances/mileage')`

**createMileageAction specifics:**
- Read IRS rate from `getPlatformSetting('finance.mileageRatePerMile', 0.70)`
- Calculate `deductionCents = Math.round(miles * ratePerMile * 100)` (server-side only)
- Store both `miles`, `ratePerMile`, and calculated `deductionCents` on the row
- The user provides `description`, `miles`, `tripDate` only. The rate and deduction are server-computed.

**updateMileageAction specifics:**
- If `miles` changes, recalculate `deductionCents` using the stored `ratePerMile` (do NOT re-read platform setting -- the rate at creation time is the rate that applies)
- Verify ownership via `getMileageById(userId, id)` before update

**deleteMileageAction specifics:**
- Verify ownership via `getMileageById(userId, id)` before delete

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- NEVER use "wallet" -- use "payout" or "earnings"
- NEVER use "Twicely Balance" -- use "Available for payout"
- NEVER use "FVF" or "Final Value Fee" -- use "Transaction Fee" or "TF"
- NEVER use "Withdraw" -- use "Request payout"

### Technology Constraints
- NEVER install `ai`, `@ai-sdk/*`, `openai`, or `anthropic` npm packages
- Receipt OCR uses raw `fetch()` to a configurable provider URL
- File storage uses Cloudflare R2 (via existing `r2-client.ts`) -- NEVER use MinIO or S3 direct
- NEVER use Prisma -- all DB operations via Drizzle ORM

### Business Logic Constraints
- Receipt scanning and mileage tracking are BOTH PRO-only (Financial Center Canonical Section 4)
- The IRS mileage rate comes from `getPlatformSetting('finance.mileageRatePerMile', 0.70)` -- NEVER hardcode the rate
- Deduction calculation (`miles * rate * 100 => cents`) is SERVER-SIDE ONLY -- the client shows a preview but the server computes the authoritative value
- OCR results stored in `receiptDataJson` are informational -- they NEVER auto-overwrite user-entered expense fields
- Staff with `finance.view` delegation can READ mileage entries but CANNOT create/update/delete them
- Ownership is via `userId` always -- `mileageEntry.userId = session.userId` (or `onBehalfOfSellerId` for delegation)
- Money values in integer cents only (`deductionCents`, `amountCents`)
- `mileageEntry.ratePerMile` and `mileageEntry.miles` are `real` (float) columns in the schema -- these are the ONLY float columns, used because they represent non-monetary values (rate per mile and distance). The deduction is stored as integer cents.
- Max 300 lines per file
- No `as any`, no `@ts-ignore`, no `@ts-expect-error`
- No `console.log` in production code (use `logger` from `@/lib/logger` if needed)
- All Zod schemas use `.strict()` to reject unknown keys

### What NOT to Build
- Do NOT build recurring mileage entries (mileage is per-trip, not recurring)
- Do NOT build AI credit tracking/billing (deferred -- platform setting exists for future use)
- Do NOT build CSV export for mileage (deferred to D4.3)
- Do NOT build QuickBooks/Xero sync (deferred to G10.3)
- Do NOT add new CASL subjects (MileageEntry already exists)
- Do NOT create new database tables or columns (schema is complete)
- Do NOT modify the mileageEntry schema

## 4. ACCEPTANCE CRITERIA

### Positive Cases
1. A seller with Finance PRO can upload a receipt photo (JPEG/PNG/WebP, max 20MB) when creating an expense, and the receipt URL is stored in `expense.receiptUrl`
2. After receipt upload, the AI OCR service is called and results are stored in `expense.receiptDataJson` as a JSON object matching `ReceiptOcrResult`
3. The expense form shows OCR-suggested values (vendor, amount, date, category) with "Apply" buttons that fill the form fields
4. The user can ignore OCR suggestions -- user-entered values always take precedence over OCR
5. A seller with Finance PRO can create a mileage entry with description, miles, and trip date
6. The `deductionCents` is calculated server-side as `Math.round(miles * ratePerMile * 100)` using the IRS rate from platform settings
7. The mileage entry stores the IRS rate that was active at creation time (`ratePerMile` column)
8. A seller can update a mileage entry -- if miles change, deduction is recalculated using the STORED rate (not current platform setting)
9. A seller can delete their own mileage entry
10. The mileage page at `/my/selling/finances/mileage` shows a paginated list of trips with summary cards
11. The mileage page shows the FinanceProGate for FREE tier users
12. The finance dashboard page has a "Mileage" quick link for PRO users
13. The P&L summary already displays mileage deductions (via existing `PnlSummary` component) -- verify this continues to work

### Negative Cases
14. A FREE tier user cannot create/update/delete mileage entries (returns "Upgrade to Finance Pro" error)
15. A FREE tier user cannot upload receipts (expense CRUD already gates on PRO)
16. An unauthenticated user cannot access `/my/selling/finances/mileage` (redirects to login)
17. A delegated staff member with `finance.view` scope can READ mileage entries but CANNOT create/update/delete them
18. A user cannot create a mileage entry for another user (CASL `userId` condition enforced)
19. Invalid input (negative miles, empty description, missing tripDate) is rejected by Zod validation

### Data Integrity
20. All monetary deductions stored as integer cents in `mileageEntry.deductionCents`
21. The IRS rate is read from `platform_settings` with key `finance.mileageRatePerMile`, never hardcoded
22. Receipt photos are stored in R2 with key pattern `receipts/{userId}/{timestamp}.{ext}`
23. OCR results in `receiptDataJson` match the `ReceiptOcrResult` interface shape

### Vocabulary
24. No banned terms appear in any UI text or code comments
25. The mileage page title uses "Mileage" (not "Mileage Tracker" in the `<title>` tag -- but can use "Mileage Tracker" as a heading)

```typescript
// src/lib/finance/receipt-ocr.ts

import { EXPENSE_CATEGORIES } from '@/lib/validations/finance-center';

export interface ReceiptOcrResult {
  vendor: string | null;
  amountCents: number | null;
  date: string | null;
  suggestedCategory: string | null;
  confidence: number;
  rawText: string | null;
}

const MOCK_RESULT: ReceiptOcrResult = {
  vendor: null,
  amountCents: null,
  date: null,
  suggestedCategory: null,
  confidence: 0,
  rawText: null,
};

export async function extractReceiptData(
  imageUrl: string,
): Promise<ReceiptOcrResult> {
  const providerUrl = process.env.RECEIPT_OCR_PROVIDER_URL;
  const apiKey = process.env.RECEIPT_OCR_API_KEY;

  if (!providerUrl || !apiKey) {
    return MOCK_RESULT;
  }

  try {
    const response = await fetch(providerUrl, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${apiKey}`,
      },
      body: JSON.stringify({
        imageUrl,
        categories: [...EXPENSE_CATEGORIES],
      }),
    });

    if (!response.ok) {
      return MOCK_RESULT;
    }

    const data = await response.json() as {
      vendor?: string;
      amount?: number;
      date?: string;
      category?: string;
      confidence?: number;
      rawText?: string;
    };

    // Convert dollar amount to integer cents
    const amountCents = typeof data.amount === 'number'
      ? Math.round(data.amount * 100)
      : null;

    // Validate suggested category is in our list
    const validCategory = EXPENSE_CATEGORIES.includes(
      data.category as typeof EXPENSE_CATEGORIES[number]
    )
      ? data.category ?? null
      : null;

    return {
      vendor: data.vendor ?? null,
      amountCents,
      date: data.date ?? null,
      suggestedCategory: validCategory,
      confidence: typeof data.confidence === 'number' ? data.confidence : 0,
      rawText: data.rawText ?? null,
    };
  } catch {
    return MOCK_RESULT;
  }
}
```

```typescript
/**
 * Upload an expense receipt image.
 */
export async function uploadReceiptImage(
  userId: string,
  buffer: Buffer
): Promise<UploadResult> {
  const bytes = new Uint8Array(buffer);
  const validation = validateImageBytes(bytes, buffer.length);
  if (!validation.valid) {
    return { success: false, error: validation.error };
  }

  const imageType = detectImageType(bytes);
  if (!imageType) {
    return { success: false, error: 'Unable to detect image type' };
  }

  const ext = getExtension(imageType);
  const contentType = `image/${imageType}`;
  const timestamp = Date.now();
  const key = `receipts/${userId}/${timestamp}.${ext}`;

  try {
    const url = await uploadToR2(key, buffer, contentType);
    return { success: true, url };
  } catch (error) {
    console.error('[ImageService] Failed to upload receipt:', error);
    return {
      success: false,
      error: error instanceof Error ? error.message : 'Upload failed',
    };
  }
}
```
