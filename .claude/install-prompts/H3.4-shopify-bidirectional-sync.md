### 2.3 Shopify Webhook HMAC Verification

**File:** `src/lib/crosslister/connectors/shopify-webhook-verify.ts`

Shopify sends webhook signatures via the `X-Shopify-Hmac-Sha256` header. The HMAC is computed over the raw request body using the app's client secret as the key. The result is Base64-encoded (NOT hex like Whatnot).

**Algorithm (per Shopify docs):**
1. Read the raw request body as a string (before JSON parsing)
2. Read `X-Shopify-Hmac-Sha256` header
3. Load the Shopify client secret from `platform_settings` key `crosslister.shopify.clientSecret`
4. Compute HMAC-SHA256 of the raw body using the client secret
5. Base64-encode the HMAC digest
6. Timing-safe compare the computed value against the header value

**Key difference from Whatnot:** Shopify uses Base64 encoding (not hex) and reads the secret from `crosslister.shopify.clientSecret` (same secret used for OAuth HMAC in the callback route). Whatnot uses a separate `crosslister.whatnot.webhookSecret` key.

**Exports:**
- `SHOPIFY_SIGNATURE_HEADER` = `'X-Shopify-Hmac-Sha256'`
- `verifyShopifyWebhookSignature(rawBody: string, signatureHeader: string): Promise<{ valid: boolean; error?: string }>`

**Pattern:** Mirror `whatnot-webhook-verify.ts` exactly but with Base64 encoding instead of hex.

### 2.6 Webhook Route

**File:** `src/app/api/crosslister/shopify/webhook/route.ts`

**Endpoint:** `POST /api/crosslister/shopify/webhook`
**Authentication:** HMAC-SHA256 signature verification ONLY. No session/CASL auth -- this is a platform-to-platform webhook.

**Headers consumed:**
- `X-Shopify-Hmac-Sha256` -- HMAC signature for verification
- `X-Shopify-Topic` -- Event type (e.g., `products/update`, `orders/create`)
- `X-Shopify-Shop-Domain` -- The `{shop}.myshopify.com` domain that sent the webhook
- `X-Shopify-Webhook-Id` -- Unique webhook delivery ID (for idempotency logging)

**Flow:**
1. Read raw body via `request.text()` (BEFORE JSON parsing -- required for HMAC verification)
2. Extract `X-Shopify-Hmac-Sha256` header. If missing, return 400.
3. Call `verifyShopifyWebhookSignature(rawBody, signatureHeader)`. If invalid, return 400.
4. Parse the raw body as JSON. If invalid JSON, return 400.
5. Read `X-Shopify-Topic` header to determine the event type.
6. Read `X-Shopify-Shop-Domain` header to identify which store sent the event.
7. Route by topic:
   - `products/update` -> `handleShopifyProductUpdate(shopDomain, body)`
   - `products/delete` -> `handleShopifyProductDelete(shopDomain, body)`
   - `orders/create` -> `handleShopifySaleWebhook(shopDomain, body)`
   - `orders/paid` -> `handleShopifySaleWebhook(shopDomain, body)` (same handler; idempotent)
   - `products/create` -> log and acknowledge (no action needed -- products created on Shopify by Twicely are already tracked)
   - `app/uninstalled` -> `handleShopifyAppUninstalled(shopDomain)`
   - Unknown topic -> log and acknowledge
8. ALWAYS return 200 `{ received: true }` even if handler throws. This prevents Shopify retry storms (Lister Canonical section 24.5: "Webhook handlers are idempotent. Second processing is a no-op.").
9. Wrap each handler call in try/catch. Log errors but do not let them bubble to the response.

**Pattern:** Mirror `POST /api/crosslister/whatnot/webhook` exactly, but:
- Use `X-Shopify-Hmac-Sha256` instead of `X-Whatnot-Signature`
- Topic comes from header, not from body field
- Body is the raw resource (product/order), not wrapped in an envelope

### 2.7 Shopify Sale Webhook Handler

**File:** Add `handleShopifySaleWebhook` to existing `src/lib/crosslister/handlers/sale-webhook-handler.ts`

This handler processes `orders/create` and `orders/paid` webhooks from Shopify.

**Flow (mirrors `handleWhatnotSaleWebhook` exactly):**

1. Parse the body using `ShopifyWebhookOrderSchema.safeParse(body)`. If invalid, log warning and return.
2. Iterate `line_items`. For each item with a non-null `product_id`:
   a. Look up `channelProjection` where `channel = 'SHOPIFY'` AND `externalId = String(product_id)`.
   b. If not found, skip silently (not managed by Twicely).
   c. Convert `total_price` to integer cents using `parseShopifyPrice` from `shopify-normalizer.ts`.
   d. Look up platform fee rate via `getPlatformFeeRate('SHOPIFY')`.
   e. Calculate platform fee via `calculatePlatformFee(salePriceCents, feeRateBps)`.
   f. Build `DetectedSale` object and call `handleDetectedSale(detectedSale)`.

**Key differences from Whatnot:**
- Shopify orders can contain MULTIPLE line items (multiple products sold in one order). Process each line item independently.
- The `externalOrderId` should be `String(order.id)`.
- `buyerUsername` comes from `order.customer?.email ?? order.customer?.first_name ?? undefined`.
- `soldAt` comes from `new Date(order.created_at)`.
- `salePriceCents`: for multi-item orders, use the per-line-item `price` (NOT `total_price` which includes all items + shipping + tax). Convert each `line_item.price` to cents.

**Idempotency:** `handleDetectedSale` already checks if the projection is already SOLD and skips if so. Multiple calls for the same order are safe.

### 2.8 Shopify Product Update Handler

**File:** `src/lib/crosslister/handlers/shopify-webhook-handlers.ts`

This is the INBOUND sync direction (External -> Canonical). Per Lister Canonical section 14.3:

> When polling/webhook detects a change on an external platform:
> 1. Change recorded as `externalDiff` on the ChannelProjection
> 2. If two-way auto-sync is DISABLED (default): seller sees "External Change Detected" notification with diff view
> 3. If two-way auto-sync is ENABLED (per field group): change auto-merges into canonical
> 4. Conflicts (both canonical and external changed same field): flagged for manual resolution

**Function:** `handleShopifyProductUpdate(shopDomain: string, body: unknown): Promise<void>`

**Flow:**
1. Parse body with `ShopifyProductSchema.safeParse(body)`. If invalid, log warning and return.
2. Build `externalId = String(product.id)`.
3. Look up `channelProjection` where `channel = 'SHOPIFY'` AND `externalId = externalId`.
4. If no projection found, log info and return (product not managed by Twicely).
5. Use `normalizeShopifyProduct(product, shopDomain)` to get normalized data.
6. Compute a diff between the normalized external data and the projection's current known state:
   - Compare `priceCents` (from normalized product) vs stored canonical price
   - Compare `quantity` (sum of variants)
   - Compare `title`
   - Compare `status` (map Shopify status to Twicely status via `mapShopifyStatus`)
7. If no meaningful changes detected, return (no-op).
8. If the Shopify product status is `'archived'` or `'draft'` (mapping to ENDED/DRAFT):
   - Update projection status to `ENDED` or `DELISTED`.
   - If the product was `ACTIVE` before, this is an externally-initiated delist. Log it.
9. Store the detected changes as `externalDiff` jsonb on the projection:
   ```typescript
   {
     detectedAt: new Date().toISOString(),
     shopDomain,
     fields: {
       priceCents?: { old: number, new: number },
       quantity?: { old: number, new: number },
       title?: { old: string, new: string },
       status?: { old: string, new: string },
     }
   }
   ```
10. Set `hasPendingSync = true` on the projection.
11. Elevate `pollTier` to `'HOT'` temporarily (per Lister Canonical section 13.3: "Price changed by seller -> Promote to WARM for 30 min"). Since this is an external change, use WARM.
12. Send a Centrifugo event `sync.external_change` on `private-user.{sellerId}` (per Lister Canonical section 21: event `sync.external_change` with payload `{ projectionId, channel, diff }`).

**Important: This handler does NOT auto-merge changes into the canonical listing.** Two-way auto-sync is OFF by default (Lister Canonical section 14.4). The seller must review and accept external changes. The diff is stored for the crosslister command center UI to display.

### 2.9 Shopify Product Delete Handler

**Function:** `handleShopifyProductDelete(shopDomain: string, body: unknown): Promise<void>`

Added to the same file `shopify-webhook-handlers.ts`.

**Flow:**
1. Parse body (Shopify sends `{ id: number }` for delete events -- only the ID).
2. Build `externalId = String(body.id)`.
3. Look up `channelProjection` where `channel = 'SHOPIFY'` AND `externalId = externalId`.
4. If no projection found, return.
5. Update projection:
   - `status = 'DELISTED'`
   - `orphanedAt = new Date()` (per Decision #112: UNMANAGED/orphaned tracking)
6. Log the external deletion.
7. Do NOT delete the canonical Twicely listing. The listing remains on Twicely even if the seller deleted it from their Shopify store. The projection is simply marked as DELISTED.

### 2.10 Shopify App Uninstalled Handler

**Function:** `handleShopifyAppUninstalled(shopDomain: string): Promise<void>`

Added to the same file `shopify-webhook-handlers.ts`.

**Flow:**
1. Look up `crosslisterAccount` where `channel = 'SHOPIFY'` AND `externalAccountId = shopDomain`.
2. If not found, return.
3. Update account: `status = 'REVOKED'`, `accessToken = null`.
4. Update all ACTIVE projections for this account to `status = 'DELISTED'`.
5. Log the uninstall event.

### 2.11 Outbound Sync Service

**File:** `src/lib/crosslister/services/outbound-sync.ts`

Per Lister Canonical section 14.2:

> When a seller edits their listing on Twicely:
> 1. Canonical listing updated
> 2. System computes hash of new canonical data
> 3. Compares against each projection's `lastCanonicalHash`
> 4. For projections where sync is enabled and hash differs -> queue SYNC job
> 5. SYNC job transforms canonical data to platform format and pushes update
> 6. Syncs do NOT count as publishes

**Function:** `computeCanonicalHash(listing: CanonicalListingData): string`

Computes a deterministic hash of the fields that matter for sync:
- `title`, `description`, `priceCents`, `quantity`, `status`, `images` (sorted URLs)
- Use `crypto.createHash('sha256')` on a JSON.stringify of the sorted field values.
- Returns hex digest string.

**Input type:**
```typescript
interface CanonicalListingData {
  title: string;
  description: string;
  priceCents: number;
  quantity: number;
  status: string;
  imageUrls: string[];  // sorted
}
```

**Function:** `detectOutboundSyncNeeded(listingId: string): Promise<ProjectionSyncTarget[]>`

1. Load the canonical listing data for `listingId`.
2. Compute the canonical hash.
3. Query all `channelProjection` rows where `listingId = listingId` AND `syncEnabled = true` AND `status = 'ACTIVE'`.
4. For each projection where `lastCanonicalHash !== canonicalHash`:
   - Include in the return array as `{ projectionId, channel, accountId }`.
5. Return the list of projections that need syncing.

**Function:** `queueOutboundSync(listingId: string, projections: ProjectionSyncTarget[]): Promise<void>`

For each projection needing sync:
1. Create a `crossJob` with:
   - `jobType: 'SYNC'`
   - `priority: 500` (per Lister Canonical section 8.2)
   - `idempotencyKey: `sync-${projectionId}-${canonicalHash}`` (dedupes redundant sync jobs)
   - `payload: { listingId, projectionId, channel, canonicalHash }`
2. Syncs do NOT count as publishes (Lister Canonical section 7.1, confirmed rule).

**Function:** `executeOutboundSync(job: SyncJobPayload): Promise<void>`

1. Load the canonical listing data.
2. Load the projection + account.
3. Get the connector for the projection's channel from the connector registry.
4. Transform canonical data to platform format (apply overrides from `overridesJson` if present).
5. Call `connector.updateListing(account, externalId, changes)`.
6. On success: update projection's `lastCanonicalHash` to the new hash, set `hasPendingSync = false`.
7. On failure: log error, crossJob status remains for retry.

**NOTE:** The outbound sync service is channel-agnostic. It works for Shopify AND any other channel that supports `updateListing`. For H3.4, wire it for Shopify. The service is designed to be reused for eBay/Etsy outbound sync later.

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- ZERO occurrences of: `SellerTier`, `SubscriptionTier`, `FVF`, `Final Value Fee`, `BASIC`, `ELITE`, `PLUS`, `MAX`, `PREMIUM`, `STANDARD`, `RISING`, `Twicely Balance`, `wallet` (in UI context), `Withdraw`

### Banned Patterns
- No `as any`, `as unknown as T`, `@ts-ignore`, `@ts-expect-error`
- No floating point for money -- integer cents only
- No hardcoded fee rates -- read from `platform_settings`
- No request body spreading into DB updates -- explicit field mapping only
- No files over 300 lines
- No `console.log` in production code

### Specific Gotchas for This Step

1. **Shopify HMAC is Base64, NOT hex.** The Whatnot webhook verify uses `.digest('hex')`. Shopify uses `.digest('base64')`. Getting this wrong will reject every webhook.

2. **Shopify does NOT wrap webhooks in an envelope.** The body IS the resource directly. The topic comes from the `X-Shopify-Topic` header. Do NOT try to parse an `eventType` from the body.

3. **Shopify product IDs are numbers.** All `externalId` values in channelProjection are stored as strings. Always convert with `String(product.id)`.

4. **Shopify orders can have multiple line items.** Unlike Whatnot (single-item BIN), a Shopify order can contain multiple products. Process each line item's `product_id` independently through `handleDetectedSale`.

5. **Two-way auto-sync is OFF by default.** The product update handler MUST NOT auto-merge external changes into the canonical listing. It records the diff and notifies the seller. This is a core architectural principle (Lister Canonical section 14.4).

6. **Syncs do NOT count as publishes.** Never call the publish meter for sync operations.

7. **The webhook route returns 200 even on handler errors.** This prevents Shopify from retrying failed deliveries repeatedly (Lister Canonical section 24.5).

8. **Use `crosslister.shopify.clientSecret` for webhook HMAC.** Shopify uses the same app secret for both OAuth HMAC and webhook HMAC. Do NOT create a separate `webhookSecret` setting (that's the Whatnot pattern, not Shopify's).

9. **Do NOT delete canonical listings when Shopify products are deleted.** The canonical listing stays on Twicely. Only the projection is marked DELISTED.

10. **Exported helpers in 'use server' files become unintended server actions.** The webhook handlers file is NOT a `'use server'` file, but be careful. Only the route file needs NextRequest/NextResponse imports.

### Tech Stack
- Drizzle ORM (never Prisma)
- Node.js `crypto` module for HMAC (no external crypto libraries)
- Zod for all webhook payload validation
- No tRPC, no Zustand, no Redis (Valkey for queues if needed)

## 4. ACCEPTANCE CRITERIA

### Positive Cases
- [ ] `POST /api/crosslister/shopify/webhook` with valid HMAC + `products/update` topic returns 200 and stores `externalDiff` on the matching projection
- [ ] `POST /api/crosslister/shopify/webhook` with valid HMAC + `orders/create` topic triggers `handleDetectedSale` which marks the listing SOLD and creates emergency delist jobs for other active projections
- [ ] `POST /api/crosslister/shopify/webhook` with valid HMAC + `products/delete` topic marks the projection as `DELISTED` and sets `orphanedAt`
- [ ] `POST /api/crosslister/shopify/webhook` with valid HMAC + `app/uninstalled` topic revokes the crosslisterAccount and delists all active projections for that store
- [ ] Multi-line-item Shopify orders process each line item independently through sale detection
- [ ] Outbound sync: `computeCanonicalHash` produces consistent hashes for identical data
- [ ] Outbound sync: `detectOutboundSyncNeeded` returns only projections where the hash differs and syncEnabled is true
- [ ] Product update handler stores diff in `externalDiff` jsonb on projection with field-level old/new values
- [ ] Product update handler sets `hasPendingSync = true` on the projection
- [ ] Sale webhook correctly converts Shopify decimal price strings to integer cents

### Negative Cases
- [ ] Webhook with missing `X-Shopify-Hmac-Sha256` header returns 400
- [ ] Webhook with invalid/tampered signature returns 400 (HMAC check fails)
- [ ] Webhook with valid signature but invalid JSON body returns 400
- [ ] Webhook handler errors do NOT propagate to the response -- always 200
- [ ] Product update handler does NOT auto-merge external changes into the canonical listing (two-way sync OFF by default)
- [ ] Product delete handler does NOT delete the canonical Twicely listing
- [ ] Sync operations are NOT counted as publishes
- [ ] Webhook for a product not managed by Twicely (no matching projection) is silently ignored
- [ ] Orders with `null` product_id in line items are skipped (Shopify can have custom line items)

### Data Integrity
- [ ] All monetary values stored as integer cents
- [ ] Fee rates loaded from `platform_settings`, never hardcoded
- [ ] Webhook idempotency: processing the same webhook twice produces the same result (no double-counting)
- [ ] `externalDiff` jsonb has consistent schema across all webhook events

### Authorization
- [ ] The webhook route does NOT require session auth or CASL -- it uses HMAC verification only
- [ ] The webhook route does NOT expose internal IDs in error messages

### Vocabulary
- [ ] No banned terms appear in any created/modified file

## 9. SPEC INCONSISTENCIES & OPEN QUESTIONS

### Confirmed: No Issues

1. **Shopify as Tier A with webhooks:** Confirmed in channel-registry.ts (`hasWebhooks: true`) and cfg/shopify page (`webhookConfig` defined). Lister Canonical section 9.1 lists "Tier A: API-First, webhooks available" for eBay/Etsy but does NOT explicitly list Shopify (it was added later). The codebase already declares Shopify as Tier A with webhooks. Follow the code.

2. **HMAC key source:** Shopify uses the app's client secret (same `crosslister.shopify.clientSecret` used in OAuth). This is confirmed by Shopify's official documentation pattern. The callback route already loads this secret for OAuth HMAC verification (line 80 of callback/route.ts).

3. **cfg/shopify page already declares all webhook topics** (`products/create`, `products/update`, `products/delete`, `orders/create`, `orders/paid`, `app/uninstalled`). The route handler should handle all of these.

### NOT SPECIFIED -- Owner Decision Needed

1. **Webhook registration timing:** Should webhooks be registered automatically during OAuth callback, or manually via admin UI? Recommendation: best-effort during OAuth callback (mirrors Shopify's recommended pattern), with admin UI as fallback. The prompt assumes this approach.

2. **`products/create` webhook handling:** When Shopify sends a `products/create` webhook, should we import the product as a new canonical listing? Or only acknowledge it? The prompt assumes acknowledge-only because:
   - Products created by Twicely's crosslist engine already have projections
   - Products created natively on Shopify would need a full import pipeline run
   - Auto-importing every new Shopify product could create unexpected listings

   If you want auto-import on `products/create`, that is a separate feature (closer to continuous sync / real-time import).

3. **Outbound sync trigger:** Should outbound sync be triggered by a DB trigger on the listing table, by the listing update server action, or by a periodic BullMQ job that scans for stale hashes? The prompt specifies the detection function (`detectOutboundSyncNeeded`) but does NOT wire it into a trigger. The installer should add a call to `detectOutboundSyncNeeded` + `queueOutboundSync` at the end of the listing update server action (wherever listings are edited on Twicely). If this wiring location is unclear, ask.
