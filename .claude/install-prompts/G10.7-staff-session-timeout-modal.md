## 2. SCOPE -- EXACTLY WHAT TO BUILD

### 2.1 Overview

The staff session timeout system has two independent timers that run on every hub page:

1. **Inactivity timer** -- resets on user interaction (mouse move, keypress, click, scroll, touch). Default: 5 minutes. When the idle threshold is approached, a warning modal appears. If the user does not interact, the session is terminated and the user is redirected to `/login?reason=inactivity`.

2. **Hard session limit** -- absolute wall-clock countdown from the session creation time. Default: 8 hours. Cannot be extended by activity. When ~5 minutes remain, a warning modal appears. On expiry, the session is terminated and the user is redirected to `/login?reason=expired`.

The Actors/Security Canonical (Section 1.2) specifies:
- Absolute session timeout: Agent = 8 hours, Admin = 4 hours
- Idle timeout: Agent = 30 min, Admin = 15 min

**SPEC INCONSISTENCY:** The Actors/Security Canonical specifies per-role timeouts (Admin: 4hr absolute / 15min idle, Agent: 8hr absolute / 30min idle). The existing implementation in `staff-auth.ts` uses a SINGLE configurable timeout for ALL staff (8hr absolute, 5min inactivity from platform_settings). The build tracker note says "configurable inactivity timeout (default 5min), 8hr hard session limit" -- no mention of per-role differentiation.

**RESOLUTION:** Follow the build tracker description and existing implementation. Use the single configurable platform settings (`general.staffInactivityTimeoutMinutes` = 5, `general.staffSessionAbsoluteHours` = 8) for ALL staff roles. Per-role differentiation is a future enhancement. Document this in the prompt so the owner can override if desired.

### 2.4 API Route: Session Heartbeat

**NEW:** `POST /api/hub/session/heartbeat`

This route allows the client to:
1. Check if the current session is still valid (not expired server-side)
2. Update `lastActivityAt` to reset the inactivity timer server-side
3. Return the remaining time for both inactivity and absolute expiry

**Request:** No body required. Auth via `twicely.staff_token` cookie.

**Response (200):**
```typescript
{
  success: true,
  sessionValid: true,
  absoluteExpiresAt: string,   // ISO 8601 timestamp
  inactivityTimeoutMs: number, // e.g., 300000 (5 min)
  warningSeconds: number,      // e.g., 60
}
```

**Response (401):**
```typescript
{
  success: false,
  sessionValid: false,
  reason: 'expired' | 'inactive' | 'not_found',
}
```

The heartbeat route MUST:
- Read the `twicely.staff_token` cookie
- Call `getStaffSession(token)` to validate + touch `lastActivityAt`
- Read `general.staffInactivityTimeoutMinutes` and `general.staffSessionWarningSeconds` from platform settings
- Return the absolute expiry time from the session row
- Return 401 if the session is invalid/expired (so the client can redirect immediately)

**NOTE:** The heartbeat route does NOT use CASL/staffAuthorize() because it only validates the session -- it does not perform any authorized action. It reads the cookie directly and calls `getStaffSession()`.

### 2.5 API Route: Session Keep-Alive (Extend Inactivity)

The heartbeat route already touches `lastActivityAt` via `getStaffSession()`, so calling the heartbeat IS the keep-alive. When the user clicks "Stay Logged In" in the warning modal, the client calls the heartbeat endpoint, which resets the inactivity timer server-side.

No separate keep-alive route is needed.

### 2.6 Client Component: `StaffSessionTimeoutProvider`

**NEW:** `src/components/admin/staff-session-timeout.tsx` -- `'use client'` component

This is a React context provider that wraps the hub layout content. It manages two independent timers:

**Inactivity Timer Logic:**
1. On mount, start a countdown timer set to `inactivityTimeoutMs` (from heartbeat response)
2. Listen for activity events on `document`: `mousemove`, `keydown`, `click`, `scroll`, `touchstart`
3. On any activity event, DEBOUNCE (300ms) and reset the inactivity countdown
4. Do NOT call the heartbeat API on every activity event -- only when the user clicks "Stay Logged In" or at periodic intervals (every 60 seconds while active, to keep server-side `lastActivityAt` fresh)
5. When `inactivityTimeoutMs - (warningSeconds * 1000)` milliseconds have elapsed without activity, show the warning modal
6. If the warning modal is showing and the remaining time reaches 0, call `logoutStaffAction()` and redirect to `/login?reason=inactivity`
7. Handle `visibilitychange` event: when the tab becomes visible again after being hidden, immediately check remaining time. If expired, redirect. If within warning threshold, show modal.

**Hard Session Limit Logic:**
1. On mount, calculate remaining time: `absoluteExpiresAt - now`
2. Start a countdown timer
3. When `warningSeconds` seconds remain before `absoluteExpiresAt`, show the warning modal (hard limit variant -- cannot be dismissed by activity, only shows remaining time)
4. When `absoluteExpiresAt` is reached, call `logoutStaffAction()` and redirect to `/login?reason=expired`

**Combined Modal Logic:**
- If BOTH timers are in warning state, show the hard limit warning (it takes priority since it cannot be extended)
- The inactivity warning has a "Stay Logged In" button. The hard limit warning does NOT (it cannot be extended)
- Only ONE modal instance is rendered at a time

**Activity Event Debouncing:**
- Activity events are debounced to prevent excessive re-renders (300ms)
- The periodic heartbeat call (every 60s while active) ensures the server-side `lastActivityAt` stays fresh without hammering the API on every mouse move

**Tab Visibility Handling:**
- When tab becomes hidden (`visibilitychange` to `hidden`): pause the UI countdown display but keep the timer tracking elapsed time
- When tab becomes visible (`visibilitychange` to `visible`): recalculate remaining time immediately. If session should have expired while hidden, redirect immediately. If within warning threshold, show modal.

## 3. CONSTRAINTS -- WHAT NOT TO DO

1. **Do NOT add any new database tables or columns.** The existing `staffSession` table has everything needed.
2. **Do NOT modify `staff-auth.ts` server-side expiry logic.** The server already correctly checks both absolute and inactivity timeouts. This feature adds CLIENT-SIDE awareness of those same timeouts.
3. **Do NOT use `setInterval` for the heartbeat call on every activity event.** Debounce activity detection (300ms) and only call the heartbeat API periodically (every 60s while user is active). This prevents excessive API calls.
4. **Do NOT store session expiry in localStorage or sessionStorage.** The `absoluteExpiresAt` is passed from the server via the layout as a prop; the inactivity timeout is returned by the heartbeat API. No client-side persistence.
5. **Do NOT use `window.alert()` or `window.confirm()`.** Use shadcn/ui `AlertDialog`.
6. **Do NOT add Zustand, Redux, or any state management library.** Use React context.
7. **Do NOT add per-role timeout differentiation.** Use single platform settings for all staff. See SPEC INCONSISTENCY note above.
8. **Do NOT add WebSocket/Centrifugo for session monitoring.** HTTP heartbeat is sufficient for this use case.
9. **Do NOT use `@ts-ignore`, `as any`, `as unknown as T`**, or `@ts-expect-error`. Fix the types.
10. **Do NOT exceed 300 lines per file.**
11. **Do NOT import Node.js modules in `'use client'` components.** The timeout provider and modal are client-only components. They must not import `db`, `drizzle`, `better-auth`, or any server module.
12. **ROUTE PREFIX:** The heartbeat API route is at `/api/hub/session/heartbeat`. Not `/api/staff/session/heartbeat`, not `/api/admin/session/heartbeat`.

## 8. IMPLEMENTATION NOTES

### Existing Infrastructure Summary

The server-side session validation is ALREADY COMPLETE in `src/lib/auth/staff-auth.ts`:
- `loginStaff()` creates a `staffSession` row with `expiresAt` set to `now + staffSessionAbsoluteHours`
- `getStaffSession()` checks absolute expiry AND inactivity timeout, deletes expired sessions, and updates `lastActivityAt` on valid requests
- `logoutStaff()` deletes the session row

The hub layout (`src/app/(hub)/layout.tsx`, 63 lines) already calls `getStaffSession(token)` and has the full `StaffSessionResult` including `expiresAt`. The timeout provider just needs this data passed to the client.

The login page (`src/app/(hub)/login/page.tsx`, 97 lines) already handles `?error=1`. Adding `?reason=inactivity|expired` is a small addition.

### Key Design Decisions

1. **One heartbeat endpoint, not two.** The heartbeat validates the session AND resets `lastActivityAt` in one call. "Stay Logged In" is just a heartbeat call.

2. **Client timers as primary, server as safety net.** The client runs countdown timers for responsive UX. The server enforces the real timeout on every request (via `getStaffSession()`). If the client timer is somehow wrong, the server catches it on the next API call.

3. **Periodic heartbeat while active (60s), not on every event.** This keeps the server-side `lastActivityAt` reasonably fresh without excessive API calls. The client-side timer tracks inactivity precisely via DOM events.

4. **AlertDialog, not Dialog.** The warning modal must not be dismissible by clicking outside or pressing Escape. The user must explicitly choose "Stay Logged In" or "Log Out Now" (inactivity) or "Log Out Now" (hard limit).

5. **Hard limit warning at 5 minutes (300s), inactivity warning at configurable seconds (default 60s).** These are different because 60 seconds warning for a hard cutoff is too short -- the user needs time to save work. The 5-minute hard-limit warning is not configurable via platform settings to keep things simple; it could be made configurable later.
