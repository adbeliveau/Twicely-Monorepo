### 3.2 New Queries

#### 3.2.1 `src/lib/queries/affiliate-payout-admin.ts` (NEW file)

Query 1: `getAffiliatePayoutList(options)` — Platform-wide payout list for `/fin/affiliate-payouts`
- Input: `{ status?: string; affiliateId?: string; limit: number; offset: number }`
- Joins: `affiliatePayout` LEFT JOIN `affiliate` LEFT JOIN `user` (to get username/email)
- Select: payoutId, affiliateId, affiliateUsername, affiliateEmail, amountCents, method, status, periodStart, periodEnd, failedReason, createdAt, completedAt, externalPayoutId
- Order: `desc(affiliatePayout.createdAt)`
- Returns: `{ rows: AffiliatePayoutAdminRow[]; total: number }`

Query 2: `getAffiliatePayoutStats()` — Aggregate stats for the stats cards on `/fin/affiliate-payouts`
- Returns: `{ totalPayoutsCents: number; totalPayoutsCount: number; pendingPayoutsCents: number; failedPayoutsCount: number; activeAffiliatesCount: number; totalCommissionsPendingCents: number }`
- Implementation:
  - `totalPayoutsCents`: SUM of amountCents WHERE status = 'COMPLETED'
  - `totalPayoutsCount`: COUNT WHERE status = 'COMPLETED'
  - `pendingPayoutsCents`: SUM of affiliate.availableBalanceCents WHERE status = 'ACTIVE'
  - `failedPayoutsCount`: COUNT WHERE affiliatePayout.status = 'FAILED'
  - `activeAffiliatesCount`: COUNT WHERE affiliate.status = 'ACTIVE'
  - `totalCommissionsPendingCents`: SUM of affiliateCommission.commissionCents WHERE status = 'PENDING'

Query 3: `getCommissionsForAdmin(options)` — Commission list for the admin affiliate detail page
- Input: `{ affiliateId: string; status?: string; limit: number; offset: number }`
- Select: commissionId, subscriptionProduct, grossRevenueCents, netRevenueCents, commissionRateBps, commissionCents, status, holdExpiresAt, paidAt, reversedAt, reversalReason, createdAt
- Order: `desc(affiliateCommission.createdAt)`
- Returns: `{ rows: CommissionAdminRow[]; total: number }`

Query 4: `getPayoutsForAdmin(options)` — Payout list for the admin affiliate detail page
- Input: `{ affiliateId: string; limit: number; offset: number }`
- Select: all affiliatePayout columns
- Order: `desc(affiliatePayout.createdAt)`
- Returns: `{ rows: PayoutAdminRow[]; total: number }`

### 3.3 New Admin Actions

#### 3.3.1 `src/lib/actions/affiliate-commission-admin.ts` (NEW file)

Action 1: `updateAffiliateCommissionRate(input)` — Adjust a specific affiliate's commission rate
- Zod schema: `updateCommissionRateSchema` = `{ affiliateId: string, commissionRateBps: number (100..5000) }`
- Auth: `staffAuthorize()` + `ability.can('manage', 'Affiliate')`
- Validation: affiliate exists. If tier = 'COMMUNITY', rate must match platform setting `affiliate.community.commissionRateBps` (community rates are uniform). If tier = 'INFLUENCER', rate must be 2000-3000 bps.
- Updates: `affiliate.commissionRateBps`, `affiliate.updatedAt`
- Audit event: `AFFILIATE_COMMISSION_RATE_UPDATED`, severity MEDIUM
- Note: this DOES NOT retroactively change existing commissions. Only affects future commissions (documented in the audit event detailsJson).

**NOT SPECIFIED — Owner Decision Needed**: The Affiliate Canonical Section 2.1 says community commission is flat 15% and influencer is "negotiated per partner" (20-30%). The schema stores `commissionRateBps` per affiliate, suggesting per-affiliate overrides are supported. This action allows admin to change the rate for individual affiliates. If community rates should NEVER be overridden, remove the community rate update path. Recommendation: allow override for both tiers (the per-affiliate column exists for this purpose) but log it clearly.

## 4. CONSTRAINTS — WHAT NOT TO DO

### Banned Terms
- NO `wallet`, `balance` (in UI copy) — use "Available for payout", "Pending commissions"
- NO `FVF`, `Final Value Fee` — use `TF` / `Transaction Fee`
- NO `SellerTier`, `SubscriptionTier` — use `StoreTier` / `ListerTier`
- NO `Withdraw` — use `Request payout`

### Tech Constraints
- NO Prisma (use Drizzle ORM)
- NO NextAuth (use Better Auth — but hub pages use `staffAuthorize()`, not Better Auth directly)
- NO tRPC (use server actions)
- NO Zustand/Redux (use React state for client components)

### Pattern Constraints
- NO `as any`, `as unknown as T`, `@ts-ignore`, `@ts-expect-error`
- NO files over 300 lines — the existing `/usr/affiliates/[id]/page.tsx` is 215 lines; keep each new/modified file under 300
- NO spreading request body into DB queries — use explicit field mapping
- NO hardcoded fee rates — commission rates come from `affiliate.commissionRateBps` per record, with defaults from platform settings
- NO `console.log` in production code — use `logger` from `@/lib/logger` if logging is needed
- Money as integer cents ONLY — format with `$${(cents / 100).toLocaleString('en-US', { minimumFractionDigits: 2 })}` for display
- Rates in basis points — format with `${(bps / 100).toFixed(1)}%` for display
- `affiliatePayout.status` is plain text (NOT an enum) — values: PENDING, PROCESSING, COMPLETED, FAILED
- `affiliateCommission.status` IS an enum (commissionStatusEnum) — values: PENDING, PAYABLE, PAID, REVERSED
- All Zod schemas use `.strict()` — unknown keys rejected
- `revalidatePath()` after mutations that affect the page
- `staffAuthorize()` for all hub pages/actions (NOT `auth.api.getSession()`)
- Hub pages use `ability.can('manage', 'AffiliatePayout')` for finance pages, `ability.can('manage', 'Affiliate')` for user management pages

### What NOT to Build
- DO NOT build `/cfg/affiliates` (affiliate program settings page) — that is a separate step
- DO NOT build `/cfg/trials` (trial configuration) — separate step
- DO NOT re-implement the existing `/usr/affiliates` list page or the existing admin actions (approve/reject/suspend/unsuspend/ban) — they already work
- DO NOT add new database columns or tables
- DO NOT build anti-fraud checks — that is G3.5

The payout service (`affiliate-payout-service.ts`) skips PayPal affiliates with a warning log. The UI should display "PayPal (not yet supported)" for method = 'paypal'. No PayPal SDK exists in the codebase.

### Gap 3: Tax Info Collection
The `taxInfoProvided` boolean on the affiliate table is checked during payout execution (must be `true`), but there is no UI to collect tax info. The `/fin/affiliate-payouts` page should show a warning icon or tooltip for affiliates where `taxInfoProvided = false`, indicating they are ineligible for payout until G5 (tax collection UI).
