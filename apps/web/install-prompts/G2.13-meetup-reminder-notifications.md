## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2.1 Schema Changes

**NO schema changes needed.** No new tables, no new columns, no new enums. The reminder system uses only existing data (`scheduledAt`, `buyerId`, `sellerId`, `orderId`) and BullMQ delayed jobs.

### 2.2 BullMQ Job — Meetup Reminder

**Create: `src/lib/jobs/local-meetup-reminder.ts`**

A single queue with two job types (24hr and 1hr) that sends reminder notifications to both parties before their meetup.

**Queue name:** `local-meetup-reminder` (hyphens per project convention)

**Job data interface:**
```typescript
interface LocalMeetupReminderData {
  localTransactionId: string;
  orderId: string;
  buyerId: string;
  sellerId: string;
  itemTitle: string;
  location: string;
  scheduledAtIso: string; // ISO string of the meetup time
  reminderType: '24hr' | '1hr';
}
```

**Enqueue helpers (both exported):**

```typescript
export async function enqueueLocalMeetupReminders(
  data: Omit<LocalMeetupReminderData, 'reminderType'>,
): Promise<void>
```

This function enqueues **two** delayed jobs:

1. **24-hour reminder:**
   - Job name: `'reminder'`
   - Job ID: `local-reminder-24hr-{localTransactionId}`
   - Delay: `scheduledAt - 24 hours - now` (in ms)
   - If `scheduledAt - now < 24 hours`, skip the 24hr reminder (already within the 24hr window)
   - `removeOnComplete: true`, `removeOnFail: { count: 100 }`

2. **1-hour reminder:**
   - Job name: `'reminder'`
   - Job ID: `local-reminder-1hr-{localTransactionId}`
   - Delay: `scheduledAt - 1 hour - now` (in ms)
   - If `scheduledAt - now < 1 hour`, skip the 1hr reminder (already within the 1hr window)
   - `removeOnComplete: true`, `removeOnFail: { count: 100 }`

Both jobs carry `reminderType: '24hr'` or `reminderType: '1hr'` respectively in the payload.

**Remove helper (exported):**

```typescript
export async function removeLocalMeetupReminders(
  localTransactionId: string,
): Promise<void>
```

Removes both pending reminder jobs by their job IDs. Best-effort — wraps each removal in try/catch, logs warnings on failure, never throws. This is called by `cleanupBullMQJobs` in `local-cancel.ts` and by the reschedule accept path before re-enqueuing.

**Worker processing logic:**

1. Parse `scheduledAtIso` into a `Date` object
2. Fetch localTransaction row by ID to check current state
3. If not found, log warning and return
4. If terminal status (`COMPLETED`, `CANCELED`, `NO_SHOW`, `DISPUTED`), log info and return (no-op)
5. If `scheduledAtConfirmedAt === null`, log info and return (scheduling was unconfirmed, perhaps the transaction regressed — should not happen but defensive)
6. If the current `scheduledAt` on the transaction does not match `scheduledAtIso` from the job payload, log info and return (a reschedule happened but the old job was not cleaned up — stale job)
7. Determine template key based on `reminderType`:
   - `'24hr'` -> `'local.reminder.24hr'`
   - `'1hr'` -> `'local.reminder.1hr'`
8. Format the meetup time for display: date string and time string using `toLocaleDateString`/`toLocaleTimeString` with `'en-US'` locale
9. Notify both parties:
   - `notify(data.buyerId, templateKey, { itemTitle, location, date, time })`
   - `notify(data.sellerId, templateKey, { itemTitle, location, date, time })`
10. Log completion

**Worker concurrency:** 1 (consistent with other local job workers)

**Graceful shutdown:** `process.on('SIGTERM', async () => { await worker.close(); })`

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Patterns
- Do NOT add any new database columns or tables — this feature is pure BullMQ + notifications
- Do NOT hardcode any fee rates or monetary values (not applicable to this feature, but enforce globally)
- Do NOT create a user-facing server action for reminders — these are automatic, system-initiated
- Do NOT duplicate the placeholder removal logic in both `local-scheduling.ts` and `local-reschedule.ts` — the `removeLocalMeetupReminders` function in the job file is the single source of truth for cleanup
- Do NOT send reminders if the meetup time has already passed (delay calculation produces negative value)
- Do NOT send reminders for terminal-status transactions (worker must check)
- Do NOT use `as any`, `@ts-ignore`, or `@ts-expect-error`
- Do NOT create files over 300 lines
- Do NOT leave `console.log` in production code — use `logger` from `@/lib/logger`
- Do NOT use `{curly}` template syntax in notification templates if the existing codebase uses `{{double-curly}}`. Check existing templates for consistency. NOTE: The `LOCAL_TEMPLATES` in `templates-local.ts` use `{single-curly}` format while `TEMPLATES` in `templates.ts` use `{{double-curly}}`. Follow the `LOCAL_TEMPLATES` pattern: `{single-curly}`.

### Gotchas
- The `enqueueSchedulingReminders` placeholder in `local-scheduling.ts` is NOT exported (it's a private function). The replacement must also remain unexported as a function call within the file. Do NOT export it — exported helpers in `'use server'` files become unintended server actions.
- The `enqueueSchedulingReminders` placeholder in `local-reschedule.ts` is also NOT exported. Same rule applies.
- Both files are `'use server'` modules. Any function defined at the top level of these files that is exported becomes a server action. Keep the `enqueueLocalMeetupReminders` import and calls private (call within unexported helper or inline).
- The delay calculation (`scheduledAt - reminderOffsetMs - Date.now()`) can produce a negative number if the meetup is sooner than the reminder window. Use `Math.max(0, delay)` or skip the job entirely. **This prompt recommends skipping** (do not enqueue) when the delay would be <= 0, because firing a "24hr before" reminder after the meetup is misleading.
- The `local-reschedule.ts` `handleAccept` function is a plain (non-exported) function, not a server action. Calling async imports from it is fine.
- `scheduledAt` on the transaction row may be a `Date` or a date-string depending on Drizzle serialization. Always construct `new Date(tx.scheduledAt)` when doing arithmetic.
- The stale-job guard (step 6 in worker logic) is critical: if a reschedule occurs but the old reminder job is not removed (race condition), the worker must detect the mismatch between job payload `scheduledAtIso` and the current DB `scheduledAt` and no-op.
- BullMQ `removeOnComplete: true` means completed jobs are auto-cleaned. But delayed jobs that haven't fired yet must be explicitly removed on cancellation/reschedule.

## 4. ACCEPTANCE CRITERIA

### Positive Cases — Enqueue
1. When `acceptMeetupTimeAction` succeeds (scheduledAt confirmed), two BullMQ jobs are enqueued: one for 24hr before and one for 1hr before
2. When `respondToRescheduleAction` accepts a reschedule, old reminder jobs are removed and new ones are enqueued for the new scheduledAt
3. The 24hr reminder job has delay = `scheduledAt - 24hr - now` (correctly calculated in milliseconds)
4. The 1hr reminder job has delay = `scheduledAt - 1hr - now` (correctly calculated in milliseconds)
5. If the meetup is less than 24 hours away when scheduling is confirmed, the 24hr job is NOT enqueued (skip, not zero delay)
6. If the meetup is less than 1 hour away when scheduling is confirmed, the 1hr job is NOT enqueued
7. Job IDs follow the pattern `local-reminder-24hr-{localTransactionId}` and `local-reminder-1hr-{localTransactionId}` for deduplication and removal

### Positive Cases — Worker Execution
8. When the 24hr reminder fires, both buyer and seller receive `local.reminder.24hr` notification
9. When the 1hr reminder fires, both buyer and seller receive `local.reminder.1hr` notification
10. Notification includes item title resolved from `listing.title` via `orderItem` join
11. Notification includes location resolved from `safeMeetupLocation.name` (or fallback if no meetup location set)
12. Worker gracefully handles missing item title or location (uses fallback strings)

### Positive Cases — Cleanup
13. When a local transaction is canceled (via `cancelLocalTransaction`), both reminder jobs are removed from the queue
14. When a reschedule is accepted, old reminder jobs are removed before new ones are enqueued
15. The `removeLocalMeetupReminders` function is best-effort — failures are logged, never thrown

### Negative Cases — Worker Guards
16. Worker no-ops if the localTransaction is not found (log warning, return)
17. Worker no-ops if the transaction is in a terminal status: COMPLETED, CANCELED, NO_SHOW, DISPUTED
18. Worker no-ops if `scheduledAtConfirmedAt` is null (scheduling was unconfirmed)
19. Worker no-ops if the current `scheduledAt` on the transaction does not match the `scheduledAtIso` from the job payload (stale job from pre-reschedule)

### Edge Cases to Cover
- Meetup scheduled exactly 24 hours from now (24hr reminder fires immediately with delay 0 — should skip OR fire, decide based on skip rule: delay <= 0 = skip)
- Meetup scheduled exactly 1 hour from now (1hr reminder fires immediately — skip)
- Meetup scheduled 25 hours from now (both reminders enqueued: 24hr at 1hr delay, 1hr at 24hr delay)
- Meetup scheduled 30 minutes from now (both reminders skipped)
- Reschedule accepted but old reminder jobs have already fired (remove is a no-op — job no longer exists)
- Worker fires but transaction was canceled between enqueue and fire (terminal status guard)
- Worker fires but transaction was rescheduled (stale scheduledAt guard)
- Location is null (meetup at custom location, not a safe spot) — worker uses fallback string

### Delay Calculation

```typescript
function calculateReminderDelay(
  scheduledAt: Date,
  offsetHours: number,
): number | null {
  const reminderTime = new Date(scheduledAt);
  reminderTime.setHours(reminderTime.getHours() - offsetHours);
  const delayMs = reminderTime.getTime() - Date.now();
  // Skip if the reminder time has already passed
  if (delayMs <= 0) return null;
  return delayMs;
}
```

### Stale Job Guard

When the worker fires, it must verify the job's `scheduledAtIso` matches the current transaction's `scheduledAt`:

```typescript
const currentScheduledAt = tx.scheduledAt?.toISOString() ?? null;
if (currentScheduledAt !== data.scheduledAtIso) {
  logger.info('[local-meetup-reminder] Stale job — scheduledAt changed, skipping', {
    localTransactionId: data.localTransactionId,
    jobScheduledAt: data.scheduledAtIso,
    currentScheduledAt,
  });
  return;
}
```

**IMPORTANT:** When comparing ISO strings, be aware that Drizzle may return `Date` objects with millisecond precision differences. Compare by converting both to dates and checking equality within a tolerance:

```typescript
const jobTime = new Date(data.scheduledAtIso).getTime();
const dbTime = tx.scheduledAt ? new Date(tx.scheduledAt).getTime() : 0;
if (Math.abs(jobTime - dbTime) > 1000) {
  // Stale job, skip
}
```

## 9. SPEC DECISIONS REQUIRING OWNER CONFIRMATION

### Decision 1: Reminder Timing Not Admin-Configurable

The addendum A10 specifies "24 hours before" and "1 hour before" without platform setting keys. The Updated Platform Settings section at the bottom of the addendum does NOT include any `commerce.local.reminderXhr` keys. This prompt implements the timing as hardcoded constants in the job file (24 and 1).

**If the owner wants these to be admin-configurable via platform_settings, two new settings would need to be seeded and the enqueue function would need to read them. This prompt does NOT do that.**

### Decision 2: Item Title and Location Resolved at Fire Time

Rather than passing `itemTitle` and `location` in the job payload at enqueue time (which would require adding DB queries to the scheduling/reschedule actions), the worker resolves these from the DB when the job fires. This is more resilient to data changes between enqueue and fire, but adds two simple DB queries to the worker.

**If the owner prefers payload-based data (no DB queries in the worker), the scheduling and reschedule actions would need to be modified to query `listing.title` and `safeMeetupLocation.name` before enqueuing.**

### Decision 3: Skip vs Fire-Immediately for Past Reminder Windows

When the meetup is less than 24 hours away (but more than 1 hour), this prompt skips the 24hr reminder entirely. An alternative would be to fire it immediately (delay 0), but sending "Your meetup is tomorrow" when the meetup is in 8 hours is misleading. The 1hr reminder will still fire correctly.

**If the owner wants immediate-fire behavior instead of skip, the delay calculation should use `Math.max(0, delay)` instead of returning null.**
