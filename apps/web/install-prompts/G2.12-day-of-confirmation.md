### 2.4 Server Actions

**Create: `src/lib/actions/local-day-of-confirmation.ts`**

Two server actions in this file:

#### Action 1: `sendDayOfConfirmationAction`

Called by the **buyer** to send the "Are we still on?" request.

**Input:** `z.infer<typeof sendDayOfConfirmationSchema>`
**Output:** `{ success: boolean; error?: string }`

**Logic (in order):**

1. `authorize()` — get session + ability
2. Parse input with `sendDayOfConfirmationSchema`
3. Fetch localTransaction row by ID
4. Validate caller is the **buyer** on this transaction (`session.userId === tx.buyerId`)
5. CASL check: `ability.can('update', sub('LocalTransaction', { buyerId: tx.buyerId }))`
6. Validate status is eligible: `['SCHEDULED', 'SELLER_CHECKED_IN', 'BUYER_CHECKED_IN']`
   - NOT `RESCHEDULE_PENDING` (a reschedule is already in flight — don't send confirmation request)
   - NOT `BOTH_CHECKED_IN` or later (both already there)
   - NOT terminal statuses
7. Validate `scheduledAtConfirmedAt !== null` (meetup time must be confirmed)
8. Validate `dayOfConfirmationSentAt === null` (one confirmation request per transaction per A9)
9. Validate we are within the confirmation window:
   - Read `commerce.local.dayOfConfirmationWindowHours` (default 12)
   - Compute `windowStart = scheduledAt - windowHours`
   - `now >= windowStart` (button only works within window)
   - `now < scheduledAt` (cannot send after meetup time has passed)
10. Update localTransaction: `{ dayOfConfirmationSentAt: now, updatedAt: now }`
11. Enqueue BullMQ delayed job: `local-day-of-confirmation-timeout` with delay = `dayOfConfirmationResponseHours * 60 * 60 * 1000` (default 2 hours)
12. Notify seller via `notify(tx.sellerId, 'local.dayof.request', { time: formatTime(tx.scheduledAt), buyerName: '...' })`
13. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`
14. Return `{ success: true }`

#### Action 2: `respondToDayOfConfirmationAction`

Called by the **seller** to confirm they are still coming.

**Input:** `z.infer<typeof respondToDayOfConfirmationSchema>`
**Output:** `{ success: boolean; error?: string }`

**Logic (in order):**

1. `authorize()` — get session + ability
2. Parse input with `respondToDayOfConfirmationSchema`
3. Fetch localTransaction row by ID
4. Validate caller is the **seller** on this transaction (`session.userId === tx.sellerId`)
5. CASL check: `ability.can('update', sub('LocalTransaction', { sellerId: tx.sellerId }))`
6. Validate `dayOfConfirmationSentAt !== null` (buyer must have sent the request)
7. Validate `dayOfConfirmationRespondedAt === null` (not already responded)
8. Validate `dayOfConfirmationExpired === false` (window has not expired)
9. Validate status is not terminal
10. Update localTransaction: `{ dayOfConfirmationRespondedAt: now, updatedAt: now }`
11. Remove pending BullMQ timeout job: `local-day-of-confirmation-timeout` queue, job ID `day-of-confirm-timeout-{txId}`
12. Notify buyer via `notify(tx.buyerId, 'local.dayof.confirmed', { time: formatTime(tx.scheduledAt) })`
13. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`
14. Return `{ success: true }`

### 2.5 BullMQ Job — Day-of Confirmation Timeout

**Create: `src/lib/jobs/local-day-of-confirmation-timeout.ts`**

A delayed BullMQ job that fires when the seller's 2-hour response window expires.

**Queue name:** `local-day-of-confirmation-timeout` (hyphens per project convention)

**Job data interface:**
```typescript
interface DayOfConfirmationTimeoutData {
  localTransactionId: string;
  orderId: string;
  sellerId: string;
  buyerId: string;
}
```

**Enqueue helper:**
```typescript
export async function enqueueDayOfConfirmationTimeout(
  data: DayOfConfirmationTimeoutData,
): Promise<void>
```

- Read `commerce.local.dayOfConfirmationResponseHours` (default 2) from platform settings
- Delay = `responseHours * 60 * 60 * 1000`
- Job ID: `day-of-confirm-timeout-{localTransactionId}`
- `removeOnComplete: true`, `removeOnFail: { count: 100 }`

**Worker processing logic:**

1. Fetch localTransaction row by ID
2. If not found, log warning and return
3. If terminal status (`COMPLETED`, `CANCELED`, `NO_SHOW`, `DISPUTED`), log info and return (no-op)
4. If `dayOfConfirmationRespondedAt !== null`, log info and return (seller already responded, race condition)
5. If `dayOfConfirmationExpired === true`, log info and return (already processed, idempotent)
6. **Mark as expired:**
   ```
   { dayOfConfirmationExpired: true, updatedAt: now }
   ```
7. **Post SELLER_DARK reliability mark:**
   - Read `commerce.local.markSellerDark` (default -1) from platform settings
   - Call `postReliabilityMark({ userId: tx.sellerId, transactionId: tx.id, eventType: 'SELLER_DARK', marksApplied: marks })`
8. **Notify buyer** that seller did not confirm:
   - `notify(tx.buyerId, 'local.dayof.expired', { time: '...' })`
9. **Notify seller** that they failed to respond:
   - `notify(tx.sellerId, 'local.dayof.expired_seller', { ... })`
10. Log the escalation

**Dark Seller Escalation (per A9):**
The addendum says: "If seller doesn't respond to day-of confirmation AND doesn't check in at scheduled time -> no-show marks apply immediately (no need to wait the full 30-minute check-in window)."

This escalation is handled NOT by this timeout job, but by the existing **no-show check worker** (`local-noshow-check.ts`). When that worker fires (30 minutes after one party checks in), it already checks if the other party is present. The key difference per A9 is: if `dayOfConfirmationExpired === true` AND seller is the no-show party, the no-show check should fire immediately rather than waiting the full 30-minute window.

**HOWEVER**, modifying the no-show check worker is beyond the scope of G2.12. The addendum says "no need to wait the full 30-minute check-in window" but there is no mechanism to trigger early no-show detection unless we add logic to the check-in flow or create a new BullMQ job. This is a future enhancement.

**SPEC AMBIGUITY — Owner decision needed:** The "Dark Seller Escalation" paragraph says no-show marks apply immediately without waiting the 30-minute check-in window. There are two interpretations:

- **Option A (minimal):** The timeout job posts the SELLER_DARK mark (-1) and sets the `dayOfConfirmationExpired` flag. If the seller subsequently no-shows at the meetup, the normal no-show detection also fires and adds SELLER_NOSHOW (-3) marks. Total = -4. The "immediately" language means the -1 mark is applied immediately at timeout expiry, not that the -3 no-show mark fires early.

- **Option B (aggressive):** If `dayOfConfirmationExpired === true` AND buyer checks in, the no-show check timer is shortened (e.g., 0 delay instead of 30 minutes). This requires modifying the check-in action.

**This prompt implements Option A** (minimal interpretation). The SELLER_DARK mark fires immediately on timeout. No-show detection runs on its existing schedule. The combined marks (-1 SELLER_DARK + -3 SELLER_NOSHOW = -4 total) are severe enough. Option B can be added in a follow-up if the owner prefers it.

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Patterns
- Do NOT allow the seller to send a day-of confirmation request. Per A9: "Buyer can send a structured meetup confirmation request" — buyer only
- Do NOT allow more than one confirmation request per transaction. Per A9: "One confirmation request per transaction"
- Do NOT allow sending the request outside the 12-hour window before `scheduledAt`
- Do NOT modify the `localTransactionStatusEnum` — day-of confirmation is a sub-flow state tracked by columns, not a status transition
- Do NOT modify the state machine — the transaction stays in its current status throughout this flow
- Do NOT hardcode the 12-hour window or 2-hour response time — read from `platform_settings`
- Do NOT hardcode the SELLER_DARK mark value (-1) — read from `platform_settings`
- Do NOT use `as any`, `@ts-ignore`, or `@ts-expect-error`
- Do NOT use floats for money — all values in integer cents
- Do NOT spread request body into DB update — explicit field mapping only
- Do NOT create files over 300 lines
- Do NOT leave `console.log` in production code — use `logger` from `@/lib/logger`

### Gotchas
- `localMeetupCard.tsx` is at 300 lines (or 301 with G2.11's cancel button). Adding the day-of confirmation import + render will push it over 300 lines. The installer MUST extract an existing section to make room. See section 2.7 for guidance.
- `scheduledAt` may be a `Date` object or a string depending on how Drizzle serializes timestamps. Always construct a `new Date(tx.scheduledAt)` when doing math.
- The BullMQ timeout job must handle race conditions: seller may respond at the exact moment the timeout fires. Always check `dayOfConfirmationRespondedAt` before posting the SELLER_DARK mark.
- The "Confirm I'm coming" action by the seller does NOT change the transaction status. It only sets `dayOfConfirmationRespondedAt` and removes the pending timeout job. The seller must still check in at the meetup location separately.
- The "reschedule" path from the seller's day-of confirmation view uses the existing `proposeRescheduleAction`. The day-of confirmation component should render a link/button that opens the reschedule flow (existing `RescheduleFlow` component), not duplicate the reschedule logic.
- If the seller responds by proposing a reschedule (via the existing reschedule flow), the day-of confirmation timeout job must still be removed. The reschedule action does NOT know about day-of confirmation. The installer has two options:
  - **Option A:** Have the `proposeRescheduleAction` check for and remove the day-of timeout job (cross-concern coupling)
  - **Option B:** Have the timeout job check if a reschedule is pending (`status === 'RESCHEDULE_PENDING'`) and skip the SELLER_DARK mark if so
  **This prompt recommends Option B** — the timeout job should check for RESCHEDULE_PENDING status and treat it as a valid seller response (no mark, no expiration flag). This avoids coupling the reschedule action to day-of confirmation state.

## 4. ACCEPTANCE CRITERIA

### Positive Cases
1. Buyer can send a day-of confirmation request when within 12 hours of `scheduledAt`
2. Buyer sees "Ask seller to confirm" button when within the confirmation window
3. Seller receives `local.dayof.request` notification with meetup time after buyer sends request
4. Seller can confirm they are coming by responding to the request
5. Buyer receives `local.dayof.confirmed` notification when seller confirms
6. `dayOfConfirmationSentAt` is set when buyer sends the request
7. `dayOfConfirmationRespondedAt` is set when seller confirms
8. BullMQ timeout job is enqueued with 2-hour delay when buyer sends the request
9. BullMQ timeout job is removed when seller confirms
10. BullMQ timeout job fires after 2 hours if seller does not respond
11. `dayOfConfirmationExpired` is set to `true` when timeout job fires
12. `SELLER_DARK` reliability mark (-1) is posted when timeout expires
13. Buyer receives `local.dayof.expired` notification when timeout fires
14. Seller receives `local.dayof.expired_seller` notification when timeout fires
15. UI correctly shows all buyer view states (not sent, waiting, confirmed, expired)
16. UI correctly shows all seller view states (requested, confirmed, expired)
17. The confirmation window hours value is read from `commerce.local.dayOfConfirmationWindowHours`
18. The response timeout value is read from `commerce.local.dayOfConfirmationResponseHours`
19. The SELLER_DARK mark value is read from `commerce.local.markSellerDark`

### Negative Cases
20. Unauthenticated user cannot send or respond to a confirmation request
21. Seller cannot send a day-of confirmation request (buyer only)
22. Buyer cannot respond to a day-of confirmation (seller only)
23. Buyer cannot send a request if one has already been sent (`dayOfConfirmationSentAt !== null`)
24. Buyer cannot send a request outside the 12-hour window (too early)
25. Buyer cannot send a request after `scheduledAt` has passed (too late)
26. Buyer cannot send a request if `scheduledAtConfirmedAt` is null (time not confirmed)
27. Buyer cannot send a request if status is terminal (COMPLETED, CANCELED, NO_SHOW)
28. Buyer cannot send a request if status is RESCHEDULE_PENDING
29. Buyer cannot send a request if status is BOTH_CHECKED_IN or later
30. Seller cannot confirm if no request was sent
31. Seller cannot confirm if already confirmed
32. Seller cannot confirm if the window already expired
33. CASL check fails for non-owner buyer (returns `Not found`)
34. CASL check fails for non-owner seller (returns `Not found`)

### Timeout / Race Condition
35. If seller responds at the exact moment the timeout fires, `dayOfConfirmationRespondedAt` check in the worker prevents double-marking
36. If transaction status is RESCHEDULE_PENDING when the timeout fires, no SELLER_DARK mark is applied (seller chose to reschedule — valid response)
37. If transaction is in a terminal status when the timeout fires, no mark is applied (no-op)
38. Timeout job is idempotent — running twice produces the same result

### Data Integrity
39. All platform setting values read via `getPlatformSetting()` with documented fallback defaults
40. `dayOfConfirmationExpired` only transitions from `false` to `true`, never back
41. `dayOfConfirmationSentAt` is immutable once set (one request per transaction)

### Vocabulary
42. No banned terms appear in UI copy or code comments
43. No references to "wallet", "withdraw", "FVF", "SellerTier", etc.
44. Notification templates use consistent language ("meetup" not "meeting" or "pickup appointment")

## 8. IMPLEMENTATION NOTES

### Window Calculation

The buyer-side window check determines when the "Ask seller to confirm" button appears:

```typescript
function isWithinConfirmationWindow(
  scheduledAt: Date,
  windowHours: number,
): boolean {
  const now = new Date();
  const windowStart = new Date(scheduledAt);
  windowStart.setHours(windowStart.getHours() - windowHours);
  return now >= windowStart && now < scheduledAt;
}
```

This must be checked both:
- **Client-side** (for button visibility in the UI component)
- **Server-side** (in the action, to prevent manipulation)

### BullMQ Job Removal on Seller Response

When the seller confirms, the pending timeout job must be removed:

```typescript
try {
  const job = await dayOfConfirmationTimeoutQueue.getJob(
    `day-of-confirm-timeout-${tx.id}`
  );
  if (job) await job.remove();
} catch (err) {
  logger.warn('[local-day-of] Could not remove timeout job', {
    transactionId: tx.id,
    error: String(err),
  });
}
```

### RESCHEDULE_PENDING Guard in Timeout Worker

```typescript
// Seller chose to reschedule — counts as a response, no mark

  return;
}
```

## 9. SPEC DECISIONS REQUIRING OWNER CONFIRMATION

### Decision 1: Schema Columns for Day-of Confirmation

The addendum A9 does not have an explicit "Schema Additions" subsection (unlike A3/A4/A7/A8). This prompt adds three columns (`dayOfConfirmationSentAt`, `dayOfConfirmationRespondedAt`, `dayOfConfirmationExpired`) to track the flow state. These follow the established pattern of other addendum sub-steps.

**If the owner wants different column names or a different approach (e.g., a separate table, or a JSON column), this prompt must be revised.**

### Decision 2: Dark Seller Escalation — Immediate No-Show

The addendum A9 says: "If seller doesn't respond to day-of confirmation AND doesn't check in at scheduled time -> no-show marks apply immediately (no need to wait the full 30-minute check-in window)."

This prompt implements the minimal interpretation (Option A): the SELLER_DARK mark (-1) fires on timeout, and the normal no-show detection runs on its 30-minute schedule. The combined marks are -1 (dark) + -3 (no-show) = -4 total.

The aggressive interpretation (Option B) would require modifying the check-in action to short-circuit the no-show timer when `dayOfConfirmationExpired === true`. This adds cross-feature coupling.

**If the owner wants Option B, the installer must also modify `checkInToMeetupAction` in `local-transaction.ts` and `enqueueNoShowCheck` in `local-noshow-check.ts`.**

### Decision 3: Seller Reschedule as Valid Response

Per A9 outcomes table, the seller can respond by "proposing reschedule" which "triggers reschedule flow." This prompt treats a pending reschedule (`RESCHEDULE_PENDING` status) as a valid response in the timeout worker, preventing the SELLER_DARK mark. The seller actively engaged by proposing a new time — they are not "dark."

**If the owner wants the SELLER_DARK mark to fire even when the seller proposes a reschedule (punishing the seller for not simply confirming), this prompt must be revised.**
