## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2.1 Schema Changes

**NO new tables are needed.**

**NO new enum values are needed.** The addendum A8 says:
> `CANCELED_BY_BUYER` and `CANCELED_BY_SELLER` as sub-states of CANCELED for reporting clarity.

These are **NOT** added to the `localTransactionStatusEnum` (which already has `CANCELED`). Instead, the "sub-state" distinction is captured via a **new nullable column** on `localTransaction`:

```
canceledByParty:  text('canceled_by_party')  // 'BUYER' | 'SELLER' | null — for reporting
```

**SPEC DECISION NOTE:** The addendum says "sub-states of CANCELED for reporting clarity" but does NOT say to add them as enum values. Adding them as enum values would break the state machine (every existing transition to CANCELED would need two targets). The correct interpretation is a metadata column. This is consistent with how `noShowParty` already works on the same table.

**New column on `localTransaction`:**

| Column | Type | Nullable | Default | Purpose |
|--------|------|----------|---------|---------|
| `canceledByParty` | `text('canceled_by_party')` | Yes | null | `'BUYER'` or `'SELLER'` — reporting sub-state of CANCELED |

**Migration:** Create migration `0014_local_canceled_by_party.ts` adding this column.

### 2.2 Server Actions

**Create: `src/lib/actions/local-cancel.ts`**

One server action:

```typescript
export async function cancelLocalTransactionAction(
  data: z.infer<typeof cancelLocalTransactionSchema>
): Promise<ActionResult>
```

**Input schema** (in `src/lib/validations/local.ts`):
```typescript
export const cancelLocalTransactionSchema = z.object({
  localTransactionId: z.string().min(1),
  reason: z.string().min(1).max(500).optional(),
}).strict();
```

**Action logic (in order):**

1. `authorize()` — get session + ability
2. Parse input with `cancelLocalTransactionSchema`
3. Fetch localTransaction row by ID
4. Validate caller is buyer or seller on this transaction
5. CASL check: `ability.can('update', sub('LocalTransaction', { buyerId/sellerId }))`
6. Validate status is in cancellable set: `['SCHEDULED', 'SELLER_CHECKED_IN', 'BUYER_CHECKED_IN', 'RESCHEDULE_PENDING']`
   - NOT `BOTH_CHECKED_IN` (per A8: "NOT available once BOTH_CHECKED_IN")
   - NOT any terminal status (COMPLETED, CANCELED, NO_SHOW, DISPUTED)
   - NOT `ADJUSTMENT_PENDING` or `RECEIPT_CONFIRMED`
7. Validate state machine: `canTransition(tx.status, 'CANCELED')`
8. Determine party: `isBuyer ? 'BUYER' : 'SELLER'`
9. Call `cancelLocalTransaction()` service (see 2.3)
10. `revalidatePath('/my/buying/orders')` and `revalidatePath('/my/selling/orders')`
11. Return `{ success: true }`

### 2.3 Commerce/Business Logic Service

**Create: `src/lib/commerce/local-cancel.ts`**

This is a plain TypeScript module (NOT `'use server'`). Contains the core cancellation orchestration.

**Function: `cancelLocalTransaction()`**

```typescript
export interface CancelLocalTransactionParams {
  transaction: LocalTransactionRow;
  cancelingParty: 'BUYER' | 'SELLER';
  cancelingUserId: string;
  reason?: string;
}

export async function cancelLocalTransaction(
  params: CancelLocalTransactionParams
): Promise<void>
```

**Orchestration steps:**

1. **Determine reliability mark** by comparing current time to `scheduledAt`:
   - If `scheduledAt` is null (never scheduled): 0 marks (graceful)
   - If `scheduledAt - now >= 24 hours`: 0 marks → event type `{PARTY}_CANCEL_GRACEFUL`
   - If `scheduledAt - now < 24 hours AND scheduledAt - now >= 2 hours`: -1 mark → event type `{PARTY}_CANCEL_LATE`
   - If `scheduledAt - now < 2 hours` (same day, under 2hr): -2 marks → event type `{PARTY}_CANCEL_SAMEDAY`

   Read mark values from platform_settings (with fallback defaults):
   - `commerce.local.markCancelLate` → default `-1`
   - `commerce.local.markCancelSameday` → default `-2`

   (Graceful = 0 marks, which causes `postReliabilityMark()` to no-op per existing logic)

2. **Update localTransaction:**
   ```typescript
   {
     status: 'CANCELED',
     canceledByParty: party,  // 'BUYER' or 'SELLER'
     updatedAt: now,
   }
   ```

3. **Update order:**
   ```typescript
   {
     status: 'CANCELED',
     canceledByUserId: cancelingUserId,
     cancelInitiator: party === 'BUYER' ? 'BUYER' : 'SELLER',
     cancelReason: reason ?? `Local meetup canceled by ${party.toLowerCase()}`,
     canceledAt: now,
     updatedAt: now,
   }
   ```

4. **Post reliability mark** via `postReliabilityMark()` (existing function in `local-reliability.ts`). Pass the computed event type and marks. The function handles 0-mark no-op internally.

5. **Issue Stripe refund** — full refund of the payment:
   - Read `orderPayment.stripePaymentIntentId` for the order
   - If `stripePaymentIntentId` exists, call Stripe refund API: `stripe.refunds.create({ payment_intent: stripePaymentIntentId })`
   - Full refund (no amount parameter = full refund)
   - If `stripePaymentIntentId` is null (cash transaction), skip refund
   - Log refund result; do NOT throw on Stripe failure (log error, continue — manual reconciliation path)

6. **Re-activate listing:**
   - Query `orderItem` for the order to get `listingId`
   - Update `listing.status` from current value to `'ACTIVE'` — but ONLY if listing.status is `'SOLD'` or `'PAUSED'` (defensive check — don't touch listings in other states like REMOVED)

   **IMPORTANT:** The addendum A8 says "Listing returns to ACTIVE status on cancellation." However, G2.14 (listing auto-reserve) has NOT been implemented yet. Currently, listings may or may not change status at order creation. The defensive check ensures we only re-activate if the listing was taken off the market. If the listing is already ACTIVE, this is a no-op.

7. **Remove pending BullMQ jobs:**
   - Remove auto-cancel job: `localAutoCancelQueue.getJob('local-auto-cancel-{txId}')` → `job.remove()`
   - Remove no-show check jobs: `localNoShowCheckQueue.getJob('noshow-{txId}-BUYER')` → `job.remove()` AND `localNoShowCheckQueue.getJob('noshow-{txId}-SELLER')` → `job.remove()`
   - Remove schedule nudge job: `localScheduleNudgeQueue.getJob('local-schedule-nudge-{txId}')` → `job.remove()` (if queue is importable)
   - Wrap each removal in try/catch — job may not exist or may have already completed. Log warnings on failure, never throw.

8. **Notify other party:**
   - Use `notify(otherPartyId, 'local.cancel', { ... })`
   - Template data: item title or orderId, party who canceled

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Patterns
- Do NOT add `CANCELED_BY_BUYER` or `CANCELED_BY_SELLER` to the `localTransactionStatusEnum` — the state machine uses `CANCELED` as the terminal status. The sub-state is metadata via `canceledByParty` column.
- Do NOT charge any fee for cancellation — full refund always, per Decision #114 and Addendum A5
- Do NOT allow cancellation once `BOTH_CHECKED_IN` — per Addendum A8
- Do NOT hardcode reliability mark values — read from `platform_settings` with fallback defaults
- Do NOT use `as any`, `@ts-ignore`, or `@ts-expect-error`
- Do NOT use floats for money — all values in integer cents
- Do NOT spread request body into DB update — explicit field mapping only
- Do NOT create files over 300 lines
- Do NOT leave `console.log` in production code — use `logger` from `@/lib/logger`

### Gotchas
- `localMeetupCard.tsx` is 299 lines — adding the cancel button import + render MUST stay under 300. If it pushes over, extract a separate wrapper or move existing elements into sub-components.
- The `cancelInitiator` column on `order` uses `cancelInitiatorEnum` which has values `['BUYER', 'SELLER', 'SYSTEM', 'ADMIN']`. Map the party to this enum.
- `scheduledAt` may be `null` if meetup time was never confirmed. In this case, cancellation is still allowed (graceful, 0 marks).
- BullMQ job removal is best-effort. Jobs may not exist (never enqueued) or may have already completed. Always wrap in try/catch.
- The Stripe refund call uses `order.paymentIntentId` NOT `orderPayment.stripePaymentIntentId`. Actually — check both. The `order` table has `paymentIntentId` and `orderPayment` table has `stripePaymentIntentId`. Use whichever is populated. Prefer `orderPayment.stripePaymentIntentId` as it's the canonical payment record.
- Cash/off-platform local transactions have no Stripe payment. Check for null `stripePaymentIntentId` before attempting refund.
- The `listing` table uses `listingStatusEnum` which has `['DRAFT', 'ACTIVE', 'PAUSED', 'SOLD', 'ENDED', 'REMOVED']`. There is NO `RESERVED` status yet (that's G2.14). When re-activating, check that the listing is in a status that makes sense to revert (SOLD or PAUSED), not REMOVED or ENDED.

### CLAUDE.md Rules That Apply
- Money as integer cents only
- Fee rates from `platform_settings`, never hardcoded
- Ownership via `userId` always
- Zod `.strict()` on all input schemas
- Explicit field mapping, never spread request body
- Max 300 lines per file
- `strict: true` TypeScript — zero `as any`

## 4. ACCEPTANCE CRITERIA

### Positive Cases
1. Buyer can cancel a SCHEDULED local transaction and receives full refund
2. Seller can cancel a SCHEDULED local transaction and buyer receives full refund
3. Buyer can cancel a SELLER_CHECKED_IN transaction (seller already at location)
4. Seller can cancel a BUYER_CHECKED_IN transaction (buyer already at location)
5. Either party can cancel a RESCHEDULE_PENDING transaction
6. Cancellation 24hr+ before meetup results in 0 reliability marks (no event recorded)
7. Cancellation under 24hr results in -1 mark on cancelling party
8. Cancellation under 2hr results in -2 marks on cancelling party
9. Cancellation with null `scheduledAt` results in 0 marks (graceful)
10. Order status changes to CANCELED with correct `canceledByUserId`, `cancelInitiator`, `cancelReason`, `canceledAt`
11. `localTransaction.canceledByParty` is set to `'BUYER'` or `'SELLER'` for reporting
12. Pending BullMQ auto-cancel job is removed on cancellation
13. Pending BullMQ no-show check jobs are removed on cancellation
14. Other party receives notification via `notify()` with template `local.cancel`
15. `revalidatePath` called for both buying and selling order pages

### Negative Cases
16. Unauthenticated user cannot cancel (returns `Unauthorized`)
17. User who is neither buyer nor seller cannot cancel (returns `Not found`)
18. Cannot cancel when status is BOTH_CHECKED_IN (returns error)
19. Cannot cancel when status is ADJUSTMENT_PENDING (returns error)
20. Cannot cancel when status is RECEIPT_CONFIRMED (returns error)
21. Cannot cancel when status is COMPLETED (returns error — terminal)
22. Cannot cancel when status is already CANCELED (returns error — terminal)
23. Cannot cancel when status is NO_SHOW (returns error — terminal)
24. Cannot cancel when status is DISPUTED (returns error — terminal)
25. CASL check fails for non-owner buyer (returns `Not found`)
26. CASL check fails for non-owner seller (returns `Not found`)

### Data Integrity
27. All monetary values are integer cents (no floats)
28. Reliability marks stored as negative integers
29. `canceledByParty` column only contains `'BUYER'` or `'SELLER'` (never null after cancellation)
30. Order `cancelInitiator` uses existing `cancelInitiatorEnum` values

### Vocabulary
31. No banned terms appear in UI copy or code comments
32. No references to "wallet", "withdraw", "FVF", "SellerTier", etc.
33. Notification template says "canceled" not "cancelled" (consistency with existing templates)

### Stripe
34. If `stripePaymentIntentId` exists, Stripe refund is issued (full refund)
35. If `stripePaymentIntentId` is null (cash transaction), no refund attempted
36. Stripe refund failure is logged but does NOT block the cancellation flow

### Listing Re-activation
37. Listing status returns to ACTIVE after cancellation (if listing was in SOLD or PAUSED status)
38. Listing in ACTIVE status is not touched (no-op)
39. Listing in REMOVED or ENDED status is not touched (defensive)

### Cancellation Window Calculation

The mark determination logic compares `now` to `scheduledAt`:

```typescript
function determineCancelEventType(
  party: 'BUYER' | 'SELLER',
  scheduledAt: Date | null,
): { eventType: LocalReliabilityEventType; marks: number } {
  if (scheduledAt === null) {
    // Never scheduled — graceful cancel
    return {
      eventType: party === 'BUYER' ? 'BUYER_CANCEL_GRACEFUL' : 'SELLER_CANCEL_GRACEFUL',
      marks: 0,
    };
  }

  const now = new Date();
  const hoursUntil = (scheduledAt.getTime() - now.getTime()) / (1000 * 60 * 60);

  if (hoursUntil >= 24) {
    // 24hr+ before — graceful
    return {
      eventType: party === 'BUYER' ? 'BUYER_CANCEL_GRACEFUL' : 'SELLER_CANCEL_GRACEFUL',
      marks: 0, // read from platform_settings for non-zero cases
    };
  }

  if (hoursUntil >= 2) {
    // Under 24hr but 2hr+ before — late
    return {
      eventType: party === 'BUYER' ? 'BUYER_CANCEL_LATE' : 'SELLER_CANCEL_LATE',
      marks: -1, // read from commerce.local.markCancelLate
    };
  }

  // Under 2hr — same day
  return {
    eventType: party === 'BUYER' ? 'BUYER_CANCEL_SAMEDAY' : 'SELLER_CANCEL_SAMEDAY',
    marks: -2, // read from commerce.local.markCancelSameday
  };
}
```

**IMPORTANT:** The mark values (-1, -2) MUST be read from `getPlatformSetting()` with these as fallback defaults. Do not hardcode.
