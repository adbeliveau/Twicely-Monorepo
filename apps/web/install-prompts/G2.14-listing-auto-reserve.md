# Install Prompt: G2.14 — Listing Auto-Reserve on Escrow Creation

**Phase & Step:** `[G2.14]`
**Feature Name:** Listing Auto-Reserve on Escrow Creation
**One-line Summary:** When a SafeTrade local transaction is created at checkout, automatically transition the listing from ACTIVE to RESERVED so other buyers cannot purchase it; reverse (RESERVED to ACTIVE) on cancellation or no-show expiry; transition RESERVED to SOLD on transaction completion. RESERVED listings are hidden from search but accessible via direct URL with a "This item is reserved" banner.

### 2.2 Reserve Listing Service

**Create: `src/lib/commerce/local-reserve.ts`**

A dedicated service module for the auto-reserve lifecycle: reserve on transaction creation, unreserve on cancellation/no-show, and transition to SOLD on completion.

**Exported functions:**

#### Function 1: `reserveListingForLocalTransaction`

Called after a local transaction is created (from `createLocalTransaction` or from the checkout finalize flow).

```typescript
export async function reserveListingForLocalTransaction(
  orderId: string,
): Promise<{ success: boolean; listingId?: string; error?: string }>
```

**Logic:**
1. Query `orderItem` by `orderId` to get `listingId` (local orders have exactly one item per order)
2. Query `listing` by `listingId` to get current `status`
3. If listing `status !== 'ACTIVE'`, log warning and return `{ success: false, error: 'Listing not in ACTIVE status' }` (may already be reserved by another transaction, or SOLD, etc.)
4. Update listing: `{ status: 'RESERVED', updatedAt: now }`
5. Decline all pending offers for this listing via `declineAllPendingOffersForListing(listingId)` — per Feature Lock-in section 1: "If another buyer purchases at full price while an offer is pending, all pending offers on that listing are auto-declined." Reserving the listing for a local transaction is equivalent: the item is committed to a buyer.
6. Log the reservation: `[local-reserve] Listing reserved for local transaction`
7. Return `{ success: true, listingId }`

#### Function 2: `unreserveListingForLocalTransaction`

Called when a local transaction is canceled, no-showed, or auto-canceled AND the listing was in RESERVED status.

```typescript
export async function unreserveListingForLocalTransaction(
  orderId: string,
): Promise<void>
```

**Logic:**
1. Query `orderItem` by `orderId` to get `listingId`
2. If no orderItem found, log warning and return
3. Query `listing` by `listingId` to get current `status`
4. If listing `status === 'RESERVED'`, update to `{ status: 'ACTIVE', updatedAt: now }`
5. If listing `status === 'SOLD'` or `status === 'PAUSED'`, update to `{ status: 'ACTIVE', updatedAt: now }` (this is the existing G2.11 behavior)
6. If listing is in any other status (DRAFT, ENDED, REMOVED), do NOT change it — something else happened (moderation action, seller ended listing, etc.)
7. Log the unreservation

**IMPORTANT:** This function REPLACES the inline listing re-activation logic currently in `cancelLocalTransaction()` in `local-cancel.ts` (lines 180-205). The cancel flow should call `unreserveListingForLocalTransaction(tx.orderId)` instead of doing the query + status check + update inline.

#### Function 3: `markListingSoldForLocalTransaction`

Called when a local transaction is confirmed (RECEIPT_CONFIRMED -> COMPLETED).

```typescript
export async function markListingSoldForLocalTransaction(
  orderId: string,
): Promise<void>
```

**Logic:**
1. Query `orderItem` by `orderId` to get `listingId`
2. If no orderItem found, log warning and return
3. Query `listing` by `listingId` to get current `status`
4. If listing `status === 'RESERVED'` or `status === 'ACTIVE'`, update to `{ status: 'SOLD', soldAt: now, updatedAt: now }`
5. Log the status change

### 2.3 Integration — Reserve at Transaction Creation

**Modify: `src/lib/commerce/local-transaction.ts`**

After the local transaction is created successfully (after the token regeneration at line 131), call `reserveListingForLocalTransaction`:

```typescript
import { reserveListingForLocalTransaction } from './local-reserve';

// After line 131 (token update), before the return:
// Auto-reserve the listing (SafeTrade only — fire-and-forget, do not block on failure)
void reserveListingForLocalTransaction(orderId).catch((err) => {
  logger.error('[local-transaction] Failed to reserve listing', {
    orderId,
    error: String(err),
  });
});
```

The reservation is fire-and-forget because the local transaction creation should not fail if the reservation fails (the transaction is the primary action; the reservation is a secondary effect). If the reservation fails, the listing remains ACTIVE — not ideal but the seller can still complete the meetup. The fraud detection system (G2.15) will catch double-sales.

### 2.4 Integration — Unreserve on Cancellation

// REPLACE lines 180-205 (the inline listing re-activation block) with:
// 6. Re-activate listing if it was reserved or taken off the market

This is cleaner because:

- The `unreserveListingForLocalTransaction` function checks for RESERVED, SOLD, and PAUSED — handling all cases.

### 2.6 Integration — Unreserve on No-Show and Auto-Cancel

The existing `cancelLocalTransaction()` in `local-cancel.ts` is called by:
- `cancelLocalTransactionAction` in `src/lib/actions/local-cancel-action.ts` (user-initiated cancel)
- The auto-cancel BullMQ worker in `src/lib/jobs/local-auto-cancel.ts`

Since we are replacing the inline listing re-activation in `cancelLocalTransaction()` with `unreserveListingForLocalTransaction()` (section 2.4 above), both user-initiated cancellation and auto-cancellation will correctly unreserve the listing.

For the NO_SHOW terminal status: the no-show check worker (`src/lib/jobs/local-noshow-check.ts`) currently marks the transaction as NO_SHOW. Per the addendum A11: "On transaction NO_SHOW + no re-schedule within 24hr -> listing transitions back to ACTIVE." This requires a BullMQ delayed job that fires 24 hours after the NO_SHOW event to check if the transaction was rescheduled or if the listing should be unreserved.

**HOWEVER**, the current no-show flow does NOT trigger cancellation — it marks the transaction as NO_SHOW (terminal) and the auto-cancel job runs separately. Looking at the addendum A11 more carefully: "On transaction NO_SHOW + no re-schedule within 24hr" — but NO_SHOW is a terminal status in the state machine (no transitions out). The "no re-schedule within 24hr" clause implies a grace period where the seller could potentially create a new transaction, but that is not how the system works currently.

**SPEC AMBIGUITY — Owner decision needed:** The addendum says "On transaction NO_SHOW + no re-schedule within 24hr -> listing transitions back to ACTIVE." But NO_SHOW is terminal — there is no reschedule from NO_SHOW. Two interpretations:

- **Option A (this prompt implements):** When a transaction reaches NO_SHOW status, immediately unreserve the listing (RESERVED -> ACTIVE). The "no re-schedule within 24hr" condition is satisfied trivially because NO_SHOW is terminal and cannot be rescheduled.

- **Option B (deferred):** Add a 24-hour delayed job after NO_SHOW that checks whether a new local transaction was created for the same listing. If yes, leave it reserved. If no, unreserve. This is more complex and the addendum does not clearly specify the mechanism.

**This prompt implements Option A** — unreserve immediately on NO_SHOW. This is simpler, consistent with cancellation behavior, and the buyer gets a full refund regardless. The listing becomes available for other buyers immediately.

To implement this, the no-show check worker must call `unreserveListingForLocalTransaction` when it marks a transaction as NO_SHOW.

**Modify: `src/lib/jobs/local-noshow-check.ts`**

Add import and call `unreserveListingForLocalTransaction(tx.orderId)` after the transaction is marked NO_SHOW.

- The `listingStatusEnum` change requires a Drizzle migration. PostgreSQL enums cannot have values added inline — it needs `ALTER TYPE listing_status ADD VALUE 'RESERVED'`. The installer must generate and run the migration.
- The `UnavailableReason` type in `availability.ts` is a TypeScript union, not a database enum. Adding `'RESERVED'` is a code-only change.

## 4. ACCEPTANCE CRITERIA

### Positive Cases — Reserve
1. When `createLocalTransaction()` succeeds, the listing transitions from ACTIVE to RESERVED
2. The `reserveListingForLocalTransaction` function queries `orderItem` to find the listing, then updates `listing.status` to `'RESERVED'`
3. All pending offers on the listing are auto-declined when the listing is reserved (via `declineAllPendingOffersForListing`)
4. The listing's `updatedAt` timestamp is set when status changes to RESERVED

### Positive Cases — Search and Browse
5. RESERVED listings do NOT appear in search results (`searchListings` filters to ACTIVE only)
6. RESERVED listings do NOT appear in category browse pages
7. RESERVED listings ARE accessible via direct URL (`/i/[slug]`)
8. Direct URL shows a blue "This item is currently reserved for a local pickup" banner

### Positive Cases — Unreserve
9. When a local transaction is canceled (user-initiated), the listing transitions from RESERVED to ACTIVE
10. When a local transaction is auto-canceled (BullMQ worker), the listing transitions from RESERVED to ACTIVE
11. When a local transaction reaches NO_SHOW, the listing transitions from RESERVED to ACTIVE
12. The listing remains ACTIVE after unreservation (no leftover RESERVED status)

### Positive Cases — Completion
13. When a local transaction is confirmed (RECEIPT_CONFIRMED), the listing transitions from RESERVED to SOLD
14. The listing's `soldAt` timestamp is set when transitioning to SOLD

### Positive Cases — Availability
15. `checkListingAvailability` returns `{ available: false, reason: 'RESERVED' }` for RESERVED listings
16. Checkout blocks purchase of RESERVED listings with "no longer available" error message

### Positive Cases — Metadata
17. RESERVED listings have `noindex` robots directive in metadata
18. RESERVED listing metadata title still renders correctly

### Negative Cases
19. Listing does NOT transition to RESERVED if current status is not ACTIVE (e.g., already SOLD, PAUSED, ENDED)
20. Reservation failure does NOT prevent local transaction creation (fire-and-forget)
21. Cash/off-platform local transactions do NOT reserve the listing (SafeTrade only — but since cash transactions do NOT create a localTransaction via `createLocalTransaction`, this is automatically handled)
22. RESERVED listings cannot be purchased at full price by another buyer
23. RESERVED listings cannot receive new offers (offers are declined on reservation)
24. The `unreserveListingForLocalTransaction` function does NOT change listings in DRAFT, ENDED, or REMOVED status
25. The `unreserveListingForLocalTransaction` function does NOT throw if no orderItem is found (graceful no-op)

### Race Conditions
26. If two buyers attempt to purchase the same listing simultaneously, only one succeeds (the `SELECT FOR UPDATE` lock in `createOrdersFromCart` prevents double-sell; the reservation is a secondary effect)
27. If the local transaction is canceled at the exact moment the listing is being reserved, the cancel flow's `unreserveListingForLocalTransaction` runs after the reserve and correctly returns it to ACTIVE

### Data Integrity
28. The `listingStatusEnum` in `enums.ts` contains `'RESERVED'` after the migration
29. The `ListingStatus` type in `listings.ts` includes `'RESERVED'`
30. The `UnavailableReason` type in `availability.ts` includes `'RESERVED'`

### Vocabulary
31. No banned terms appear in UI copy, banner text, or code comments
32. The banner uses "reserved" (not "held", "locked", or "pending")
33. No references to "wallet", "withdraw", "FVF", "SellerTier", etc.

## 8. IMPLEMENTATION NOTES

### Drizzle Migration

To add a value to an existing PostgreSQL enum, the migration SQL is:

```sql
ALTER TYPE listing_status ADD VALUE IF NOT EXISTS 'RESERVED';
```

Generate the migration using `pnpm drizzle-kit generate` after updating the enum definition. If Drizzle Kit does not handle enum value additions properly (known issue with some versions), the installer may need to create a custom migration file.

### Fire-and-Forget Pattern for Reservation

The reservation call in `createLocalTransaction` uses `void ... .catch()` pattern (fire-and-forget):

```typescript
void reserveListingForLocalTransaction(orderId).catch((err) => {
  logger.error('[local-transaction] Failed to reserve listing', {
    orderId,
    error: String(err),
  });
});
```

This pattern is used because:
1. Transaction creation is the primary operation — it must succeed even if reservation fails
2. The reservation is a secondary side effect
3. If reservation fails, the listing remains ACTIVE — other buyers could theoretically purchase it, but the no-show/fraud detection (G2.15) will catch this
4. The `createLocalTransaction` function does not have access to the Drizzle transaction context (the DB insert is already committed), so the reservation runs as a separate query

## 9. SPEC DECISIONS REQUIRING OWNER CONFIRMATION

### Decision 1: Immediate Unreserve on NO_SHOW

The addendum A11 says: "On transaction NO_SHOW + no re-schedule within 24hr -> listing transitions back to ACTIVE."

This prompt implements immediate unreserve on NO_SHOW (Option A) rather than a 24-hour delayed check (Option B). Rationale: NO_SHOW is a terminal status with no rescheduling possible, so the "no re-schedule within 24hr" condition is trivially satisfied.

**If the owner wants the 24-hour delay, a new BullMQ delayed job is needed (similar to the auto-cancel pattern). The job would fire 24 hours after NO_SHOW and call `unreserveListingForLocalTransaction` only if no new local transaction exists for that listing.**

### Decision 2: Offers Declined on Reservation

Per Feature Lock-in section 1, pending offers are auto-declined when a listing is no longer available. This prompt declines all pending offers when the listing is reserved (not just when it's sold). This prevents buyers from having offers accepted on a listing that's committed to another buyer.

**If the owner wants offers to remain pending during RESERVED status (and only be declined on actual SOLD), remove the `declineAllPendingOffersForListing` call from `reserveListingForLocalTransaction`.**

### Decision 3: No New Platform Settings

The addendum A11 does not specify any platform settings for the auto-reserve feature. The behavior is entirely mechanical (create -> reserve, cancel -> unreserve, complete -> sold). No admin-configurable knobs are needed.

**If the owner wants an admin toggle to enable/disable auto-reserve, a `commerce.local.autoReserveEnabled` boolean setting would need to be added.**

### Decision 4: SafeTrade-Only Enforcement

Per the addendum A0 feature matrix, "Auto-reserve listing" is a SafeTrade-only feature. Cash transactions do not create a `localTransaction` via `createLocalTransaction()` — they are tracked manually. Therefore, the auto-reserve logic is naturally limited to SafeTrade transactions without needing an explicit check.

**If the codebase ever adds a `createLocalTransaction` call for cash transactions, an explicit SafeTrade check would be needed in `reserveListingForLocalTransaction`.**
