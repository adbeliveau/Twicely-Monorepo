## G9.1 Helpdesk Wiring — Root Causes (2026-03-16)

### Bug 1: Replies Not Appearing
- case-workspace.tsx uses raw `fetch()` to POST `/api/hub/helpdesk/cases/[id]/messages` instead of `addAgentReply` server action
- No `router.refresh()` after send — message written to DB but page data never re-fetched
- DUPLICATE: Both API route and server action `addAgentReply` exist doing same thing
- Fix: Replace fetch with server action, delete API route

### Bug 2: Context Panel "No context"
- `getCaseContext` doesn't null-coalesce `tags` field — if DB returns null, `data.tags.length` throws TypeError
- Promise.all in getCaseContext has no error handling — any sub-query failure crashes whole render
- Fix: Add `tags: caseRecord.tags ?? []`, wrap in try-catch with fallback

### Bug 3: Badges Always 0
- Drawer sidebar in `helpdesk-layout-client.tsx` line 76-79 renders HelpdeskSidebar WITHOUT `badges` prop
- Seed data may only have RESOLVED/CLOSED cases (no open status cases)
- Fix: Pass badges to drawer sidebar, verify seed has open-status cases

### Additional Issues Found
- `ON_HOLD` missing from `CaseStatus` type in helpdesk-badges.tsx (would crash StatusBadge)
- `addAgentReply` doesn't transition NEW -> OPEN on first outbound reply (per Canonical section 3.4)
- Sidebar role-gating broken: permission strings like `helpdesk.macros.view` don't match platformRole values like `HELPDESK_LEAD`
- Macro query hardcodes `category: 'General'` instead of reading from DB

**Why:** G9 built all the pieces but several wiring connections were missed. Common pattern: parallel code paths (fetch vs server action), missing null safety on optional DB fields, and prop forwarding missed in conditional render branches.

**How to apply:** When reviewing helpdesk code, check for: (1) fetch vs server action consistency, (2) null safety on all DB fields, (3) prop forwarding in all conditional render branches.
