## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2.1 Schema Change: Add `localHandlingFlags` to `listing` table

In `src/lib/db/schema/listings.ts`, add after the `localPickupRadiusMiles` column:

```typescript
localHandlingFlags: text('local_handling_flags').array().notNull().default(sql`'{}'::text[]`),
```

**Valid flag values (enforced by Zod, not by DB enum):**
- `NEEDS_VEHICLE` — "Buyer must bring own transport (won't fit in standard car)"
- `NEEDS_HELP` — "Loading help required (heavy/bulky — bring someone)"
- `NEEDS_DISASSEMBLY` — "Disassembly required before transport"
- `NEEDS_EQUIPMENT` — "Special equipment needed (dolly, straps, etc.)"

These are NOT a Postgres enum — they are text values validated by Zod at the application layer. This follows the existing pattern used by `tags` (also `text[].default(sql\`'{}'::text[]\`)`).

## 3. CONSTRAINTS — WHAT NOT TO DO

- **No new database enum** for handling flags — use `text[]` validated by Zod, matching the pattern of `tags`.
- **No new tables** — this is a column addition to the existing `listing` table.
- **No fee calculation changes** — handling flags are informational only.
- **No blocking of checkout** — the acknowledgment checkbox prevents proceeding but does not reject the order server-side. The flags are informational.
- **No changes to the order table** — the flags live on the listing, not the order.
- **No `as any`** or `@ts-ignore`.
- **No files over 300 lines** — if `listing-form.tsx` exceeds 300 lines after changes, extract the AI suggestion handler or submit logic into a utility.
- **Handling flags are only relevant when fulfillmentType is LOCAL_ONLY or SHIP_AND_LOCAL.** When fulfillmentType is SHIP_ONLY, flags MUST be empty array `[]`.
- **Do NOT create a separate "handling flags" page** — the flags are a section within the existing fulfillment section of the listing form.

### Server Actions
13. `createListing` persists `localHandlingFlags` to the database
14. `updateListing` persists `localHandlingFlags` to the database

16. When a listing has `fulfillmentType !== 'SHIP_ONLY'` and `localHandlingFlags.length > 0`, an amber warning box shows the flags near the badge area
17. When a listing has no handling flags, no warning box renders
18. When a listing is `SHIP_ONLY`, no handling flags display regardless of data

### Checkout
19. When the buyer selects local pickup on a listing with handling flags, a warning box appears listing the flags
20. An "I understand the pickup requirements" checkbox appears below the warning
21. The acknowledgment state is exposed to the parent checkout flow (via callback prop)

23. A listing with fulfillmentType `SHIP_ONLY` and non-empty handling flags is NOT possible (server clears flags)
24. Invalid flag values (e.g., `NEEDS_CRANE`) are rejected by Zod

## 8. IMPLEMENTATION NOTES

### Migration Generation
After adding the column to the schema file, run:
```bash
pnpm drizzle-kit generate
```

This generates the SQL migration. Verify it contains ONLY the `ALTER TABLE "listing" ADD COLUMN "local_handling_flags"` statement and nothing else.

### Handling Flags Only Apply to Local-Eligible Listings
The flags section in the listing form only appears when `fulfillmentType` is `LOCAL_ONLY` or `SHIP_AND_LOCAL`. When the seller switches back to `SHIP_ONLY`, the flags should be visually hidden but the form data should reset `localHandlingFlags` to `[]`. The server actions also enforce this — if `fulfillmentType` is `SHIP_ONLY`, they save `[]` regardless.

### Checkout Acknowledgment is UI-Only
The acknowledgment checkbox state is managed by the checkout flow component. There is NO server-side validation that the buyer acknowledged the flags. The flags are informational — they help the buyer prepare, but failing to acknowledge them does not prevent the order. The checkbox prevents the "Proceed" button from being clickable, which is sufficient.

### ListingDetailData Update is a Prerequisite for Display
`fulfillmentType` is currently NOT in `ListingDetailData` — it must be added. This is needed not just for handling flags but for displaying the "Local Pickup Available" badge on listing detail (which may already be handled elsewhere via a separate field/component — check if so and avoid duplication).

### Existing Test Pattern for Server Actions
Follow the pattern in `src/components/local/__tests__/local-meetup-card.test.ts` and `src/lib/actions/__tests__/price-history.test.ts` — test the logic (data transformation, validation) rather than DOM rendering. Use `vi.mock` for database and auth dependencies.

### SafeTrade Feature Gating (From Addendum A0)
Per the SafeTrade feature matrix in Addendum Section A0, large item handling flags are a SafeTrade-only feature. However, the flags are set at LISTING CREATION time (before any transaction exists), so the gating applies at DISPLAY time — the flags should only be shown on the meetup screen and checkout when SafeTrade is the payment method. For this step, we display the flags unconditionally on the listing detail page (informational) and in the checkout fulfillment selector (with acknowledgment). The SafeTrade-gating of the flags display on the actual meetup screen is part of a later step when SafeTrade payment is implemented.
