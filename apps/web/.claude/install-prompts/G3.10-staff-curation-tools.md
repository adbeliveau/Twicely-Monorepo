### 2.3 Validation Schemas

File: `src/lib/validations/curated-collections.ts` (NEW)

Define these Zod schemas:

```typescript
import { z } from 'zod';

export const createCollectionSchema = z.object({
  title: z.string().min(1).max(200),
  description: z.string().max(1000).optional(),
  coverImageUrl: z.string().url().max(500).optional(),
  startDate: z.string().datetime().optional(),
  endDate: z.string().datetime().optional(),
  sortOrder: z.number().int().min(0).default(0),
}).strict();

export const updateCollectionSchema = z.object({
  collectionId: z.string().min(1),
  title: z.string().min(1).max(200).optional(),
  description: z.string().max(1000).nullable().optional(),
  coverImageUrl: z.string().url().max(500).nullable().optional(),
  isPublished: z.boolean().optional(),
  startDate: z.string().datetime().nullable().optional(),
  endDate: z.string().datetime().nullable().optional(),
  sortOrder: z.number().int().min(0).optional(),
}).strict();

export const deleteCollectionSchema = z.object({
  collectionId: z.string().min(1),
}).strict();

export const addCollectionItemSchema = z.object({
  collectionId: z.string().min(1),
  listingId: z.string().min(1),
  sortOrder: z.number().int().min(0).default(0),
}).strict();

export const removeCollectionItemSchema = z.object({
  collectionId: z.string().min(1),
  listingId: z.string().min(1),
}).strict();

export const reorderCollectionItemsSchema = z.object({
  collectionId: z.string().min(1),
  items: z.array(z.object({
    listingId: z.string().min(1),
    sortOrder: z.number().int().min(0),
  })).min(1),
}).strict();
```

**Key rules:**
- All schemas use `.strict()` -- unknown keys rejected.
- `slug` is NOT an input field -- it is auto-generated from `title` using `slugify()` on create, and derived on update if title changes.
- `curatedBy` / `addedBy` fields come from `session.staffUserId`, NEVER from client input.
- `isPublished` is only settable via update (defaults to `false` on create).
- `startDate` and `endDate` are ISO 8601 strings in the schema, converted to `Date` objects before DB insert.

### 2.4 Server Actions

File: `src/lib/actions/admin-curated-collections.ts` (NEW)

**`'use server'` directive at top.**

All actions use `staffAuthorize()` and check `ability.can('manage', 'CuratedCollection')`.
All actions insert an `auditEvent` row on success.
All actions use `revalidatePath('/mod/collections')` on success.

**Actions to implement:**

1. **`createCollectionAction(input: unknown)`**
   - Validates with `createCollectionSchema`
   - Generates slug via `slugify(title)` from `src/lib/utils/format.ts`
   - If slug collision (unique constraint), append CUID2 suffix: `${slug}-${createId().slice(0, 8)}`
   - Sets `curatedBy` to `session.staffUserId`
   - Sets `isPublished` to `false`
   - Inserts into `curatedCollection`
   - Audit: action `CREATE_CURATED_COLLECTION`, subject `CuratedCollection`, severity `MEDIUM`
   - Returns `{ success: true, collectionId: string }` or `{ error: string }`

2. **`updateCollectionAction(input: unknown)`**
   - Validates with `updateCollectionSchema`
   - Verifies collection exists (return `{ error: 'Not found' }` if not)
   - If `title` is updated, regenerate slug (same collision logic)
   - Explicit field mapping -- only set fields that are present in parsed input
   - Sets `updatedAt` to `new Date()`
   - Audit: action `UPDATE_CURATED_COLLECTION`, subject `CuratedCollection`, severity `MEDIUM`
   - `revalidatePath('/mod/collections')` AND `revalidatePath('/mod/collections/' + collectionId)`
   - Returns `{ success: true }` or `{ error: string }`

3. **`deleteCollectionAction(input: unknown)`**
   - Validates with `deleteCollectionSchema`
   - Verifies collection exists
   - Deletes from `curatedCollection` (cascade deletes items via FK)
   - Audit: action `DELETE_CURATED_COLLECTION`, subject `CuratedCollection`, severity `HIGH`
   - Returns `{ success: true }` or `{ error: string }`

4. **`addCollectionItemAction(input: unknown)`**
   - Validates with `addCollectionItemSchema`
   - Verifies collection exists
   - Verifies listing exists and has status `ACTIVE`
   - Inserts into `curatedCollectionItem` with `addedBy` = `session.staffUserId`
   - On unique constraint violation (duplicate collection+listing), return `{ error: 'Listing already in collection' }`
   - Audit: action `ADD_COLLECTION_ITEM`, subject `CuratedCollection`, severity `LOW`
   - `revalidatePath('/mod/collections/' + collectionId)`
   - Returns `{ success: true }` or `{ error: string }`

5. **`removeCollectionItemAction(input: unknown)`**
   - Validates with `removeCollectionItemSchema`
   - Deletes from `curatedCollectionItem` where `collectionId` AND `listingId` match
   - Audit: action `REMOVE_COLLECTION_ITEM`, subject `CuratedCollection`, severity `LOW`
   - `revalidatePath('/mod/collections/' + collectionId)`
   - Returns `{ success: true }` or `{ error: string }`

6. **`reorderCollectionItemsAction(input: unknown)`**
   - Validates with `reorderCollectionItemsSchema`
   - For each item in the array, update `sortOrder` in `curatedCollectionItem` where `collectionId` AND `listingId` match
   - Audit: action `REORDER_COLLECTION_ITEMS`, subject `CuratedCollection`, severity `LOW`
   - `revalidatePath('/mod/collections/' + collectionId)`
   - Returns `{ success: true }` or `{ error: string }`

**Pattern reference:** Follow `src/lib/actions/admin-moderation.ts` exactly for the `staffAuthorize()` + CASL check + Zod parse + DB operation + audit event pattern.

### 2.5 Query Functions

File: `src/lib/queries/admin-curated-collections.ts` (NEW)

**NOT a `'use server'` file. Pure query functions.**

1. **`getAdminCollections(page: number, pageSize: number, filter?: 'all' | 'published' | 'draft' | 'seasonal')`**
   - Returns `{ collections: AdminCollectionRow[], total: number }`
   - `AdminCollectionRow` type:
     ```typescript
     type AdminCollectionRow = {
       id: string;
       title: string;
       slug: string;
       description: string | null;
       coverImageUrl: string | null;
       isPublished: boolean;
       startDate: Date | null;
       endDate: Date | null;
       sortOrder: number;
       itemCount: number;
       curatedByName: string | null;
       createdAt: Date;
       updatedAt: Date;
     };
     ```
   - Joins `curatedCollection` with a subquery count of `curatedCollectionItem` rows per collection
   - Joins with `user` table on `curatedBy` to get `curatedByName` (staff display name, if available; note: `curatedBy` references `user.id` -- for staff users this is the `staffUser` mapped to a user record. If no user record match, return `null` for name)
   - Filter logic:
     - `'all'`: no filter
     - `'published'`: `isPublished = true`
     - `'draft'`: `isPublished = false`
     - `'seasonal'`: `startDate IS NOT NULL AND endDate IS NOT NULL`
   - Order by `sortOrder ASC, createdAt DESC`
   - Paginate with LIMIT/OFFSET

2. **`getAdminCollectionById(collectionId: string)`**
   - Returns the full collection row with its items
   - Return type:
     ```typescript
     type AdminCollectionDetail = {
       id: string;
       title: string;
       slug: string;
       description: string | null;
       coverImageUrl: string | null;
       isPublished: boolean;
       startDate: Date | null;
       endDate: Date | null;
       sortOrder: number;
       curatedBy: string;
       createdAt: Date;
       updatedAt: Date;
       items: AdminCollectionItemRow[];
     };

     type AdminCollectionItemRow = {
       id: string;
       listingId: string;
       listingTitle: string | null;
       listingSlug: string | null;
       listingPriceCents: number | null;
       listingStatus: string;
       primaryImageUrl: string | null;
       sortOrder: number;
       addedByName: string | null;
       createdAt: Date;
     };
     ```
   - Inner query: `curatedCollectionItem` JOIN `listing` JOIN `listingImage` (primary) JOIN `user` (on `addedBy`)
   - Order items by `sortOrder ASC`
   - Returns `null` if collection not found

3. **`searchListingsForCollection(query: string, excludeListingIds: string[])`**
   - Simple text search on `listing.title` using `ILIKE '%query%'`
   - Filter: `listing.status = 'ACTIVE'`
   - Exclude any listing IDs already in the collection
   - Return `{ id, title, slug, priceCents, primaryImageUrl }[]`
   - Limit 20 results
   - This is used by the "Add listing" UI in the collection editor

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- No `SellerTier`, `SubscriptionTier`, `FVF`, `BASIC`, `ELITE`, `PLUS`, `MAX`, `PREMIUM`, `STANDARD`, `RISING` (as PerformanceBand), `Twicely Balance`, `wallet`, `Withdraw` anywhere in UI text or code comments.

### Banned Tech
- No Prisma, NextAuth, Redis, Zustand, tRPC, Meilisearch, MinIO, Soketi, Nodemailer/SES.

### Code Constraints
- Zero `as any`, `as unknown as T`, `@ts-ignore`, `@ts-expect-error`.
- All files under 300 lines.
- Money values in integer cents.
- No request body spreading into DB updates -- explicit field mapping only.
- `curatedBy` and `addedBy` derived from `session.staffUserId`, NEVER from request body.
- `slug` auto-generated server-side, NEVER from request body.
- All Zod schemas use `.strict()`.
- All server actions in `admin-curated-collections.ts` must use `staffAuthorize()` (NOT `authorize()`).
- Hub routes do NOT use `revalidatePath` with marketplace paths -- only hub paths.

### Business Logic Constraints
- `curatedCollection.curatedBy` references `user.id`. For staff users, this is the staff user's ID. The schema FK references `user.id` -- the staff auth system maps staff users to user records. If the staffUserId does not have a matching `user.id`, the insert will fail. This is the existing pattern (used in seed data with `USER_IDS.seller1`). For the hub tool, use `session.staffUserId` directly.
- Collections default to `isPublished: false`. Staff must explicitly publish.
- Published collections with both `startDate` and `endDate` set appear as seasonal on the Explore page (per Personalization Canonical Section 5).
- Published collections without date range appear as Staff Picks on the Explore page.
- Only ACTIVE listings should be addable to collections. The collection item query already filters `listing.status = 'ACTIVE'` (see `explore-collections.ts` line 25).
- Deleting a collection cascade-deletes its items (FK `onDelete: 'cascade'` on `curatedCollectionItem.collectionId`).
- The `uniqueItem` constraint on `curatedCollectionItem` (collectionId + listingId) prevents adding the same listing twice.

## 4. ACCEPTANCE CRITERIA

### Functional
1. Staff with MODERATION or ADMIN role can access `/mod/collections` and see all collections.
2. Staff with MODERATION or ADMIN role can create a new collection with title, description, cover image URL, date range, and sort order.
3. Newly created collections have `isPublished: false` by default.
4. `slug` is auto-generated from `title` via `slugify()` and is unique (appends CUID2 suffix on collision).
5. Staff can edit any collection's title, description, cover image, date range, sort order, and published status.
6. Staff can delete a collection (with confirmation), which cascade-deletes all items.
7. Staff can add ACTIVE listings to a collection via search.
8. Staff can remove listings from a collection.
9. Staff can reorder items within a collection.
10. Publishing a collection makes it appear on the Explore page (verified by existing `getStaffPickCollections()` query).
11. Collections with both `startDate` and `endDate` set appear as seasonal on the Explore page (verified by existing `getSeasonalCollections()` query).

### Authorization
12. Unauthenticated users cannot access any `/mod/collections` pages -- `staffAuthorize()` throws.
13. Staff without MODERATION or ADMIN role cannot access `/mod/collections` -- CASL check fails.
14. FINANCE, DEVELOPER, SRE, HELPDESK_* roles CANNOT manage collections.
15. All mutation actions create audit events with correct `actorType: 'STAFF'`, `actorId`, `action`, `subject`, `subjectId`, and `severity`.

### Data Integrity
16. `curatedBy` is always set to `session.staffUserId`, never from client input.
17. `addedBy` on collection items is always set to `session.staffUserId`, never from client input.
18. Duplicate listing in same collection is rejected (unique constraint).
19. Only ACTIVE listings can be added to collections.
20. All Zod schemas reject unknown fields (`.strict()`).

### Vocabulary & UI
21. Page title: "Curated Collections | Twicely Hub" (list page).
22. No banned terms appear in any UI text or code.
23. Hub sidebar shows "Collections" under the Moderation section for ADMIN/MODERATION roles.

### Negative Cases
24. Adding a listing with status other than ACTIVE returns an error.
25. Adding a listing that is already in the collection returns `{ error: 'Listing already in collection' }`.
26. Updating a non-existent collection returns `{ error: 'Not found' }`.
27. Deleting a non-existent collection returns an appropriate error.
28. Search for listings with empty query is rejected by Zod (min 1 char).

## 8. SPEC NOTES AND DECISIONS

### NOT SPECIFIED -- Owner Decisions Documented

1. **Cover image upload**: The `coverImageUrl` column is a text URL. For V1, staff enters a URL manually (e.g., from R2 or placeholder). A proper image upload component (using the existing `/api/upload` route pattern) can be added in a follow-up step. This is consistent with how seed data uses placeholder URLs.

2. **curatedBy references user.id**: The schema FK on `curatedCollection.curatedBy` references `user.id`, not `staffUser.id`. The staff auth system stores `staffUserId` which may or may not have a corresponding `user.id` row. In seed data, `curatedBy` uses `USER_IDS.seller1` (a user ID). For hub tool usage, `session.staffUserId` from `staffAuthorize()` is used. If the staff user's ID doesn't exist in the `user` table, the DB insert will fail on the FK constraint. This is an existing design constraint -- if it becomes a problem, the FK should be changed to reference `staffUser.id` or made nullable. For now, document and proceed.

3. **Collection type field**: The schema has no explicit `type` column (e.g., 'STAFF_PICK' vs 'SEASONAL'). Type is derived: collections with both `startDate` and `endDate` are seasonal; all others are staff picks. This matches the existing query logic in `explore-collections.ts`.

4. **Listing search**: For V1, listing search uses simple `ILIKE` on the listing title. Typesense integration for the collection listing picker would be more performant but is deferred -- the admin tool is low-volume and `ILIKE` with a LIMIT 20 is adequate.
