### 2.3 Ed25519 Token Service (NEW FILE -- Server-Side)

**File:** `src/lib/commerce/local-token.ts` (NEW -- ~200 lines)

This is the core cryptographic module. ALL signing happens server-side using Node.js `crypto`.

#### Token Payload Type
```typescript
interface LocalTransactionTokenPayload {
  transactionId: string;
  amountCents: number;
  buyerId: string;
  sellerId: string;
  role: 'BUYER' | 'SELLER';
  expiresAt: string; // ISO 8601
  nonce: string;     // cuid2 -- unique per token, marked used on first submission
}
```

#### Functions to Implement

1. **`getSigningKey(): Buffer`** -- Reads `LOCAL_TX_SIGNING_KEY` env var, base64-decodes to 64-byte Ed25519 private key. Throws if missing or wrong length.

2. **`getVerifyKey(): string`** -- Returns `NEXT_PUBLIC_LOCAL_TX_VERIFY_KEY` env var (base64-encoded public key). Throws if missing.

3. **`signToken(payload: LocalTransactionTokenPayload): string`** -- JSON-encodes the payload, signs with Ed25519 using `crypto.sign('ed25519', payloadBuffer, privateKey)`, returns base64url-encoded `${base64url(payloadJSON)}.${base64url(signature)}` format.

4. **`verifyTokenServer(token: string): { valid: boolean; payload?: LocalTransactionTokenPayload; error?: string }`** -- Splits token on `.`, base64url-decodes both parts, verifies Ed25519 signature using `crypto.verify('ed25519', payloadBuffer, publicKey, signature)`. If valid, parses payload JSON. Checks `expiresAt` is not past. Returns parsed payload if valid.

5. **`generateTokenPair(params: { transactionId: string; amountCents: number; buyerId: string; sellerId: string; expiresAt: Date }): { sellerToken: string; buyerToken: string; sellerOfflineCode: string; buyerOfflineCode: string; sellerNonce: string; buyerNonce: string }`** -- Generates both tokens + both 6-digit offline codes + both nonces.

6. **`generateOfflineCode(): string`** -- Generates a 6-digit numeric code (100000-999999). Use `crypto.randomInt(100000, 999999)` for cryptographic randomness (NOT `Math.random()`).

#### Key Design Notes
- Use `crypto.sign(null, data, { key: privateKey, format: 'der', type: 'pkcs8' })` for Ed25519 signing (Node.js 16+ API).
- The token format is: `base64url(JSON payload).base64url(64-byte Ed25519 signature)`
- The public key is ALSO used server-side for verification (in `verifyTokenServer`).
- Nonces are cuid2 values stored alongside the transaction. On first valid submission, the nonce is marked as used to prevent replay.
- `expiresAt` is calculated as `scheduledAt + tokenExpiryHours` (read from platform settings).

#### 2.8.2 Update `confirmReceiptOfflineAction()` -- Major Rework

**Offline mode (dual-token submission):**

Per Addendum A4: "Both parties scan each other's QR or enter each other's 6-digit codes. Each phone verifies the other's token signature offline using the embedded public key. Either phone submits the combined payload `{ sellerToken, buyerToken }` when signal returns. First submission wins. Second is idempotent no-op."

New schema:
```typescript
export const confirmOfflineDualSchema = z.object({
  localTransactionId: z.string().min(1),
  sellerToken: z.string().min(1),
  buyerToken: z.string().min(1),
  offlineTimestamp: z.string().datetime(),
}).strict();
```

Logic:
1. Validate both tokens server-side via `verifyTokenServer()`.
2. Verify both tokens reference the SAME `transactionId`.
3. Verify one has `role: 'SELLER'` and the other has `role: 'BUYER'`.
4. Look up transaction, verify `confirmedAt` is null (idempotent: if already confirmed, return `{ success: true }` without error).
5. Set `confirmationMode` to `'QR_DUAL_OFFLINE'`.
6. Run `postConfirmationEffects()`.

**Also support offline 6-digit dual code submission:**

```typescript
export const confirmOfflineDualCodeSchema = z.object({

```

Logic: same as above but validates 6-digit codes against DB fields. Sets `confirmationMode` to `'CODE_DUAL_OFFLINE'`.

#### 2.8.3 Nonce Tracking

Per Addendum A4: "Nonce -- server marks used on first submission."

After successful token validation on any confirmation action:
- Extract nonce(s) from the token payload(s).
- The nonce check is implicit: once `confirmedAt` is set, the transaction cannot be confirmed again. The `confirmedAt !== null` check effectively prevents replay. No separate nonce table needed -- the transaction's `confirmedAt` field serves as the "nonce used" marker.

**SPEC CLARIFICATION:** The addendum says "Nonce -- server marks used on first submission." Since each token pair is unique to a transaction and the transaction can only be confirmed once (checked by `confirmedAt === null`), there is no need for a separate nonce storage table. The per-transaction single-use nature of the confirmation is the replay prevention. If the spec owner disagrees and wants an explicit nonce table, flag this for discussion.

### 2.16 Token Preload API Route (NEW)

**File:** `src/app/api/local/tokens/route.ts` (NEW)

GET endpoint to retrieve preloaded tokens for a transaction. Called by client on meetup screen mount.

```typescript
// GET /api/local/tokens?transactionId=xxx
// Auth: authorize() -- must be buyer or seller of the transaction
// Returns: { sellerToken, buyerToken, sellerOfflineCode, buyerOfflineCode, expiresAt }
```

Logic:
1. `authorize()` -- must be authenticated.
2. Read `transactionId` from query params.
3. Look up transaction. Verify `session.userId === buyerId || session.userId === sellerId`.
4. CASL check: `ability.can('read', sub('LocalTransaction', { buyerId, sellerId }))`.
5. Return the role-appropriate data:
   - Both parties get BOTH tokens (needed for offline dual verification).
   - This is per Addendum A4: "All four values pushed to both phones."

**Security consideration:** Both tokens are available to both parties. This is by design -- the buyer needs the seller's token to verify offline, and vice versa. ## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- No "wallet", "withdraw", "Twicely Balance" anywhere in code or UI strings.
- No "FVF", "Final Value Fee" -- always "Transaction Fee" / "TF".
- No "escrow" in user-facing UI strings (use "SafeTrade" per Addendum A0). Code comments and variable names may use "escrow" internally.

### Banned Tech
- Do NOT use `crypto-js` or `noble-ed25519` or any other crypto library for server-side signing. Use Node.js built-in `crypto`.
- Do NOT use `localStorage` for token storage. Use IndexedDB via `idb-keyval`.
- Do NOT use Zustand/Redux for offline state. Use React state + IndexedDB.
- Do NOT install Centrifugo/Soketi/Pusher client libraries in this step.

### Banned Patterns
- No `as any`, `@ts-ignore`, `@ts-expect-error`.
- No hardcoded fee rates or expiry values -- read from `platform_settings`.
- No `Math.random()` for cryptographic purposes -- use `crypto.randomInt()` for offline codes.
- No files over 300 lines.
- No spreading request body into DB updates.

### Gotchas
1. **Token size in QR codes:** A signed Ed25519 token (base64url JSON payload + 64-byte signature) will be ~400-600 characters. QR codes handle up to ~4000 chars, so this is fine, but the QR image may need a larger `width` (280-320px) for reliable scanning.
2. **The `confirmationCode` column is referenced in 15+ files.** Every reference must be updated to the new dual-column names. Use a thorough search.
3. **The `offlineCode` column is referenced in 10+ files.** Same treatment.
4. **`local-price-adjustment.ts` imports from `local-transaction.ts`:** The old `generateConfirmationCode` and `generateOfflineCode` are imported there. These must be replaced with `generateTokenPair` from `local-token.ts`.
5. **Test mock shapes must include the new columns.** All `makeTransaction()` helpers in test files must add `sellerConfirmationCode`, `sellerOfflineCode`, `buyerConfirmationCode`, `buyerOfflineCode`, `confirmationMode` and remove `confirmationCode`, `offlineCode`, `confirmationMethod`.
6. **Ed25519 key format matters.** Node.js `crypto.sign` with ed25519 expects specific key object formats. Use `crypto.createPrivateKey({ key: Buffer.from(base64, 'base64'), format: 'der', type: 'pkcs8' })` to construct the key object.
7. **`tweetnacl` expects raw 32-byte public keys**, not DER-encoded. The SPKI DER encoding adds a 12-byte header. Extract the raw 32 bytes by slicing off the header: `derBytes.slice(12)`.

## 4. ACCEPTANCE CRITERIA

### Positive Cases
1. `generateTokenPair()` produces two valid Ed25519-signed tokens with distinct nonces and roles.
2. `verifyTokenServer()` accepts a freshly signed token and returns the correct payload.
3. `verifyTokenClient()` (tweetnacl) accepts the same token that `verifyTokenServer()` accepts -- cross-library compatibility.
4. `createLocalTransaction()` stores 4 code fields (sellerConfirmationCode, sellerOfflineCode, buyerConfirmationCode, buyerOfflineCode) in the database.
5. Online QR confirmation: buyer scans seller's QR (containing sellerToken) -> server validates Ed25519 signature + nonce -> transaction marked RECEIPT_CONFIRMED -> `confirmationMode` set to `QR_ONLINE`.
6. Online manual code: buyer enters seller's 6-digit code -> server validates against `sellerOfflineCode` -> `confirmationMode` set to `CODE_ONLINE`.
7. Offline dual-token submission: combined payload `{ sellerToken, buyerToken }` submitted -> both verified server-side -> `confirmationMode` set to `QR_DUAL_OFFLINE`.
8. Offline dual-code submission: `{ sellerOfflineCode, buyerOfflineCode }` submitted -> both verified -> `confirmationMode` set to `CODE_DUAL_OFFLINE`.
9. Price adjustment acceptance regenerates ALL 4 token/code values (old ones invalidated).
10. Token preload endpoint returns all 4 values to authenticated buyer or seller.
11. IndexedDB stores and retrieves preloaded tokens correctly.
12. 3 new platform settings seeded: `offlineModeEnabled`, `preloadTokensOnEscrow`, `tokenExpiryHours`.
13. `tokenExpiryHours` is read from platform_settings (default 48), not hardcoded.
14. `.env.example` has both new env var placeholders with comments.
15. Key generation script produces valid Ed25519 key pair.

### Negative Cases
16. Tampered token (any byte modified) is rejected by both server and client verifiers.
17. Expired token (expiresAt in the past) is rejected.
18. Wrong-role token rejected: scanner rejects token with matching role (buyer scanning buyer token fails).
19. Already-confirmed transaction rejects new submissions (confirmedAt not null).
20. Unauthenticated user cannot access token preload endpoint (401).
21. User who is neither buyer nor seller of a transaction cannot access its tokens (returns "Not found").
22. Invalid 6-digit code format rejected (non-numeric, wrong length).
23. Missing `LOCAL_TX_SIGNING_KEY` env var causes clear error on token generation (not a silent failure).
24. Missing `NEXT_PUBLIC_LOCAL_TX_VERIFY_KEY` causes clear error on client verification.

### Data Integrity
25. All old `confirmationCode` / `offlineCode` / `confirmationMethod` references are removed from codebase (except migration file).
26. New `sellerConfirmationCode` and `buyerConfirmationCode` columns have unique constraints.
27. Offline codes generated with `crypto.randomInt()`, not `Math.random()`.
28. Token expiry calculated from `scheduledAt + tokenExpiryHours` (platform setting), not hardcoded 48.

### Authorization
29. Token preload API route checks CASL `ability.can('read', sub('LocalTransaction', ...))`.
30. Confirmation actions check CASL `ability.can('update', sub('LocalTransaction', ...))`.
31. Only the buyer can call `confirmReceiptAction` (online mode).
32. Either party can submit the dual-token offline payload (first wins).

### Vocabulary
33. No "wallet", "withdraw", "FVF", "Final Value Fee", "Twicely Balance" in any file.
34. No "escrow" in UI-facing strings (code comments are fine).

## 8. SPEC AMBIGUITIES AND DECISIONS NEEDED

### 8.1 Nonce Storage Strategy
**Ambiguity:** Addendum A4 says "Nonce -- server marks used on first submission." This implies a separate nonce tracking mechanism. However, since each token pair is unique to a transaction and the transaction can only be confirmed once (`confirmedAt === null` check), the nonce is implicitly single-use.

**Recommendation:** Use the implicit approach (no separate nonce table). The `confirmedAt` field is the replay-prevention gate. If the spec owner wants an explicit `usedNonces` table for defense-in-depth, that can be added but is not strictly necessary.

### 8.2 Token Format -- DER vs Raw Key Bytes
**Ambiguity:** Addendum A4 says "Ed25519 private key" and "public key" without specifying encoding format.

**Decision made in this prompt:** Use DER PKCS8 for private key, DER SPKI for public key (Node.js standard formats). The key generation script outputs base64-encoded DER. For `tweetnacl` client-side, extract raw 32-byte public key from SPKI DER by slicing the 12-byte ASN.1 header.

### 8.3 Offline Mode Activation
**Ambiguity:** Addendum A4 says "Activated by: manual 'No Internet Access' button, `navigator.onLine === false`, or submission timeout."

**Decision for this prompt:** The offline mode is an explicit opt-in by the user (clicking a button or automatic detection). The client component detects `navigator.onLine === false` and shows the offline UI automatically. A manual button is also available. Submission timeout (e.g., 5-second fetch timeout) triggers fallback to offline mode. Exact timeout value is NOT SPECIFIED -- recommend 5 seconds.

### 8.4 Who Scans Whom in Offline Mode
**Ambiguity:** In online mode, the buyer scans the seller's QR. In offline mode, the addendum says "Both parties scan each other's QR." This means the SELLER also needs to scan the BUYER's QR. But the seller's app currently shows their own QR -- it does not have a scanner.

**Decision for this prompt:** In offline mode, BOTH phones show their own QR AND have a scanner available. The UI switches to a split view: top half shows own QR, bottom half has the scanner. Both phones verify the other's token client-side. Either phone can submit the combined payload on reconnect.

### 8.5 amountCents in Token Payload
**Ambiguity:** The addendum token payload includes `amountCents`. The `localTransaction` table does NOT have an `amountCents` column. The original price is on `order.itemSubtotalCents`.

**Decision for this prompt:** When generating tokens, read `order.itemSubtotalCents` as the `amountCents`. If a price adjustment has been accepted, use `localTransaction.adjustedPriceCents` instead. This matches the existing pattern in `postConfirmationEffects()`.

### 8.6 ADJUSTMENT_PENDING Status Already Exists
**Confirmed:** `ADJUSTMENT_PENDING` was already added to `localTransactionStatusEnum` in G2.6. No enum change needed for that value. The state machine in `local-state-machine.ts` already handles `BOTH_CHECKED_IN -> ADJUSTMENT_PENDING -> BOTH_CHECKED_IN/RECEIPT_CONFIRMED` transitions.

### 8.7 TF Rate for Tokens
**SPEC INCONSISTENCY NOTE (from MEMORY.md):** Addendum A0 deprecates `commerce.local.transactionFeePercent` (5% flat), says TF = same as shipped (category-based feeBucket). But CLAUDE.md still says "5% flat". The `commerce.local.tfRateBps: 500` key is still seeded.

**Impact on G2.7:** The `amountCents` in the token payload is the ITEM price, not the fee. Token generation does not calculate fees. This inconsistency does not affect G2.7 implementation. The TF calculation happens in `postConfirmationEffects()` which is NOT being modified in G2.7 (only the confirmation flow changes).
