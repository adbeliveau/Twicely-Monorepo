For Posh sharing and offer-to-likers (which are platform actions that don't create/modify projections), we record them as crossJob entries with jobType `SYNC` and a payload marker `{ automationType: 'POSH_SHARE' }`.

## STREAM D: Four Automation Engines

Each engine is a pure service module (NOT a `'use server'` file) that finds eligible listings and creates crossJob entries for the automation scheduler to process.

### D.1 Auto-Relist Engine
**File:** `src/lib/crosslister/automation/auto-relist-engine.ts`

**Source:** Lister Canonical Section 17.1 -- "End and relist stale listings to refresh search ranking."

**Logic:**
1. Query all sellers WHERE `hasAutomation = true` AND `automationSetting.autoRelistEnabled = true`
2. For each seller, find `channelProjection` rows WHERE:
   - `status = 'ACTIVE'`
   - `channel` is in `automationSetting.autoRelistChannels`
   - `channelProjection.createdAt` is older than `automationSetting.autoRelistDays` days
   - No existing PENDING/QUEUED `crossJob` with `automationEngine = 'AUTO_RELIST'` for this projection
3. Check automation meter: `canPerformAutomationAction(sellerId)`
4. For each eligible projection, create a `crossJob`:
   - `jobType = 'RELIST'`
   - `priority = 700` (AUTOMATION priority from Lister Canonical Section 8.1)
   - `payload = { automationEngine: 'AUTO_RELIST', projectionId, listingId, channel }`
   - `idempotencyKey = auto-relist:${projectionId}:${monthKey}`
5. Record automation action via meter
6. The job executor (existing `job-executor.ts`) handles RELIST by: calling `connector.delistListing()` then `connector.createListing()` to end+relist

**Connector support:**
- All platforms support delist + create (which IS a relist)
- Check `ConnectorCapabilities.canAutoRelist` before creating job
- Platforms: eBay (Tier A), Poshmark (Tier C), Mercari (Tier B) -- all launch connectors

### D.2 Smart Price Drop Engine
**File:** `src/lib/crosslister/automation/price-drop-engine.ts`

**Source:** Lister Canonical Section 17.1 -- "Scheduled price reductions to trigger notification to watchers."
**Source:** Decision Rationale -- "Auto price drops are time-based ('drop 10% after 14 days')."

**Logic:**
1. Query sellers with `hasAutomation = true` AND `automationSetting.priceDropEnabled = true`
2. For each seller, find listings WHERE:
   - `listing.status = 'ACTIVE'`
   - `listing.priceCents > 0`
   - At least one ACTIVE `channelProjection` exists
   - Last price drop was >= `priceDropIntervalDays` ago (or never dropped)
   - Current price is above floor: `listing.priceCents > (listing.originalPriceCents * priceDropFloorPercent / 100)`
   - **NOTE:** `listing.originalPriceCents` may not exist. Use `listing.priceCents` at listing creation as the baseline. If no `originalPriceCents` column exists, track the original price in the automation action payload.
3. Calculate new price: `currentPriceCents * (100 - priceDropPercent) / 100`, rounded to nearest cent
4. Enforce floor: `max(newPrice, originalPrice * priceDropFloorPercent / 100)`
5. Check automation meter
6. Create crossJob entries for each affected projection:
   - `jobType = 'UPDATE'`
   - `priority = 700`
   - `payload = { automationEngine: 'PRICE_DROP', projectionId, listingId, channel, newPriceCents, oldPriceCents }`
7. Also update the canonical listing price (Twicely listing is source of truth)
8. Record automation actions

**Important:** Price drop updates the CANONICAL Twicely listing price AND all active projections. This is NOT a per-channel override -- it's a real price reduction.

**Floor enforcement:** Never drop below `priceDropFloorPercent` of the ORIGINAL listing price (price at first creation). Track original price via `listing.priceCents` at creation time. Since we don't have a dedicated `originalPriceCents` column, use the listing's `createdAt` price. To avoid complex history tracking, **store the original price in the automationSetting or as the first entry's payload**.

**SIMPLER V1:** Use `listing.priceCents` as current price. Calculate floor as `currentInitialPrice * floorPercent / 100`. For V1, store `initialPriceCents` in the crossJob payload when the first drop fires. This is imperfect but avoids schema changes.

**BEST V1:** Check if listing has a `priceHistory` or COGS field. Existing schema has `listing.costOfGoodsCents` (COGS). That's purchase cost, not initial list price. For V1, compute floor against current price at the time automation was enabled. Store this as `priceDropBasePriceCents` on the `automationSetting` per-listing... but that's per-listing state on a per-seller table.

**FINAL DECISION:** For V1, enforce the floor against the CURRENT price at each interval. So `priceDropFloorPercent = 50` means "stop dropping when total cumulative drops reach 50% of the price when automation was first enabled." Track per-listing state via the crossJob payload chain. On each drop, the payload records `{ originalPriceCents, newPriceCents }`. The engine checks: has this listing already been dropped below floor? If so, skip.

**NOT SPECIFIED -- Owner decision needed:** Should smart price drops track per-listing original price? The schema has no column for it. Options: (A) Add `automationBasePriceCents` to listing table, (B) Track in automation job payloads, (C) Use first crossJob for this listing to determine baseline. Recommend (B) for V1.

### D.3 Offer-to-Likers Engine
**File:** `src/lib/crosslister/automation/offer-to-likers-engine.ts`

**Source:** Lister Canonical Section 17.1 -- "Send automatic offers to interested buyers."
**Source:** Decision Rationale #68 -- "C2.2 Offer to Watchers: manual seller action. F6.1 Offer-to-Likers: automated. Different feature, different infrastructure."

**Platforms:** eBay, Poshmark, Mercari, Depop (Lister Canonical Section 17.1)

**Logic:**
1. Query sellers with `hasAutomation = true` AND `automationSetting.offerToLikersEnabled = true`
2. For each seller, find projections WHERE:
   - `status = 'ACTIVE'`
   - `channel` supports offers (`ConnectorCapabilities.canMakeOffers = true`)
   - `channelProjection.createdAt` is older than `offerMinDaysListed` days
   - No offer-to-likers job fired for this projection in the last 7 days (cooldown)
3. Calculate offer price: `listing.priceCents * (100 - offerDiscountPercent) / 100`
4. Check automation meter
5. Create crossJob:
   - `jobType = 'SYNC'` (offers are a sync-type action, not a publish)
   - `priority = 700`
   - `payload = { automationEngine: 'OFFER_TO_LIKERS', projectionId, listingId, channel, offerPriceCents, discountPercent }`
6. The job executor dispatches to the connector's future `sendOfferToLikers()` method

### D.4 Posh Sharing Engine

## STREAM E: Automation Scheduler (BullMQ Cron)

### E.1 Automation Scheduler Service
**File:** `src/lib/crosslister/automation/automation-scheduler.ts`

A BullMQ repeatable job that runs on a schedule and dispatches the 4 automation engines.

**Schedule:**
- Auto-relist: Once daily at 03:00 UTC (low traffic time)
- Smart price drops: Once daily at 04:00 UTC
- Offer-to-likers: Once daily at 10:00 UTC (peak engagement time)
- Posh sharing: Multiple times daily (based on `poshShareTimesPerDay` setting per seller)

**Implementation:**
```typescript
export async function runAutomationTick(): Promise<void> {
  const now = new Date();
  const hour = now.getUTCHours();

  // Run the appropriate engine based on the hour
  if (hour === 3) await runAutoRelistEngine();
  if (hour === 4) await runPriceDropEngine();
  if (hour === 10) await runOfferToLikersEngine();

  // Posh sharing runs every tick, but the engine self-limits per seller
  await runPoshShareEngine();
}
```

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- Do NOT use `wallet`, `balance`, `withdraw` in any UI copy
- Do NOT use `SellerTier`, `SubscriptionTier`
- Do NOT use `FVF`, `Final Value Fee`
- Do NOT use `BASIC`, `ELITE`, `PLUS`, `MAX`, `PREMIUM` as tier names

### Banned Patterns
- Do NOT hardcode the 2,000 action limit -- read from `automation.actionsPerMonth` platform setting
- Do NOT hardcode pricing ($9.99/$12.99) -- read from `automation.pricing.annualCents`/`automation.pricing.monthlyCents`
- Do NOT calculate fees on the frontend
- Do NOT use `as any`, `@ts-ignore`, `@ts-expect-error`
- Do NOT spread request body into DB operations -- use explicit field mapping
- Do NOT create files over 300 lines -- split into sub-modules
- Do NOT use Prisma, NextAuth, Redis, tRPC, Zustand, or any other banned tech

### Business Rules to NOT Violate
- Automation requires Crosslister LITE or above (Section 17.3). Lister FREE and NONE cannot subscribe.
- No fees on off-platform sales (Decision #31) -- automation actions on external platforms are subscription-funded
- Auto-relist does NOT consume publish credits (Lister Canonical Section 7.3: "Syncs, relists, and delists NEVER count as publishes")
- Price drops and offers do NOT consume publish credits either
- Poshmark sharing requires explicit risk acknowledgment before enabling (Section 16.3)
- All automation actions consume automation action quota (Section 17.2), NOT publish credits

### Gotchas from Canonicals
- `sellerProfile.id` is CUID2 PK, NOT `userId`. Always use `sellerId` (which equals `userId` from session) when querying, not `sellerProfile.id`
- The `automationSetting.sellerId` references `user.id`, not `sellerProfile.id` -- this is correct for ownership model
- Poshmark is Tier C (session-based automation) -- all actions have elevated risk
- Automation vs Store auto-reprice: Two different features. Auto-reprice is market intelligence-driven (Store Power+). Automation price drops are time-based. No overlap. (Decision Rationale near #68)
- Page Registry says Build Phase F5 for automation page, but Build Tracker says F6 -- use F6
