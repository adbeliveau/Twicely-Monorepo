### 3.2 Video Validation Module

Create `src/lib/upload/validate-video.ts`:

**Constants:**
- `MAX_VIDEO_SIZE` = 100 * 1024 * 1024 (100MB, matches platform setting default but used as fallback)
- `MIN_VIDEO_DURATION` = 15 (seconds)
- `MAX_VIDEO_DURATION` = 60 (seconds)
- `ALLOWED_VIDEO_TYPES` = `['video/mp4', 'video/quicktime', 'video/webm']`
- `ALLOWED_VIDEO_EXTENSIONS` = `['mp4', 'mov', 'webm']`

**Functions:**
- `detectVideoType(bytes: Uint8Array): 'mp4' | 'mov' | 'webm' | null` -- magic byte detection
  - MP4/MOV: bytes 4-7 = `ftyp` (0x66, 0x74, 0x79, 0x70) -- both use ISO base media file format
  - WebM: bytes 0-3 = `0x1A, 0x45, 0xDF, 0xA3` (EBML header)
  - MP4 vs MOV distinction: after `ftyp`, check brand bytes. If brand is `qt  ` = MOV. Otherwise MP4.
- `getVideoExtension(type: 'mp4' | 'mov' | 'webm'): string` -- returns file extension
- `getVideoContentType(type: 'mp4' | 'mov' | 'webm'): string` -- returns MIME type (`video/mp4`, `video/quicktime`, `video/webm`)
- `validateVideoBytes(bytes: Uint8Array, size: number): ValidationResult` -- checks magic bytes + file size

**Important:** Duration validation cannot happen server-side without ffmpeg (which we are NOT adding). Duration is validated client-side before upload and sent as form data. The server trusts the client-reported duration for V1 (stored in `videoDurationSeconds`). This is acceptable because:
- The video itself is stored as-is; duration metadata is informational
- Malicious duration spoofing is low-value (no financial impact)
- Server-side ffprobe validation is a Phase H optimization

### 3.3 Video Storage Service

Create `src/lib/storage/video-service.ts`:

**Interface:**
```typescript
export interface VideoUploadResult {
  success: boolean;
  videoUrl?: string;
  thumbnailUrl?: string;
  error?: string;
}
```

**Functions:**
- `uploadListingVideo(listingId: string, buffer: Buffer, videoType: 'mp4' | 'mov' | 'webm'): Promise<VideoUploadResult>` -- uploads video to R2 at key `videos/listings/{listingId}/{timestamp}.{ext}`. Returns R2 public URL. Does NOT generate thumbnail (client handles that).
- `uploadVideoThumbnail(listingId: string, buffer: Buffer): Promise<UploadResult>` -- uploads thumbnail image to R2 at key `videos/listings/{listingId}/thumb-{timestamp}.jpg`. Returns R2 public URL. Uses existing `UploadResult` type from image-service.
- `deleteListingVideo(videoUrl: string, thumbUrl: string | null): Promise<{ success: boolean; error?: string }>` -- deletes both video and thumbnail from R2.

R2 key prefix: `videos/listings/` (per Social Discovery Lock-in: "same bucket as images, different prefix").

### 3.4 Upload API Route Modification

Modify `src/app/api/upload/route.ts`:

Add a new `type: 'video'` case that:
1. CASL gate: `ability.can('create', 'Listing')` (same as listing images)
2. Reads `file` from form data
3. Validates file using `validateVideoBytes()` from the new validation module
4. Reads `listingId` from form data (optional, for edit flow; generate CUID2 if absent)
5. Reads `durationSeconds` from form data (integer, client-reported)
6. Validates `durationSeconds` is between `MIN_VIDEO_DURATION` and `MAX_VIDEO_DURATION`
7. Uploads video via `uploadListingVideo()`
8. Returns `{ success: true, video: { id, url, durationSeconds } }`

Add a new `type: 'video-thumbnail'` case that:
1. CASL gate: `ability.can('create', 'Listing')` (same gate)
2. Reads image file (the client-extracted thumbnail frame)
3. Validates using existing `validateImageBytes()`
4. Reads `listingId` from form data
5. Uploads via `uploadVideoThumbnail()`
6. Returns `{ success: true, image: { id, url } }`

Rate limit: same 20/minute as image uploads (shared counter -- videos and images use the same rate limiter).

Local filesystem fallback: for dev mode (R2 not configured), save video to `public/uploads/videos/` and thumbnail to `public/uploads/video-thumbnails/`.

**CRITICAL:** The upload route is already 239 lines. Adding video support will push it beyond 300 lines. The installer MUST extract the existing upload logic into helper functions or split the route. Options:
- Extract each upload type handler into separate files under `src/app/api/upload/handlers/` and import them
- Or extract the video handler into `src/app/api/upload/video-handler.ts`
- The route file itself should only contain the POST function that dispatches to handlers

### 3.5 Video Upload Component

Create `src/components/pages/listing/video-uploader.tsx`:

**Props:**
```typescript
interface VideoUploaderProps {
  videoUrl: string | null;
  videoThumbUrl: string | null;
  videoDurationSeconds: number | null;
  onVideoChange: (video: { url: string; thumbUrl: string; durationSeconds: number } | null) => void;
  disabled?: boolean;
}
```

**Behavior:**
1. Shows a drop zone (similar to image uploader) with text: "Drop a video here or click to upload" and "MP4, MOV, or WebM up to 100MB, 15-60 seconds"
2. On file select/drop:
   a. Client-side validation: check file type (accept attribute + MIME type check), check file size
   b. Load video into a hidden `<video>` element to extract duration
   c. Validate duration: 15-60 seconds. Show error if outside range.
   d. Extract first frame as thumbnail: draw video frame at t=0.5s onto a `<canvas>`, convert to JPEG blob
   e. Upload video to `/api/upload` with `type: 'video'` + `durationSeconds`
   f. Upload thumbnail to `/api/upload` with `type: 'video-thumbnail'`
   g. Call `onVideoChange({ url, thumbUrl, durationSeconds })`
3. When a video is uploaded, show:
   - Video preview (small `<video>` element with controls, NOT autoplay in the form)
   - Duration badge (e.g., "0:32")
   - Remove button (X icon)
4. Remove: calls `onVideoChange(null)` and clears the preview. Does NOT delete from R2 (orphan cleanup is deferred).
5. Only 1 video allowed per listing (per Social Discovery Lock-in: "1 video per listing, optional")
6. Accept attribute: `video/mp4,video/quicktime,video/webm`

**Thumbnail extraction logic (client-side):**
```
1. Create HTMLVideoElement, set src to object URL
2. Wait for 'loadeddata' event
3. Seek to 0.5 seconds (or 0 if video is < 1s)
4. Wait for 'seeked' event
5. Draw current frame to canvas (match video dimensions, max 1280px width)
6. canvas.toBlob('image/jpeg', 0.85) -> upload as thumbnail
```

## 4. CONSTRAINTS -- WHAT NOT TO DO

- **NO ffmpeg/ffprobe dependency.** Thumbnail extraction is client-side via canvas. Duration validation is client-side. Server-side ffmpeg processing is a Phase H optimization.
- **NO separate `listingVideo` table.** The schema doc puts video fields directly on the `listing` table. One video per listing, three nullable columns.
- **NO MinIO or S3 direct.** Use Cloudflare R2 via existing `r2-client.ts`.
- **NO video transcoding.** Store the original file as-is. Transcoding to HLS/DASH is post-launch.
- **NO video streaming optimization.** Standard `<video>` tag with direct R2 URL. CDN/adaptive streaming is post-launch.
- **NO mobile camera recording UI.** That is G1.7 (depends on G3.11). This step only adds upload from file picker.
- **NO `as any`, `@ts-ignore`, `@ts-expect-error`.** Fix types properly.
- **NO files over 300 lines.** Split listing-form.tsx and upload route.ts if needed.
- **NO hardcoded size/duration limits** in upload validation. Read from platform settings where possible. Use constants as fallback defaults.
- **NO orphan video cleanup.** If a user uploads a video then removes it from the form before saving, the R2 file remains. Orphan cleanup is deferred (use `cleanup` queue in a future step).
- **DO NOT change the existing image upload flow.** Video upload is additive.

## 5. ACCEPTANCE CRITERIA

### Functional
1. A seller can upload a single video (MP4, MOV, or WebM) during listing creation
2. A seller can upload a single video during listing editing
3. A seller can remove a previously uploaded video from a listing
4. The video upload rejects files over 100MB with a clear error message
5. The video upload rejects files shorter than 15 seconds or longer than 60 seconds with a clear error message
6. The video upload rejects non-video files (e.g., .exe, .pdf, .gif) with a clear error message
7. A thumbnail is extracted from the video (at ~0.5 seconds) and uploaded separately
8. The listing detail page (`/i/[slug]`) displays the video ABOVE the photo gallery
9. The video autoplays muted on page load
10. The video shows standard browser controls (play/pause, seek, volume/unmute)
11. The video has a poster image (thumbnail) displayed before the video loads
12. When no video is attached, the listing detail page shows only the photo gallery (no empty video container)
13. Video URL, thumbnail URL, and duration are stored on the listing record in the database
14. The listing creation action persists video fields to the database
15. The listing update action persists video fields to the database (including setting to null on removal)
16. The edit form loads existing video data and displays the video preview

### Authorization
17. Unauthenticated users cannot upload videos (401)
18. Users without `create Listing` ability cannot upload videos (403)
19. Rate limiting applies to video uploads (shared with image uploads, 20/minute)

### Data Integrity
20. All video URLs are R2 public URLs (or mock URLs in dev mode)
21. Duration is stored as integer seconds (not milliseconds, not fractional)
22. Video fields are nullable -- existing listings without video are unaffected
23. No new database migration is required (columns already exist)

### Vocabulary
24. No banned terms appear anywhere in the implementation
25. No references to MinIO, S3, Prisma, or other banned tech stack items

### UI/UX
26. `prefers-reduced-motion` is respected (show poster + play button, no autoplay)
27. Video player is responsive (fills container width)
28. The listing form clearly labels the video section as "optional"
29. Upload progress feedback is shown during video upload (the video file can be large)
30. All `<video>` elements have `playsInline` attribute (mobile Safari requirement)

### Edge Cases to Cover
- Upload a 0-byte file -- rejected
- Upload a file exactly at 100MB boundary -- accepted
- Upload a file at 100MB + 1 byte -- rejected
- Upload with duration exactly 15 seconds -- accepted
- Upload with duration exactly 60 seconds -- accepted
- Upload with duration 14 seconds -- rejected
- Upload with duration 61 seconds -- rejected
- Upload a video, then remove it, then save the listing -- video fields should be null
- Upload a video, replace it with a different video -- new URL stored
- Listing with video displays video player; listing without video does not render video element

## 9. SPEC INCONSISTENCIES

### 9.1 Column Name Mismatch (MINOR)
- Social Discovery Lock-in says `listing.videoDuration` (line 109)
- Schema doc says `listing.videoDurationSeconds` (line 818)
- Codebase has `listing.videoDurationSeconds` (listings.ts line 86)
- **Resolution:** Use `videoDurationSeconds` (matches schema doc and codebase). Lock-in was written before schema finalization.

### 9.2 Thumbnail Extraction Method
- Social Discovery Lock-in says "first frame extracted on upload, stored as image" (line 97)
- This implies server-side extraction, but no ffmpeg dependency is specified
- **Resolution for V1:** Client-side extraction via `<canvas>` + `<video>` element. The client extracts the frame at ~0.5s, converts to JPEG, and uploads as a separate image. This is how Depop and similar apps handle it without server-side processing. Server-side ffmpeg extraction can be added in Phase H if needed.

### 9.3 "Displayed ABOVE photo gallery"
- Lock-in says "Displayed on listing detail page ABOVE the photo gallery as hero content" (line 95)
- Current page layout has ImageGallery in a 3-column left section
- **Resolution:** Video player goes directly above ImageGallery in the same left column. When video is present, the page flow is: Video -> Photos -> (right column: title, price, etc.)

- The Platform Settings Canonical (`TWICELY_V3_PLATFORM_SETTINGS_CANONICAL.md`) does not define video settings
- The Social Discovery Lock-in specifies hard values (100MB, 15-60s)
- **Resolution:** Create 3 new platform settings under `listing.video.*` category. This follows the established pattern of making limits admin-configurable even when the lock-in gives specific defaults.
