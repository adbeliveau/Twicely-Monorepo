#### Page 2: Conversation Thread (`/my/messages/[conversationId]`) -- `src/app/(hub)/my/messages/[conversationId]/page.tsx`

**Metadata:** Dynamic title `{ title: '{subject} -- Message | Twicely' }` (from Page Registry row 78).

**Server Component Logic:**
1. Get session. Redirect if unauthenticated.
2. Call `getConversationMessages(conversationId, userId)`. If returns null (non-participant or not found), call `notFound()`.
3. Fire-and-forget `markAsRead({ conversationId })` to clear unread count on page load.
4. Render `ConversationThread` client component.

**Page States (from Page Registry Section 6.1):**
- **LOADING:** Message thread skeleton (use `loading.tsx`)
- **POPULATED:** Message bubbles (sender aligned left/right), listing card at top, compose area at bottom, attachment support (placeholder -- actual attachment upload is deferred)
- **READ_ONLY:** "This conversation is closed" banner. No compose area. (Page Registry: "listing ended, user blocked")

**Page Structure:**
- **Top:** Listing context card (image thumbnail, title, price, link to `/i/[slug]`). If conversation has an `orderId`, show "Order linked" badge.
- **Middle:** Scrollable message thread. Own messages right-aligned (purple-ish bg). Other party messages left-aligned (gray bg). Each message shows: sender name, body text, timestamp (relative), read status indicator ("Read" if `isRead` and sender is current user). Auto-generated messages styled differently (system message: centered, muted text, italic).
- **Bottom:** Compose area with textarea and send button. Disabled when `conversation.status !== 'OPEN'` with a banner: "This conversation is closed."

**Off-platform warning banner (Feature Lock-in Section 19):**
If `conversation.isFlagged` is true, show a yellow warning banner at the top of the thread:
> "For your protection, complete transactions on Twicely."
This is purely display -- the flagging already happened server-side in E2.2.

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- NEVER use "wallet", "balance", "withdraw" anywhere
- NEVER use "FVF", "Final Value Fee", "Commission"
- NEVER use "SellerTier", "SubscriptionTier"
- NEVER use "BASIC", "ELITE", "PLUS", "MAX", "PREMIUM" as tier names

### Banned Technology
- NO Centrifugo/WebSocket integration in this step. Real-time message delivery is deferred. The page uses standard Next.js server components + revalidation. Users will see new messages on page refresh or navigation.
- NO typing indicators (requires Centrifugo -- deferred)
- NO tRPC, NO Zustand/Redux
- NO `as any`, NO `@ts-ignore`, NO `@ts-expect-error`

### Route Constraints
- Pages live at `/my/messages` and `/my/messages/[conversationId]` (inside the `(hub)` route group, under the `my/` layout which provides the hub shell with sidebar/topbar/bottom-nav).
- `/m` and `/m/[id]` are 308 redirects to the above (defined in `next.config.ts`).
- NEVER use `/m/` as the actual page path -- it is a convenience redirect ONLY.
- NEVER use `/dashboard`, `/messages`, `/inbox` or any other route prefix.

### Architecture Constraints
- All files under 300 lines. The listing detail page refactor (extracting `ListingActionButtons`) is required to stay under the limit.
- No new server actions in this step -- use only the existing ones from E2.2.
- No new queries except for the admin flagged messages query (if needed for the mod page update).
- Client components use `'use client'` directive.
- Server components (pages) do NOT use `'use client'`.
- Money displayed using `formatPrice(cents)` from `@/lib/utils/format`.
- Ownership via `userId` always. The conversation's `sellerId` IS the seller's `user.id`.

### Business Logic Constraints
- The "Message seller" button on listing detail MUST NOT appear on the user's own listings.
- The "Message seller" button MUST NOT appear on unavailable (SOLD/ENDED) listings.
- Guests see the button but are redirected to login on click (Actors Canonical 3.1: "Cannot message sellers").
- Existing conversation is reopened, NOT duplicated (Decision #38).
- The compose area is hidden when `conversation.status !== 'OPEN'` (Feature Lock-in Section 19: "Read-only = both parties can see full history but cannot send new messages").
- Auto-generated messages are styled differently from user messages (Feature Lock-in Section 19).
- Off-platform flag banner shown on flagged conversations (Feature Lock-in Section 19).

### What This Step Does NOT Include
- NO Centrifugo real-time message delivery (messages appear on page refresh)
- NO typing indicators ("Seller is typing...")
- NO read receipts display (the `markAsRead` action is called server-side on page load, but we do NOT show "Read" indicators in the UI in this step -- the data exists but UI display is deferred with real-time)
- NO quick replies / canned responses (seller feature, separate step)
- NO auto-messages UI configuration (seller manages auto-message templates -- separate step)
- NO image attachment upload (the paperclip button is a placeholder; actual R2 upload is deferred)
- NO conversation lifecycle CRON job (auto-close after 30 days)
- NO helpdesk case creation from message reports (Phase G9)
