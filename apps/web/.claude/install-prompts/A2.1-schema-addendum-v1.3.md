### 3C. user Table -- referredByAffiliateId FK Fix (auth.ts)

The canonical `user` table (SS2.1) has:
```typescript
referredByAffiliateId: text('referred_by_affiliate_id').references(() => affiliate.id),
```

The Drizzle `auth.ts` has:
```typescript
referredByAffiliateId: text('referred_by_affiliate_id'),
```

The FK reference is missing. However, this is a deliberate omission because `auth.ts` cannot import from `affiliates.ts` without creating a circular dependency (auth exports `user`, affiliates imports `user`). The canonical includes the FK for documentation but Drizzle cannot express it inline.

**Action:** Leave `auth.ts` as-is (no FK reference). Add a comment explaining the FK is enforced at the application layer, not via DB constraint, due to circular import. The canonical FK is aspirational; the DB column exists and application code enforces the relationship.

### 3F. Migration SQL (drizzle/0008_schema-addendum-v1-3.sql)

Generate a migration file that applies the changes from 3A and 3B:

```sql
-- Migration: 0008_schema-addendum-v1-3.sql
-- A2.1: Schema Addendum v1.3 — Reconcile Drizzle with canonical v2.0.7

-- 3A: Add missing sellerProfile fields
ALTER TABLE "seller_profile"
  ADD COLUMN IF NOT EXISTS "is_authenticated_seller" boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS "max_meetup_distance_miles" integer,
  ADD COLUMN IF NOT EXISTS "payout_frequency" text NOT NULL DEFAULT 'WEEKLY';

-- 3B: Add missing ledger_entry_type enum values
-- Postgres requires ALTER TYPE ADD VALUE for each new enum value
ALTER TYPE "ledger_entry_type" ADD VALUE IF NOT EXISTS 'AUTH_FEE_BUYER';
ALTER TYPE "ledger_entry_type" ADD VALUE IF NOT EXISTS 'AUTH_FEE_SELLER';
ALTER TYPE "ledger_entry_type" ADD VALUE IF NOT EXISTS 'AUTH_FEE_REFUND';
ALTER TYPE "ledger_entry_type" ADD VALUE IF NOT EXISTS 'LOCAL_TRANSACTION_FEE';
ALTER TYPE "ledger_entry_type" ADD VALUE IF NOT EXISTS 'FINANCE_SUBSCRIPTION_CHARGE';
ALTER TYPE "ledger_entry_type" ADD VALUE IF NOT EXISTS 'AFFILIATE_COMMISSION_PAYOUT';
```

**IMPORTANT MIGRATION NOTES:**
- `ALTER TYPE ADD VALUE IF NOT EXISTS` requires Postgres 9.3+ (which we have).
- Each `ALTER TYPE ADD VALUE` must be in its own transaction OR the migration runs outside a transaction. Drizzle Kit migrations run each statement separately, so this is fine.
- The `IF NOT EXISTS` guard is important for idempotency.
- Do NOT remove `ORDER_LOCAL_FEE` or `ORDER_AUTH_FEE` from the Postgres enum -- they may be referenced by existing rows.

6. **Do NOT modify any table definitions beyond sellerProfile and the enum.** No other tables have discrepancies that need fixing in this step.

11. **Do NOT run `drizzle-kit generate`** to auto-generate the migration. Write it by hand as specified in section 3F. Auto-generated migrations often produce destructive ALTER TYPE statements that drop and recreate enums.
