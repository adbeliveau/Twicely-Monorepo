## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2.1 NO New Database Tables or Columns

All metrics data already exists on the `user` table (`localTransactionCount`, `localCompletionRate`, `localReliabilityMarks`) and can be derived from `localTransaction` and `dayOfConfirmation*` columns. No schema migration needed.

The one metric that does NOT exist yet is `localAvgResponseTimeHours`. This must be computed at query time from existing `dayOfConfirmationSentAt` and `dayOfConfirmationRespondedAt` columns on `localTransaction`.

### 2.2 New Query: `getSellerLocalMetrics()`

**File:** `src/lib/queries/local-metrics.ts`

This query fetches local-specific metrics for display on storefront and listing detail pages.

**Input:** `userId: string`

**Output interface:**
```typescript
export interface SellerLocalMetrics {
  localTransactionCount: number;    // Total local transactions attempted (from user cache)
  localCompletedCount: number;      // Completed count (derived from count * rate)
  localCompletionRate: number;      // 0.0 to 1.0 (from user cache)
  localReliabilityTier: 'RELIABLE' | 'INCONSISTENT' | 'UNRELIABLE'; // derived from marks
  localAvgResponseLabel: string | null; // "Same day" / "Within X hours" / null if no data
  hasLocalActivity: boolean;        // true if localTransactionCount > 0
}
```

**Logic:**
1. Read `localTransactionCount`, `localCompletionRate`, `localReliabilityMarks` from `user` table (cached values, no joins needed).
2. Derive `localCompletedCount` as `Math.round(localTransactionCount * localCompletionRate)`.
3. Derive `localReliabilityTier` from marks: use same thresholds as `getReliabilityDisplay()` — read `commerce.local.inconsistentMarkThreshold` (default 3) and `commerce.local.suspensionMarkThreshold` (default 9) from platform settings. 0-2 = RELIABLE, 3-8 = INCONSISTENT, 9+ = UNRELIABLE.
4. Compute `localAvgResponseLabel` from `localTransaction` rows where BOTH `dayOfConfirmationSentAt IS NOT NULL` AND `dayOfConfirmationRespondedAt IS NOT NULL` AND `sellerId = userId`. Calculate the average `(respondedAt - sentAt)` in hours across all such rows. Convert to a human label:
   - < 1 hour: "Within 1 hour"
   - 1-4 hours: "Within a few hours"
   - 4-24 hours: "Same day"
   - 24-48 hours: "Within 1 day"
   - 48+ hours: "Within 2 days"
   - No data (0 rows): `null`
5. Set `hasLocalActivity = localTransactionCount > 0`.

**IMPORTANT:** The `localAvgResponseTimeHours` metric from the addendum spec (A14) is NOT a column on any table. It must be computed at query time. Do NOT add a new column for this.

```

For `variant: 'listing-detail'` (compact display on listing detail):
```

**Design constraints:**
- Use shadcn Card for storefront variant.
- Use inline text for listing-detail variant (matches SellerCard compact style).

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Terms
- Never use "wallet", "withdraw", "Twicely Balance", "FVF", "Final Value Fee"
- Never use "SellerTier", "SubscriptionTier"
- These are unlikely to appear in this feature but scan your output before committing

### Banned Patterns
- No `as any`, `@ts-ignore`, `@ts-expect-error`
- No files over 300 lines
- No hardcoded thresholds — reliability tier thresholds come from platform settings
- No new database columns or tables — all data already exists
- No new pages or routes — this feature only adds display components to existing pages
- No new server actions — this is read-only display

### Gotchas
1. **Privacy rule from A14:** "Reliability marks are private — only the tier display is public." The raw mark count and specific event types MUST NOT be shown on storefront or listing detail. Only the TIER (RELIABLE/INCONSISTENT/UNRELIABLE) is visible on the meetup screen (already built). On storefront and listing detail, show only `completedCount`, `completionRate`, and `responseLabel`. Do NOT show the tier badge on storefront or listing pages.
2. **The `localCompletionRate` on user table is nullable** (`real` type, nullable). If null, treat as 0. The `localTransactionCount` defaults to 0.
3. **`StorefrontHeader` is a `'use client'` component.** The `LocalMeetupStats` component it renders must also be client-safe (no server imports). Since it is pure display (no data fetching), this is fine as a regular component.
4. **`getListingPageData()` in `listing-page.ts` does NOT currently fetch local metrics.** The parallel data fetch in the listing page (`page.tsx`) must add the call.
5. **Day-of confirmation response time is seller-only.** Only count rows where `sellerId = userId` for the response time calculation. Buyers do not have a "response time" metric.
6. **`localMeetupCard.tsx` is 287 lines** — close to the 300-line limit. The `LocalMeetupStats` component MUST be a separate file.
7. **`StorefrontHeader` currently imports `StorefrontSeller` and `StorefrontStats` from `storefront-public.ts`.** After adding `localMetrics` to `StorefrontStats`, the import remains unchanged but the prop type expands.

## 8. SPEC NOTES & CLARIFICATIONS

### NOT SPECIFIED — Owner Decision Needed

1. **Response time label thresholds:** The addendum A14 specifies `localAvgResponseTimeHours` as a metric but does not specify the exact label mapping (e.g., what range maps to "Same day"). The prompt above proposes reasonable thresholds. Owner: confirm or adjust the response time label ranges.

2. **Storefront section positioning:** A14 says "On seller storefront" but does not specify whether the local meetup stats appear in the header stats bar or as a separate section. The prompt places it below the header stats row. Owner: confirm placement.

3. **Minimum threshold for display:** Should local metrics be shown if the seller has only 1 completed transaction? The spec says "47 completed" as an example but does not specify a minimum. The prompt shows metrics for any seller with `localTransactionCount > 0`. Owner: confirm or set a minimum (e.g., 3 transactions).

### Spec Inconsistency

The addendum A14 defines `localAvgResponseTimeHours` as a tracked metric, but there is no column for it on any table. This prompt computes it at query time from `dayOfConfirmationSentAt` and `dayOfConfirmationRespondedAt` on `localTransaction`. This is consistent with Decision #120 which defines reliability tiers as derived values.

The schema doc (v2.1.2) does NOT include `localReliabilityMarks`, `localTransactionCount`, `localCompletionRate`, or `localSuspendedUntil` on the user table, but these columns EXIST in the actual code (`src/lib/db/schema/auth.ts:36-39`, added by G2.8). The schema doc is behind the implementation. This does not affect this prompt — we use what exists in code.
