## SPEC GAPS — READ BEFORE STARTING

The following items are NOT specified in any canonical document. The owner must confirm the decisions marked below before implementation. This prompt documents the most conservative safe interpretation where decisions are needed.

**GAP 1 — Route not in Page Registry (CRITICAL)**
`/my/settings/payments` does NOT appear in the Page Registry (v1.8, which goes up to page 78 at `/my/settings/privacy`). The build tracker mentions it by name but it has not been added as an official route entry. This prompt adds it as page 78-A in the settings group. NOT SPECIFIED — owner decision needed: Should the page be accessible to ALL authenticated users (buyers only), or only sellers (isSeller === true), or only sellers with a sellerProfile? This prompt implements AUTH (any authenticated user) which aligns with how a buyer without a seller profile needs to save a card for checkout. The CASL ability `can('update', 'User', { id: userId })` already exists for this case.

**GAP 2 — Stripe Customer ID for buyers (CRITICAL)**
The `user` table has NO `stripeCustomerId` column. Only `sellerProfile` has `stripeCustomerId` (used for billing subscriptions, not marketplace purchases). Buyers who are not sellers have no Stripe Customer object. For saved payment methods to work, a Stripe Customer must be created or retrieved for any authenticated user.

NOT SPECIFIED — owner decision needed: Should `stripeCustomerId` be added to the `user` table (used for buyer-side payment methods), or should it only be fetched from `sellerProfile` (limiting saved cards to sellers)? This prompt adds `stripeCustomerId` to the `user` table with a Drizzle migration, as this is the correct separation of concerns: `sellerProfile.stripeCustomerId` = billing customer for subscriptions, `user.stripeCustomerId` = payment customer for marketplace purchases. Owner must confirm before proceeding.

**GAP 3 — No PaymentMethod CASL subject**
`PaymentMethod` does not exist in `SUBJECTS` array in `subjects.ts`. Saved payment methods are managed entirely via Stripe's API (no local DB record). The correct CASL check for this page is `can('update', 'User', { id: userId })` — the user is managing their own payment preferences. No new CASL subject is required.

**GAP 4 — No `defaultPaymentMethodId` on user table**
There is no column to track which saved payment method is the user's default for marketplace purchases. The default is stored on the Stripe Customer object itself (`customer.invoice_settings.default_payment_method`). This prompt reads and writes the default via Stripe API only. No DB column is added. Owner decision: this is the correct approach per Stripe docs; confirm before implementing.

### 2.2 API Routes (Server Actions)

All server actions live in `src/lib/actions/payment-methods.ts`. This file is `'use server'`.

**Action: `createSetupIntent()`**
- Purpose: Creates a Stripe SetupIntent so the frontend can collect and save a new card via Stripe Elements.
- Input: none (user is derived from session)
- Flow:
  1. `const { session } = await authorize()` — FIRST operation
  2. If `!session` → `return { success: false, error: 'Unauthorized' }`
  3. Check CASL: `ability.can('update', 'User', { id: session.userId })` — if false → `return { success: false, error: 'Forbidden' }`
  4. Get or create Stripe Customer for this user (see `getOrCreateStripeCustomer` helper below)
  5. Call `stripe.setupIntents.create({ customer: customerId, payment_method_types: ['card'] })`
  6. Return `{ success: true, clientSecret: setupIntent.client_secret }`
- Return type: `{ success: boolean; clientSecret?: string; error?: string }`
- Revalidates: nothing (read-only creation of Stripe objects)

**Action: `listPaymentMethods()`**
- Purpose: Lists saved payment methods for the current user from Stripe.
- Input: none
- Flow:
  1. `const { session } = await authorize()` — FIRST operation
  2. If `!session` → `return { success: false, paymentMethods: [] }`
  3. Check CASL: `ability.can('read', 'User', { id: session.userId })` — if false → forbidden
  4. Look up `stripeCustomerId` from `user` table where `id = session.userId`
  5. If no `stripeCustomerId` → return `{ success: true, paymentMethods: [], defaultPaymentMethodId: null }`
  6. Call `stripe.paymentMethods.list({ customer: customerId, type: 'card' })`
  7. Call `stripe.customers.retrieve(customerId)` to get default PM
  8. Return `{ success: true, paymentMethods: SerializedPaymentMethod[], defaultPaymentMethodId: string | null }`
- `SerializedPaymentMethod` type (plain object, safe to pass to client):
  ```typescript
  interface SerializedPaymentMethod {
    id: string;
    brand: string;        // 'visa' | 'mastercard' | 'amex' | ...
    last4: string;
    expMonth: number;
    expYear: number;
    isDefault: boolean;
  }
  ```
- Return type: `{ success: boolean; paymentMethods: SerializedPaymentMethod[]; defaultPaymentMethodId: string | null; error?: string }`

**Action: `removePaymentMethod(paymentMethodId: string)`**
- Purpose: Detaches a saved card from the user's Stripe Customer.
- Input schema (inline, strict):
  ```typescript
  const RemovePaymentMethodSchema = z.object({ paymentMethodId: z.string().min(1).max(100) }).strict();
  ```
- Flow:
  1. `const { session } = await authorize()` — FIRST operation
  2. If `!session` → forbidden
  3. CASL check: `ability.can('update', 'User', { id: session.userId })`
  4. Load `stripeCustomerId` from DB
  5. If no `stripeCustomerId` → `return { success: false, error: 'No payment methods on file' }`
  6. Retrieve PM from Stripe: `stripe.paymentMethods.retrieve(paymentMethodId)` — verify `pm.customer === customerId` to prevent IDOR (a user must not be able to detach another user's card)
  7. If ownership mismatch → `return { success: false, error: 'Payment method not found' }` (do NOT reveal ownership details)
  8. Call `stripe.paymentMethods.detach(paymentMethodId)`
  9. Return `{ success: true }`
  10. Revalidate: `revalidatePath('/my/settings/payments')`
- Return type: `{ success: boolean; error?: string }`

**Action: `setDefaultPaymentMethod(paymentMethodId: string)`**
- Purpose: Sets a saved card as the default for this Stripe Customer.
- Input schema (inline, strict):
  ```typescript
  const SetDefaultSchema = z.object({ paymentMethodId: z.string().min(1).max(100) }).strict();
  ```
- Flow:
  1. `const { session } = await authorize()` — FIRST operation
  2. If `!session` → forbidden
  3. CASL check: `ability.can('update', 'User', { id: session.userId })`
  4. Load `stripeCustomerId` from DB
  5. If no `stripeCustomerId` → error
  6. Retrieve PM from Stripe and verify `pm.customer === customerId` (IDOR guard, same pattern as remove)
  7. Call `stripe.customers.update(customerId, { invoice_settings: { default_payment_method: paymentMethodId } })`
  8. Return `{ success: true }`
  9. Revalidate: `revalidatePath('/my/settings/payments')`
- Return type: `{ success: boolean; error?: string }`

**Internal helper (NOT exported — not a server action):**

```typescript
async function getOrCreateStripeCustomer(userId: string, email: string): Promise<string>
```
- Looks up `user.stripeCustomerId` for `userId`
- If exists: returns it
- If not: creates new Stripe Customer `stripe.customers.create({ email, metadata: { userId, platform: 'twicely' } })`, then writes it back to DB: `db.update(user).set({ stripeCustomerId: customer.id }).where(eq(user.id, userId))`
- Returns the customer ID string
- This function MUST NOT be exported. Exporting a helper from a `'use server'` file creates an unintended server action.

**API Route: `POST /api/payments/setup-intent`**

File: `src/app/api/payments/setup-intent/route.ts`

This is needed because `createSetupIntent()` is a server action — but the `StripeProvider` wrapper requires the `clientSecret` before mounting. The page fetches the client secret via the server action on page load (SSR). Alternatively, if the page is a Server Component, call the action directly. See Pages section for the SSR fetch approach — no separate API route is needed if the page is an async Server Component.

Decision: Use a Server Component that calls `createSetupIntent()` on the server. NO separate API route needed. The `clientSecret` is passed as a prop to the client component.

However, "add a new card" must be triggered client-side on demand (user clicks "Add card"). Therefore, a lightweight API route is still needed for the client to request a fresh SetupIntent when adding a card:

File: `src/app/api/payments/setup-intent/route.ts`
- Method: POST
- Auth: Uses `authorize()` (Better Auth session from cookies)
- Returns: `{ success: boolean; clientSecret?: string; error?: string }`
- Max lines: Must stay under 300 lines
- No request body needed (user identity from session)

### 2.6 Stripe SetupIntent Flow

The SetupIntent is the correct Stripe mechanism for saving a card without charging it.

Flow:
1. User clicks "Add a card"
2. Client posts to `POST /api/payments/setup-intent`
3. Server calls `getOrCreateStripeCustomer(userId, email)` → gets or creates Stripe Customer
4. Server calls `stripe.setupIntents.create({ customer: customerId, payment_method_types: ['card'] })`
5. Returns `clientSecret` to client
6. Client mounts `<Elements stripe={stripePromise} options={{ clientSecret }}>` with the SetupIntent client secret
7. User fills in `<CardElement>` and clicks "Save card"
8. Client calls `stripe.confirmCardSetup(clientSecret, { payment_method: { card: cardElement } })`
9. On success, the card is saved to the Stripe Customer's payment methods
10. Client calls `listPaymentMethods()` to refresh the displayed list

### Gotchas
- The `user` table does NOT currently have `stripeCustomerId`. A Drizzle migration is required. Generate it with `pnpm drizzle-kit generate` after editing `auth.ts`.

- The `StripeProvider` component in `src/components/providers/stripe-provider.tsx` takes a `clientSecret` tied to a PaymentIntent. The `<Elements>` component accepts both — no change to `StripeProvider` needed, but the `options` must pass the SetupIntent's client secret.
- SetupIntents created on page load (even before user clicks "Add card") are wasted Stripe API calls and count against API usage. Do NOT pre-create a SetupIntent on server render. Create it only when the user clicks "Add card".
- Better Auth session is accessed via `authorize()` from `@/lib/casl`, not via `useSession()` in server actions.
- `revalidatePath('/my/settings/payments')` must be called after remove and setDefault actions.
- Do NOT use `export const dynamic = 'force-dynamic'` on the payments page if data is fetched in the Server Component at render time — the page is already dynamic because it calls `authorize()`. Add `export const dynamic = 'force-dynamic'` explicitly to match the privacy page pattern.

## 8. ADDITIONAL CONTEXT FOR THE INSTALLER

### How existing settings pages work

The existing settings pages in `src/app/(hub)/my/settings/` use two different patterns:
- **Server Component with `authorize()` redirect** (privacy page): Async server component, calls `authorize()` first, redirects if no session, then fetches data and passes to client component. This is the correct pattern for this feature.
- **Client Component with `useSession()`** (profile and security pages): Uses `useSession()` hook. This is a less secure pattern for data-heavy pages. Do NOT use this pattern for the payments page.

Follow the **privacy page pattern** (`src/app/(hub)/my/settings/privacy/page.tsx`) for the payments page.

### How Stripe customer creation works in V3

Multiple existing actions (`create-subscription-checkout.ts`, `create-bundle-checkout.ts`, `purchase-automation.ts`, `purchase-overage-pack.ts`) all use the pattern:
```typescript
let stripeCustomerId = profile.stripeCustomerId;
if (!stripeCustomerId) {
  const customer = await stripe.customers.create({ email, metadata: { userId } });
  stripeCustomerId = customer.id;
  await setStripeCustomerId(sellerProfileId, stripeCustomerId);
}
```

The difference: those actions write to `sellerProfile.stripeCustomerId` (billing customer). The new `payment-methods.ts` writes to `user.stripeCustomerId` (payment customer). The internal helper must write to the `user` table, NOT the `sellerProfile` table.

### The `InlineCardElement` component

`src/components/offers/inline-card-element.tsx` already demonstrates the `CardElement` + `useStripe` + `useElements` pattern. The `add-card-form.tsx` can follow the same pattern but uses `stripe.confirmCardSetup()` instead of `stripe.createPaymentMethod()`.

### Schema migration safety

The new `stripeCustomerId` column on `user` is nullable (no `.notNull()`). This is intentional — existing users who haven't yet added a card will have null. The column is populated lazily when the user first clicks "Add a card".
