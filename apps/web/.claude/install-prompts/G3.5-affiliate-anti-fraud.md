## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2.1 Fraud Signal Checks (Canonical Section 2.9)

Six fraud checks from the canonical, mapped to implementation:

| # | Check (Canonical) | Signal Type | Data Source | Automated Action |
|---|---|---|---|---|
| 1 | Same IP/device for affiliate + referred user | `SELF_REFERRAL_IP` | `referral.ipAddress` vs `session.ipAddress` where `session.userId = affiliate.userId` | Block attribution, flag affiliate |
| 2 | Same payment method across accounts | `SAME_PAYMENT_METHOD` | Stripe API: compare payment method fingerprints for affiliate's Stripe customer and referred user's Stripe customer | Block attribution, flag affiliate |
| 3 | Referred user cancels within 48 hours repeatedly | `RAPID_CHURN` | `referral.churnedAt` within 48h of `referral.convertedAt`, counted per affiliate | Flag affiliate after 3 occurrences |
| 4 | Bot traffic patterns (high clicks, zero signups) | `BOT_TRAFFIC` | `referral` rows: high count of CLICKED status with zero SIGNED_UP/CONVERTED in a rolling window | Throttle link, notify admin |
| 5 | Affiliate creates fake accounts | `FAKE_ACCOUNTS` | Cross-reference: multiple referred users sharing IPs, device characteristics, or signup patterns | Flag for admin review (manual confirmation required) |
| 6 | Geographic anomaly (clicks from bot farms) | `GEO_ANOMALY` | IP clustering: many clicks from a narrow IP range in a short window | Auto-pause affiliate, admin review |

### 2.2 Three Strikes Policy (Canonical Section 2.9)

| Strike | Action | `affiliate.warningCount` | `affiliate.status` |
|---|---|---|---|
| 1st | Warning: in-app notification + audit log | Increment to 1 | Remains ACTIVE |
| 2nd | Suspension: 30-day suspension | Increment to 2 | Set to SUSPENDED |
| 3rd | Permanent ban: ban + pending balance forfeiture | Increment to 3+ | Set to BANNED |

The existing `suspendAffiliate` and `banAffiliate` admin actions handle the status changes. G3.5 adds the automated escalation path that calls them programmatically based on signal accumulation.

### 2.3 Attribution Blocking

When a fraud signal is detected at referral-click time (Check #1: same IP), the system must:
1. Still insert the `referral` record (for analytics visibility)
2. NOT set the attribution cookie
3. Mark the referral record as fraud-blocked (store in a new `fraudBlocked` flag — see NOT SPECIFIED section below)

When a fraud signal is detected at signup-attribution time (Check #2: same payment method), the system must:
1. NOT set `user.referredByAffiliateId`
2. NOT update `referral.status` to SIGNED_UP
3. Log an audit event

### 2.4 Platform Settings (6 new keys)

| Key | Type | Default | Description |
|---|---|---|---|
| `affiliate.fraud.enabled` | boolean | true | Master switch for fraud detection |
| `affiliate.fraud.botClickThreshold` | number | 100 | Clicks with zero signups in rolling window triggers bot flag |
| `affiliate.fraud.botWindowHours` | number | 24 | Rolling window for bot traffic detection |
| `affiliate.fraud.rapidChurnThreshold` | number | 3 | Number of rapid churns before flagging |
| `affiliate.fraud.rapidChurnWindowHours` | number | 48 | Hours after conversion to count as rapid churn |
| `affiliate.fraud.geoClusterThreshold` | number | 50 | Clicks from same /24 subnet in window triggers geo anomaly |

### 2.5 Server Actions / Services

**Service: `src/lib/affiliate/fraud-detection.ts`** (core fraud detection logic, NOT a server action)

Functions:
1. `checkSelfReferralByIp(affiliateUserId: string, referralIpAddress: string): Promise<FraudCheckResult>` — Queries `session` table for sessions belonging to the affiliate's userId that share the same IP as the referral click.
2. `checkSamePaymentMethod(affiliateUserId: string, referredUserId: string): Promise<FraudCheckResult>` — Compares Stripe payment method fingerprints. Requires `sellerProfile.stripeCustomerId` for both users. If either has no Stripe customer, returns `{ flagged: false }` (cannot check).
3. `checkRapidChurn(affiliateId: string): Promise<FraudCheckResult>` — Counts referrals where `churnedAt` is within N hours of `convertedAt`. Returns flagged if count >= threshold.
4. `checkBotTraffic(affiliateId: string): Promise<FraudCheckResult>` — Counts CLICKED referrals with no signup in rolling window. Returns flagged if count >= threshold.
5. `checkGeoAnomaly(affiliateId: string): Promise<FraudCheckResult>` — Groups referral IPs by /24 subnet in rolling window. Returns flagged if any subnet exceeds threshold.
6. `runAllFraudChecks(affiliateId: string): Promise<FraudScanResult>` — Runs checks 3-6 (periodic checks, not per-click). Returns aggregate result.

Type definitions:
```typescript
interface FraudCheckResult {
  flagged: boolean;
  signalType: string;      // e.g., 'SELF_REFERRAL_IP', 'BOT_TRAFFIC'
  details: string;         // Human-readable explanation
  severity: 'WARNING' | 'SUSPEND' | 'BAN';
}

interface FraudScanResult {
  affiliateId: string;
  signals: FraudCheckResult[];
  highestSeverity: 'NONE' | 'WARNING' | 'SUSPEND' | 'BAN';
}
```

**Service: `src/lib/affiliate/fraud-escalation.ts`** (consequences engine)

Functions:
1. `escalateAffiliate(affiliateId: string, signal: FraudCheckResult, actorId: string): Promise<void>` — Applies three-strikes policy:
   - Reads current `affiliate.warningCount`
   - If warningCount === 0: increment warningCount to 1, leave status ACTIVE, send warning notification, audit log
   - If warningCount === 1: increment warningCount to 2, set status SUSPENDED, suspendedAt, suspendedReason, send suspension notification, audit log
   - If warningCount >= 2: increment warningCount, set status BANNED, forfeit pending balance (set `pendingBalanceCents = 0`, `availableBalanceCents = 0`), deactivate all promo codes, send ban notification, audit log
   - For severity BAN (e.g., fake accounts confirmed): skip to permanent ban regardless of current warningCount

**Enhancement to existing referral route: `src/app/ref/[code]/route.ts`**

Modify the GET handler to call `checkSelfReferralByIp` BEFORE inserting the referral and setting the cookie. If flagged:
- Still insert the referral record (with a flag indicating it was fraud-blocked)
- Do NOT set the attribution cookie
- Log audit event with signal details
- Call `escalateAffiliate` with the signal

**Server action: `src/lib/actions/affiliate-fraud-scan.ts`** (admin-triggered manual scan)

1. `runAffiliateFraudScan(input: unknown): Promise<FraudScanActionResult>` — Staff action (CASL: `manage Affiliate`). Takes `{ affiliateId: string }`. Calls `runAllFraudChecks`. Returns results. Does NOT auto-escalate (admin reviews results and decides).

**BullMQ cron job: affiliate fraud scan** (periodic automated scan)

A cron job that runs every 6 hours, scanning all ACTIVE affiliates with recent referral activity (last 24 hours) for bot traffic, rapid churn, and geo anomaly patterns. For each flagged affiliate, calls `escalateAffiliate`.

### 2.6 Queries

**`src/lib/queries/affiliate-fraud.ts`**

1. `getAffiliateFraudSignals(affiliateId: string): Promise<FraudSignalRow[]>` — Returns audit events of type `AFFILIATE_FRAUD_*` for this affiliate, ordered by createdAt DESC, limited to last 100.
2. `getRelatedAccountsByIp(userId: string): Promise<RelatedAccount[]>` — Finds other user accounts that share session IP addresses with the given user. Returns user ID, username, email (masked), shared IPs, and match type. This powers the "Related Accounts (Fraud Investigation)" section of the user detail page described in the Feature Lock-in.
3. `getAffiliateFraudSummary(affiliateId: string): Promise<FraudSummary>` — Aggregate stats: total signals, warnings issued, current risk level, last scan date.

## 3. NOT SPECIFIED — OWNER DECISIONS NEEDED

### 3.A Referral Fraud-Blocked Flag

The `referral` table has no `fraudBlocked` column. When a referral click is detected as self-referral, the current schema has no way to mark it as blocked (the record is still inserted for analytics). Options:

- **Option A (recommended):** Add a boolean `fraudBlocked` column to `referral` table (requires schema change + migration). Simple, queryable.
- **Option B:** Store the fraud-blocked status in the audit event only. The referral record has no marker, but the audit log records the block. Harder to query.
- **Option C:** Use a specific `status` value. But `referralStatusEnum` has no fraud-related value (CLICKED, SIGNED_UP, TRIALING, CONVERTED, CHURNED). Adding one requires enum migration.

**Recommendation for installer:** Use Option B (audit event only) to avoid schema changes. The referral record stays as CLICKED (since no signup occurs), and the audit event records why attribution was blocked. If the owner later wants a queryable flag, it can be added.

### 3.B Payment Method Fingerprint — Stripe API Dependency

Check #2 (same payment method) requires calling Stripe API to compare payment method fingerprints. This means:
- Both the affiliate and the referred user must have `sellerProfile.stripeCustomerId` set
- Stripe API call: `stripe.paymentMethods.list({ customer })` then compare `fingerprint` fields
- If either user has no Stripe customer, this check is skipped (returns not-flagged)
- This check cannot run at referral-click time (referred user doesn't have an account yet). It runs at signup-attribution time or during periodic scans.

**Decision needed:** Should the payment method check run:
- (A) At signup-attribution time (when referral converts to SIGNED_UP), OR
- (B) During periodic scans only (every 6 hours), OR
- (C) Both?

**Recommendation for installer:** Option B (periodic scans only). At signup time the referred user rarely has a payment method yet. Check during periodic scans when the user has had time to add payment info.

### 3.C Device Fingerprint

The canonical's Feature Lock-in mentions "same device fingerprints" in the Related Accounts section, but there is no `deviceFingerprint` column on any table in the schema. The `session` table has `userAgent` but that is a weak proxy (many users share the same user-agent string).

**Recommendation for installer:** Use `userAgent` as a weak signal only (same UA alone does not trigger a flag — it must combine with IP overlap). True device fingerprinting requires a client-side library (e.g., FingerprintJS) and a new schema column, which is out of scope for G3.5.

## 5. ACCEPTANCE CRITERIA

### Positive Cases
1. When a referral click comes from the SAME IP as a recent session of the affiliate, the attribution cookie is NOT set, but the referral record IS created.
2. When a referral click comes from a DIFFERENT IP than the affiliate, the attribution cookie IS set normally.
3. When an affiliate accumulates 3+ rapid churns (referred users cancel within 48h of converting), the affiliate is flagged.
4. When an affiliate has 100+ clicks in 24 hours with zero signups, the bot traffic check flags them.
5. When 50+ clicks come from the same /24 subnet in the bot window, the geo anomaly check flags them.
6. First fraud flag on a clean affiliate (warningCount=0) increments warningCount to 1, sends warning notification, and leaves status ACTIVE.
7. Second fraud flag on a warned affiliate (warningCount=1) suspends the affiliate (status=SUSPENDED, warningCount=2).
8. Third fraud flag on a suspended affiliate (warningCount=2) bans the affiliate (status=BANNED, warningCount=3), forfeits pending+available balance, deactivates all promo codes.
9. Confirmed fake accounts trigger immediate ban regardless of current warningCount.
10. Admin can run a manual fraud scan from `/usr/affiliates/[id]` page and see results.
11. The periodic cron job runs every 6 hours and scans only ACTIVE affiliates with recent referral activity.
12. All fraud detections create audit events with severity HIGH or CRITICAL.

### Negative Cases
13. Already-PAID commissions are NEVER clawed back or reversed by the fraud system.
14. An affiliate with status BANNED or SUSPENDED is NOT scanned by the periodic job.
15. Fraud thresholds are NOT hardcoded — they come from platform_settings.
16. The referral click handler does NOT call Stripe API (performance requirement — must redirect fast).
17. The self-referral IP check does NOT flag on common/shared IPs (e.g., localhost, 127.0.0.1) — these should be excluded.
18. A fraud scan that finds zero signals returns `highestSeverity: 'NONE'` and does NOT escalate.
19. The `affiliate.fraud.enabled` platform setting being `false` disables ALL automated fraud checks (manual scan still works).

### Data Integrity
20. All audit events use `actorType: 'SYSTEM'` for automated fraud checks and `actorType: 'STAFF'` for manual scans.
21. `warningCount` only ever increases (never decremented by the fraud system).
22. When balance is forfeited on ban, both `pendingBalanceCents` and `availableBalanceCents` are set to 0.
23. All PENDING and PAYABLE commissions are reversed (status set to REVERSED, reversalReason set) on ban.

### Authorization
24. Only staff with `manage Affiliate` CASL ability can trigger manual fraud scans.
25. The cron job uses system-level authentication (no user session required).
26. Regular users (affiliates) cannot see fraud signals or trigger scans.

### Vocabulary
27. No banned terms appear in any file created or modified.

## 9. IMPLEMENTATION NOTES

### IP Subnet Parsing

For the geo anomaly check, extract /24 subnet from IPv4 addresses:
```typescript
function getSubnet24(ip: string): string | null {
  const parts = ip.split('.');
  if (parts.length !== 4) return null; // Skip IPv6 or invalid
  return `${parts[0]}.${parts[1]}.${parts[2]}.0/24`;
}
```

### Self-Referral IP Check Integration Point

The referral route (`src/app/ref/[code]/route.ts`) currently:
1. Checks affiliate exists and is ACTIVE
2. Extracts IP and user-agent from request headers
3. Inserts referral record
4. Sets attribution cookie if none exists
5. Redirects to /auth/signup

G3.5 inserts a check between steps 2 and 3:
- Call `checkSelfReferralByIp(affiliate.userId, ipAddress)`
- If flagged: insert referral (step 3) but skip cookie (step 4), log audit, call escalateAffiliate
- If not flagged: proceed normally

### Cron Job Pattern

Follow the existing pattern in `src/lib/jobs/` (e.g., `local-noshow-check.ts`):
- Export a processor function
- Register in `instrumentation.ts` using the existing BullMQ factory
- Schedule: every 6 hours (`'0 */6 * * *'`)
- Job name: `affiliate-fraud-scan`
- Queue name: `affiliate-fraud-scan` (hyphens, not colons — per known bug fix)

### Commission Reversal on Ban

When banning an affiliate, use the existing `reverseCommission` function from `src/lib/affiliate/commission-reversal.ts` to reverse all PENDING and PAYABLE commissions. Query:
```sql
SELECT * FROM affiliate_commission
WHERE affiliate_id = ? AND status IN ('PENDING', 'PAYABLE')
```
Then call `reverseCommission` for each.

### Hub UI: Fraud Panel

The fraud panel component (`affiliate-fraud-panel.tsx`) should be a self-contained client component that:
1. Displays fraud signal history (from `getAffiliateFraudSignals`)
2. Shows current risk summary (from `getAffiliateFraudSummary`)
3. Has a "Run Fraud Scan" button that calls `runAffiliateFraudScan`
4. Displays related accounts (from `getRelatedAccountsByIp`)

This component is imported into the existing `/usr/affiliates/[id]/page.tsx` as a new section below the existing content.
