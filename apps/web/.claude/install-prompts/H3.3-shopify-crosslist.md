### 2.2 Shopify Admin REST API Endpoints

All calls use the `X-Shopify-Access-Token` header (NOT Bearer token). The shop domain is stored as `account.externalAccountId`. The API version comes from `loadShopifyConfig().apiVersion` (default `'2024-01'`).

| Operation | HTTP Method | Endpoint | Request Body |
|---|---|---|---|
| Create product | `POST` | `https://{shop}/admin/api/{version}/products.json` | `{ product: { ... } }` |
| Update product | `PUT` | `https://{shop}/admin/api/{version}/products/{id}.json` | `{ product: { ... } }` |
| Delete product | `DELETE` | `https://{shop}/admin/api/{version}/products/{id}.json` | (none) |
| Fetch single product | `GET` | `https://{shop}/admin/api/{version}/products/{id}.json` | (none) |

#### 2.3.1 `toShopifyProductInput(listing: TransformedListing): ShopifyProductInput`

Converts a `TransformedListing` (Twicely canonical format) into the Shopify REST API product creation payload shape.

**Field mapping rules:**
- `listing.title` --> `product.title` (truncate to 255 chars per SHOPIFY_CAPABILITIES.maxTitleLength)
- `listing.descriptionHtml` --> `product.body_html` if present; otherwise wrap `listing.description` in `<p>` tags (Shopify is Tier A in CHANNEL_REGISTRY, supports HTML descriptions). Truncate underlying text to 5000 chars (maxDescriptionLength from capabilities). NOTE: Shopify body_html is rendered as HTML in the storefront, so HTML is preferred.
- `listing.brand` --> `product.vendor` (empty string if null)
- `listing.category.externalCategoryName` --> `product.product_type` (empty string if no mapping)
- `listing.itemSpecifics.tags` --> `product.tags` (comma-separated string). If no tags key, empty string.
- Product `status` is always `'active'` (we only crosslist ACTIVE listings)
- `listing.priceCents` --> `variants[0].price` as decimal string (`(priceCents / 100).toFixed(2)`) -- Shopify uses decimal string prices
- `listing.quantity` --> `variants[0].inventory_quantity`
- `listing.itemSpecifics.sku` --> `variants[0].sku` (null if not present)
- `listing.itemSpecifics.barcode` --> `variants[0].barcode` (null if not present)
- `listing.shipping.weightOz` --> `variants[0].weight` + `variants[0].weight_unit = 'oz'` (null if no weight)
- `listing.images` --> `product.images[]` sorted by `sortOrder`, limited to 10 (maxImagesPerListing). Each image: `{ src: img.url, position: idx + 1 }`. Shopify positions are 1-based.
- `listing.condition` --> `product.metafields` array with a single metafield: `{ namespace: 'twicely', key: 'condition', value: listing.condition, type: 'single_line_text_field' }` (only if condition is non-empty). This preserves condition data that Shopify doesn't natively track.

**Return type -- define in shopify-types.ts:**
```typescript
export interface ShopifyProductInput {
  title: string;
  body_html: string;
  vendor: string;
  product_type: string;
  tags: string;
  status: 'active' | 'draft';
  variants: Array<{
    price: string;
    inventory_quantity: number;
    sku: string | null;
    barcode: string | null;
    weight: number | null;
    weight_unit: string | null;
  }>;
  images: Array<{
    src: string;
    position: number;
  }>;
  metafields?: Array<{
    namespace: string;
    key: string;
    value: string;
    type: string;
  }>;
}
```

#### 2.3.2 `toShopifyPartialInput(changes: Partial<TransformedListing>): Partial<ShopifyProductInput>`

Converts only the changed fields. Only include properties in the output that are present in the `changes` input. Rules:
- If `changes.title` present: set `title` (truncated to 255)
- If `changes.description` or `changes.descriptionHtml` present: set `body_html`
- If `changes.priceCents` present: set `variants[0].price`
- If `changes.quantity` present: set `variants[0].inventory_quantity`
- If `changes.brand` present: set `vendor`
- If `changes.images` present: set `images[]`
- If `changes.category` present: set `product_type`
- If `changes.itemSpecifics` present: set `tags` from `changes.itemSpecifics.tags`

For the `variants` array in partial updates: if either `priceCents` or `quantity` is present, include a variants array with one variant object containing only the changed fields. The variant is NOT identified by ID here -- the update endpoint auto-maps to the first variant when only one exists.

#### 2.3.3 `createShopifyProduct(config, shopDomain, accessToken, input): Promise<ShopifyCreateResult>`

Makes the HTTP POST to Shopify.

```typescript
interface ShopifyCreateResult {
  success: boolean;
  productId: string | null;
  handle: string | null;
  error?: string;
  retryable: boolean;
  statusCode: number;
}
```

**Implementation:**
1. POST to `https://{shopDomain}/admin/api/{config.apiVersion}/products.json`
2. Headers: `{ 'Content-Type': 'application/json', 'X-Shopify-Access-Token': accessToken }`
3. Body: `JSON.stringify({ product: input })`
4. Parse response:
   - On `response.ok`: parse body as `{ product: { id: number; handle: string; ... } }`. Return `{ success: true, productId: String(product.id), handle: product.handle }`.
   - On 401: return `{ success: false, error: 'Unauthorized', retryable: false, statusCode: 401 }`.
   - On 422 (validation error): parse body for `{ errors: Record<string, string[]> }`. Return `{ success: false, error: first error message, retryable: false }`.
   - On 429: return `{ success: false, error: 'Rate limited', retryable: true, statusCode: 429 }`.
   - On 5xx: return `{ success: false, error: ..., retryable: true }`.
   - On network error (catch): return `{ success: false, error: String(err), retryable: true }`.

#### 2.3.4 `updateShopifyProduct(config, shopDomain, accessToken, externalId, input): Promise<ShopifyUpdateResult>`

Makes the HTTP PUT to Shopify.

```typescript
interface ShopifyUpdateResult {
  success: boolean;
  error?: string;
  retryable: boolean;
  statusCode: number;
}
```

**Implementation:**
1. PUT to `https://{shopDomain}/admin/api/{config.apiVersion}/products/${externalId}.json`
2. Headers: same as create
3. Body: `JSON.stringify({ product: input })`
4. Error handling same pattern as create (401=non-retryable, 422=non-retryable, 429=retryable, 5xx=retryable)

#### 2.3.5 `deleteShopifyProduct(config, shopDomain, accessToken, externalId): Promise<ShopifyDeleteResult>`

Makes the HTTP DELETE to Shopify.

```typescript
interface ShopifyDeleteResult {
  success: boolean;
  error?: string;
  retryable: boolean;
  statusCode: number;
}
```

**Implementation:**
1. DELETE to `https://{shopDomain}/admin/api/{config.apiVersion}/products/${externalId}.json`
2. Headers: `{ 'X-Shopify-Access-Token': accessToken }`
3. Response handling:
   - `response.ok` or 200: success
   - 404: **success** (idempotent -- product already deleted)
   - 401: non-retryable failure
   - 429: retryable failure
   - 5xx: retryable failure
   - Network error: retryable failure

#### 2.3.6 `fetchShopifyProductForVerify(config, shopDomain, accessToken, externalId): Promise<ShopifyVerifyResult>`

Fetches a single product to verify its existence and current state.

```typescript
interface ShopifyVerifyResult {
  exists: boolean;
  status: 'ACTIVE' | 'ENDED' | 'DRAFT' | 'REMOVED';
  priceCents: number | null;
  quantity: number | null;
  lastModifiedAt: Date | null;
}
```

**Implementation:**
1. GET `https://{shopDomain}/admin/api/{config.apiVersion}/products/${externalId}.json`
2. Headers: `{ 'X-Shopify-Access-Token': accessToken }`
3. Response handling:
   - `response.ok`: parse with ShopifyProductSchema, use `parseShopifyPrice` for price, `mapShopifyStatus` for status, parse `updated_at` for lastModifiedAt, sum variant inventory_quantity for quantity
   - 404: `{ exists: false, status: 'REMOVED', ... nulls }`
   - Other errors: `{ exists: false, status: 'REMOVED', ... nulls }`

### 2.4 Modify: `shopify-connector.ts`

Replace the four stub methods with real implementations that delegate to `shopify-crosslist.ts` helpers. The connector methods are thin wrappers -- all HTTP logic lives in `shopify-crosslist.ts`.

#### 2.4.1 `createListing(account, listing) -> PublishResult`

```
1. Guard: if !account.accessToken || !account.externalAccountId -> return { success: false, error: 'No credentials', retryable: false }
2. Load config via loadShopifyConfig()
3. Convert listing via toShopifyProductInput(listing)
4. Call createShopifyProduct(config, shopDomain, accessToken, input)
5. On success: return { success: true, externalId: result.productId, externalUrl: `https://${shopDomain}/products/${result.handle}` }
6. On failure: return { success: false, externalId: null, externalUrl: null, error: result.error, retryable: result.retryable }
```

IMPORTANT: The `externalUrl` uses the storefront URL format (`/products/{handle}`), NOT the admin URL. The handle comes from the Shopify API response.

If the handle is null/empty (edge case), fall back to `https://${shopDomain}/admin/products/${productId}`.

#### 2.4.2 `updateListing(account, externalId, changes) -> UpdateResult`

```
1. Guard: if !account.accessToken || !account.externalAccountId -> return { success: false, error: 'No credentials', retryable: false }
2. Load config via loadShopifyConfig()
3. Convert changes via toShopifyPartialInput(changes)
4. Call updateShopifyProduct(config, shopDomain, accessToken, externalId, partialInput)
5. Return { success: result.success, error: result.error, retryable: result.retryable }
```

#### 2.4.3 `delistListing(account, externalId) -> DelistResult`

```
1. Guard: if !account.accessToken || !account.externalAccountId -> return { success: false, error: 'No credentials', retryable: false }
2. Load config via loadShopifyConfig()
3. Call deleteShopifyProduct(config, shopDomain, accessToken, externalId)
4. Return { success: result.success, error: result.error, retryable: result.retryable }
```

NOTE: Shopify does NOT have a "draft" or "unpublish" concept that maps cleanly to delist. The DELETE endpoint permanently removes the product. This is intentional -- when Twicely delists from Shopify, the product is removed entirely. The `channelProjection` record retains the externalId for audit purposes.

#### 2.4.4 `verifyListing(account, externalId) -> VerificationResult`

```
1. Guard: if !account.accessToken || !account.externalAccountId -> return { exists: false, status: 'UNKNOWN', priceCents: null, quantity: null, lastModifiedAt: null, diff: null }
2. Load config via loadShopifyConfig()
3. Call fetchShopifyProductForVerify(config, shopDomain, accessToken, externalId)
4. Return { exists: result.exists, status: result.status, priceCents: result.priceCents, quantity: result.quantity, lastModifiedAt: result.lastModifiedAt, diff: null }
```

## 3. CONSTRAINTS -- WHAT NOT TO DO

### 3.1 Banned Patterns
- Do NOT use `Bearer` token auth -- Shopify uses `X-Shopify-Access-Token` header
- Do NOT use GraphQL -- Shopify crosslist uses the REST Admin API (H3.1 comment incorrectly says "GraphQL" in the connector class doc comment -- ignore that; the actual code uses REST)
- Do NOT hardcode API version -- read from `loadShopifyConfig().apiVersion`
- Do NOT create any new database tables or columns
- Do NOT modify the `PlatformConnector` interface
- Do NOT add `as any`, `@ts-ignore`, or `@ts-expect-error`
- Do NOT use floating point for money -- integer cents only in Twicely types; convert to Shopify decimal strings only at the API boundary
- Do NOT spread request body -- explicit field mapping in `toShopifyProductInput` and `toShopifyPartialInput`
- Do NOT export helpers from any `'use server'` file (these files are NOT `'use server'` files)
- No file may exceed 300 lines

### 3.2 Shopify-Specific Gotchas
- Shopify product IDs are **numbers** in the API response but must be stored as **strings** in `channelProjection.externalId` (via `String(product.id)`)
- Shopify prices are **decimal strings** ("29.99"), NOT numbers. Always use `.toFixed(2)` when converting from cents.
- Shopify `body_html` renders as HTML in the storefront. Send HTML, not plain text.
- Shopify variant `weight_unit` accepts: `lb`, `oz`, `kg`, `g`. Use `'oz'` to match Twicely's `weightOz` field.
- Shopify image `position` is **1-based**, not 0-based. Map from 0-based sortOrder via `sortOrder + 1` or `index + 1`.
- Shopify `tags` is a single comma-separated string, not an array.
- Shopify DELETE is permanent (no "archive" via this flow). This is intentional for delist.
- 404 on DELETE is success (idempotent).
- 404 on GET (verify) means product was removed.

### 3.3 CLAUDE.md Rules That Apply
- Money as integer cents -- convert to decimal string only at Shopify API boundary
- Max 300 lines per file -- `shopify-crosslist.ts` extraction handles this
- Ownership via `userId` -- connector receives `account` from job-executor which already verified ownership
- No fees on off-platform sales -- crosslisting is subscription-funded, no TF applies

## 4. ACCEPTANCE CRITERIA

### 4.1 Functional Criteria

1. **createListing succeeds**: When given a valid account and TransformedListing, the connector POSTs to `https://{shop}/admin/api/{version}/products.json` with correctly mapped fields and returns `{ success: true, externalId: "<product-id>", externalUrl: "https://{shop}/products/{handle}" }`.

2. **createListing maps all fields correctly**:
   - title truncated to 255 characters
   - body_html contains HTML (wrapped in `<p>` tags if descriptionHtml is null)
   - vendor = brand (empty string if null)
   - product_type = category.externalCategoryName
   - tags = itemSpecifics.tags (comma-separated string)
   - status = 'active'
   - variants[0].price = decimal string (e.g. "49.99" for 4999 cents)
   - variants[0].inventory_quantity = quantity
   - variants[0].sku from itemSpecifics (null if absent)
   - variants[0].barcode from itemSpecifics (null if absent)
   - variants[0].weight from shipping.weightOz (null if absent)
   - variants[0].weight_unit = 'oz' (when weight is present)
   - images sorted by sortOrder, limited to 10, position is 1-based
   - condition stored as metafield (namespace: 'twicely', key: 'condition')

3. **createListing retryable logic**: 429 and 5xx return `retryable: true`. 401 and 422 return `retryable: false`. Network errors return `retryable: true`.

4. **updateListing sends only changed fields**: When `changes = { priceCents: 1999 }`, only `variants[0].price` is sent. Title, images, vendor, etc. are NOT included in the PUT body.

5. **delistListing sends DELETE**: Correctly sends DELETE to `/admin/api/{version}/products/{externalId}.json`. Returns success on 200/204/404. Returns retryable on 429/5xx.

6. **delistListing is idempotent**: 404 response is treated as success (product already gone).

7. **verifyListing returns correct status**: Maps Shopify `status: 'active'` to `ACTIVE`, `status: 'archived'` to `ENDED`, `status: 'draft'` to `DRAFT`. 404 returns `{ exists: false, status: 'REMOVED' }`.

8. **verifyListing returns correct price**: Parses variant price from decimal string to integer cents using `parseShopifyPrice`.

9. **verifyListing returns correct quantity**: Sums all variant `inventory_quantity` values. Minimum 1.

10. **verifyListing returns lastModifiedAt**: Parses `updated_at` ISO string to Date.

11. **No credentials guard**: All four methods return early with appropriate failure when `accessToken` or `externalAccountId` is null/empty.

## 8. IMPLEMENTATION NOTES

### 8.1 Pattern Reference: Whatnot Crosslist (H2.2)

The Whatnot connector (`whatnot-connector.ts` lines 425-613) is the closest pattern match. Key differences for Shopify:

| Aspect | Whatnot (H2.2) | Shopify (H3.3) |
|---|---|---|
| API type | GraphQL mutations | REST Admin API |
| Create flow | 2-step (create draft, then publish) | 1-step (POST with status: 'active') |
| Price format | Money type `{ amount, currencyCode }` | Decimal string on variant `"29.99"` |
| Auth header | `Authorization: Bearer {token}` | `X-Shopify-Access-Token: {token}` |
| Delist | Unpublish mutation (preserves listing) | DELETE (permanently removes product) |
| Transform file | `whatnot-transform.ts` | `shopify-crosslist.ts` |
| IDs | String IDs from GraphQL | Number IDs from REST (convert to string) |
| Verify | GraphQL listing query | REST GET /products/{id}.json |

For the crosslist helpers, use the value from the TransformedListing which has already been limited by listing-transform. **Resolution:** Use the incoming TransformedListing's images as-is (they are already limited by listing-transform.ts which uses channel-registry). In `toShopifyProductInput`, slice to 250 as a defensive cap matching `SHOPIFY_CAPABILITIES.maxImagesPerListing`.

### 8.4 Price Conversion Safety

The cents-to-decimal conversion MUST use:
```typescript
(priceCents / 100).toFixed(2)
```
This avoids floating point issues (e.g., `1999 / 100 = 19.990000000000002`). `.toFixed(2)` handles rounding correctly.

The reverse conversion (Shopify decimal string to cents) already exists in `shopify-normalizer.ts` as `parseShopifyPrice()`:
```typescript
export function parseShopifyPrice(priceStr: string): number {
  const parsed = parseFloat(priceStr);
  if (isNaN(parsed)) return 0;
  return Math.round(parsed * 100);
}
```

### 8.5 Config Loading

All four connector methods need the Shopify config (specifically `apiVersion`). The existing `loadShopifyConfig()` function (defined in `shopify-connector.ts`, unexported) loads from `platform_settings` on every call.

For `shopify-crosslist.ts` to call the API, it needs `apiVersion`, `shopDomain`, and `accessToken`. These should be passed as parameters from the connector methods -- do NOT re-export `loadShopifyConfig()`. The crosslist helpers should accept a simple config interface:

```typescript
interface ShopifyCrosslistConfig {
  apiVersion: string;
}
```

This mirrors the `ShopifyImportConfig` pattern in `shopify-import.ts` (line 18-20).
