## 2. SCOPE — EXACTLY WHAT TO BUILD

### 2A. Canonical Spec (Addendum §A12 — verbatim)

The spec defines four detection triggers:

| # | Signal | Action |
|---|--------|--------|
| 1 | Seller manually marks a **different order** for the **same listing** as COMPLETED while localTransaction is SCHEDULED | **Automatic fraud flag** |
| 2 | pHash duplicate listing created while transaction is SCHEDULED (from A11) | **Fraud investigation trigger** |
| 3 | Seller no-show + listing immediately relisted within 24hr | **Fraud investigation trigger** |
| 4 | Buyer files fraud claim with evidence (screenshot of item sold elsewhere) | **Manual review** |

The spec defines three severity tiers:

| Severity | What Happened | Consequence |
|----------|---------------|-------------|
| `CONFIRMED` | Sold item after escrow, buyer no-showed | Full refund from escrow + local transaction **permanent ban** + account review for full suspension |
| `STRONG_SIGNAL` | Duplicate listing while SCHEDULED | Listing removed + formal warning + reliability marks |
| `PATTERN` (2nd offense) | Second confirmed fraud | Full account suspension |

Additional spec statements:
- **Buyer Protection:** "Full refund from escrow always. Escrow funds have not moved to the seller — Stripe reverses cleanly. Buyer is made whole regardless of fraud investigation outcome."
- **New Ledger Entry:** `LOCAL_FRAUD_REVERSAL` — escrow reversed to buyer on confirmed fraud. Seller payout blocked.

### 2C. Business Logic — Detection Service

Create `src/lib/commerce/local-fraud-detection.ts` (~200 lines).

#### Signal 1: Same-Listing Sold While SCHEDULED (AUTOMATIC)

**When to check:** This check runs inside `markOrderCompleted()` and `autoCompleteDeliveredOrders()` in `src/lib/commerce/order-completion.ts`. Before marking an order COMPLETED, check if the listing involved has any ACTIVE (non-terminal) localTransaction with status in `['SCHEDULED', 'SELLER_CHECKED_IN', 'BUYER_CHECKED_IN', 'BOTH_CHECKED_IN', 'ADJUSTMENT_PENDING', 'RESCHEDULE_PENDING', 'RECEIPT_CONFIRMED']`.

**Detection query:**
```typescript
async function detectSameListingSold(
  orderId: string,
  listingId: string,
  sellerId: string,
): Promise<{ fraudDetected: boolean; conflictingTransactionId?: string }>
```

1. Get `listingId` from `orderItem` where `orderId = {completingOrderId}`
2. Find any `localTransaction` where:
   - There exists an `orderItem` for that `localTransaction.orderId` with the same `listingId`
   - `localTransaction.status` NOT IN (`COMPLETED`, `CANCELED`, `NO_SHOW`, `DISPUTED`)
   - `localTransaction.orderId != {completingOrderId}` (don't flag the same order)
3. If found: create `localFraudFlag` with severity `CONFIRMED`, trigger `SAME_LISTING_SOLD`
4. Auto-cancel the local transaction (full refund to buyer)
5. Create `LOCAL_FRAUD_REVERSAL` ledger entry
6. Set `user.localFraudBannedAt` = now (permanent local ban)
7. Notify buyer: "Your local meetup for {item} has been canceled. A full refund has been issued."
8. Notify seller: "Your local transaction for {item} has been flagged for review."
9. Create audit event with severity `CRITICAL`

**IMPORTANT:** This is the only AUTOMATIC consequence. Signals 2-4 create flags for staff review only.

#### Signal 3: No-Show + Relist Within 24hr (INVESTIGATION TRIGGER)

**When to check:** After a local transaction reaches `NO_SHOW` status. Check if the seller relists the same listing (changes status back to `ACTIVE`) within 24 hours. This is detected by a BullMQ delayed job.

**Implementation:**
1. When `localTransaction.status` transitions to `NO_SHOW`, enqueue a BullMQ job `local-fraud-noshow-relist-check` with 24-hour delay
2. At job fire time, check if the listing associated with the no-show transaction has `status = 'ACTIVE'` AND `updatedAt > noShowTimestamp`
3. If so: create `localFraudFlag` with severity `STRONG_SIGNAL`, trigger `NOSHOW_RELIST`
4. Flag the listing (`enforcementState = 'FLAGGED'`)
5. Notify seller: "Your listing has been flagged for review."

#### Signal 2: pHash Duplicate (DEFERRED)

**NOT BUILT in G2.15.** pHash infrastructure is deferred (it's referenced in D6.4 as built for authentication, but listing-to-listing pHash comparison is not wired). This trigger is specked in the addendum but depends on a pHash comparison pipeline that doesn't exist yet.

**What to do:** Add a placeholder function `checkPhashDuplicate()` that returns `{ duplicate: false }`. Add a TODO comment: `// G2.15: pHash duplicate detection deferred — requires listing pHash comparison pipeline`. Add a platform setting `commerce.local.phashFraudCheckEnabled: false` so this can be toggled on when ready.

#### Signal 4: Buyer Fraud Claim (MANUAL)

**Implementation:** A server action that allows a buyer to report suspected fraud on their local transaction. This creates a `localFraudFlag` with severity `MANUAL_REVIEW`, trigger `BUYER_CLAIM`, and the `detailsJson` includes the buyer's description. NO automatic consequences — staff reviews.

### 2D. Server Actions

Create `src/lib/actions/local-fraud.ts` (~150 lines, `'use server'`).

#### Action 1: `reportLocalFraudAction`

**Who:** Buyer (the buyer of the local transaction)
**When:** After a local transaction reaches a terminal state OR the buyer suspects fraud during SCHEDULED
**CASL:** `ability.can('update', sub('LocalTransaction', { buyerId: session.userId }))`
**Input Zod schema:**
```typescript
const reportLocalFraudSchema = z.object({
  localTransactionId: z.string().min(1),
  description: z.string().min(10).max(2000),
}).strict();
```
**Logic:**
1. Authorize: buyer must own the transaction
2. Validate input
3. Look up the localTransaction and associated listingId (via orderItem)
4. Create `localFraudFlag` with severity `MANUAL_REVIEW`, trigger `BUYER_CLAIM`
5. Notify staff (create audit event with severity `HIGH`)
6. Return `{ success: true }`

**Validation rules:**
- One fraud report per buyer per localTransaction (check existing `localFraudFlag` with same `localTransactionId` and `trigger = 'BUYER_CLAIM'`)
- Transaction must belong to the authenticated buyer

#### Action 2: `resolveLocalFraudFlagAction` (Hub Staff)

**Who:** Staff with MODERATION or ADMIN role
**CASL:** `staffAuthorize()` + `ability.can('manage', 'LocalTransaction')`
**Input Zod schema:**
```typescript
const resolveLocalFraudFlagSchema = z.object({
  flagId: z.string().min(1),
  resolution: z.enum(['CONFIRMED', 'DISMISSED']),
  note: z.string().min(1).max(2000),
  applyConsequences: z.boolean(),
}).strict();
```
**Logic:**
1. Staff authorize
2. Validate input
3. Load the flag (must be `status = 'OPEN'`)
4. If `resolution = 'CONFIRMED'` and `applyConsequences = true`:
   - Issue full refund (if not already refunded) — call `stripe.refunds.create()` with `reverse_transfer: true, refund_application_fee: true`
   - Create `LOCAL_FRAUD_REVERSAL` ledger entry
   - Set `user.localFraudBannedAt = now`
   - If this is the seller's 2nd confirmed fraud: suspend the entire account (`user.sellerStatus = 'SUSPENDED'`)
   - Update listing `enforcementState = 'REMOVED'`
   - Update flag: status `CONFIRMED`, resolvedByStaffId, resolvedAt, resolutionNote
5. If `resolution = 'DISMISSED'`:
   - Update flag: status `DISMISSED`, resolvedByStaffId, resolvedAt, resolutionNote
   - If listing was `FLAGGED`, restore to previous state
6. Create audit event
7. `revalidatePath('/mod/fraud')`
8. Return `{ success: true }`

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Terms
- Never use "escrow" in UI copy — use "SafeTrade" (Addendum §A0)
- Never use "wallet", "balance", "withdraw" — use correct payout language per CLAUDE.md
- Never use any banned tier names (SellerTier, BASIC, ELITE, etc.)

### Gotchas
- **Do NOT build pHash comparison** — it's deferred. Add a placeholder only.
- **Do NOT auto-suspend accounts** on Signal 2/3/4 — only Signal 1 has automatic consequences (permanent local ban). Signals 2-4 create flags for staff review. Full account suspension (PATTERN) is staff-only.
- **Signal 1 does NOT block the shipped order from completing** — the shipped order is a legitimate sale. The fraud is against the LOCAL buyer who had funds escrowed.
- **Refund flows through Stripe** — `stripe.refunds.create()` with `reverse_transfer: true`. Escrow funds have NOT moved to the seller yet, so this is a clean reversal.
- **`listing.ownerUserId`** is the ownership column on the listing table (NOT `sellerId`). The `localTransaction.sellerId` must match `listing.ownerUserId`.
- **`orderItem.listingId`** is the bridge between `order` and `listing`. Always go through `orderItem` to find the listing for an order.
- **Money in integer cents only** — all amounts as integer cents.
- **Fee rates from `platform_settings`** — read `commerce.local.fraudNoshowRelistCheckHours` and `commerce.local.fraudPatternOffenseCount` at runtime.
- **No `as any`, no `@ts-ignore`** — fix the types.
- **Max 300 lines per file** — split if longer.
- **Explicit field mapping** — never spread request body into DB inserts.
- **Zod `.strict()`** on all input schemas.

### What NOT to build
- No hub UI page for `/mod/fraud` — that is a separate step (not in G2.15 scope). G2.15 builds the backend detection + actions + queries. The hub page will consume these.
- No Centrifugo real-time push — detection is server-side only.
- No email templates — use `notify()` with template keys. Template content implementation is deferred.

## 4. ACCEPTANCE CRITERIA

### Positive Cases

1. **AC-1:** When a shipped order for listing X completes while listing X has an active local SafeTrade transaction in SCHEDULED status, a `localFraudFlag` is created with severity `CONFIRMED`, trigger `SAME_LISTING_SOLD`.
2. **AC-2:** On Signal 1 auto-detection, the buyer of the local transaction receives a full Stripe refund.
3. **AC-3:** On Signal 1 auto-detection, a `LOCAL_FRAUD_REVERSAL` ledger entry is created for the seller.
4. **AC-4:** On Signal 1 auto-detection, `user.localFraudBannedAt` is set to the current timestamp.
5. **AC-5:** On Signal 1 auto-detection, the local transaction status transitions to `CANCELED`.
6. **AC-6:** On Signal 1 auto-detection, the listing is unreserved (status returns to ACTIVE for the legitimate shipped buyer).
7. **AC-7:** After a local transaction reaches NO_SHOW, a BullMQ job is enqueued with a 24-hour delay to check for relisting.
8. **AC-8:** If the no-show relist check finds the listing re-activated within 24 hours, a `localFraudFlag` is created with severity `STRONG_SIGNAL`, trigger `NOSHOW_RELIST`, and the listing `enforcementState` is set to `FLAGGED`.
9. **AC-9:** A buyer can submit a fraud report via `reportLocalFraudAction`, creating a `localFraudFlag` with severity `MANUAL_REVIEW` and trigger `BUYER_CLAIM`.
10. **AC-10:** Staff can resolve a fraud flag via `resolveLocalFraudFlagAction` with either `CONFIRMED` (applies consequences) or `DISMISSED`.
11. **AC-11:** When staff confirms a fraud flag with `applyConsequences = true`, a full refund is issued and the seller is permanently banned from local transactions.
12. **AC-12:** When staff confirms a 2nd fraud for the same seller, the seller's `sellerStatus` is set to `SUSPENDED`.
13. **AC-13:** `isUserSuspendedFromLocal()` returns `{ suspended: true, reason: 'fraud_ban' }` when `user.localFraudBannedAt` is non-null.
14. **AC-14:** `getLocalFraudFlags()` returns paginated fraud flags with seller name, listing title, and transaction details.

### Negative Cases

15. **AC-15:** Signal 1 does NOT fire when the completing order IS the same order as the local transaction (same listing, same order = normal completion, not fraud).
16. **AC-16:** Signal 1 does NOT fire when the local transaction is already in a terminal state (COMPLETED, CANCELED, NO_SHOW).
17. **AC-17:** Cash/off-platform local transactions do NOT have fraud detection (SafeTrade only — per Addendum §A0 feature matrix).
18. **AC-18:** Signal 3 (no-show relist check) does NOT fire if the listing is NOT re-activated within 24 hours.
19. **AC-19:** A buyer cannot submit more than one fraud report per local transaction.
20. **AC-20:** A non-buyer cannot submit a fraud report on a local transaction they don't own.
21. **AC-21:** Only staff with MODERATION or ADMIN role can resolve fraud flags.
22. **AC-22:** A fraud flag that is already CONFIRMED or DISMISSED cannot be resolved again.

### Data Integrity

23. **AC-23:** All monetary values stored as integer cents.
24. **AC-24:** `LOCAL_FRAUD_REVERSAL` exists in `ledgerEntryTypeEnum`.
25. **AC-25:** `localFraudFlag` table has FK constraints to `user`, `localTransaction`, `listing`.
26. **AC-26:** Audit events with severity `CRITICAL` are created for confirmed fraud.

### Vocabulary

27. **AC-27:** No banned terms appear in any code or UI strings (no "escrow", "wallet", "withdraw", "FVF", "SellerTier", etc.).

## 8. OWNER DECISIONS NEEDED

The following items are NOT fully specified in the canonical documents and require owner input:

### Decision A: No-Show Source Location

**Question:** Where exactly is the NO_SHOW status applied to a localTransaction? The no-show check worker needs to be identified so the relist-check BullMQ job can be enqueued there. The build tracker says G2.3 (no-show detection) is complete and updated in G2.8. The installer needs to find the exact file/function.

**Recommendation:** Look in `src/lib/jobs/local-noshow-check.ts` for the handler that sets `status = 'NO_SHOW'`. Enqueue the relist check job there.

### Decision B: Account Suspension Column

**Question:** The spec says PATTERN (2nd offense) results in "Full account suspension." The `sellerStatusEnum` has `SUSPENDED` as a value, and `user` has no `accountSuspendedAt` column. Should we set `sellerProfile.status = 'SUSPENDED'` or add a dedicated column? The existing `sellerStatusEnum` ('ACTIVE', 'RESTRICTED', 'SUSPENDED') maps cleanly.

**Recommendation:** Use `sellerProfile.status = 'SUSPENDED'` — it already exists and is the standard enforcement path.

### Decision C: SafeTrade-Only Gate

**Question:** The addendum says fraud detection is SafeTrade-only (not Cash). How do we distinguish SafeTrade vs Cash local transactions in the current schema? There is no `paymentMethod` column on `localTransaction`. The presence of a Stripe PaymentIntent (via `orderPayment.stripePaymentIntentId`) for the localTransaction's order is the indicator.

**Recommendation:** The detection function checks whether `orderPayment.stripePaymentIntentId` exists for the order. If no Stripe payment exists, it's a cash transaction and fraud detection is skipped.

### Decision D: Listing Status After Signal 1

**Question:** When Signal 1 fires (same listing sold via shipped order while local transaction is SCHEDULED), what should the listing status be? The shipped order completion normally marks it SOLD. The local transaction cancellation calls `unreserveListingForLocalTransaction()` which restores to ACTIVE. These conflict.

**Recommendation:** Let the shipped order mark it SOLD first (that's the legitimate sale). The local transaction cancellation should NOT call unreserve in the fraud case — the listing is legitimately sold. The fraud detection function should skip the unreserve step and instead just cancel the local transaction + refund the local buyer.
