### CRITICAL BUG IN ORCHESTRATOR

Line 56 of `src/lib/db/seed.ts` calls `seedOffers(db)` but `seedOffers` is defined inline at line 101 as a regular function. This works because of JavaScript hoisting, BUT the function signature takes `ReturnType<typeof drizzle>` which is the Drizzle instance type from line 30 -- this is consistent.

The real issue: the `seedOffers` function inserts 3 platform settings for offer limits (keys `commerce.offer.maxPerBuyerPerListing`, `commerce.offer.maxPerBuyerPerSeller`, `commerce.offer.maxPerBuyerGlobal`). These may CONFLICT with the 104 settings inserted by `seedPlatform` if those keys overlap. The `seedOffers` uses `onConflictDoNothing()` so this is safe but potentially redundant. Check whether the `V32_PLATFORM_SETTINGS` array already contains these keys. If so, the `seedOffers` platform settings inserts are unnecessary.

## CONSTRAINTS -- WHAT NOT TO DO

1. **DO NOT seed tables for unbuilt features.** The "DO NOT SEED" list above is definitive. No affiliate tables, no local tables, no authentication tables, no market intelligence tables, no live session tables.
2. **DO NOT use `as any` or `@ts-ignore`.** Type every insert value correctly using Drizzle's inferred insert types.
3. **DO NOT use float/dollar values for money.** All monetary values as integer cents (e.g., `45000` not `450.00`).
4. **DO NOT hardcode fee rates.** Fee rates for promoted listings come from `platform_settings` conceptually, but seed data for `totalFeeCents` on promoted listings is static demo data, which is acceptable.
5. **DO NOT use `console.log` in production code.** The seed script is a dev tool and may use `console.log` for progress output (this is the established pattern).
6. **DO NOT create files over 300 lines.** Each seed file must be under 300 lines.
7. **DO NOT invent table columns.** Only use columns defined in the Drizzle schema files.
8. **DO NOT break referential integrity.** Every FK must point to a valid seed record:
   - `userId` references must use values from `USER_IDS` (seed-users.ts)
   - `listingId` references must use values from `LISTING_IDS` (seed-listings.ts)
   - `orderId` references must use values from the ORDER_IDS array in seed-orders.ts
   - `sellerProfileId` references must use `SELLER_IDS` (seed-users.ts)
   - `promotedListingId` references must use IDs defined in seed-promotions.ts
   - `storefrontId` references must use `STOREFRONT_SEED_IDS` (seed-storefronts.ts)
   - `resolvedByStaffId` must use `SEED_IDS.staffAdminId` (seed-system.ts)
9. **DO NOT change existing seed data.** Only add new data. Exception: fixing the orchestrator bug (Task 1) and adding storefront_page to seed-storefronts.ts.

## IMPLEMENTATION NOTES FOR THE INSTALLER

### Pattern to Follow

Every existing seed file follows this exact pattern. Follow it precisely:

```typescript
import type { PostgresJsDatabase } from 'drizzle-orm/postgres-js';
import { tableNameHere } from '../schema';
import { USER_IDS } from './seed-users';
// ... other imports

// Hardcoded IDs for idempotency
const SOME_IDS = {
  item1: 'seed-prefix-001',
  item2: 'seed-prefix-002',
};

export async function seedSomething(db: PostgresJsDatabase): Promise<void> {
  await db.insert(tableNameHere).values([
    { id: SOME_IDS.item1, /* ... */ },
    { id: SOME_IDS.item2, /* ... */ },
  ]).onConflictDoNothing();
}

// Export IDs for use in other seeders
export const EXPORTED_IDS = SOME_IDS;
```

Key conventions:
- **Type:** Use `PostgresJsDatabase` from `drizzle-orm/postgres-js` (not the raw `drizzle` return type).
- **IDs:** Hardcoded strings with `seed-` prefix for idempotency. Format: `seed-{abbreviation}-{number}`.
- **Conflict:** Always `.onConflictDoNothing()` for idempotent re-runs.
- **Exports:** Export ID constants so downstream seeders can reference them.
- **Date helpers:** Use `daysAgo(n)` pattern established in seed-engagement.ts.
