### 2. Update Location Action

**File:** `src/lib/actions/admin-meetup-locations.ts` (MODIFY)

Add a new action for updating existing locations:

**Action: `updateMeetupLocationAction`**
- Auth: `staffAuthorize()` with CASL check `can('update', 'Setting')`
- Input (Zod `.strict()`):
  ```typescript
  const updateLocationSchema = z.object({
    locationId: z.string().min(1),
    name: z.string().min(1).max(200).optional(),
    address: z.string().min(1).max(500).optional(),
    city: z.string().min(1).max(100).optional(),
    state: z.string().min(1).max(50).optional(),
    zip: z.string().min(1).max(20).optional(),
    latitude: z.number().min(-90).max(90).optional(),
    longitude: z.number().min(-180).max(180).optional(),
    type: z.string().min(1).max(50).optional(),
    verifiedSafe: z.boolean().optional(),
    operatingHoursJson: z.unknown().optional(),
  }).strict();
  ```
- Logic: Update only the provided fields (explicit field mapping, no spread)
- Audit: Create audit event with action='UPDATE_MEETUP_LOCATION'
- Returns: `{ success: true }` or `{ error: string }`

## CONSTRAINTS -- WHAT NOT TO DO

- Do NOT create a separate `/cfg/safe-spots` route -- per Page Registry, meetup locations are a TAB on `/cfg` (tab key: `meetup-locations`, entry 114k)
- Do NOT add community suggestion/review workflow -- Local Canonical Section 6 mentions "Community can suggest -> staff reviews and approves" but this is a Phase G polish feature. For now, admin creates all locations manually.
- The admin actions use `staffAuthorize()` (hub staff auth), NOT `authorize()` (marketplace auth)
- CASL subject for admin actions is `Setting` (same as existing meetup actions) -- the SafeMeetupLocation CASL subject is for marketplace user access, the admin manages them under the `Setting` subject
- Do NOT modify the hub sidebar -- the `/cfg` page is already in the hub sidebar
- Keep all files under 300 lines
- Use existing hub component patterns (DataTable, Card, Badge, etc.)
