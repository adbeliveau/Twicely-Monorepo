## 2. SCOPE -- EXACTLY WHAT TO BUILD

This step builds the **framework layer only** — no external API calls, no BullMQ queues, no UI pages. It creates the TypeScript interfaces, connector registry, CASL integration, delegation scopes, Zod validations, platform settings, seed data, and shared types that F1 through F6 will consume. Think of this as the "contract definition" step.

### 2.8 Connector Registry (Runtime)

Create `src/lib/crosslister/connector-registry.ts` — runtime registry where concrete connector implementations register themselves:

```typescript
import type { PlatformConnector } from './connector-interface';
import type { ExternalChannel } from './types';

// Map of channel -> connector instance
const connectors = new Map<ExternalChannel, PlatformConnector>();

export function registerConnector(connector: PlatformConnector): void {
  connectors.set(connector.channel, connector);
}

export function getConnector(channel: ExternalChannel): PlatformConnector {
  const connector = connectors.get(channel);
  if (!connector) {
    throw new Error(`No connector registered for channel: ${channel}`);
  }
  return connector;
}

export function hasConnector(channel: ExternalChannel): boolean {
  return connectors.has(channel);
}

export function getRegisteredChannels(): ExternalChannel[] {
  return Array.from(connectors.keys());
}
```

This is a simple in-memory registry. F1 (eBay Import) will call `registerConnector(ebayConnector)` when the eBay connector is built. The registry pattern allows connectors to be added incrementally without modifying framework code.

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- NO `SellerTier` or `SubscriptionTier` — use `StoreTier` or `ListerTier`
- NO `FVF` or `Final Value Fee` — use `TF` or `Transaction Fee`
- NO `BASIC`, `ELITE`, `PLUS`, `MAX`, `PREMIUM` as tier names
- NO `wallet`, `withdraw`, `Twicely Balance` in any UI-facing strings
- NO `Redis` references — Valkey + BullMQ only (but do NOT build BullMQ queues in this step)

### Tech Stack
- NO BullMQ queue creation in this step (queues are built in F1/F3)
- NO external API calls (connector implementations are built in F1/F2)
- NO Centrifugo setup (real-time events are wired in F1)
- NO UI pages or components (pages are built in F1.3)
- NO Prisma, NextAuth, Zustand, tRPC, or other banned tech

### Code Rules
- All monetary values as integer cents (`priceCents`, not `price`)
- No `as any`, `as unknown as T`, `@ts-ignore`, `@ts-expect-error`
- Max 300 lines per file — split if needed
- Explicit field mapping only — never spread request body
- Ownership always via `userId` or `sellerId` (which equals `userId`)
- All Zod schemas use `.strict()`

### Architecture Rules
- Connectors are registered at runtime, NOT imported statically. The registry pattern allows incremental connector addition.
- Channel metadata is STATIC (compile-time). It does not read from the database.
- Platform settings for feature flags and rate limits are READ from `platform_settings` table at runtime. The channel registry has `enabled: boolean` as a compile-time DEFAULT but the actions layer should always check platform_settings.
- The `connectorTierEnum` from the Lister Canonical Section 5.7 is NOT in the actual codebase enums.ts. Do NOT add it as a Drizzle pgEnum — it is a TypeScript-only type (`ConnectorTier = 'A' | 'B' | 'C'`). The tier is metadata on the connector, not a database column.

### Business Logic Rules
- Imports are ALWAYS free, go ACTIVE immediately, exempt from insertion fees (Decision #16)
- Crosslister does NOT require Store subscription (Decision #15)
- Crosslister does NOT require BUSINESS status (CLAUDE.md subscription axes)
- Crosslister works without a store (Lister Canonical Section 1)
- No per-push credits — subscription-based (Lister Canonical Section 1)
- Syncs, relists, and delists NEVER count as publishes (Lister Canonical Section 7.1)

## 9. NOTES FOR INSTALLER

### Why This Step Exists Before F1

The crosslister connector framework is infrastructure that ALL Phase F steps depend on. F1 (eBay Import) needs the connector interface, channel registry, CASL subjects, and delegation scopes. F2 (Poshmark/Mercari Import) needs the same. F3 (Crosslist Outbound) needs TransformedListing types and the connector registry. Building this framework separately means:
1. F1 can focus purely on eBay-specific logic without defining shared types
2. Multiple F-phase steps can be built in parallel because they share a stable interface
3. CASL subjects and delegation scopes are wired once, tested once, not duplicated

### Connector Implementation is NOT Part of This Step

The `PlatformConnector` interface is defined here. The eBay implementation of that interface is built in F1. The Poshmark implementation is built in F2. This step only defines the contract, not the concrete implementations.

### No BullMQ Queues

The BullMQ queue architecture (Lister Canonical Section 4.3) is built in F1/F3 when actual jobs need to be dispatched. This step does not create any queues.

### Schema Is Already Complete

All 9 crosslister tables already exist in `src/lib/db/schema/crosslister.ts`. This step does NOT modify the schema. It builds the TypeScript layer on top of the existing schema.

### NOT SPECIFIED -- Owner Decisions Needed

1. **Connector encryption at rest:** Lister Canonical Section 25.1 specifies AES-256-GCM for tokens. The actual encryption implementation (which library, key management) is not specified. This framework step stores the interface contract; the encryption implementation should be decided before F1.1 (eBay OAuth).

2. **Crosslister delegation scopes not in Actors Canonical:** The Actors Canonical Section 3.4 delegation scopes table does NOT include crosslister scopes. The Lister Canonical Section 22.3 defines them (`crosslister:read`, `crosslister:publish`, `crosslister:import`, `crosslister:manage`). This prompt follows the Lister Canonical (it's the authority for crosslister). But the Actors Canonical delegation scope table should be updated to include these four scopes for consistency. The colon-separated format in Lister Canonical (`crosslister:read`) is normalized to dot-separated format (`crosslister.read`) to match existing scope conventions in the codebase (e.g., `dashboard.view`, `listings.manage`).

3. **Kill-switch implementation:** Lister Canonical Section 23.1 defines per-platform, per-seller, and global kill-switches. The kill-switch mechanism is not built in this step. Placeholder platform settings keys are seeded, but the actual enforcement logic belongs in F3 (Crosslist Outbound) when the scheduler is built.
