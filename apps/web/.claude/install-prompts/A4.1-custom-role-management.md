#### 2a. `createCustomRoleAction(input: unknown)`

CASL check: `ability.can('manage', 'CustomRole')`

Additional check: `session.platformRoles.includes('SUPER_ADMIN')` -- Only SUPER_ADMIN can manage custom roles per Section 4.3.5. The CASL check alone is not sufficient because `definePlatformAdminAbilities` grants `can('manage', 'all')` to both ADMIN and SUPER_ADMIN. We need the explicit role check.

Zod schema:
```typescript
export const createCustomRoleSchema = z.object({
  name: z.string().min(3).max(50).regex(/^[a-zA-Z0-9 ]+$/, 'Name must be alphanumeric with spaces only'),
  description: z.string().max(500).optional(),
  permissions: z.array(z.object({
    subject: z.string().min(1),
    action: z.string().min(1),
  }).strict()).default([]),
}).strict();
```

Business rules:
- **Max 20 custom roles.** Count active (`isActive = true`) custom roles. If >= 20, return `{ error: 'Maximum 20 custom roles allowed' }`.
- **Auto-generate code from name:** `name.trim().toUpperCase().replace(/\s+/g, '_')`. Example: "Returns Specialist" becomes `RETURNS_SPECIALIST`.
- **Unique name check.** Query `customRole` WHERE `name = input.name`. If exists, return `{ error: 'A role with this name already exists' }`.
- **Unique code check.** Query `customRole` WHERE `code = generatedCode`. If exists, return `{ error: 'A role with this code already exists' }`.
- **Validate permissions against registry.** Import `validatePermissions` from the permission registry. If any permission pair is not in the registry, return `{ error: 'Invalid permission: {subject}.{action}' }`.
- INSERT into `customRole` with explicit field mapping: `{ name, code, description, permissionsJson: permissions, createdByStaffId: session.staffUserId }`.
- INSERT `auditEvent` with: `actorType='STAFF'`, `actorId=session.staffUserId`, `action='CREATE_CUSTOM_ROLE'`, `subject='CustomRole'`, `subjectId=newRoleId`, `severity='CRITICAL'`, `detailsJson={ name, code, permissionCount: permissions.length }`.
- `revalidatePath('/roles')`.
- Return `{ success: true, customRoleId: newRoleId }`.

### 5. Wire Custom Roles into CASL Ability Factory

This is the critical integration step from Actors Canonical Section 4.3.3.

#### 5a. Update `staffAuthorize()` to load custom roles

**File:** `src/lib/casl/staff-authorize.ts` (MODIFY)

Currently `staffAuthorize()` builds a `CaslSession` from the staff session but does NOT include custom roles. Update the flow:

1. After loading the staff session (which gives `staffSession.roles` for system roles), also load custom role permissions for this staff user.
2. Query: SELECT `customRole.permissionsJson` FROM `staffUserCustomRole` JOIN `customRole` ON `staffUserCustomRole.customRoleId = customRole.id` WHERE `staffUserCustomRole.staffUserId = staffSession.staffUserId` AND `staffUserCustomRole.revokedAt IS NULL` AND `customRole.isActive = true`.
3. Parse each `permissionsJson` into `Array<{ subject: string; action: string }>`.
4. Pass the combined custom role permissions to the ability factory.

IMPORTANT: The current `CaslSession` type in `types.ts` does NOT have a field for custom role permissions. You have two options:

**Option A (Preferred):** Do NOT modify the `CaslSession` type. Instead, apply custom role permissions directly in `staffAuthorize()` before calling `defineAbilitiesFor()`. After building the ability via `defineAbilitiesFor(caslSession)`, apply additional `can()` calls for each custom role permission. However, since `defineAbilitiesFor` returns a built ability (not a builder), you cannot add to it after building.

**Option B (Required):** Apply custom role permissions inside the ability factory. Add an optional `customRolePermissions` field to the `CaslSession` type:

```typescript
// In types.ts, add to CaslSession:
customRolePermissions?: Array<{ subject: string; action: string }>;
```

Then in `ability.ts`, after the platform staff block that calls `definePlatformAdminAbilities` or `definePlatformAgentAbilities`, apply custom role permissions:

```typescript
// After system role abilities are applied, add custom role permissions
if (session.customRolePermissions?.length) {
  for (const perm of session.customRolePermissions) {
    can(perm.action, perm.subject);
  }
}
```

Use Option B. It keeps all ability definitions in the factory where they belong.

import { db } from '@/lib/db';
import { staffUserCustomRole, customRole } from '@/lib/db/schema';
import { eq, and, isNull } from 'drizzle-orm';

async function loadCustomRolePermissions(
  staffUserId: string
): Promise<Array<{ subject: string; action: string }>> {
  const rows = await db
    .select({ permissionsJson: customRole.permissionsJson })
    .from(staffUserCustomRole)
    .innerJoin(customRole, eq(staffUserCustomRole.customRoleId, customRole.id))
    .where(
      and(
        eq(staffUserCustomRole.staffUserId, staffUserId),
        isNull(staffUserCustomRole.revokedAt),
        eq(customRole.isActive, true)
      )
    );

  const allPermissions: Array<{ subject: string; action: string }> = [];
  for (const row of rows) {
    const perms = row.permissionsJson as Array<{ subject: string; action: string }>;
    if (Array.isArray(perms)) {
      allPermissions.push(...perms);
    }
  }
  return allPermissions;
}
