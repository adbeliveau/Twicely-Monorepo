## SCOPE -- EXACTLY WHAT TO BUILD

### 1. Escrow Release on Receipt Confirmation

**Modify:** `src/lib/actions/local-transaction.ts`

When `confirmReceiptAction` or `confirmReceiptOfflineAction` completes successfully:

1. Update `localTransaction.status` to `RECEIPT_CONFIRMED` (not immediately COMPLETED -- escrow hold still applies)
2. Update `order.status` to `DELIVERED` (receipt confirmation = delivery equivalent for local)
3. Create ledger entries:
   - `LOCAL_TRANSACTION_FEE` -- the 5% TF charged to seller (read rate from `commerce.local.tfRateBps`)
   - `ORDER_PAYMENT_CAPTURED` -- the full payment amount
   - `ORDER_STRIPE_PROCESSING_FEE` -- Stripe processing fee (pass-through)
4. After escrow hold period expires (per `commerce.escrow.holdHours` = 72 hours), mark as COMPLETED
   - NOTE: The actual timed release is handled by a BullMQ job (see step 3 below). The action just records RECEIPT_CONFIRMED and enqueues the release job.

**IMPORTANT Business Rules (from Pricing Canonical Section 2.2):**
- Local sales use 5% flat fee, NOT progressive brackets
- Local sales do NOT count toward seller's monthly GMV for bracket progression
- No minimum TF for local (unlike $0.50 minimum for shipped orders)
- Stripe fee shown as separate line item

### 2. Refine State Machine

The local transaction status flow is:
```
SCHEDULED
  -> SELLER_CHECKED_IN (seller checks in first)
  -> BUYER_CHECKED_IN (buyer checks in first)
  -> BOTH_CHECKED_IN (both checked in)
  -> RECEIPT_CONFIRMED (buyer scans QR / enters code)
  -> COMPLETED (after escrow hold expires, escrow released)

SCHEDULED -> CANCELED (by either party before meetup, or auto-cancel at 48h)
SCHEDULED -> NO_SHOW (one party checks in, other doesn't within 30 min)
RECEIPT_CONFIRMED -> DISPUTED (buyer files local claim within 7 days)
```

**File:** `src/lib/commerce/local-state-machine.ts` (NEW)

Create a state machine validator:

```typescript
const VALID_TRANSITIONS: Record<string, string[]> = {
  SCHEDULED: ['SELLER_CHECKED_IN', 'BUYER_CHECKED_IN', 'BOTH_CHECKED_IN', 'CANCELED'],
  SELLER_CHECKED_IN: ['BOTH_CHECKED_IN', 'RECEIPT_CONFIRMED', 'NO_SHOW', 'CANCELED'],
  BUYER_CHECKED_IN: ['BOTH_CHECKED_IN', 'RECEIPT_CONFIRMED', 'NO_SHOW', 'CANCELED'],
  BOTH_CHECKED_IN: ['RECEIPT_CONFIRMED', 'NO_SHOW', 'CANCELED'],
  RECEIPT_CONFIRMED: ['COMPLETED', 'DISPUTED'],
  COMPLETED: [], // terminal
  CANCELED: [], // terminal
  NO_SHOW: [], // terminal
  DISPUTED: ['COMPLETED', 'CANCELED'], // resolved by admin
};

export function canTransition(from: string, to: string): boolean {
  return VALID_TRANSITIONS[from]?.includes(to) ?? false;
}
```

Update all server actions to validate transitions via `canTransition()` before updating status.

### 3. BullMQ Job for Escrow Release

**File:** `src/lib/jobs/local-escrow-release.ts` (NEW)

A BullMQ job that fires after the escrow hold period (72 hours from RECEIPT_CONFIRMED):

- Job name: `local-escrow-release`
- Enqueued when: `confirmReceiptAction` sets status to RECEIPT_CONFIRMED
- Delay: reads `commerce.escrow.holdHours` from platform settings (default: 72 hours), converts to ms
- Job data: `{ localTransactionId: string, orderId: string }`
- Processing logic:
  1. Fetch localTransaction, verify still RECEIPT_CONFIRMED (not disputed)
  2. If still RECEIPT_CONFIRMED:
     - Update localTransaction.status to COMPLETED
     - Update order.status to COMPLETED
     - NOTE: Actual Stripe payout happens via the existing payout system (seller balance becomes available). Do NOT call Stripe directly here.
  3. If DISPUTED: skip (dispute resolution handles completion)

**Enqueue helper:**
```typescript
export async function enqueueLocalEscrowRelease(
  localTransactionId: string,
  orderId: string,
): Promise<void> {
  const holdHours = await getPlatformSetting<number>('commerce.escrow.holdHours', 72);
  const delayMs = holdHours * 60 * 60 * 1000;

  const queue = createQueue('local-escrow-release');
  await queue.add('release', { localTransactionId, orderId }, { delay: delayMs });
}
```

Use the existing BullMQ factory at `src/lib/jobs/queue.ts` (createQueue, createWorker).

### 5. QR Scanner Component

**File:** `src/components/local/qr-scanner.tsx` (NEW)

Client component that uses the device camera to scan QR codes:

- Uses the `html5-qrcode` library (or similar lightweight scanner)
- On successful scan: extracts the code string, calls `confirmReceiptAction` with `method: 'QR'`
- Shows scanning viewfinder with instructions: "Point your camera at the seller's QR code"
- Error states: camera permission denied, invalid code, expired code
- Fallback link: "Can't scan? Enter the code manually" -> switches to manual code entry

**npm dependency:** Check if `html5-qrcode` or equivalent is already installed. If not, install it.

### 6. Offline Confirmation UX

**File:** `src/components/local/offline-confirm.tsx` (NEW)

When the buyer has no connectivity:
- Shows the 6-digit offline code entry form
- Stores confirmation locally using localStorage:
  ```
  key: `twicely_offline_confirm_${localTransactionId}`
  value: { offlineCode, timestamp: new Date().toISOString() }
  ```
- Shows banner: "You're offline. Your confirmation will sync when you reconnect."
- On reconnect (detect via `navigator.onLine` event): calls `confirmReceiptOfflineAction` with stored data
- Clears localStorage after successful sync
- If sync doesn't happen within 2 hours (per `commerce.local.offlineGraceHours`): show persistent notification "Please connect to the internet to complete your transaction"

## CONSTRAINTS -- WHAT NOT TO DO

- Do NOT implement actual Stripe payment capture in this step. The existing order/payment pipeline handles Stripe. Local transactions follow the same "payment captured at checkout" model -- the 5% TF is calculated at checkout time. What this step does is create the ledger entries and mark the order COMPLETED for payout.
- Do NOT modify the existing ledger entry insertion helpers -- call them
- Do NOT implement no-show detection -- that is G2.5
- Do NOT implement safety timer -- that is G2.6
- BullMQ queue name: use hyphens not colons (`local-escrow-release`, NOT `local:escrow:release`)
- Worker runs in-process (same as existing BullMQ workers)
- Keep all files under 300 lines

## ACCEPTANCE CRITERIA

1. Receipt confirmation (QR or manual) sets localTransaction to RECEIPT_CONFIRMED and order to DELIVERED
2. Ledger entries created: ORDER_PAYMENT_CAPTURED, LOCAL_TRANSACTION_FEE, ORDER_STRIPE_PROCESSING_FEE
3. LOCAL_TRANSACTION_FEE uses 5% rate from `commerce.local.tfRateBps` platform setting
4. Local sale does NOT increment seller's monthly GMV counter
5. BullMQ job enqueued with correct delay (72h default from `commerce.escrow.holdHours`)
6. BullMQ job marks RECEIPT_CONFIRMED -> COMPLETED after delay, unless disputed
7. State machine validates all transitions (invalid transitions rejected with error)
8. QR scanner reads code from camera and triggers confirmation
9. Offline confirmation stores locally and syncs on reconnect
10. 2-hour offline grace period warning shown
11. No banned terms in any new/modified files
