# [G1.6] Referral Link Handler (`/ref/{code}`)

**One-line Summary:** Build the public `/ref/{code}` route that records a referral click, sets an attribution cookie with the affiliate's configured duration, and redirects the visitor to `/auth/signup`. If the affiliate is invalid or inactive, redirect to the homepage instead.

## 2. SCOPE -- EXACTLY WHAT TO BUILD

### 2.1 Database

No schema changes. The `referral` table already has all required columns.

**Referral table columns used by this step** (from `affiliates.ts` lines 65-89):
- `id` -- text PK, CUID2 auto-generated
- `affiliateId` -- text, NOT NULL, FK to affiliate.id
- `referredUserId` -- text, nullable, FK to user.id (NULL at click time -- set later on signup)
- `status` -- referralStatusEnum, NOT NULL, default 'CLICKED'
- `clickedAt` -- timestamp with timezone, NOT NULL, defaultNow()
- `expiresAt` -- timestamp with timezone, NOT NULL (computed: now + affiliate.cookieDurationDays)
- `ipAddress` -- text, nullable (from request headers, for fraud detection)
- `userAgent` -- text, nullable (from request headers, for fraud detection)
- `utmSource` -- text, nullable (from query params)
- `utmMedium` -- text, nullable (from query params)
- `utmCampaign` -- text, nullable (from query params)
- `promoCodeId` -- text, nullable, FK to promoCode.id (NULL for link-only referrals -- set when promo code is applied)

**Affiliate table columns read by this step** (from `affiliates.ts` lines 14-40):
- `id` -- PK, referenced as `affiliateId` on referral
- `status` -- affiliateStatusEnum: must be 'ACTIVE' to accept clicks
- `tier` -- affiliateTierEnum: 'COMMUNITY' or 'INFLUENCER' (determines cookie duration lookup)
- `cookieDurationDays` -- integer, stored on the affiliate record itself (was set from platform settings at join time)
- `referralCode` -- text, unique, used for lookup

### 2.2 Route Handler

**Route:** `src/app/ref/[code]/route.ts` -- Next.js App Router dynamic route handler

This is a **GET** route handler (not a page component). It:
1. Extracts the `{code}` parameter from the URL
2. Looks up the affiliate by `referralCode` (case-insensitive -- uppercase the input before querying, since codes are stored uppercased)
3. Validates the affiliate exists and has `status === 'ACTIVE'`
4. Checks the `affiliate.enabled` platform setting (master switch)
5. Reads the cookie duration from the affiliate record's `cookieDurationDays` field (NOT from platform settings -- the affiliate-specific value was already set from settings at join time, and may differ for influencers with custom terms)
6. Creates a `referral` record with status CLICKED
7. Sets the `twicely_ref` cookie with the referral ID
8. Extracts UTM parameters from query string (`?utm_source=...&utm_medium=...&utm_campaign=...`)
9. Redirects to `/auth/signup`

**Why route handler, not page:** This is a redirect endpoint. No UI is rendered. A route handler (`route.ts`) is the correct Next.js pattern for this.

### 2.3 Attribution Cookie

**Cookie name:** `twicely_ref`

**Cookie value:** The `referral.id` (CUID2). NOT the affiliate ID, NOT the referral code. Storing the referral ID lets us look up the exact click event later during signup attribution.

**Cookie settings:**
- `httpOnly: true` -- not accessible from client JS (security)
- `secure: process.env.NODE_ENV === 'production'` -- matches existing pattern in `staff-login.ts`
- `sameSite: 'lax'` -- allows the cookie to be sent on top-level navigations from external sites (required for referral links shared on social media)
- `path: '/'` -- available across the entire site
- `maxAge: cookieDurationDays * 24 * 60 * 60` -- in seconds, derived from the affiliate's `cookieDurationDays`

### 2.4 First-Touch Attribution Logic

Per Affiliate Canonical Section 2.3:
- **First-touch attribution**: The FIRST affiliate link a user clicks gets the credit. No last-touch override.
- **Multiple affiliates**: First click wins. If user clicks Affiliate A's link, then Affiliate B's link 5 days later, Affiliate A gets credit.

**Implementation:** Before setting the cookie, check if a `twicely_ref` cookie already exists. If it does, the route handler should:
1. Still create a new `referral` record (for click tracking analytics -- every click is recorded)
2. NOT overwrite the existing cookie (first-touch wins)
3. Still redirect to `/auth/signup`

This means the route handler does NOT check for cookie existence before inserting a referral row (clicks are always tracked), but it DOES check before setting the cookie (attribution is first-touch only).

### 2.5 Fraud Detection Data Capture

Per Affiliate Canonical Section 2.9, the referral record captures:
- `ipAddress`: Read from `request.headers.get('x-forwarded-for')` (first IP in chain if comma-separated) or `request.headers.get('x-real-ip')` or `null`
- `userAgent`: Read from `request.headers.get('user-agent')` or `null`

These are stored for later anti-fraud analysis (G3.5). No fraud checks are performed at click time in G1.6 -- that is deferred to G3.5.

### 2.6 Error/Edge Cases

| Scenario | Behavior |
|----------|----------|
| Code does not match any affiliate | Redirect to `/` (homepage). No cookie, no referral record. |
| Affiliate exists but `status !== 'ACTIVE'` (PENDING, SUSPENDED, BANNED) | Redirect to `/` (homepage). No cookie, no referral record. |
| `affiliate.enabled` platform setting is `false` | Redirect to `/` (homepage). No cookie, no referral record. |
| User already has `twicely_ref` cookie | Create referral record (click tracking). Do NOT overwrite cookie. Redirect to `/auth/signup`. |
| Code contains invalid characters | The route handler will find no matching affiliate. Redirect to `/`. |
| User is already authenticated | Still create referral record, still set cookie (if no existing one). Redirect to `/auth/signup` (signup page redirects authenticated users to `/my` per proxy.ts). Note: Affiliate Canonical Section 2.3 says "Existing users -- No commission. Only NEW account signups qualify." This is enforced at signup attribution time (future step), not at click time. |

### 2.7 Signup Page Integration

The existing signup page (`src/app/auth/signup/page.tsx`) is a `'use client'` component that calls `signUp.email()` from Better Auth. The referral attribution (reading the `twicely_ref` cookie and updating the referral record with the new user ID) happens at signup time, NOT during the click.

**For G1.6, we add a cookie reader utility** that will be consumed later when the signup attribution step is built. Specifically:

Create a server-side utility `src/lib/affiliate/referral-cookie.ts` that exports:
- `REFERRAL_COOKIE_NAME = 'twicely_ref'` -- constant for the cookie name
- `getReferralIdFromCookie(): Promise<string | null>` -- reads the `twicely_ref` cookie using `cookies()` from `next/headers` and returns the referral ID or null
- `clearReferralCookie(): Promise<void>` -- clears the cookie (for use after attribution is complete)

This utility is NOT wired into the signup flow in G1.6. The signup attribution step is a separate task that will:
1. Read the cookie via `getReferralIdFromCookie()`
2. Look up the referral record
3. Set `referral.referredUserId` and `referral.signedUpAt`
4. Set `user.referredByAffiliateId`
5. Clear the cookie via `clearReferralCookie()`

That wiring is NOT part of G1.6. G1.6 only handles the click -> cookie -> redirect flow.

## 4. ACCEPTANCE CRITERIA

### Positive Cases
1. Visiting `twicely.co/ref/SELLER1CODE` (where SELLER1CODE is a valid, ACTIVE affiliate's referralCode) creates a `referral` record with status CLICKED and the correct `affiliateId`.
2. The `twicely_ref` cookie is set with the referral record's ID, httpOnly, correct maxAge (30 days for COMMUNITY, 60 days for INFLUENCER).
3. The visitor is redirected to `/auth/signup` with HTTP 307 (temporary redirect).
4. UTM parameters (`?utm_source=twitter&utm_medium=social&utm_campaign=spring26`) are stored on the referral record.
5. The `expiresAt` timestamp on the referral record equals `now + affiliate.cookieDurationDays` in days.
6. The `ipAddress` and `userAgent` fields are populated from request headers.
7. Visiting with a case-insensitive code (e.g., `seller1code`) still matches the affiliate (code is uppercased before lookup).

### Negative Cases
8. Visiting `/ref/NONEXISTENT` (no matching affiliate) redirects to `/` (homepage). No referral record created. No cookie set.
9. Visiting `/ref/SUSPENDEDCODE` (affiliate with status SUSPENDED) redirects to `/`. No referral record. No cookie.
10. When `affiliate.enabled` platform setting is `false`, ALL referral links redirect to `/`. No referral record. No cookie.
11. If the visitor already has a `twicely_ref` cookie, a new referral CLICKED record IS created (for analytics), but the cookie is NOT overwritten (first-touch attribution preserved).

### Data Integrity
12. All referral records have a non-null `affiliateId` that references a valid affiliate.
13. All referral records have `status = 'CLICKED'` at creation time.
14. The `clickedAt` timestamp is set to `defaultNow()` (database default).
15. The `expiresAt` is computed correctly from the affiliate's `cookieDurationDays` field.
16. No `referredUserId` is set at click time (it remains NULL until signup).

### Security
17. The cookie is `httpOnly: true` (not readable by client-side JavaScript).
18. The cookie is `secure: true` in production.
19. The cookie `sameSite` is `lax` (allows cross-site navigation but blocks cross-site POST).
20. No internal IDs are exposed in error messages or redirects.

## 7. IMPLEMENTATION DETAILS

### 7.1 Route Handler: `src/app/ref/[code]/route.ts`

```
Purpose: Handle GET /ref/{code}
Auth: PUBLIC (no auth required)
Input: URL parameter `code`, optional query params `utm_source`, `utm_medium`, `utm_campaign`
Output: HTTP 307 redirect to /auth/signup (success) or / (failure)
Side effects: Inserts referral record, sets twicely_ref cookie
```

**Step-by-step logic:**

1. Extract `code` from `params.code` (Next.js App Router dynamic segments).
2. Uppercase the code: `const normalizedCode = code.toUpperCase()`.
3. Read `affiliate.enabled` via `getPlatformSetting('affiliate.enabled', true)`. If false, redirect to `/`.
4. Call `getAffiliateByReferralCode(normalizedCode)`. If null, redirect to `/`.
5. Check `affiliate.status === 'ACTIVE'`. If not, redirect to `/`.
6. Compute `expiresAt`: `new Date(Date.now() + affiliate.cookieDurationDays * 24 * 60 * 60 * 1000)`.
7. Extract fraud data:
   - `ipAddress`: `request.headers.get('x-forwarded-for')?.split(',')[0]?.trim() ?? request.headers.get('x-real-ip') ?? null`
   - `userAgent`: `request.headers.get('user-agent') ?? null`
8. Extract UTM params from `request.nextUrl.searchParams`:
   - `utmSource`: `searchParams.get('utm_source') || null`
   - `utmMedium`: `searchParams.get('utm_medium') || null`
   - `utmCampaign`: `searchParams.get('utm_campaign') || null`
9. Insert referral record with explicit field mapping:
   ```
   const [newReferral] = await db.insert(referral).values({
     affiliateId: aff.id,
     status: 'CLICKED',
     expiresAt,
     ipAddress,
     userAgent,
     utmSource,
     utmMedium,
     utmCampaign,
   }).returning({ id: referral.id });
   ```
10. Check for existing `twicely_ref` cookie:
    ```
    const cookieStore = await cookies();
    const existingRef = cookieStore.get(REFERRAL_COOKIE_NAME);
    ```
11. If NO existing cookie, set the cookie:
    ```
    cookieStore.set(REFERRAL_COOKIE_NAME, newReferral.id, {
      httpOnly: true,
      secure: process.env.NODE_ENV === 'production',
      sameSite: 'lax',
      path: '/',
      maxAge: affiliate.cookieDurationDays * 24 * 60 * 60,
    });
    ```
12. Redirect to `/auth/signup`:
    ```
    return NextResponse.redirect(new URL('/auth/signup', request.url));
    ```

**Important:** Use `NextResponse.redirect()` with status 307 (default for GET redirects in Next.js route handlers). Do NOT use `redirect()` from `next/navigation` -- that throws an error and is for server components/actions, not route handlers.

### 7.2 Cookie Utility: `src/lib/affiliate/referral-cookie.ts`

This is a simple server-side utility (NOT marked `'use server'` -- it's imported by other server code, not called from clients):

```
import { cookies } from 'next/headers';

export const REFERRAL_COOKIE_NAME = 'twicely_ref';

export async function getReferralIdFromCookie(): Promise<string | null> {
  const cookieStore = await cookies();
  return cookieStore.get(REFERRAL_COOKIE_NAME)?.value ?? null;
}

export async function clearReferralCookie(): Promise<void> {
  const cookieStore = await cookies();
  cookieStore.delete(REFERRAL_COOKIE_NAME);
}
```

## 9. SPEC GAPS AND OPEN QUESTIONS

### Resolved
- **Cookie name:** Canonical does not specify the cookie name. Using `twicely_ref` (underscore-separated, matches platform's `twicely.` cookie prefix convention but avoids dots for cookie name simplicity). This is a reasonable implementation detail.
- **Redirect target:** Canonical Section 7.1 says "sets cookie, redirects to signup". Using `/auth/signup`.
- **HTTP status code:** Using 307 (temporary redirect) which is the Next.js default for `NextResponse.redirect()` on GET requests. Appropriate because the referral URL is stable and should be re-processable.

### NOT SPECIFIED -- Owner decisions potentially needed
- **Rate limiting on /ref/ endpoint:** Canonical Section 2.9 mentions "Bot traffic patterns (high clicks, zero signups) -- Throttle link, notify admin." However, the rate limiting and throttling implementation is explicitly deferred to G3.5 (Anti-fraud checks). G1.6 does NOT implement any rate limiting. If you want basic rate limiting at click time, flag it and we can add it.
- **Signup page query param:** Should the redirect to `/auth/signup` include a query param like `?ref=1` to visually indicate the user came from a referral link (e.g., to show a welcome message)? Canonical does not specify this. Current implementation does a clean redirect with no query params. The cookie carries the attribution.
