## 2. CRITICAL FINDING: DATA SOURCE INCONSISTENCY

### Problem

The canonical schema (`TWICELY_V3_SCHEMA_v2_0_7.md` section 2.5b) defines branding fields on the **`storefront`** table:
- `bannerUrl`, `logoUrl`, `accentColor`, `announcement`, `aboutHtml`, `socialLinksJson`, `featuredListingIds`, `defaultView`, `returnPolicy`, `shippingPolicy`, `isPublished`, `vacationMode`, `vacationMessage`

However, the current codebase stores these same fields on the **`sellerProfile`** table (`src/lib/db/schema/identity.ts`):
- `bannerUrl`, `logoUrl`, `accentColor`, `announcement`, `aboutHtml`, `socialLinks`, `featuredListingIds`, `isStorePublished`, `defaultStoreView`

The `storefront` table (`src/lib/db/schema/storefront.ts`) EXISTS in the schema with the correct canonical fields, but all existing queries (`storefront-public.ts`, `storefront-owner.ts`) and all existing actions (`storefront.ts`) read/write from `sellerProfile`, not `storefront`.

### Decision Required

**NOT SPECIFIED -- owner decision needed:** Should D1/D1.1 migrate the data source from `sellerProfile` to `storefront` table (matching the canonical schema), or should it continue using `sellerProfile` and defer migration? The canonical schema is the authority, but migration has risk with 1009+ tests and existing D1.2/D1.3 code.

**Recommendation for this prompt:** Proceed WITHOUT migration. Use the existing `sellerProfile`-based pattern for now. The storefront table is used only for `storefrontCustomCategory` (via foreign key) and `storefrontPage` (Puck). A separate migration prompt can be written later. This keeps D1/D1.1 focused on the user-visible deliverable.

**The installer should NOT migrate data sources unless the owner explicitly approves.** The rest of this prompt assumes the existing `sellerProfile`-based pattern continues.
