## 2. SCHEMA CHANGES NEEDED

### 2.1 Add `vacationAutoReplyMessage` to `sellerProfile` (identity.ts)

The spec (Feature Lock-in §30) defines TWO distinct messages:
- **Vacation message** (displayed on storefront banner) — `vacationMessage` column EXISTS
- **Auto-reply message** (sent as automatic response to new messages) — NOT in schema

**OWNER DECISION NEEDED:** The schema doc does NOT include a `vacationAutoReplyMessage` column. The Feature Lock-in clearly describes auto-reply as a separate customizable message with a different default text. Options:

- **Option A (Recommended):** Add `vacationAutoReplyMessage text('vacation_auto_reply_message')` to `sellerProfile`. This keeps the storefront banner message and auto-reply message independent as the spec intends.
- **Option B:** Reuse `vacationMessage` for both purposes. Simpler but loses spec-intended separation.

**This prompt assumes Option A is approved.** If Option B is chosen, remove the new column and use `vacationMessage` for auto-reply as well.

### 2.2 Migration Required

A Drizzle migration is needed to add the one new column:

```sql
ALTER TABLE seller_profile ADD COLUMN vacation_auto_reply_message text;
```

No other schema changes. All other columns already exist. No new enums needed (`vacationModeType` is stored as plain text, which matches the existing codebase pattern).

### 3.2 Zod Validation Schema (new file: `src/lib/validations/vacation.ts`)

```typescript
// activateVacationSchema
```

### 3.3 Server Actions (new file: `src/lib/actions/vacation.ts`)

**ACTION 1: `activateVacation(data)`**

Steps:
1. `authorize()` — get session, check `ability.can('update', sub('SellerProfile', { userId }))`
2. Parse input with `activateVacationSchema`
3. Load `vacation.maxPauseDays` and `vacation.maxAllowSalesDays` from `platform_settings`
4. Validate date range:
   - `endAt` must be in the future
   - `startAt` must be before `endAt`
   - Duration must not exceed max for the chosen mode type (30 days for PAUSE_SALES/CUSTOM, 15 days for ALLOW_SALES)
5. Check for unfulfilled orders (`order.sellerId = userId AND order.status = 'PAID'`). Return count in response as a warning (do NOT block activation — spec says "seller sees warning", not "seller is blocked")
6. Update `sellerProfile`: set `vacationMode=true`, `vacationModeType`, `vacationMessage`, `vacationStartAt`, `vacationEndAt`, auto-reply message, `updatedAt`
7. Sync `storefront.vacationMode=true` and `storefront.vacationMessage` for the same ownerUserId
8. If `vacation.autoDeclinePendingOffers` is true: batch-decline all PENDING offers for ALL of the seller's active listings. For each listing: call existing `declineAllPendingOffersForListing(listingId)`. Collect total declined count.
9. **Offer decline message**: Feature Lock-in says "Your offer has been declined. Seller is currently away. You can resubmit when they return on [date]." — pass decline message context. Since `declineAllPendingOffersForListing()` doesn't send notifications, fire notification for each declined offer via `notifyOfferEvent('declined', offerId)` (existing helper).
10. `revalidatePath('/my/selling/store')` and `revalidatePath('/st/[slug]')` if slug exists
11. Return `{ success: true, unfulfilledOrderCount: number, offersDeclined: number }`

**ACTION 2: `deactivateVacation()`**

Steps:
1. `authorize()` — same CASL check
2. Verify seller is actually on vacation (`vacationMode === true`)
3. Clear ALL vacation fields on `sellerProfile`: `vacationMode=false`, `vacationModeType=null`, `vacationMessage=null`, `vacationAutoReplyMessage=null`, `vacationStartAt=null`, `vacationEndAt=null`, `updatedAt=now`
4. Sync `storefront.vacationMode=false`, `storefront.vacationMessage=null`
5. `revalidatePath` both paths
6. Return `{ success: true }`

**ACTION 3: `adminForceDeactivateVacation(data)`**

Steps:
1. `staffAuthorize()` — must be ADMIN or SUPER_ADMIN scope (use `ability.can('manage', 'SellerProfile')` from platform-abilities)
2. Parse with `adminForceDeactivateVacationSchema`
3. Verify target seller exists and is on vacation
4. Clear all vacation fields (same as deactivateVacation)
5. Create `auditEvent` with action='admin.vacation.forceDeactivate', details including reason
6. Return `{ success: true }`

### 3.4 Query: Get Unfulfilled Order Count (new file: `src/lib/queries/vacation.ts`)

```typescript
export async function getUnfulfilledOrderCount(userId: string): Promise<number>
```

Count `order` rows where `sellerId = userId AND status = 'PAID'`. Used by the activation action and the vacation settings UI to show the warning.

Also:


Count `listingOffer` rows where `sellerId = userId AND status = 'PENDING'`. Shown in UI as "X pending offers will be auto-declined."

### 3.5 Messaging Auto-Reply (modify: `src/lib/actions/messaging-actions.ts`)

In `sendMessage()`, after inserting the new message, check if the OTHER participant is a seller on vacation:

1. After message insert succeeds, determine if the message sender is the buyer (sender !== conv.sellerId)
2. If sender is the buyer, look up the seller's vacation status
3. If `sellerProfile.vacationMode === true` AND `sellerProfile.vacationAutoReplyMessage` is not null:
   - Insert an auto-reply `message` row with:
     - `conversationId`: same conversation
     - `senderUserId`: the seller's userId (the auto-reply comes FROM the seller)
     - `body`: the seller's `vacationAutoReplyMessage`. If null, use default: `"Thanks for your message! I'm currently away until [return date]. I'll respond when I return."` (interpolate `vacationEndAt`)
     - `isAutoGenerated: true`
     - `attachments: []`
   - Update conversation `lastMessageAt` and `sellerUnreadCount` stays the same (it's the seller's auto-reply), `buyerUnreadCount` increments by 1

Also apply the same logic in `createConversation()` — if the first message triggers a conversation to a seller on vacation, insert the auto-reply immediately after.

**IMPORTANT:** Auto-reply should fire at MOST once per conversation while vacation is active. To prevent spam: check if the conversation already has an `isAutoGenerated = true` message from the seller within the current vacation period (since `vacationStartAt`). If yes, skip the auto-reply. This avoids flooding the buyer with the same auto-reply on every message.

## 4. CONSTRAINTS — WHAT NOT TO DO

### Banned Terms
- Do NOT use "wallet" anywhere — use "payout"
- Do NOT use "Twicely Balance" — use "Available for payout"
- Do NOT use "FVF" — use "TF" / "Transaction Fee"

### Gotchas
- **Listing status does NOT change during vacation.** Listings stay ACTIVE. The `checkListingAvailability()` function handles the vacation blocking at the purchase layer. PAUSE_SALES mode means hidden from search, not status=PAUSED. Do NOT change any listing statuses.
- **Storefront vacation state is a MIRROR of sellerProfile.** Always update both in the same action. The storefront `vacationMode`/`vacationMessage` exist for query convenience (the public storefront query joins storefront, not sellerProfile for these fields). On any vacation state change, sync both.
- **`vacationModeType` is plain TEXT, not an enum.** The schema already has this as `text('vacation_mode_type')`. Do NOT create a pgEnum for it.
- **Offers are blocked in ALL vacation modes** (Feature Lock-in §30). This is already handled by `offer-create.ts` checking `sellerProfile.vacationMode`. No changes needed to the offer creation path.
- **The vacation cron job already exists.** Do NOT create a new cron job. The existing `processVacationAutoEnd()` handles auto-ending expired vacations.
- **Ownership via userId.** All CASL checks use `sellerProfile.userId`, not `sellerProfile.id`.
- **`sellerProfile.id` is CUID2 PK, NOT userId.** When querying sellerProfile, always filter by `eq(sellerProfile.userId, userId)`, never by `eq(sellerProfile.id, userId)`.
- **Max 300 lines per file.** The vacation settings component may be complex — split into sub-components if needed.
- **Never hardcode fee rates or limits.** Max vacation durations come from `platform_settings`.
- **Zod `.strict()` on all schemas.**
- **Auto-reply is from the SELLER, not from "Twicely System".** The `senderUserId` is the seller's userId, and `isAutoGenerated: true` distinguishes it.

### Routes
- Vacation settings are part of existing `/my/selling/store` page — do NOT create a new route
- No new hub admin page for vacation in G3.7. Admin force-deactivate is an action only (can be triggered from existing `/usr/` user detail pages)

## 5. ACCEPTANCE CRITERIA

### Positive Cases
1. Seller can activate PAUSE_SALES vacation mode with message, start date, end date
2. Seller can activate ALLOW_SALES vacation mode (max 15 days enforced)
3. Seller can activate CUSTOM vacation mode (max 30 days enforced)
4. On activation, all PENDING offers across seller's listings are auto-declined (when `vacation.autoDeclinePendingOffers` is true)
5. On activation, seller sees warning about unfulfilled orders (count of PAID orders)
6. Storefront shows vacation banner with return date
7. Public storefront shows vacation warning (existing functionality, enhanced with return date and mode-specific messaging)
8. New buyer messages to a vacation seller receive auto-reply with vacation message
9. Auto-reply fires only ONCE per conversation during the vacation period
10. Seller can deactivate vacation early
11. Vacation auto-ends when `vacationEndAt` passes (existing cron, already works)
12. Admin can force-deactivate any seller's vacation mode with a reason
13. Both `sellerProfile` and `storefront` vacation fields stay in sync
14. `vacationModeType` stores one of: PAUSE_SALES, ALLOW_SALES, CUSTOM (as plain text)
15. Platform settings `vacation.maxPauseDays` (30) and `vacation.maxAllowSalesDays` (15) are enforced

### Negative Cases
16. Activation WITHOUT an end date is rejected (Zod validation)
17. Activation with duration exceeding max for the mode type is rejected
18. End date in the past is rejected
19. Start date after end date is rejected
20. Non-seller cannot activate vacation mode (CASL blocks)
21. Seller staff (delegated) CAN activate vacation on behalf of their delegating seller (if delegation has appropriate scope)
22. Unauthenticated users cannot activate/deactivate vacation
23. Auto-reply does NOT fire when the seller themselves send a message
24. Auto-reply does NOT fire more than once per conversation per vacation period
25. Admin force-deactivate creates an auditEvent

### Data Integrity
26. All date fields stored with timezone
27. `vacationModeType` is one of exactly three values or null
28. `vacationMessage` and `vacationAutoReplyMessage` max 500 characters (Zod enforced)
29. `vacationAutoReplyMessage` is nullable — if null, default text is used at runtime

### Vocabulary
30. No banned terms in any UI text or code comments
31. VacationBanner uses "on vacation" not "on holiday" or "away mode"
