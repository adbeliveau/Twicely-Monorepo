### CRITICAL: CASL Condition Mismatch (Must Address)

The CASL rules in `ability.ts` lines 230-236 use conditions like `{ affiliateId: userId }`, meaning they compare `affiliateId` to the session's `userId`. However, in the actual schema:

- `referral.affiliateId` references `affiliate.id` (a CUID2 PK), NOT `user.id`
- `promoCode.affiliateId` references `affiliate.id` (a CUID2 PK), NOT `user.id`
- `affiliateCommission.affiliateId` references `affiliate.id` (a CUID2 PK), NOT `user.id`

This means the CASL conditions as written will **never match** for referral/promoCode/commission subjects, because `affiliateId` (e.g., `seed-aff-001`) will never equal `userId` (e.g., `user-seller-1`).

**Resolution for G1.4:** Since the dashboard queries are all scoped by first fetching the affiliate record by `userId`, then querying child tables by `affiliate.id`, the installer should:
1. Use `authorize()` + `ability.can('read', sub('Affiliate', { userId: session.userId }))` for the top-level affiliate access check (this DOES work correctly since `affiliate.userId` matches).
2. For child table reads (referrals, commissions, payouts), the authorization is implicitly covered by the fact that we only query children of the affiliate record that belongs to the session user. No additional per-row CASL check is needed for read queries that are already scoped.
3. Do NOT attempt to fix the CASL conditions in this step. That is a broader fix that should be tracked separately.

#### `getAffiliateStats(affiliateId: string)`
Returns aggregated stats for the dashboard. Query:
- **This month clicks**: COUNT referrals WHERE `affiliateId` = param AND `clickedAt` >= start of current calendar month (UTC)
- **This month signups**: COUNT referrals WHERE `affiliateId` = param AND `signedUpAt` >= start of current month AND `signedUpAt` IS NOT NULL
- **This month conversions**: COUNT referrals WHERE `affiliateId` = param AND `convertedAt` >= start of current month AND `convertedAt` IS NOT NULL
- **All-time clicks**: COUNT referrals WHERE `affiliateId` = param
- **All-time signups**: COUNT referrals WHERE `affiliateId` = param AND `signedUpAt` IS NOT NULL
- **All-time conversions**: COUNT referrals WHERE `affiliateId` = param AND `convertedAt` IS NOT NULL
- **This month earnings**: SUM `affiliateCommission.commissionCents` WHERE `affiliateId` = param AND `createdAt` >= start of current month
- **All-time earnings**: SUM `affiliateCommission.commissionCents` WHERE `affiliateId` = param (this should match `affiliate.totalEarnedCents` but calculate live)

Return shape:
```typescript
interface AffiliateStats {
  thisMonth: {
    clicks: number;
    signups: number;
    conversions: number;
    earningsCents: number;
    conversionRate: number; // signups > 0 ? (conversions / signups) * 100 : 0
  };
  allTime: {
    clicks: number;
    signups: number;
    conversions: number;
    earningsCents: number;
    conversionRate: number;
  };
}
```

Use `Promise.all` to parallelize the two time-period queries (this month vs all-time). Each period can be a single query with conditional aggregation using Drizzle's `sql` template, or two separate queries — whichever stays cleaner under 300 lines.

#### `getAffiliateReferrals(affiliateId: string, options: { limit: number; offset: number })`
Returns paginated list of referred users for the "View Referred Users" section.

Query: SELECT from `referral` LEFT JOIN `user` ON `referral.referredUserId = user.id` WHERE `referral.affiliateId` = param, ORDER BY `referral.clickedAt` DESC.

Return shape per row:
```typescript
interface ReferralRow {
  id: string;
  status: string; // 'CLICKED' | 'SIGNED_UP' | 'TRIALING' | 'CONVERTED' | 'CHURNED'
  clickedAt: Date;
  signedUpAt: Date | null;
  convertedAt: Date | null;
  churnedAt: Date | null;
  // Referred user info (anonymized per Canonical Section 2.7: "username anonymized after 30 days")
  referredUsername: string | null; // Show actual username if signup < 30 days ago, else "User ***"
  referredSignupDate: Date | null;
}
```

Also return `total: number` (COUNT of all referrals for this affiliate) for pagination.

**Anonymization rule (Canonical Section 2.7):** Referred user list shows username — anonymized after 30 days from signup. Implement: if `signedUpAt` is more than 30 days ago, return `referredUsername` as `null` (the component will display "Anonymous user"). If `signedUpAt` is within 30 days or `signedUpAt` is null (not yet signed up), show the actual username.

#### `getAffiliateCommissions(affiliateId: string, options: { limit: number; offset: number })`
Returns paginated commission history.

Query: SELECT from `affiliateCommission` WHERE `affiliateId` = param, ORDER BY `createdAt` DESC.

Return shape per row:
```typescript
interface CommissionRow {
  id: string;
  subscriptionProduct: string;
  grossRevenueCents: number;
  netRevenueCents: number;
  commissionRateBps: number;
  commissionCents: number;
  status: string; // 'PENDING' | 'PAYABLE' | 'PAID' | 'REVERSED'
  holdExpiresAt: Date;
  createdAt: Date;
}
```

Also return `total: number` for pagination.

#### `getAffiliatePayouts(affiliateId: string, options: { limit: number; offset: number })`
Returns paginated payout history.

Query: SELECT from `affiliatePayout` WHERE `affiliateId` = param, ORDER BY `createdAt` DESC.

Return shape per row: the full `affiliatePayout` row.

Also return `total: number` for pagination.

## SPEC INCONSISTENCIES & OPEN QUESTIONS

### 1. Referral CASL Conditions (Known Issue)
The CASL conditions in `ability.ts` use `{ affiliateId: userId }` for Referral, PromoCode, AffiliateCommission, and AffiliatePayout subjects. But these tables' `affiliateId` column references `affiliate.id` (a CUID2 PK), not `user.id`. This means per-row CASL checks on child tables will always fail. **For G1.4, this is not a blocker** because the dashboard queries are scoped by first fetching the affiliate by `userId`, then querying children by `affiliate.id`.
