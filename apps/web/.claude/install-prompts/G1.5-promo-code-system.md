### 2.4 Server Actions

**File:** `src/lib/actions/promo-codes.ts` (NEW)

All actions use `'use server'`, Zod `.strict()` validation, `authorize()` for CASL, and explicit field mapping.

#### Action 1: `createAffiliatePromoCode(input: unknown)`

Creates a promo code for the calling seller's affiliate account.

Business rules:
- Caller must be a seller with an ACTIVE affiliate record
- CASL check: `ability.can('create', sub('PromoCode', { affiliateId: affiliate.id }))` -- but NOTE the CASL issue (affiliateId in ability.ts is userId, not affiliate.id). Work around this: look up the affiliate by userId, verify it exists and is ACTIVE, then proceed. Do NOT rely on CASL per-row check for create -- use the ability check `ability.can('create', 'PromoCode')` as subject-level, then manually verify affiliate ownership.
- Code must be unique across platform (query `getPromoCodeByCode`)
- Community affiliates: `discountValue` for PERCENTAGE type must be <= `affiliate.maxPromoDiscountBps` (read from platform settings, default 2000 = 20%). For FIXED type, no setting exists -- use 5000 cents ($50) as a hardcoded max. NOT SPECIFIED in canonical -- flag as: "Fixed discount max for community affiliates not specified in canonical. Using $50 hardcoded max. Owner decision needed."
- Influencer affiliates: `discountValue` for PERCENTAGE type must be <= `affiliate.maxInfluencerDiscountBps` (read from platform settings, default 5000 = 50%)
- Community affiliates can have max 3 months of duration per Canonical Section 2.4. Influencer max 6 months.
- Set `type: 'AFFILIATE'`, `affiliateId: affiliate.id`, `createdByUserId: session.userId`
- Sync to Stripe: create Stripe Coupon + PromotionCode (see Section 2.6)
- Write audit event: `PROMO_CODE_CREATED`
- Revalidate `/my/selling/affiliate`

#### Action 2: `updateAffiliatePromoCode(input: unknown)`

Updates an existing promo code owned by the calling affiliate.

Business rules:
- Only `isActive`, `usageLimit`, and `expiresAt` are mutable (code, discountType, discountValue are immutable after creation)
- CASL check: look up promo code, verify `promoCode.affiliateId === affiliate.id` where affiliate is looked up by `session.userId`
- If deactivating (`isActive: false`), deactivate the Stripe PromotionCode too
- Write audit event: `PROMO_CODE_UPDATED`
- Revalidate `/my/selling/affiliate`

#### Action 3: `deleteAffiliatePromoCode(id: string)`

Soft-deletes by setting `isActive: false`. No hard delete.

Business rules:
- Verify ownership via affiliate lookup
- Deactivate Stripe PromotionCode
- Write audit event: `PROMO_CODE_DEACTIVATED`
- Revalidate `/my/selling/affiliate`

#### Action 4: `createPlatformPromoCode(input: unknown)` (Admin/Staff)

Creates a platform promo code (no affiliate attribution).

Business rules:
- Auth: use `staffAuthorize()` (not regular `authorize()`) -- hub staff only
- CASL: this is a staff action. Currently NO staff CASL rule for PromoCode exists in `platform-abilities.ts`. Add `can('manage', 'PromoCode')` for `finance.manage` scope per Canonical Section 8.
- Set `type: 'PLATFORM'`, `affiliateId: null`, `createdByUserId: staffSession.staffUserId`
- Sync to Stripe (same as affiliate)
- Write audit event: `PLATFORM_PROMO_CODE_CREATED`
- Revalidate `/fin/promo-codes`

#### Action 5: `updatePlatformPromoCode(input: unknown)` (Admin/Staff)

Updates a platform promo code.

Business rules:
- Auth: `staffAuthorize()` with `finance.manage` scope
- Same mutability rules as affiliate codes
- Sync Stripe if deactivating
- Write audit event: `PLATFORM_PROMO_CODE_UPDATED`

#### Action 6: `validatePromoCode(input: unknown)`

Validates a promo code for a subscription checkout. Does NOT create a redemption -- just checks eligibility.

Business rules per Canonical Section 2.4 and 5.2:
- Code must exist and `isActive === true`
- Code must not be expired (`expiresAt === null || expiresAt > now`)
- Usage limit check: if `usageLimit` is set, `usageCount < usageLimit`
- Scope check: if `scopeProductTypes` is set, the requested product must be in the array
- User must not have already redeemed this code for this product (unique constraint)
- Stacking rule: promo code CANNOT stack with another promo code. One discount per checkout.
- Return: `{ valid: true, discountType, discountValue, durationMonths, type }` or `{ valid: false, error }`

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- No `SellerTier`, `SubscriptionTier`, `FVF`, `Final Value Fee`, `BASIC`, `ELITE`, `PLUS`, `MAX`, `PREMIUM`, `wallet`, `Withdraw`, `Twicely Balance`
- Use `StoreTier` / `ListerTier` / `Transaction Fee` / `Request payout` / `Available for payout`

### Banned Patterns
- No `as any`, `@ts-ignore`, `@ts-expect-error`
- No float arithmetic for money -- integer cents only
- No hardcoded fee rates (read from `platform_settings`)
- No spreading request body into DB updates
- No files over 300 lines
- No `console.log` in production code

### Gotchas
1. **CASL affiliateId mismatch**: CASL ability.ts lines 231-234 use `{ affiliateId: userId }` but the promoCode table's `affiliateId` references `affiliate.id` (CUID2 PK), NOT `user.id`. Per-row CASL checks like `ability.can('update', sub('PromoCode', existingCode))` WILL FAIL because `existingCode.affiliateId` is the affiliate's PK, not the userId. **Workaround**: For all PromoCode mutations, look up the affiliate by userId first, verify `promoCode.affiliateId === affiliate.id`, and use subject-level CASL check only (e.g., `ability.can('update', 'PromoCode')` without conditions). This is the same pattern used in G1.4 queries.

2. **Stripe Coupon vs PromotionCode**: A Stripe `Coupon` defines the discount parameters. A `PromotionCode` is the user-facing code that maps to a Coupon. One Coupon can have multiple PromotionCodes. Create 1 Coupon per promoCode, 1 PromotionCode per Coupon.

3. **Discount value semantics**: For PERCENTAGE type, `discountValue` is in BPS (1000 = 10%). Stripe's `percent_off` expects a number like `10` (meaning 10%). Convert: `percent_off = discountValue / 100`. For FIXED type, `discountValue` is in cents. Stripe's `amount_off` also uses cents. No conversion needed.

4. **Stacking prohibition**: Per Canonical Section 2.4 and Feature Lock-in Section 2, promo codes CANNOT stack with each other. Only ONE promotional discount per checkout. The existing `applyCoupon` action in `apply-coupon.ts` handles seller coupons for marketplace checkout (cart items). The new promo code system is for SUBSCRIPTION checkout only. These are different systems -- seller coupons reduce item prices, promo codes reduce subscription prices. No stacking logic needed between them because they apply to different checkout flows.

5. **Platform code admin attribution**: Admin promo codes do NOT earn affiliate commission (Canonical Section 2.5). However, if a user has BOTH an affiliate cookie AND enters an admin promo code, the affiliate still gets attribution but commission is calculated on the discounted amount. This interaction is for G3.3 (affiliate payouts), not G1.5.

6. **`createdByUserId` for platform codes**: This should be the staff user's ID. Staff auth uses `staffUser` table (separate from `user` table). The `createdByUserId` column in promoCode is `text NOT NULL` with no FK. Store `staffSession.staffUserId`.

7. **Existing `applyCoupon` action**: This is for SELLER coupons on marketplace checkout (item purchases), NOT for subscription promo codes. Do NOT modify or conflate with subscription promo codes.
