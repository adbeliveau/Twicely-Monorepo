## SCOPE -- EXACTLY WHAT TO BUILD

### 1. Seller Local Settings Action

**File:** `src/lib/actions/seller-local-settings.ts` (NEW)

Server action for sellers to configure their local pickup settings.

**Action: `updateSellerLocalSettings`**
- Auth: `authorize()` with CASL check `can('update', 'SellerProfile', { userId })`
- Input (Zod `.strict()`):
  ```typescript
  const updateSellerLocalSettingsSchema = z.object({
    maxMeetupDistanceMiles: z.number().int().min(1).max(50).nullable(),
  }).strict();
  ```
- Logic: Updates `sellerProfile.maxMeetupDistanceMiles` for the current user
- Returns: `{ success: true }` or `{ error: string }`
- Revalidates: `/my/selling/settings/local`
- NOTE: max radius of 50 miles comes from `commerce.local.maxRadiusMiles` platform setting. However, for the Zod schema validation, hardcode `.max(50)` as a safety guard -- the server action ALSO reads the platform setting and enforces it dynamically (the Zod max is a ceiling, the setting is the actual limit).

### 2. Local Transaction Check-In Action

**File:** `src/lib/actions/local-transaction.ts` (NEW)

Server actions for the meetup check-in and confirmation flow.

**Action: `checkInToMeetupAction`**
- Auth: `authorize()` with CASL check: buyer checks `can('update', sub('LocalTransaction', { buyerId: session.userId }))`, seller checks `can('update', sub('LocalTransaction', { sellerId: session.userId }))`
- Input (Zod `.strict()`):
  ```typescript
  const checkInSchema = z.object({
    localTransactionId: z.string().min(1),
  }).strict();
  ```
- Logic:
  1. Fetch the localTransaction by ID
  2. Verify the caller is the buyer OR seller on this transaction
  3. Determine party: if `session.userId === transaction.buyerId` then 'BUYER', if `session.userId === transaction.sellerId` then 'SELLER'
  4. Verify transaction status allows check-in (must be SCHEDULED, SELLER_CHECKED_IN, or BUYER_CHECKED_IN -- NOT already BOTH_CHECKED_IN, COMPLETED, CANCELED, NO_SHOW, DISPUTED, RECEIPT_CONFIRMED)
  5. Call `recordCheckIn(transactionId, party)` from existing `local-transaction.ts`
  6. Return `{ success: true, bothCheckedIn: boolean }`
- Error: "Not found" (generic, no ID leak), "Already checked in", "Transaction not in valid state"

**Action: `confirmReceiptAction`**
- Auth: `authorize()` -- ONLY the buyer can confirm receipt
- CASL: `can('update', sub('LocalTransaction', { buyerId: session.userId }))`
- Input (Zod `.strict()`):
  ```typescript
  const confirmReceiptSchema = z.object({
    localTransactionId: z.string().min(1),
    code: z.string().min(1),
    method: z.enum(['QR', 'MANUAL']),
  }).strict();
  ```
- Logic:
  1. Fetch the localTransaction by ID
  2. Verify `session.userId === transaction.buyerId` (ONLY buyer confirms receipt)
  3. Verify transaction status is BOTH_CHECKED_IN or SELLER_CHECKED_IN or BUYER_CHECKED_IN or SCHEDULED (flexible -- both don't need to check in for confirmation)
  4. If `method === 'QR'`: call `validateConfirmationCode(code)` from existing helper
  5. If `method === 'MANUAL'`: call `validateOfflineCode(code)` from existing helper
  6. If validation fails: return error from validation
  7. If validation succeeds:
     - Call `confirmLocalTransaction(transactionId, method)` from existing helper
     - This sets status to COMPLETED, confirmedAt, and confirmationMethod
     - NOTE: Escrow release (payment capture) is NOT handled here -- it will be wired to the payment system in a future step. For now, just update the local transaction status.
  8. Return `{ success: true }`

**Action: `confirmReceiptOfflineAction`**
- Auth: `authorize()` -- ONLY the buyer
- CASL: `can('update', sub('LocalTransaction', { buyerId: session.userId }))`
- Input (Zod `.strict()`):
  ```typescript
  const confirmOfflineSchema = z.object({
    localTransactionId: z.string().min(1),
    offlineCode: z.string().regex(/^\d{6}$/),
    offlineTimestamp: z.string().datetime(), // ISO 8601 from client
  }).strict();
  ```
- Logic:
  1. Validate offline code matches transaction's `offlineCode` field
  2. Validate code not expired (scheduledAt + configurable hours from `commerce.local.confirmationCodeExpiryHours`)
  3. Update transaction: status = 'COMPLETED', confirmationMethod = 'OFFLINE', offlineConfirmedAt = parsed offlineTimestamp, syncedAt = now()
  4. Return `{ success: true }`
- NOTE: Per Local Canonical Section 5, offline grace period is 2 hours. If device doesn't sync within 2 hours, buyer gets push notification to confirm online. This action IS the sync moment -- the client stores the offline confirmation and calls this when connectivity returns.

### 3. Local Transaction Queries

**File:** `src/lib/queries/local-transaction.ts` (NEW)

**Query: `getLocalTransactionByOrderId(orderId: string)`**
- Returns full localTransaction row with joined safeMeetupLocation data
- Used by order detail page to show meetup info

**Query: `getLocalTransactionById(transactionId: string)`**
- Returns full localTransaction row
- Used by server actions for validation

**Query: `getActiveLocalTransactionsForUser(userId: string)`**
- Returns localTransactions where `(buyerId = userId OR sellerId = userId) AND status IN ('SCHEDULED', 'SELLER_CHECKED_IN', 'BUYER_CHECKED_IN', 'BOTH_CHECKED_IN')`
- Ordered by scheduledAt ASC
- Used by dashboard to show upcoming meetups

**Query: `getCompletedLocalTransactionsForUser(userId: string, limit: number = 20)`**
- Returns localTransactions where `(buyerId = userId OR sellerId = userId) AND status IN ('COMPLETED', 'RECEIPT_CONFIRMED')`
- Ordered by confirmedAt DESC
- Used by local history view

### 4. Safe Meetup Location Queries

**File:** `src/lib/queries/safe-meetup-locations.ts` (NEW)

**Query: `getActiveSafeMeetupLocations(city?: string, state?: string)`**
- Returns all active safe meetup locations
- Optional filter by city/state
- Ordered by meetupCount DESC (popular first)
- Used by meetup scheduling UI

**Query: `getSafeMeetupLocationById(locationId: string)`**
- Returns single location or null
- Used by transaction detail

**Query: `getNearbyMeetupLocations(latitude: number, longitude: number, radiusMiles: number)`**
- Calculates approximate distance using Haversine formula in SQL
- Returns locations within radius, ordered by distance
- Used by location selection UI
- NOTE: Use a simplified Haversine approximation. Do NOT import any geospatial library. Example:
  ```sql
  -- Approximate distance in miles using Haversine
  (3959 * acos(cos(radians(:lat)) * cos(radians(latitude)) * cos(radians(longitude) - radians(:lng)) + sin(radians(:lat)) * sin(radians(latitude))))
  ```
  Use Drizzle's `sql` template tag for this.

## CONSTRAINTS -- WHAT NOT TO DO

- Do NOT modify the existing `local-transaction.ts` -- call its functions from the server actions
- Do NOT modify `local-fee.ts` -- it is already complete and tested
- Do NOT implement escrow release / Stripe payment capture in this step -- that is G2.3
- Do NOT implement no-show detection / penalties -- that is G2.5
- Do NOT implement safety timer -- that is G2.6
- Do NOT build UI components -- those are in later slices
- The `confirmLocalTransaction` helper sets status to 'COMPLETED' -- but per the canonical flow, the correct state after buyer scans QR is 'RECEIPT_CONFIRMED' (then 'COMPLETED' after escrow releases). For now, using 'COMPLETED' as the helper does is fine -- we will refine the state machine in G2.3 when escrow release is wired.
- Ownership: all queries filter by `userId` matching `buyerId` or `sellerId` on the transaction. NEVER expose data from other users' transactions.
- Never hardcode fee rates -- read from `platform_settings` via `getPlatformSetting()`
- Use `sub()` from `@casl/ability` for per-instance CASL checks
- All monetary values in integer cents
- Generic error messages: "Not found" never "You don't own transaction X"
