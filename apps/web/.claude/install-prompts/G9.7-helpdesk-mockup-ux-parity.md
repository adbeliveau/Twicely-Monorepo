**2.1.2 Unread Dots (Blue Indicator)**

Cases with messages the current agent hasn't read need a blue dot indicator. Since there is NO read-tracking column in the schema, implement a **lightweight approach** using existing data:

- A case has "unread" messages if `lastActivityAt > firstResponseAt` AND the last message in the case has `direction = 'INBOUND'` (user sent something the agent hasn't replied to yet).
- Add a `hasUnread` boolean to `CaseRowData` type.
- Compute `hasUnread` server-side in the cases API route query: compare the most recent message's direction to see if the latest message is INBOUND and there's no subsequent OUTBOUND/INTERNAL message from an agent.
- In `CaseRow`, render a small blue dot (`w-2 h-2 rounded-full bg-blue-500`) next to the case number when `hasUnread` is true.

**NOT SPECIFIED -- Owner Decision Needed:** The Helpdesk Canonical does not define per-agent read tracking. This lightweight "last message is inbound" heuristic avoids a new table. A proper `lastReadAt` per agent per case would require a schema change. Recommend the lightweight approach for now.

### G9.7.2 -- AI Suggestion Card + AI Assist Button

**2.2.1 Platform Settings (2 new)**

|-----|------|---------|-------------|

| `helpdesk.ai.suggestionEnabled` | boolean | `true` | Enable AI-generated suggested reply on case workspace |
| `helpdesk.ai.assistEnabled` | boolean | `true` | Enable AI assist button in reply composer |

**2.2.2 AI Service (`src/lib/helpdesk/ai-service.ts`)**

Create a server-side AI service:

```typescript
// Two functions, both read model from platform_settings:

// 1. generateSuggestion(caseContext) -> Promise<string | null>
//    Input: case type, priority, subject, description, last 5 messages, linked entity summaries
//    Output: a suggested reply the agent can review/edit/send
//    Prompt: "You are a customer support agent for Twicely, a peer-to-peer resale marketplace.
//             Based on the case context below, draft a helpful, professional reply to the customer.
//             Be concise. Do not make commitments the agent can't keep. Format: plain text."
//    Catches ALL errors gracefully (API down, rate limit, bad response) -> return null

// 2. assistReply(body, action) -> Promise<string | null>
//    Input: the agent's draft text + action (REWRITE | SUMMARIZE | TRANSLATE_ES | TRANSLATE_FR)
//    Output: transformed text
//    Prompt varies by action:
//      REWRITE: "Rewrite this support reply to be more professional and empathetic. Keep the same meaning."
//      SUMMARIZE: "Summarize this support reply into 2-3 concise sentences."
//      TRANSLATE_ES: "Translate this support reply into Spanish. Keep the professional tone."
//      TRANSLATE_FR: "Translate this support reply into French. Keep the professional tone."
//    Catches ALL errors gracefully -> return null
```

- Use `@anthropic-ai/sdk` with `process.env.ANTHROPIC_API_KEY`
- Max tokens: 500 for suggestion, 500 for assist
- Temperature: 0.3 (low for consistency)
- Both functions are server-side only (NOT exported from a `'use server'` file)
- Read `helpdesk.ai.model` from platform_settings via the settings loader. If setting is missing, fall back to `claude-haiku-4-5-20251001`.
- If `helpdesk.ai.suggestionEnabled` or `helpdesk.ai.assistEnabled` is false, return null immediately.

**2.2.3 AI Server Actions (`src/lib/actions/helpdesk-ai.ts`)**

Two server actions:

```typescript
'use server'

// 1. getAiSuggestion({ caseId: string }): Promise<ActionResult<{ suggestion: string }>>
//    - staffAuthorize() + ability.can('read', 'HelpdeskCase')
//    - Load case + last 5 messages + context entities
//    - Call generateSuggestion() from ai-service
//    - Return { success: true, data: { suggestion } } or { success: false, error: '...' }

// 2. getAiAssist({ body: string, action: 'REWRITE' | 'SUMMARIZE' | 'TRANSLATE_ES' | 'TRANSLATE_FR' }): Promise<ActionResult<{ result: string }>>
//    - staffAuthorize() + ability.can('read', 'HelpdeskCase')
//    - Call assistReply() from ai-service
//    - Return { success: true, data: { result } } or { success: false, error: '...' }
```

Zod schemas with `.strict()` on all inputs. Explicit field mapping (never spread).

**2.2.4 AI Suggestion Card (`src/components/helpdesk/ai-suggestion-card.tsx`)**

A client component displayed in the case workspace between the timeline and the reply composer:

- On mount, calls `getAiSuggestion({ caseId })` (lazy load, not blocking page render)
- Shows loading skeleton while fetching
- Once loaded, displays in a card with:
  - Header: "AI Suggested Reply" with a sparkle icon and "Powered by AI" badge
  - Body: the suggested reply text
  - Footer: "Use Suggestion" button (inserts into reply composer) + "Dismiss" button
  - "Regenerate" button (re-calls the action)
- If suggestion returns null or error: show a muted "AI suggestion unavailable" message, not an error banner
- If `helpdesk.ai.suggestionEnabled` is false: don't render the card at all (check via a prop passed from the server)

**2.2.5 AI Assist Button in Reply Composer**

Add to `ReplyComposer` (`src/components/helpdesk/reply-composer.tsx`):

## 3. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- No `Twicely Balance` -- use `Available for payout` (not relevant here but check all UI strings)
- No `wallet` in any user-facing string
- No `FVF` or `Final Value Fee`
- No `Withdraw` -- use `Request payout`

### Tech Stack
- AI calls use `@anthropic-ai/sdk` (Anthropic SDK), NOT OpenAI, NOT a generic LLM wrapper
- Cache/queue: Valkey + BullMQ (not Redis/Bull) -- but no queue needed for this step
- No tRPC, no Zustand, no Redux
- No Prisma -- use Drizzle

### Code Patterns
- `strict: true` TypeScript -- zero `as any`, zero `@ts-ignore`
- Max 300 lines per file. If `dashboard-widgets.tsx` exceeds 300 after additions, extract the new widget to `team-status-grid.tsx` and/or `live-activity-feed.tsx`.
- Zod `.strict()` on all server action input schemas
- Explicit field mapping in all actions (never spread request body)
- `staffAuthorize()` on all server actions and API routes
- Error messages: "Not found", never "You don't have access to case X"
- All API routes return `{ success: boolean, error?: string, data?: ... }` shape
- Exported helpers in `'use server'` files = unintended server actions. Keep helpers unexported.

### Business Logic
- AI suggestions are ASSISTIVE ONLY. The agent ALWAYS reviews before sending. AI never auto-sends.
- AI suggestion failures are silent (return null, show "unavailable"), never error banners.
- The AI model is read from platform_settings, NEVER hardcoded in the AI service.
- `ANTHROPIC_API_KEY` comes from `process.env`, NOT from platform_settings (secrets are env vars per Decision Rationale).

### Routes
- All helpdesk routes under `/hd/*` -- never `/helpdesk/`, never `/support/admin/`
- New API route: `/api/hub/helpdesk/activity` -- uses the `/api/hub/` prefix for hub API routes

## 8. SPEC INCONSISTENCIES (Document for Owner)

1. **AI Model ID**: Build Tracker says `claude-haiku-4-5-20251001`. This appears to be a future model identifier. The code reads the model from `helpdesk.ai.model` platform setting, so it can be updated without code changes. The seed default is exactly what the Build Tracker specifies.

2. **Per-Agent Signatures**: The Schema doc does not include `signatureHtml` on `staffUser`. The Build Tracker G9.7 says "message signatures." The existing `helpdeskEmailConfig.signatureHtml` is global (for email). This prompt adds per-agent signatures on `staffUser` for workspace display. Owner should confirm.

3. **Unread Tracking**: The Helpdesk Canonical does not define per-agent read tracking. G9.7 implements a lightweight heuristic (last message is inbound = unread). Proper per-agent `lastReadAt` tracking would require a new junction table `caseAgentReadState(caseId, staffUserId, lastReadAt)`. Deferred.

4. **Live Activity Feed**: Feature Lock-in section 15 specifies Centrifugo channels (`private-hd-queue`, `private-hd-agent.{staffUserId}`) for real-time updates. G9.7 implements polling (30-second interval) as a pragmatic approach. Full Centrifugo integration for helpdesk is not in any specific G-phase step and would be part of G10 production readiness or post-launch.

5. **Platform Settings Canonical Gap**: `helpdesk.ai.*` settings are NOT in the Platform Settings Canonical document. They are defined solely by the Build Tracker G9.7 description. These settings should be added to the canonical during the next doc sync.
