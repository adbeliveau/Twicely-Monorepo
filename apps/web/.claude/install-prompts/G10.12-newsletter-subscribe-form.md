## 3. SCOPE — EXACTLY WHAT TO BUILD

### 3.1 Database

**New table: `newsletter_subscriber`** — to be added as a new file `src/lib/db/schema/newsletter.ts`.

This table is NOT in schema v2.1.0. It is a new addition required by G10.12. The schema doc §23 covers BUYER ACQUISITION and ends at §23.2 `buyer_referral`. This table belongs conceptually to §23 but is added as a new schema file (acquisition.ts already exists and is complete — add the new table to a separate file to stay modular).

```typescript
// src/lib/db/schema/newsletter.ts
import { pgTable, text, timestamp, index, unique } from 'drizzle-orm/pg-core';
import { pgEnum } from 'drizzle-orm/pg-core';
import { createId } from '@paralleldrive/cuid2';

export const newsletterSourceEnum = pgEnum('newsletter_source', [
  'HOMEPAGE_SECTION',
  'HOMEPAGE_FOOTER',
]);

export const newsletterSubscriber = pgTable('newsletter_subscriber', {
  id:                text('id').primaryKey().$defaultFn(() => createId()),
  email:             text('email').notNull(),
  source:            newsletterSourceEnum('source').notNull().default('HOMEPAGE_SECTION'),
  unsubscribeToken:  text('unsubscribe_token').notNull().$defaultFn(() => createId()),
  confirmedAt:       timestamp('confirmed_at', { withTimezone: true }).notNull().defaultNow(),
  unsubscribedAt:    timestamp('unsubscribed_at', { withTimezone: true }),
  welcomeSentAt:     timestamp('welcome_sent_at', { withTimezone: true }),
  createdAt:         timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
}, (table) => ({
  emailUnique:       unique('newsletter_subscriber_email_unique').on(table.email),
  tokenUnique:       unique('newsletter_subscriber_token_unique').on(table.unsubscribeToken),
  emailIdx:          index('newsletter_subscriber_email_idx').on(table.email),
  createdAtIdx:      index('newsletter_subscriber_created_at_idx').on(table.createdAt),
}));
```

**Column notes:**
- `email`: unique constraint prevents duplicate subscriptions. Case-insensitive matching handled at application layer (lowercase before insert).
- `unsubscribeToken`: CUID2. Sent in welcome email as URL param. Never exposed in GET responses.
- `confirmedAt`: set to `NOW()` immediately on subscribe (single opt-in per OD-1 default). If OD-1 changes to double opt-in, this becomes nullable and is set only when the confirmation email link is clicked.
- `unsubscribedAt`: set when GET `/api/newsletter/unsubscribe?token=xxx` is called. Subscriber record is kept (soft delete) for audit and suppression.
- `welcomeSentAt`: set after the welcome email is sent. Allows retry logic if send fails.
- `source`: tracks where the subscriber came from for analytics.

**Schema file must export:**
- `newsletterSubscriber` (table)
- `newsletterSourceEnum` (enum)

**Drizzle migration:** A new migration file must be generated. Run `pnpm drizzle-kit generate` after creating the schema file.

### 3.2 API Routes

**Route 1: POST `/api/newsletter/subscribe`**

- **File:** `src/app/api/newsletter/subscribe/route.ts`
- **Auth:** PUBLIC — no session required. Guests can subscribe.
- **Method:** POST
- **Content-Type:** `application/json`
- **Input schema (Zod, `.strict()`):**
  ```typescript
  const subscribeSchema = z.object({
    email: z.string().email().max(254).transform((e) => e.toLowerCase().trim()),
    source: z.enum(['HOMEPAGE_SECTION', 'HOMEPAGE_FOOTER']).default('HOMEPAGE_SECTION'),
  }).strict();
  ```
- **Logic:**
  1. Parse and validate body with `subscribeSchema.safeParse()`.
  2. If parse fails: return `{ success: false, error: 'Invalid email address' }` with status 400.
  3. Check platform setting `newsletter.enabled` (boolean). If false: return `{ success: false, error: 'Newsletter subscriptions are currently unavailable' }` with status 503.
  4. Query DB: `SELECT id, unsubscribedAt FROM newsletter_subscriber WHERE email = $email LIMIT 1`.
  5. If existing record found AND `unsubscribedAt IS NULL`: return `{ success: true, alreadySubscribed: true }` with status 200. Do NOT error — treat as idempotent.
  6. If existing record found AND `unsubscribedAt IS NOT NULL` (previously unsubscribed): update `unsubscribedAt = NULL`, `confirmedAt = NOW()`, `welcomeSentAt = NULL`. Treat as re-subscribe. Continue to step 7.
  7. If no record found: insert new row. `source` from validated input. `unsubscribeToken` generated by `createId()`. `confirmedAt = NOW()`.
  8. Send welcome email (see §3.4). Set `welcomeSentAt = NOW()` after successful send.
  9. Return `{ success: true }` with status 200.
- **Error handling:** Wrap in try/catch. On unexpected DB error: return `{ success: false, error: 'Something went wrong' }` with status 500. Log with `logger.error()`.
- **Response shape:**
  ```typescript
  type SubscribeResponse =
    | { success: true; alreadySubscribed?: true }
    | { success: false; error: string }
  ```
- **No CASL check** — this is a pure public write endpoint. No `authorize()` call needed.
- **No rate limiting in this implementation** (per OD-3 recommended default). Add a TODO comment.

**Route 2: GET `/api/newsletter/unsubscribe`**

- **File:** `src/app/api/newsletter/unsubscribe/route.ts`
- **Auth:** PUBLIC — token is the only credential.
- **Method:** GET
- **Query params:** `?token=<unsubscribeToken>`
- **Logic:**
  1. Read `token` from `request.nextUrl.searchParams.get('token')`.
  2. If token is missing or empty: redirect to `/?unsubscribed=error`.
  3. Query DB: `SELECT id, unsubscribedAt FROM newsletter_subscriber WHERE unsubscribe_token = $token LIMIT 1`.
  4. If not found: redirect to `/?unsubscribed=error`.
  5. If already unsubscribed (`unsubscribedAt IS NOT NULL`): redirect to `/?unsubscribed=already` (idempotent).
  6. Update: `SET unsubscribed_at = NOW() WHERE id = $id`.
  7. Redirect to `/?unsubscribed=success`.
- **Response:** HTTP redirect (302). Never return JSON — this is called from email links.
- **No new pages needed.** The homepage already handles query params (or ignores them safely). A toast/banner on the homepage can optionally read the `unsubscribed` param to show a confirmation message — this is an enhancement left to the installer's judgment based on line count budget.

### 3.3 Server Action (alternative to API route for the subscribe form)

**Decision:** Use a **Next.js API route** (not a server action) for the subscribe form.

**Reason:** The form is a `'use client'` component. Server actions require importing from `'use server'` files. The existing pattern for public anonymous writes (e.g., affiliate tracking at `/api/affiliate/listing-click/route.ts`) uses API routes. Using an API route also allows the `List-Unsubscribe` header to be added to the welcome email without needing the server action to return a special value.

## 4. CONSTRAINTS — WHAT NOT TO DO

### Banned Tech
- Never use `as any`, `@ts-ignore`, or `@ts-expect-error`
- Never use floating-point for money (not applicable here — no money involved)
- Never use Redis or Bull — use Valkey + BullMQ if queues are needed (not needed for this feature)

### Banned Patterns
- Do NOT store emails in plaintext and then log them anywhere. Email is PII.
- Do NOT expose the `unsubscribeToken` in any API response other than the welcome email itself.
- Do NOT add the subscriber form to the `'use server'` actions files. The form submission goes to an API route.
- Do NOT use `router.refresh()` after subscribe — the form shows a success state inline, no page refresh needed.
- Do NOT spread the request body into the DB insert. Use explicit field mapping only:
  ```typescript
  // CORRECT
  await db.insert(newsletterSubscriber).values({
    email: parsed.data.email,
    source: parsed.data.source,
  });

  // WRONG
  await db.insert(newsletterSubscriber).values(parsed.data);
  ```

### Route Constraints
- The subscribe endpoint is `/api/newsletter/subscribe` (not `/api/newsletter` with a POST — keep the resource-action URL structure consistent with the codebase)
- The unsubscribe endpoint is `/api/newsletter/unsubscribe` (GET, uses query param `?token=`)
- There is NO hub admin page for newsletter in this prompt. A future step could add `/cfg/newsletter` but it is NOT in the Page Registry and NOT in scope here.

### Email Constraints
- From address MUST be `Twicely <noreply@twicely.co>` (matches existing templates)
- Subject MUST NOT use banned vocabulary. No "Balance", "Wallet", "FVF", "SellerTier", etc.

### UI Copy Constraints

- Do NOT use the word "subscribe" on the button — use "Get updates" or similar active phrasing
