```typescript
// Macro update
export const updateMacroSchema = z.object({
  macroId: z.string().cuid2(),
  name: z.string().min(1).max(200).optional(),
  description: z.string().max(500).optional().nullable(),
  bodyTemplate: z.string().min(1).max(10000).optional(),
  actionsJson: z.array(z.object({ type: z.string(), value: z.unknown() })).optional(),
  isShared: z.boolean().optional(),
}).strict();

// Team
export const createTeamSchema = z.object({
  name: z.string().min(1).max(100),
  description: z.string().max(500).optional(),
  maxConcurrentCases: z.number().int().min(1).max(100).default(25),
  roundRobinEnabled: z.boolean().default(true),
}).strict();

export const updateTeamSchema = z.object({
  teamId: z.string().cuid2(),
  name: z.string().min(1).max(100).optional(),
  description: z.string().max(500).optional().nullable(),
  maxConcurrentCases: z.number().int().min(1).max(100).optional(),
  roundRobinEnabled: z.boolean().optional(),
}).strict();

// Routing rule condition/action shapes
const routingConditionSchema = z.object({
  field: z.enum(['type', 'channel', 'priority', 'subject', 'tags', 'requesterType']),
  operator: z.enum(['eq', 'in', 'contains', 'gte', 'lte', 'startsWith']),
  value: z.union([z.string(), z.array(z.string())]),
});

const routingActionSchema = z.object({
  assignTeamId: z.string().cuid2().optional(),
  assignAgentId: z.string().cuid2().optional(),
  setPriority: z.enum(['CRITICAL', 'URGENT', 'HIGH', 'NORMAL', 'LOW']).optional(),
  addTags: z.array(z.string()).optional(),
  setCategory: z.string().max(100).optional(),
});

export const createRoutingRuleSchema = z.object({
  name: z.string().min(1).max(200),
  conditionsJson: z.array(routingConditionSchema).min(1),
  actionsJson: routingActionSchema,
  sortOrder: z.number().int().min(0).optional(),
  isActive: z.boolean().default(true),
}).strict();

export const updateRoutingRuleSchema = z.object({
  ruleId: z.string().cuid2(),
  name: z.string().min(1).max(200).optional(),
  conditionsJson: z.array(routingConditionSchema).min(1).optional(),
  actionsJson: routingActionSchema.optional(),
  isActive: z.boolean().optional(),
}).strict();

// Automation rule
const automationTriggerEnum = z.enum([
  'CASE_CREATED', 'STATUS_CHANGED', 'PRIORITY_CHANGED',
  'SLA_WARNING', 'SLA_BREACHED', 'NO_RESPONSE',
  'AGENT_ASSIGNED', 'MESSAGE_RECEIVED', 'CASE_REOPENED',
]);

const automationConditionSchema = z.object({
  field: z.string().min(1),
  operator: z.enum(['eq', 'in', 'gte', 'lte', 'contains']),
  value: z.union([z.string(), z.number(), z.array(z.string())]),
});

const automationActionSchema = z.object({
  type: z.enum([
    'SET_PRIORITY', 'ASSIGN_TEAM', 'ASSIGN_AGENT',
    'ADD_TAGS', 'REMOVE_TAGS', 'SET_STATUS',
    'SEND_NOTIFICATION', 'ADD_NOTE',
  ]),
  value: z.unknown(),
});

export const createAutomationRuleSchema = z.object({
  name: z.string().min(1).max(200),
  triggerEvent: automationTriggerEnum,
  conditionsJson: z.array(automationConditionSchema),
  actionsJson: z.array(automationActionSchema).min(1),
  sortOrder: z.number().int().min(0).optional(),
  isActive: z.boolean().default(true),
}).strict();

export const updateAutomationRuleSchema = z.object({
  ruleId: z.string().cuid2(),
  name: z.string().min(1).max(200).optional(),
  triggerEvent: automationTriggerEnum.optional(),
  conditionsJson: z.array(automationConditionSchema).optional(),
  actionsJson: z.array(automationActionSchema).min(1).optional(),
  isActive: z.boolean().optional(),
}).strict();

// SLA policy (extended)
export const updateSlaPolicySchema = z.object({
  policyId: z.string().cuid2(),
  firstResponseMinutes: z.number().int().min(1).optional(),
  resolutionMinutes: z.number().int().min(1).optional(),
  businessHoursOnly: z.boolean().optional(),
  escalateOnBreach: z.boolean().optional(),
}).strict();
```

## 3. CONSTRAINTS — WHAT NOT TO DO

### Banned Patterns
- No external drag-and-drop library (use native HTML5 DnD for routing reorder)
- No new CASL subjects or permissions — everything needed already exists
- No schema changes — all tables are complete
- No new platform settings — all helpdesk.* settings are already seeded
- No `as any`, `@ts-ignore`, `@ts-expect-error`
- No files over 300 lines
- No fee calculations (irrelevant to this feature)
- No `console.log` in production code

### Gotchas
1. **`helpdesk-agent.ts` is 297 lines.** Do NOT add more actions to this file. All new actions go in the new `helpdesk-manage.ts` file.
2. **`validations/helpdesk.ts` is 168 lines.** Adding ~80 lines of new schemas will bring it to ~248 lines. Within the 300-line limit.
3. **Routing rule `conditionsJson` and `actionsJson`** — In the actual schema, `conditionsJson` is `jsonb` and `actionsJson` is `jsonb`. In seed data, conditions is an array of `{field, operator, value}` objects, and actions is a flat object `{assignTeamId, setPriority, ...}`. The Canonical sketch (section 25.4) uses `assignTeamId`, `assignAgentId`, `setPriority`, `addTags`, `setCategory` as separate columns, but the actual schema stores them as a single `actionsJson` object. Match the actual schema, not the canonical sketch.
4. **helpdeskRoutingRule actual schema** — Does NOT have `description`, `assignTeamId`, `assignAgentId`, `setPriority`, `addTags`, `setCategory`, `createdByStaffId` as separate columns. Everything is inside `conditionsJson` and `actionsJson`. Match this.
5. **helpdeskAutomationRule actual schema** — Has `conditionsJson` (array) and `actionsJson` (object — but in seed it's an array of action objects). The `actionsJson` column definition says `default(sql'[]')` — it's treated as an array in seed data. Match seed data pattern.

## 9. SPEC INCONSISTENCIES (Document for Owner Decision)

1. **Schema vs Canonical: `helpdeskRoutingRule` columns.** The Canonical sketch (section 25.4) shows `assignTeamId`, `assignAgentId`, `setPriority`, `addTags`, `setCategory`, `createdByStaffId` as separate columns. The actual schema stores everything in `conditionsJson` (array) and `actionsJson` (jsonb object). **Resolution: Match the actual schema.** The conditions/actions JSON approach is more flexible and already in use in seed data.

2. **Schema vs Canonical: `helpdeskTeam.displayName`.** The Canonical sketch has both `name` (internal) and `displayName` (UI). The actual schema only has `name`. **Resolution: Use `name` for display.** No need for both.

3. **Settings page CASL gate.** The Canonical says settings are "Accessible to HELPDESK_MANAGER and ADMIN" (section 24). But `updateSettingAction` uses `ability.can('update', 'Setting')` which is ADMIN-only. **Resolution: Create a helpdesk-specific settings update action that gates on `HelpdeskEmailConfig` manage permission and restricts key prefix to `helpdesk.*`.**

4. **`helpdeskSlaPolicy.escalateToTeamId`.** The Canonical sketch has this column. The actual schema does not. **Resolution: Omit. Escalation team routing is handled by automation rules, not SLA policy configuration.**
