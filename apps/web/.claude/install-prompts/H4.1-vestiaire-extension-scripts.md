## 1. FEATURE OVERVIEW: Vestiaire Collective Model

Vestiaire Collective is a **luxury fashion resale marketplace** headquartered in Paris. Unlike TheRealReal (which is a full consignment model where TRR sets prices), Vestiaire is a **peer-to-peer marketplace where sellers set their own prices**. The platform handles authentication of luxury goods and processes transactions.

Key characteristics:
- **Multi-currency**: Vestiaire operates globally. Prices may be in EUR, GBP, USD, or other currencies depending on the user's locale. The scraper must handle currency detection and conversion to USD cents.
- **Seller-set pricing**: Unlike TRR, sellers control the listing price. Auto-fill DOES include price.
- **Authentication service**: Vestiaire authenticates luxury items (similar to TRR). Relevant for scraping condition/authentication status but does not affect the content script's behavior.
- **React-based SPA**: Vestiaire Collective's site (vestiairecollective.com) uses a React SPA. DOM selectors must be resilient to class name changes. JSON-LD Product schema markup is present on product pages (similar to TRR pattern).
- **Session-based**: No public API. Classified as **Tier C** (session-based), same tier as Poshmark and TheRealReal.

**Impact comparison with existing scripts:**

| Capability | Poshmark (H1.2) | FB Marketplace (H1.2) | TheRealReal (H1.3) | Vestiaire (H4.1) |
|---|---|---|---|---|
| Session capture | JWT from cookies/\_\_NEXT\_DATA\_\_ | c\_user cookie | session\_id + CSRF | Auth token from cookies |
| Listing scrape | \_\_NEXT\_DATA\_\_ + DOM | DOM only (no structured data) | JSON-LD + DOM | JSON-LD + DOM |
| Auto-fill price | Yes (seller sets price) | Yes (seller sets price) | NO (TRR sets price) | Yes (seller sets price) |
| Auto-fill photos | No (browser security) | No (browser security) | No (browser security) | No (browser security) |
| Multi-currency | No (USD only) | No (USD only) | No (USD only) | YES (EUR/GBP/USD/etc.) |

### 3.1 Extension Content Script: `vestiaire.ts`

**File:** `extensions/chrome/src/content/vestiaire.ts`

This follows the exact same module pattern as `poshmark.ts`, `fb-marketplace.ts`, and `therealreal.ts`:
- NOT an IIFE -- exports a handler object consumed by bridge.ts

#### 3.1.1 Session Capture

**Function:** `captureSession(): Promise<Record<string, unknown> | null>`

Strategy (in order of priority):
1. **Cookie-based**: Look for an auth/session token in cookies. Vestiaire uses cookies like `_vc_session`, `_vestiaire_session`, or a JWT-shaped cookie. Try all known names via `getCookieValue()`.
2. **Embedded script data**: Search inline `<script>` tags for patterns like `window.__INITIAL_STATE__`, `window.__PRELOADED_STATE__`, or JSON containing `"user"` with an `"id"` or `"email"` field. Parse with JSON.parse in try/catch. (Same pattern as `therealreal.ts` `extractUserFromPageScripts()`.)
3. **DOM fallback**: Check for logged-in indicators (account link, user avatar, profile dropdown). Same pattern as `therealreal.ts` `extractUserFromDom()`.

Return shape (matches what the server-side session route will store):
```typescript
{
  sessionToken: string;   // Auth cookie value
  userId: string;         // Vestiaire user ID
  email: string;          // User email (if extractable)
  detectedAt: number;     // Date.now() timestamp
}
```

Return `null` if the user is not logged in (no session cookie found).

#### 3.1.2 Listing Scrape

**Function:** `scrapeListing(): ScrapedListing | null`

Only scrape on product detail pages. Vestiaire product URLs follow patterns:
- `https://www.vestiairecollective.com/*/p-XXXXXXX.html` (product pages)
- `https://www.vestiairecollective.com/*/XXXXXXX.shtml` (older format)

The `externalId` is the numeric product ID extracted from the URL.

Strategy (in order of priority):
1. **JSON-LD Product markup** (most reliable): Parse `<script type="application/ld+json">` containing `"@type": "Product"`. Extract: name, description, offers.price, offers.priceCurrency, brand.name, image. This is the same pattern as TheRealReal's `scrapeFromJsonLd()` in `therealreal.ts` (lines 259-315).
2. **DOM selectors fallback**: Title from `h1` or `[data-testid="product-title"]`. Price from elements containing currency symbols (EUR, GBP, USD, $). Brand from breadcrumb or dedicated brand element. Description from product description section. Images from product gallery.

**CRITICAL: Multi-currency price handling.**

Vestiaire displays prices in the user's local currency. The `ScrapedListing.priceCents` field must be in the **displayed currency's cents** (e.g., EUR 150.00 = 15000). The extension content script does NOT perform currency conversion -- that is the server's responsibility during import.

The scrape data should include currency info via the new optional `currency` field on `ScrapedListing` (see Section 3.3). If currency cannot be determined, default to `'USD'`.

For the DOM scrape:
- Parse price string with currency symbol detection using the new `parsePriceWithCurrency()` helper (see Section 3.4)
- Known currency symbols: `$` (USD), `EUR` or euro sign (EUR), `GBP` or pound sign (GBP), `CHF`, `SEK`, `DKK`, etc.
- If JSON-LD `offers.priceCurrency` is present, use that (ISO 4217 code)
- Extract images from product gallery: look for `img` elements inside gallery/carousel containers. Prefer srcset highest resolution. Deduplicate (same `[...new Set(urls)]` pattern as `therealreal.ts` line 252).
- Condition: Vestiaire uses condition labels like "Never worn", "Very good condition", "Good condition", "Fair condition"
- Brand: typically in breadcrumb navigation or a dedicated `.product-brand` / `[data-testid="product-brand"]` element
- Category: from breadcrumb navigation (same `extractTrrCategoryFromDom()` pattern)
- Size: from product attributes section

#### 3.1.3 Auto-fill

**Function:** `autofillListing(payload: AutofillPayload): Promise<boolean>`

Vestiaire's listing creation form is at `https://www.vestiairecollective.com/sell/`. Fields to fill:

- **Title** (required): `input[name="title"]`, `input[placeholder*="title" i]`, `input[data-testid="sell-title"]`
- **Description** (optional): `textarea[name="description"]`, `textarea[placeholder*="description" i]`, `textarea[data-testid="sell-description"]`
- **Price** (required): `input[name="price"]`, `input[placeholder*="price" i]`, `input[data-testid="sell-price"]`. Convert `priceCents` to decimal string (priceCents / 100, formatted to 2 decimal places). Since we cannot know which currency the sell form expects, just use dollar equivalent. The seller will adjust.
- **Brand** (optional): `input[name="brand"]`, `input[placeholder*="brand" i]`, `input[data-testid="sell-brand"]`. May trigger autocomplete dropdown -- use longer delay (500-1200ms), same pattern as `therealreal.ts` line 482.
- **Size** (optional): Look for size selector. May be `select` element or clickable size buttons. Same pattern as `therealreal.ts` lines 487-508.
- **Condition** (optional): Look for condition selector/radio buttons. Same pattern as `therealreal.ts` lines 511-524.

Use `setReactInputValue()` for all input/textarea fills. Use `randomDelay(300, 800)` between fields per Lister Canonical Section 9.4. Return `true` if all required fields (title + price) were filled.

NOTE: Photos are NOT automated (browser security restriction). Same as all other platforms.

**Handler export:**
```typescript
export const vestiaireHandler = {
  captureSession,
  scrapeListing,
  autofillListing: async (payload: AutofillPayload) => autofillListing(payload),
};
```

### 3.4 Shared Utils Update: `shared-utils.ts`

**File:** `extensions/chrome/src/content/shared-utils.ts`

Add a new helper function for multi-currency price parsing (append after `safeGet()`):

```typescript
/**
 * Parse a price string that may include a currency symbol/code.
 * Returns { cents: number, currency: string } or null.
 *
 * Handles:
 *   "$25.00"        -> { cents: 2500, currency: 'USD' }
 *   "EUR 150,00"    -> { cents: 15000, currency: 'EUR' }
 *   "150,00 EUR"    -> { cents: 15000, currency: 'EUR' }
 *   "GBP 99.99"     -> { cents: 9999, currency: 'GBP' }
 *   "CHF 200.00"    -> { cents: 20000, currency: 'CHF' }
 *   "1 299,00 EUR"  -> { cents: 129900, currency: 'EUR' }
 *   "$1,299.00"     -> { cents: 129900, currency: 'USD' }
 */
export function parsePriceWithCurrency(raw: string): { cents: number; currency: string } | null
```

Implementation notes:
- Currency detection order: check for `$` -> USD, check for `EUR` or euro sign -> EUR, check for `GBP` or pound sign -> GBP, check for `CHF` -> CHF, check for `SEK`/`DKK`/`NOK` etc.
- European format uses comma as decimal separator: `150,00` = 150.00. Detect by checking if the string ends with `,\d{2}` (comma followed by exactly 2 digits at end) with no period after the comma.
- Thousands separators: European uses period or space (`1.299,00` or `1 299,00`), US uses comma (`1,299.00`). Strip thousands separators before parsing.
- Fall back to `parsePriceStringToCents()` with `'USD'` currency if no currency symbol detected.
- Return null for empty strings, "Free", "Contact for price", etc.

## 4. CONSTRAINTS -- WHAT NOT TO DO

### Banned Terms
- Do NOT use `SellerTier`, `SubscriptionTier`, `FVF`, `Final Value Fee`, `Twicely Balance`, `wallet`, `Withdraw`
- Do NOT use `BASIC`, `ELITE`, `PLUS`, `MAX`, `PREMIUM` as tier names
- These should not appear anywhere, but verify before committing

### Tech Stack
- Do NOT use Prisma, NextAuth, Redis, Zustand, tRPC, or any other banned tech
- The extension content scripts use plain TypeScript with Chrome APIs -- no React, no Next.js imports
- shared-utils.ts must remain pure functions (no Chrome API dependencies) for testability

### Business Logic
- Do NOT create a full `vestiaire-connector.ts` server-side connector (that is H4.2+)
- Do NOT implement import functionality (that is H4.2)
- Do NOT implement crosslist publish functionality (future phase)
- The content script is the ONLY new functional code -- everything else is enum/config wiring
- No fetch() or XHR calls from the content script -- all data flows through `chrome.runtime.sendMessage`
- No direct HTTP calls to Vestiaire APIs from the content script

### Code Patterns
- No `as any`, no `@ts-ignore`, no `@ts-expect-error`
- Money values as integer cents (priceCents)
- Files under 300 lines
- Use explicit field mapping in all handlers
- Human-like random delays between form field interactions (Lister Canonical Section 9.4)

### Extension-Specific
- Content scripts cannot make cross-origin fetch calls (that is the service worker's job)
