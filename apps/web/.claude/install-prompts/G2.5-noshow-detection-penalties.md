## SCOPE -- EXACTLY WHAT TO BUILD

### 1. No-Show Detection BullMQ Job

**File:** `src/lib/jobs/local-noshow-check.ts` (NEW)

A BullMQ job that checks for no-shows after one party checks in.

**When enqueued:** Every time a check-in is recorded (buyer or seller), enqueue a delayed job.

**Job data:**
```typescript
interface LocalNoShowCheckData {
  localTransactionId: string;
  checkedInParty: 'BUYER' | 'SELLER';
  checkedInAt: string; // ISO timestamp
}
```

**Delay:** `commerce.local.meetupAutoCancelMinutes` minutes (default: 30) converted to ms.

**Processing logic:**
1. Fetch the localTransaction by ID
2. Check current status:
   - If BOTH_CHECKED_IN, RECEIPT_CONFIRMED, COMPLETED, CANCELED, or DISPUTED: **skip** (not a no-show)
   - If SELLER_CHECKED_IN and `checkedInParty === 'SELLER'`: buyer is the no-show
   - If BUYER_CHECKED_IN and `checkedInParty === 'BUYER'`: seller is the no-show
3. If a no-show is detected:
   - Determine the no-show party (opposite of who checked in)
   - Read `commerce.local.noShowFeeCents` from platform settings (default: 500 = $5.00)
   - Update localTransaction:
     - `status` = 'NO_SHOW'
     - `noShowParty` = 'BUYER' or 'SELLER'
     - `noShowFeeCents` = fee amount
     - `noShowFeeChargedAt` = now()
   - Update `order.status` to 'CANCELED' (with cancelInitiator='SYSTEM', cancelReason='No-show at local meetup')
   - Create ledger entry for the no-show fee (if paying via platform):
     - NOTE: The $5 no-show fee is paid to the party who DID show up. This is a compensation, not Twicely revenue. Per Decision 43: "The $5 goes to the party who DID show up -- compensating their time and gas. It's not a Twicely revenue stream; it's a fairness mechanism."
     - For now, record the no-show fee in the localTransaction record. Actual fee collection is deferred (requires charging the no-show party's payment method, which is a sensitive Stripe operation). Mark with noShowFeeCents for future processing.
   - Check strike count (see step 2)

**Enqueue helper:**
```typescript
export async function enqueueNoShowCheck(
  localTransactionId: string,
  checkedInParty: 'BUYER' | 'SELLER',
  checkedInAt: Date,
): Promise<void> {
  const autoCancelMinutes = await getPlatformSetting<number>(
    'commerce.local.meetupAutoCancelMinutes', 30
  );
  const delayMs = autoCancelMinutes * 60 * 1000;

  const queue = createQueue('local-noshow-check');
  await queue.add('check', {
    localTransactionId,
    checkedInParty,
    checkedInAt: checkedInAt.toISOString(),
  }, {
    delay: delayMs,
    jobId: `noshow-${localTransactionId}-${checkedInParty}`,
  });
}
```

### 2. Strike Tracking and Suspension

**File:** `src/lib/commerce/local-noshow-strikes.ts` (NEW)

**Function: `getNoShowStrikeCount(userId: string, windowDays: number)`**
- Query localTransaction table for rows where:
  - `noShowParty = 'BUYER'` and `buyerId = userId`, OR
  - `noShowParty = 'SELLER'` and `sellerId = userId`
  - AND `noShowFeeChargedAt` is within the last `windowDays` days
- Returns the count

**Function: `checkAndSuspendIfNeeded(userId: string)`**
- Reads `commerce.local.noShowStrikeLimit` (default: 3)
- Reads `commerce.local.noShowSuspensionDays` (default: 90)
- Gets strike count for the last 90 days
- If strikes >= limit:
  - The user is "suspended from local transactions" for `noShowSuspensionDays` days
  - NOT SPECIFIED in schema: There is no `localSuspendedUntil` column on user or sellerProfile.
  - **Implementation approach:** Create a platform-level user restriction record. Options:
    - Option A: Add a check in the local transaction creation flow that queries recent no-shows. If count >= limit within 90 days, reject.
    - Option B: Use a feature flag or setting per-user. But this doesn't exist in the schema.
  - **RECOMMENDED: Option A** -- no schema change needed. When a local order is being created, check the user's no-show count. If >= limit within the window, block with message: "Local transactions are temporarily unavailable for your account due to missed meetups. They will be re-enabled on [date]."
  - This check goes in the order creation flow for local orders, NOT in the no-show handler.

**Function: `isUserSuspendedFromLocal(userId: string): Promise<{ suspended: boolean; resumesAt?: Date }>`**
- Checks if user has >= `noShowStrikeLimit` no-shows in the last `noShowSuspensionDays` days
- If yes, calculates when suspension ends (90 days from the Nth no-show)
- Returns suspension status and resume date

### 3. Both-No-Show Handling

Per Local Canonical Section 7: "Both don't show -> Order auto-canceled, no penalty"

**File:** `src/lib/jobs/local-auto-cancel.ts` (NEW)

A separate BullMQ job that auto-cancels if NEITHER party checks in within the auto-cancel window.

**When enqueued:** When a local transaction is created (in `createLocalTransaction`)

**Delay:** `commerce.local.confirmationCodeExpiryHours` hours (default: 48) from `scheduledAt`

**Processing logic:**
1. Fetch the localTransaction
2. If status is still 'SCHEDULED' (neither party checked in):
   - Update status to 'CANCELED'
   - Update order status to 'CANCELED' (cancelInitiator='SYSTEM', cancelReason='No meetup activity within 48 hours')
   - NO penalty (per canonical: "Both don't show -> auto-canceled, no penalty")
3. If any other status: skip (something already happened)

### 5. Suspension Check in Local Order Flow

**File:** `src/lib/commerce/local-eligibility.ts` (NEW)

```typescript
export async function checkLocalEligibility(userId: string): Promise<{
  eligible: boolean;
  reason?: string;
  resumesAt?: Date;
}> {
  const { suspended, resumesAt } = await isUserSuspendedFromLocal(userId);
  if (suspended) {
    return {
      eligible: false,
      reason: 'Local transactions are temporarily unavailable for your account due to missed meetups.',
      resumesAt,
    };
  }
  return { eligible: true };
}
```

This function should be called when:
- Buyer selects local pickup at checkout
- Seller enables local pickup in settings

## CONSTRAINTS -- WHAT NOT TO DO

- Do NOT add new columns to the schema -- the `localTransaction` table already has all needed no-show fields
- Do NOT charge the no-show fee via Stripe in this step -- just record it. Actual fee collection requires careful Stripe integration (charging a saved payment method) which is out of scope for G2.
- Do NOT implement safety timer / emergency features -- that is G2.6
- BullMQ queue names: `local-noshow-check`, `local-auto-cancel` (hyphens, not colons)
- Use `jobId` on no-show check to prevent duplicate jobs for the same check-in
- Money values in integer cents (noShowFeeCents = 500 for $5)
- Read ALL configurable values from platform_settings, never hardcode
- Generic error messages for suspension: do NOT reveal exact strike count

## ACCEPTANCE CRITERIA

1. When one party checks in and the other doesn't within 30 minutes, transaction marked NO_SHOW
2. No-show party identified correctly (opposite of who checked in)
3. noShowFeeCents set from `commerce.local.noShowFeeCents` platform setting (500 = $5.00)
4. noShowFeeChargedAt timestamp recorded
5. Order canceled with cancelInitiator='SYSTEM' on no-show
6. When both parties don't show within 48 hours, transaction auto-canceled with NO penalty
7. 3 no-shows in 90 days blocks user from local transactions
8. Suspension check returns correct resume date
9. No-show check skips transactions that have progressed (BOTH_CHECKED_IN, COMPLETED, etc.)
10. BullMQ jobs use correct delay from platform settings
11. All configurable values read from platform_settings
12. No banned terms in any new/modified files
13. TypeScript: 0 errors
